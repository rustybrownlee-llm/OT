# POC-001.0 - Modbus Library Validation

**Status**: Approved
**Created**: 2026-03-01
**Priority**: High
**OT Domain Review**: Complete (11 findings, all addressed)

## 1. Objective

Validate that `simonvetter/modbus` (MIT) can serve as the Modbus TCP server foundation for the OT Simulator plant binary. This POC exercises the four capabilities required by SOW-002.0: TCP server lifecycle, all four Modbus object types, unit ID-based routing (gateway pattern), and handler-level response delays. The POC produces throwaway code in `poc/modbus-validation/` and a pass/fail verdict that gates SOW-002.0.

## 2. Background

ADR-002 D1 selects `simonvetter/modbus` as the primary Modbus library, with the explicit requirement: "Must be validated in a POC before production use." The milestone spec (Beta 0.1) lists POC-001 as the first unblocked work item. SOW-002.0 (Modbus TCP Server) depends on this POC's verdict.

The library exposes a `RequestHandler` interface with four methods (`HandleCoils`, `HandleDiscreteInputs`, `HandleHoldingRegisters`, `HandleInputRegisters`). Each request struct carries a `UnitId` field, enabling a single TCP listener to dispatch requests to different register maps based on the Modbus unit ID -- the exact pattern needed for the Moxa gateway simulation.

**Key unknowns this POC resolves:**

1. Can `time.Sleep()` in handler methods reliably simulate per-device response delays without blocking other connections?
2. Does the `UnitId` field correctly propagate through all four handler methods?
3. Does the library return proper Modbus exception responses when handlers return error constants (e.g., `ErrIllegalDataAddress`)?
4. Can multiple concurrent clients connect and receive correct responses?
5. Does the library correctly handle both gateway exception codes (0x0A path unavailable vs 0x0B target failed to respond)?

**Fallback**: If the library fails validation, `goburrow/modbus` (BSD-3, client-only) is the fallback for monitoring-side reads. A custom TCP server using `encoding/binary` and `net` would be needed for the plant side.

## 3. Scope

### In Scope

- Single Go binary in `poc/modbus-validation/` with its own `go.mod`
- TCP server on `localhost:5020` serving all four Modbus object types
- Unit ID routing modeling the Moxa NPort gateway pattern: unit ID 10 returns gateway status registers (educational abstraction), unit ID 1 returns SLC-500 registers (serial address 1), unit ID 2 returns Modicon 984 registers (serial address 2)
- Handler-level `time.Sleep()` to simulate additive response delays (gateway overhead + serial transmission + device processing)
- Built-in client test that verifies reads, writes, unit ID routing, delays, exception responses, and write-reject on read-only registers
- Pass/fail output to stdout with timing measurements

### Out of Scope

- Docker or containerized deployment
- YAML configuration loading
- Design layer integration or device atom parsing
- Modbus RTU or serial (library does not support RTU server; this is a known limitation per ADR-002)
- TLS/MBAPS (future concern, not relevant to Beta 0.1)
- Production code structure, packages, or interfaces
- Persistent state or process simulation
- Concurrent server stress testing beyond basic multi-client verification

## 4. Requirements

### Functional Requirements

1. **FR-1**: Start a Modbus TCP server on `localhost:5020` using `simonvetter/modbus.NewServer`
2. **FR-2**: Implement `RequestHandler` with in-memory register maps for three unit IDs modeling the Moxa NPort gateway pattern:
   - Unit ID 10: "gateway" -- 9 holding registers modeling observable gateway state. NOTE: This is a simulation abstraction. A real Moxa NPort 5150 exposes no Modbus registers. This unit models observable gateway state for educational purposes only.
   - Unit ID 1: "slc-500" -- 7 holding registers (addresses 1-7), 4 coils (addresses 1-4), 3 input registers, 2 discrete inputs. One-based addressing per ProSoft MVI46-MCM module convention. Mimics the mfg-line-a variant from the SLC-500 device atom.
   - Unit ID 2: "modicon-984" -- 7 holding registers (addresses 1-7), 4 coils (addresses 1-4), 3 input registers, 2 discrete inputs. One-based addressing per original Modbus convention (this device family created the protocol). Mimics the mfg-cooling variant from the Modicon 984 device atom.
3. **FR-3**: Serve all four Modbus object types:
   - FC01 (Read Coils) / FC05 (Write Single Coil) / FC0F (Write Multiple Coils)
   - FC02 (Read Discrete Inputs) -- return static values
   - FC03 (Read Holding Registers) / FC06 (Write Single Register) / FC10 (Write Multiple Registers)
   - FC04 (Read Input Registers) -- return static values
4. **FR-4**: Apply additive response delays in handlers modeling real serial gateway behavior:
   - Unit ID 10 (gateway own registers): 15ms (gateway processing only, no serial RTT)
   - Unit ID 1 (SLC-500 via gateway): 81ms total (15ms gateway + 8ms serial TX at 9600 baud + 50ms device processing + 8ms serial RX)
   - Unit ID 2 (Modicon 984 via gateway): 111ms total (15ms gateway + 8ms serial TX at 9600 baud + 80ms device processing + 8ms serial RX)
5. **FR-5**: Return `ErrIllegalDataAddress` for out-of-range register/coil addresses. For one-based units (1 and 2), reading address 0 must return this error.
6. **FR-6**: Return `ErrGWPathUnavailable` for unit IDs with no configured route (e.g., unit ID 99 -- the gateway has no path to this address)
7. **FR-7**: Return `ErrGWTargetFailedToRespond` for a unit ID that is configured but simulates a non-responding downstream device (e.g., unit ID 3 -- configured route exists but device is offline)
8. **FR-8**: Reject writes to read-only registers with an appropriate Modbus exception. A write to a sensor feedback register (e.g., supply_temp on the Modicon 984) must not succeed.
9. **FR-9**: Built-in client test sequence that runs automatically after server starts

### Non-Functional Requirements

1. **NFR-1**: POC binary completes all tests and exits within 15 seconds
2. **NFR-2**: Measured response times must be at least 80% of the configured delay target with no upper bound. This avoids flaky results from OS scheduler jitter while confirming the sleep is applied.
3. **NFR-3**: Two concurrent client connections must both receive correct responses

## 5. Design

### Architecture

Single `main.go` file with embedded handler and client test. No packages, no abstractions. This is throwaway validation code.

```
poc/modbus-validation/
  go.mod       # independent module
  main.go      # server + handler + client tests, single file
```

### Unit ID Assignment Rationale

[OT-REVIEW #1] Unit ID 0 is the Modbus broadcast address (Modbus Application Protocol V1.1b3, Section 4.1). Slaves do not respond to broadcasts. It is never used as a device address.

The unit ID assignments in this POC match the `serial_address` fields in the environment template (`environment.yaml`):
- Unit ID 1 = SLC-500 (`serial_address: 1` in mfg-plc-01 placement)
- Unit ID 2 = Modicon 984 (`serial_address: 2` in mfg-plc-02 placement)
- Unit ID 10 = Gateway status (educational abstraction, arbitrary address outside the serial bus range)
- Unit ID 3 = Configured but offline device (tests gateway timeout exception)

### Handler Design

```go
// handler implements modbus.RequestHandler
type handler struct {
    // Per-unit-ID register maps. In-memory arrays use zero-based indexing internally.
    // For one-based devices (units 1, 2), the handler subtracts the base address offset
    // during range checking and array access.
    holding    map[uint8][]uint16        // unitID -> holding register values
    coils      map[uint8][]bool          // unitID -> coil values
    dinputs    map[uint8][]bool          // unitID -> discrete input values (static)
    iregs      map[uint8][]uint16        // unitID -> input register values (static)
    writable   map[uint8]map[uint16]bool // unitID -> address -> writable flag (holding regs)
    coilWrite  map[uint8]map[uint16]bool // unitID -> address -> writable flag (coils)
    baseAddr   map[uint8]uint16          // unitID -> base address (0 for zero-based, 1 for one-based)
    delays     map[uint8]time.Duration   // unitID -> response delay
    configured map[uint8]bool            // unitID -> true if route exists (even if offline)
    mu         sync.RWMutex
}
```

Each handler method:
1. Checks if `req.UnitId` is in `configured` map; returns `ErrGWPathUnavailable` if not
2. Checks if unit has register data (nil = configured but offline); returns `ErrGWTargetFailedToRespond` if nil
3. Applies `time.Sleep(h.delays[req.UnitId])`
4. Adjusts address by subtracting `baseAddr[req.UnitId]`; returns `ErrIllegalDataAddress` if below base
5. Validates adjusted address + quantity against array bounds; returns `ErrIllegalDataAddress` if out of range
6. For writes: checks `writable` map; returns `ErrIllegalDataAddress` if register is read-only
7. For reads: returns the requested slice from the in-memory map
8. For writes: updates the in-memory map and returns nil

### Client Test Sequence

The binary starts the server, waits 100ms, then runs these tests sequentially using `simonvetter/modbus` client:

| # | Test | Method | Expected |
|---|------|--------|----------|
| T1 | Read holding registers, unit 1 (SLC-500) | FC03, addr 1, qty 7 | 7 values matching mfg-line-a initial map |
| T2 | Read coils, unit 1 (SLC-500) | FC01, addr 1, qty 4 | 4 bool values matching mfg-line-a |
| T3 | Write single coil, unit 1 | FC05, addr 1 (conveyor_run), value true | Success, re-read confirms |
| T4 | Write single register, unit 1 | FC06, addr 1 (conveyor_speed), value 500 | Success, re-read confirms |
| T5 | Read input registers, unit 2 (Modicon 984) | FC04, addr 1, qty 3 | 3 static values |
| T6 | Read discrete inputs, unit 2 | FC02, addr 1, qty 2 | 2 static bool values |
| T7 | Unit ID routing | FC03, addr 1, qty 1 to units 1, 2, 10 | Different register values per unit |
| T8 | Unknown unit ID (no route) | FC03, unit 99 | ErrGWPathUnavailable (exception 0x0A) |
| T9 | Configured but offline unit | FC03, unit 3 | ErrGWTargetFailedToRespond (exception 0x0B) |
| T10 | Out-of-range address | FC03, unit 1, addr 999 | ErrIllegalDataAddress |
| T11 | Below-base address (one-based device) | FC03, unit 2, addr 0 | ErrIllegalDataAddress |
| T12 | Response delay measurement | FC03 to units 1, 2, 10 with timing | unit10 >=12ms, unit1 >=65ms, unit2 >=89ms |
| T13 | Concurrent clients | Two goroutines reading unit 1 and unit 2 simultaneously | Both get correct values |
| T14 | Write multiple registers | FC10, unit 2, addr 6 (setpoint_temp), 1 value | Success, re-read confirms |
| T15 | Write multiple coils | FC0F, unit 1, addr 1, 2 values | Success, re-read confirms both |

## 6. Implementation Plan

### Phase 1: Setup

3. Add dependency: `go get github.com/simonvetter/modbus`

### Phase 2: Server Implementation

4. Define handler struct with in-memory register maps (see Design section)
5. Initialize register maps with values from the design layer device atoms:
   - Unit ID 10 (gateway): 9 holding registers from moxa-nport-5150.yaml serial-gateway variant (serial_port_status through uptime_hours). Zero-based addressing. NOTE: These registers are a simulation abstraction. A real Moxa NPort 5150 exposes no Modbus registers. This unit models observable gateway state for educational purposes only.
   - Unit ID 1 (SLC-500): 7 holding registers from slc-500-05.yaml mfg-line-a variant (conveyor_speed through status_word), 4 coils (conveyor_run through jam_detected), 3 input registers (static sensor values), 2 discrete inputs (static status bits). One-based addressing per ProSoft MVI46-MCM module convention. Mark conveyor_speed (addr 1) as writable, all other holding registers as read-only. Mark conveyor_run (addr 1) and conveyor_direction (addr 2) as writable coils, e_stop_active (addr 3) and jam_detected (addr 4) as read-only.
   - Unit ID 2 (Modicon 984): 7 holding registers from modicon-984.yaml mfg-cooling variant (supply_temp through pump_runtime_hours), 4 coils (pump_1_run through high_temp_alarm), 3 input registers (static), 2 discrete inputs (static). One-based addressing per Modbus convention originated by this device family. Mark setpoint_temp (addr 6) as writable, all other holding registers as read-only. Mark pump_1_run (addr 1) and pump_2_run (addr 2) as writable coils, alarm coils (addr 3-4) as read-only.
   - Unit ID 3 (offline device): Add to `configured` map as true, but set all register maps to nil. This simulates a serial device that is configured in the gateway routing table but is not responding (powered off, cable disconnected, etc.).
6. Implement `HandleHoldingRegisters` with unit ID dispatch, configured/online check, delay, base-address adjustment, range check, writable check, read/write
7. Implement `HandleCoils` with same pattern

9. Implement `HandleDiscreteInputs` (read-only)
10. Create and start `ModbusServer` with `ServerConfiguration{URL: "tcp://localhost:5020"}`

### Phase 3: Client Tests

11. Create Modbus TCP client connecting to `localhost:5020`
12. Implement test runner executing T1-T16 sequentially
13. Implement timing measurement for T12 (average of 5 reads per unit ID)
14. Implement concurrent client test for T13 (two goroutines with sync.WaitGroup)
15. Print results in the specified format
16. Exit with code 0 on all-pass, code 1 on any failure

### Phase 4: Validation

17. Run `go build && ./modbus-validation` and verify all 16 tests pass
18. Verify response delay measurements meet the 80%-of-target floor
19. Document any library limitations or surprises encountered

## 8. Success Criteria

1. `cd poc/modbus-validation && go build ./...` compiles without errors
2. `./modbus-validation` runs and prints VERDICT: PASS with 16/16 tests passing
3. All four Modbus object types (holding registers, coils, input registers, discrete inputs) serve correct values
4. Unit ID routing dispatches to different register maps per unit ID
5. Unknown unit ID (no route) returns ErrGWPathUnavailable (Modbus exception 0x0A)
6. Configured-but-offline unit ID returns ErrGWTargetFailedToRespond (Modbus exception 0x0B)
7. Out-of-range address returns ErrIllegalDataAddress exception
8. One-based device rejects address 0 with ErrIllegalDataAddress
9. Response delay measurements for each unit meet the 80%-of-target floor
10. Two concurrent clients both receive correct responses
11. Write operations (FC05, FC06, FC0F, FC10) persist and are readable on subsequent read
12. Write to a read-only register returns a Modbus exception (does not silently succeed)

**Go/No-Go Decision**: If all 12 criteria pass, SOW-002.0 proceeds with `simonvetter/modbus`. If criteria 1-7 fail, evaluate the fallback (custom server). If only criteria 8-12 fail, SOW-002.0 proceeds with noted limitations and workarounds documented.

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| `time.Sleep` in handler blocks the server's goroutine pool, serializing all responses | High | Medium | T13 (concurrent clients) validates this; if blocked, investigate library's concurrency model -- does it spawn a goroutine per connection or use a shared worker pool? |
| Library silently drops exception errors instead of returning Modbus exception PDUs | High | Low | T8, T9, T10, T11 validate error propagation for three different exception codes; capture Wireshark trace if ambiguous |
| Library maps custom errors to wrong Modbus exception codes (e.g., 0x0A mapped as 0x04) | Medium | Medium | Client tests check for the specific error type returned by the client; if mapping is wrong, document the actual mapping for SOW-002.0 |
| Library version incompatibility with Go 1.24 | Low | Low | Pin version in go.mod; check library's CI matrix |
| macOS scheduler jitter causes delay measurements to fail intermittently | Low | Medium | 80% floor tolerance avoids sub-millisecond precision requirements; no upper bound eliminates scheduler-spike failures |
