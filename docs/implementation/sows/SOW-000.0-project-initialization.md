# SOW-000.0: Project Initialization

**Status**: Complete
**Created**: 2026-02-28

## Objective

Initialize the OT Simulator project with Go module scaffolding, build tooling, Docker Compose skeleton, and initial configuration files. This SOW creates the foundation that all subsequent SOWs build upon.

## Scope

### In Scope

**Plant module (`/plant/`)**:
- Go module initialization (`go.mod`)
- Thin `cmd/plant/main.go` entry point with flag parsing and graceful shutdown
- Package scaffolding with `doc.go` files for all planned packages
- Makefile with standard targets (build, test, fmt, lint, clean)
- Dockerfile for containerized deployment
- Default YAML configuration file defining initial plant topology

**Monitoring module (`/monitoring/`)**:
- Go module initialization (`go.mod`)
- Thin `cmd/monitor/main.go` entry point with flag parsing
- Makefile with standard targets
- Dockerfile for containerized deployment

**Top-level infrastructure**:
- Root Makefile orchestrating plant and monitoring builds
- Docker Compose file with network definitions per ADR-003
- Scenario directory scaffolding per ADR-007

**Configuration**:
- `plant/config/plant.yaml` -- default plant topology definition
- `plant/config/devices/` -- device profile directory with one example profile

### Out of Scope
- Modbus TCP implementation (SOW-001.0)
- Process simulation engine (SOW-002.0)
- HMI dashboard (SOW-003.0)
- Manufacturing floor simulation (SOW-004.0)
- Any monitoring tool implementation
- Scenario content (READMEs, hints, solutions)

## Success Criteria

1. `cd plant && go build ./cmd/plant/` compiles without errors
2. `cd plant && go test ./...` passes (logger tests)
3. `cd monitoring && go build ./cmd/monitor/` compiles without errors
4. `make -C plant build` succeeds
5. `make -C monitoring build` succeeds
6. `make build` (root) builds both plant and monitoring
7. `docker compose config` validates without errors
8. Plant binary starts, logs initialization message, responds to SIGINT with graceful shutdown
9. Monitoring binary starts and exits cleanly
10. All doc.go files have valid package-level godoc comments
11. Plant config file loads and validates without error
12. Device profile file loads and validates without error

## Execution Order

1. **Initialize plant Go module** (`plant/go.mod`)
   - `go mod init github.com/rustybrownlee/ot-simulator/plant`
   - Add `gopkg.in/yaml.v3` dependency

2. **Create logging package** (`plant/internal/logging/`)
   - Implement Logger interface matching vektera-v2/cyber-generator pattern
   - `Debug`, `Info`, `Warn`, `Error`, `Fatal` methods
   - `WithFields`, `WithLevel` methods
   - `NewLogger()` constructor with slog JSON handler
   - `NewTestLogger()` for quiet tests
   - Write unit tests

3. **Create package doc.go files** (`plant/internal/*/doc.go`)
   - One doc.go per package with package-level godoc
   - Document intended purpose and scope per ADR references

4. **Create plant configuration** (`plant/config/`)
   - `plant.yaml` with typed config struct and `DefaultConfig()` + `Validate()`
   - `devices/compactlogix-l33er.yaml` as reference device profile

5. **Create plant entry point** (`plant/cmd/plant/main.go`)
   - Flag parsing (--config, --addr)
   - YAML config loading
   - Logger initialization
   - Signal handling (SIGINT, SIGTERM)
   - Graceful shutdown with 30-second timeout

6. **Create plant build tooling** (`plant/Makefile`, `plant/Dockerfile`)
   - Makefile with standard targets
   - Multi-stage Dockerfile

7. **Initialize monitoring Go module** (`monitoring/go.mod`, `monitoring/cmd/monitor/main.go`)
   - Thin entry point, flag parsing, signal handling
   - Makefile and Dockerfile

8. **Create root infrastructure** (`Makefile`, `docker-compose.yml`)
   - Root Makefile delegating to plant and monitoring
   - Docker Compose with network definitions per ADR-003

9. **Create scenario scaffolding** (`scenarios/`)
   - Placeholder README.md in each scenario directory

10. **Validate all success criteria**
    - Run builds, tests, docker compose config
    - Verify binary startup and shutdown behavior

## Notes

- The logging package is duplicated between plant and monitoring (separate Go modules). This is intentional -- they are independent modules that cannot share code (per CLAUDE.md separation principle). If the logger interface proves identical long-term, consider extracting to a shared module in a future SOW.
- Device profile YAML schema is initial and will evolve as ADR-004 is implemented in subsequent SOWs.
- Docker Compose network definitions follow ADR-003 but containers are minimal in this SOW (just the plant and monitoring binaries). Additional containers (individual PLCs, HMIs) are added in subsequent SOWs.
- Scenario README.md files are intentionally minimal placeholders. Full scenario content is authored in a later SOW after the plant is functional.
