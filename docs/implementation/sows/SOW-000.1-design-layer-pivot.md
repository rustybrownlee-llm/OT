# SOW-000.1: Design Layer Pivot

**Status**: Complete
**Created**: 2026-03-01
**Priority**: High

## 1. Objective

Refactor the SOW-000.0 foundation to align with ADR-009 (Design Layer and Composable Environments). Create the `design/` directory with device library, network atoms, and the reference environment template. Generalize plant config structs from facility-specific to topology-generic. Update the plant binary to load environments from the design layer instead of a hardcoded facility config.

## 2. Background

SOW-000.0 built the project foundation assuming a single hardcoded facility (water treatment + manufacturing). After architectural review, ADR-009 introduced a three-layer model (design/plant/monitoring) with composable atomic elements. The existing plant config structs (`WaterTreatmentConfig`, `ManufacturingConfig`, `CrossPlantConfig`) and the `plant/config/` directory must be replaced with generic topology loading from `design/`.

This SOW is intentionally narrow: it restructures the config layer and creates the design directory. It does not add new functionality -- the plant binary still starts, loads config, logs, and shuts down. The difference is that it now loads a generic environment definition instead of a facility-specific one.

**Key files affected**:
- `plant/internal/config/config.go` -- complete rewrite (facility-specific -> generic topology)
- `plant/internal/config/config_test.go` -- complete rewrite (new structs, new test data paths)
- `plant/cmd/plant/main.go` -- update flags and startup log
- `plant/config/` -- removed (replaced by design/)
- `design/` -- new directory tree

## 4. Requirements

### Functional Requirements

1. Plant binary accepts `--environment` flag pointing to a design layer environment directory
2. Plant binary loads environment.yaml, resolves device refs from `design/devices/`, resolves network refs from `design/networks/`
3. Plant binary validates the loaded environment (non-empty name, valid schema version, all device refs resolve, all network refs resolve, no duplicate placement IDs, ports in valid range)
4. Plant binary logs environment name and placement count on startup
5. All design layer YAML files include `schema_version: "0.1"`
6. Device atom separates register capabilities from register map variants
7. The `compactlogix-l33er` device atom includes two register map variants: `water-intake` and `water-distribution`
8. The reference environment template (`greenfield-water-mfg`) defines the same topology as the original `plant.yaml` (same devices, same IPs, same ports)

### Non-Functional Requirements

1. All existing success criteria from SOW-000.0 that still apply must continue to pass (build, test, lint, docker compose config, binary start/shutdown)
2. Config package remains under 500 lines per file
3. All functions remain under 60 lines
4. Design layer YAML files are human-readable without documentation (self-documenting comments)

## 5. Design

### Config Package Rewrite

Replace facility-specific structs with generic topology structs that map to ADR-009 schemas:

```go
// Top-level: what the plant binary loads
type Environment struct {
    SchemaVersion string      `yaml:"schema_version"`
    Environment   EnvMeta     `yaml:"environment"`
    Networks      []NetworkRef `yaml:"networks"`
    Placements    []Placement  `yaml:"placements"`
}

// Resolved from design/devices/{id}.yaml
type Device struct {
    SchemaVersion    string            `yaml:"schema_version"`
    Device           DeviceMeta        `yaml:"device"`
    Connectivity     Connectivity      `yaml:"connectivity"`
    Registers        RegisterCapacity  `yaml:"registers"`
    RegisterMapVariants map[string]RegisterMap `yaml:"register_map_variants"`
    Diagnostics      Diagnostics       `yaml:"diagnostics"`
}

// Resolved from design/networks/{id}.yaml
type Network struct {
    SchemaVersion string         `yaml:"schema_version"`
    Network       NetworkMeta    `yaml:"network"`
    Properties    NetworkProps   `yaml:"properties"`
}
```

### Loading Strategy

The config package exposes a single entry point:

```go
func LoadEnvironment(envDir string) (*ResolvedEnvironment, error)
```

This function:
1. Reads `{envDir}/environment.yaml`
2. Walks up to find the `design/` root (parent of `environments/`)
3. Resolves each network ref from `design/networks/{ref}.yaml`
4. Resolves each device ref from placements via `design/devices/{device}.yaml`
5. Validates cross-references (device connectivity matches network type, etc.)
6. Returns a `ResolvedEnvironment` with all YAML fully loaded

### Plant Binary Changes

```
Before: plant --config config/plant.yaml
After:  plant --environment ../../design/environments/greenfield-water-mfg
```

The `--environment` flag takes a path to an environment directory containing `environment.yaml`. The default is `../../design/environments/greenfield-water-mfg` (relative to the plant binary's working directory in `plant/`).

Logging config (level, format) moves to a plant-specific runtime config flag `--log-level` since it is not part of the environment definition. Default: `info`.

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Config rewrite breaks binary startup | High | Low | Tests validate loading before removing old config |
| Design layer path resolution fragile across environments (Docker vs local) | Medium | Medium | LoadEnvironment accepts absolute path; Dockerfile COPY adjusts accordingly |
| Register map variant structure too complex for v0.1 | Low | Low | Variants are parsed into structs but not wired to runtime yet -- just data |
