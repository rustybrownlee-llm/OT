# SOW-001.0: Full Device Atom Profiles

**Status**: Approved
**Created**: 2026-03-01
**Priority**: High

## 1. Objective

Complete the three stub device atoms (moxa-nport-5150, slc-500-05, modicon-984) with full register map variants, add the missing water-treatment variant to the CompactLogix device atom, correct OT domain inaccuracies in existing device atoms identified by the OT Domain Reviewer agent, update the greenfield environment template to assign all variants, and add register_map_variant cross-reference validation. This resolves TD-003 from SOW-000.1 and ensures every placement in the reference environment has a defined, OT-realistic register map for future Modbus simulation.

## 2. Background

SOW-000.1 created the design layer with one complete device atom (compactlogix-l33er) and three stubs (moxa-nport-5150, slc-500-05, modicon-984). The stubs have identity, connectivity, register capacity, and diagnostics metadata but empty register map variants -- they parse and resolve but define no registers. Additionally, the CompactLogix has `water-intake` and `water-distribution` variants but no variant for wt-plc-02, which handles filtration and chemical treatment.

Without full register maps, the plant binary cannot serve meaningful Modbus data. This SOW fills in every register map variant referenced by the greenfield environment, preparing the design layer for the Modbus server implementation in a subsequent SOW.

### OT Domain Review

Before finalizing this SOW, all register map designs and existing device atom metadata were reviewed by the **ot-domain-reviewer** agent (`.claude/agents/ot-domain-reviewer.md`). The review identified 6 critical corrections and 6 important improvements. All findings are incorporated in this revision, tagged with `[OT-REVIEW]` in the Design section for traceability.

**Critical corrections caught by the OT Domain Reviewer:**

| # | Finding | Impact |
|---|---------|--------|
| 1 | CompactLogix L33ER has no RS-485 port -- it is Ethernet-native | Would teach wrong device architecture |
| 2 | SLC-500/05 natively speaks DH-485, not Modbus RTU -- Modbus requires a ProSoft MVI46-MCM module | Core teaching point about legacy devices |
| 3 | Moxa baud rate register value 115200 exceeds 16-bit Modbus register range (0-65535) | Would break Modbus simulation |
| 4 | Modicon 984 register capacity (256) is far too low -- should be ~999 holding, ~2000 coils | Misrepresents the Modbus originator device |
| 5 | UV intensity unit mJ/cm2 is dose, not intensity -- real-time sensor reads mW/cm2 | Would confuse engineers referencing UV specs |
| 6 | SLC-500 float byte order is NOT little-endian natively -- it uses big-endian (AB convention) via ProSoft module | Conflates two vendors' byte order conventions |

**Important improvements caught by the OT Domain Reviewer:**

| # | Finding | Impact |
|---|---------|--------|
| 7 | Chlorine dosing and filter backwash registers belong in water-treatment, not water-intake | Misplaces treatment-stage operations at intake |
| 8 | Modicon 984 cooling system needs a writable temperature setpoint -- otherwise it is a monitor, not a controller | Reduces attack surface realism |
| 9 | Filter differential pressure range 100 kPa would damage the vessel -- use 50 kPa | Unrealistic physical range |
| 10 | CompactLogix concurrent connections should be 32, not 10 | Understates device capability |
| 11 | Modicon 984 (Modbus originator) should support full 125 registers per read | Paradoxical to limit the device that defined the spec |
| 12 | SLC-500 missing status_word bitmask register -- the most common register type in OT assessments | Missed key educational element |

**Key context from ADR-004**:
- SLC-500-05 requires a ProSoft MVI46-MCM module for Modbus RTU -- one-based addressing is a property of the module, not the processor `[OT-REVIEW #2]`
- SLC-500-05 uses big-endian float byte order (Allen-Bradley native); Modicon 984 uses little-endian (CDAB word order, classic "Modicon byte swap") `[OT-REVIEW #6]`
- The Modicon 984 is from the processor family whose company (Modicon) created the Modbus protocol in 1979. This specific unit was installed circa 1988. `[OT-REVIEW #4]`
- Response times vary: CompactLogix 5ms, SLC-500 50ms (through ProSoft module), Modicon 984 80ms, Moxa 15ms (gateway overhead only)
- Water treatment side uses metric units (L/s, kPa, NTU, degC); manufacturing side uses imperial (ft/min, PSI, GPM, degF). This mixed-unit split is realistic and intentional -- many US facilities have this pattern.

**Register map design principle**: Register maps should be realistic enough to teach IT engineers what real OT register scans look like, but small enough to be comprehensible. Each variant defines 5-9 holding registers and 0-4 coils, mirroring the pattern established in the CompactLogix water-intake variant.

## 4. Requirements

### Functional Requirements

1. `moxa-nport-5150.yaml` defines a `serial-gateway` register map variant with holding registers representing gateway operational status (serial port status, baud rate as coded enum, serial mode, data format, connection count, traffic counters, error count, uptime)
2. `slc-500-05.yaml` defines a `mfg-line-a` register map variant with holding registers and coils for a conveyor/assembly line (speed, motor current, product count, reject count, temperature, cycle time, bitmask status word, run/stop/e-stop/jam controls)
3. `modicon-984.yaml` defines a `mfg-cooling` register map variant with holding registers and coils for a cooling water system (temperatures, flow rate, pump pressure, tank level, temperature setpoint, pump runtime hours, pump controls, alarms)
4. `compactlogix-l33er.yaml` defines a `water-treatment` register map variant with holding registers and coils for filtration and chemical treatment (filter pressures, UV intensity, chemical feed rate, chlorine residual, post-filter turbidity, backwash/dosing controls, alarms)
5. All register addresses in SLC-500-05 and Modicon 984 variants use one-based addressing (starting at address 1). SLC-500 addressing is documented as a ProSoft MVI46-MCM module convention.
6. All register addresses in CompactLogix and Moxa variants use zero-based addressing (starting at address 0)
7. All register addresses fall within the device's declared register capacity limits
8. Moxa NPort 5150 variant defines holding registers only (max_coils is 0)
9. Moxa NPort baud rate uses a coded enum (0-7 mapping to 1200-115200) that fits within 16-bit register range
10. Every placement in the greenfield environment that uses a PLC or gateway device has a register_map_variant assigned
11. Validation rejects placements that specify a register_map_variant not found in the device's RegisterMapVariants map
12. CompactLogix device atom has no RS-485 port (Ethernet-native only)
13. SLC-500 device atom documents DH-485 as native protocol with ProSoft module for Modbus RTU
14. All existing tests continue to pass after changes

### Non-Functional Requirements

1. Each device atom YAML file remains human-readable without documentation (self-documenting comments on every register and coil, including OT context)
2. Register map variants contain 5-9 holding registers and 0-4 coils each (educational focus: comprehensible, not exhaustive)
3. Device atom files remain under 500 lines
4. All functions remain under 60 lines
5. Safety-critical registers (e-stop, alarms) are always read-only with comments explaining why

### Corrections to Existing Device Atoms

#### CompactLogix L33ER Connectivity Fix `[OT-REVIEW #1, #10]`

The CompactLogix 5380 L33ER is Ethernet-native. It has no physical RS-485 port. Modbus TCP is supported via EtherNet/IP explicit messaging. Remove the RS-485 port entry and update concurrent connections.

```yaml
# Before (incorrect):
connectivity:
  ports:
    - type: "ethernet"
      protocols: ["modbus-tcp"]
    - type: "rs485"
      protocols: ["modbus-rtu"]
  concurrent_connections: 10

# After (correct):
connectivity:
  ports:
    # EtherNet/IP native; Modbus TCP via Add-On Instruction or explicit messaging
    - type: "ethernet"
      protocols: ["modbus-tcp"]
  concurrent_connections: 32
```

#### SLC-500-05 Connectivity Fix `[OT-REVIEW #2, #6]`

The SLC-500/05 natively speaks DH-485 (Data Highway 485), NOT Modbus RTU. Modbus communication requires a third-party ProSoft MVI46-MCM module installed in the SLC chassis. This is a critical teaching point: legacy Allen-Bradley devices do not natively speak Modbus, and an IT engineer who tries to send Modbus RTU to an SLC's Channel 0 port will get no response.

```yaml
# Before (incorrect):
connectivity:
  ports:
    - type: "rs485"
      protocols: ["modbus-rtu"]

# After (correct):
connectivity:
  ports:
    # Native processor Channel 0 -- DH-485 protocol, NOT Modbus.
    # IT engineers must understand: RS-485 electrical standard != Modbus protocol.
    - type: "dh485"
      protocols: ["dh-485"]
    # ProSoft MVI46-MCM communication module in chassis slot.
    # This module provides Modbus RTU capability and maps SLC data files (N7:, F8:)
    # into Modbus register addresses. One-based addressing is the module's convention.
    - type: "rs485"
      protocols: ["modbus-rtu"]
```

Also fix:
- `float_byte_order`: change from `"little-endian"` to `"big-endian"` with comment `# Allen-Bradley native byte order; ProSoft module preserves this by default`
- `max_registers_per_read`: change from `60` to `125` with comment `# Modbus spec maximum; ProSoft MVI46 supports full range`
- `addressing` comment: add `# One-based addressing via ProSoft MVI46-MCM module Modbus mapping`

#### Modicon 984 Capacity Fix `[OT-REVIEW #4, #11]`

The Modicon 984 uses the classic Modbus 5-digit addressing convention. With 16KB memory, it supports ~999 holding registers (4xxxx range) and ~2000 coils (0xxxx range). 256 is characteristic of a Modicon Micro 84, not the full 984. As the Modbus originator device, it supports the full 125 registers per read.

```yaml
# Before (too low):
registers:
  max_holding: 256
  max_coils: 256
  max_input_registers: 256
  max_discrete_inputs: 256
  max_registers_per_read: 60

# After (correct for 16KB 984):
registers:
  max_holding: 999            # 4xxxx range (40001-40999 in Modbus convention)
  max_coils: 2000             # 0xxxx range (00001-02000)
  max_input_registers: 999    # 3xxxx range (30001-30999)
  max_discrete_inputs: 2000   # 1xxxx range (10001-12000)
  max_registers_per_read: 125 # Full Modbus spec -- this device defined the spec
```

Also fix `response_jitter_ms` from `30` to `50` (models scan-time variability on legacy processors).

#### Moxa NPort 5150 Capacity Fix `[OT-REVIEW #3]`

The real NPort 5150 has NO Modbus register map -- it is a transparent serial-to-TCP converter. The `serial-gateway` variant is a simulated educational abstraction. Register capacity should match the simulated variant size, not imply 125 real registers.

```yaml
# Before (misleading):
registers:
  max_holding: 125
  max_input_registers: 125

# After (honest about simulated abstraction):
# NOTE: Real NPort 5150 devices have NO Modbus registers. They are transparent
# serial-to-TCP converters. The register capacity below matches the simulated
# serial-gateway variant, which models gateway status as an educational abstraction.
registers:
  max_holding: 9              # Matches serial-gateway variant (simulated, not real hardware)
  max_coils: 0
  max_input_registers: 0
  max_discrete_inputs: 0
```

### Register Map Variant: Moxa NPort 5150 `serial-gateway`

The Moxa NPort 5150 is a serial-to-Ethernet gateway, not a PLC. Real NPort devices manage configuration via web/SNMP, not Modbus registers. For educational purposes, the simulator models the gateway's operational status as read-only Modbus holding registers -- this teaches engineers that infrastructure devices have observable state on the network. All values are simulated.

```yaml
register_map_variants:
  serial-gateway:
    # SIMULATED: Real NPort 5150 does not expose Modbus registers.
    # This variant models gateway status as an educational abstraction,
    # teaching engineers that infrastructure devices have observable state.
    holding:
      - address: 0
        name: "serial_port_status"
        description: "Serial port operational state"
        unit: "enum"            # 0=offline, 1=online, 2=error
        scale_min: 0
        scale_max: 2
        writable: false
      - address: 1
        name: "serial_baud_rate"
        description: "Serial port baud rate (coded enum, not raw bps)"
        unit: "enum"            # 0=1200, 1=2400, 2=4800, 3=9600, 4=19200, 5=38400, 6=57600, 7=115200
        scale_min: 0
        scale_max: 7
        writable: false         # [OT-REVIEW #3] Coded enum avoids exceeding 16-bit register range
      - address: 2
        name: "serial_data_format"
        description: "Serial data format (coded enum)"
        unit: "enum"            # 0=8N1, 1=8E1, 2=8O1, 3=7E1
        scale_min: 0
        scale_max: 3
        writable: false
      - address: 3
        name: "serial_mode"
        description: "Serial interface electrical mode"
        unit: "enum"            # 0=RS-232, 1=RS-422, 2=RS-485-2wire, 3=RS-485-4wire
        scale_min: 0
        scale_max: 3
        writable: false
      - address: 4
        name: "active_tcp_connections"
        description: "Current TCP client connections (max 4, but serial access is serialized)"
        unit: "count"
        scale_min: 0
        scale_max: 4
        writable: false
      - address: 5
        name: "serial_tx_count"
        description: "Messages forwarded from TCP to serial port"
        unit: "msgs"
        scale_min: 0
        scale_max: 65535
        writable: false
      - address: 6
        name: "serial_rx_count"
        description: "Messages received from serial port"
        unit: "msgs"
        scale_min: 0
        scale_max: 65535
        writable: false
      - address: 7
        name: "serial_error_count"
        description: "Serial parity, framing, and overrun errors"
        unit: "count"
        scale_min: 0
        scale_max: 65535
        writable: false

        name: "uptime_hours"

```

All registers are read-only (`writable: false`) -- a gateway's status is observed, not commanded via Modbus. Also add a comment to `concurrent_connections` in connectivity: `# 4 TCP connections accepted, but serial port access is serialized -- only one active at a time`
