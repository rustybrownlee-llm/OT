# SOW-002.0 - Modbus TCP Server

**Status**: Approved
**Created**: 2026-03-01
**Priority**: High
**OT Domain Review**: Complete (12 findings, all addressed)

## 1. Objective

Make the plant binary serve Modbus TCP registers from the design layer. When `plant` starts, it reads the environment template, spawns one Modbus TCP listener per Ethernet placement, serves register values from device atom variants, routes gateway requests by unit ID to serial device register maps, and applies per-device response delays. Register values are static (initial values from scale ranges) -- process simulation is deferred to SOW-003.0.

After this SOW, an IT engineer can use any Modbus client to discover and read registers from every device in the greenfield-water-mfg environment on ports 5020-5022 and 5030.

## 2. Background

POC-001.0 validated that `simonvetter/modbus` supports TCP server lifecycle, all four Modbus object types (FC01-FC06, FC0F, FC10), unit ID routing, configurable response delays, and proper exception codes (0x0A, 0x0B, 0x02). The library spawns a goroutine per connection, so `time.Sleep()` in handlers does not block other clients. This SOW integrates the library into the plant binary's production architecture.

The plant binary already loads the environment template via `config.LoadEnvironment()` and returns a `ResolvedEnvironment` with all placements, device atoms, and network atoms resolved. SOW-002.0 bridges the gap between config loading and protocol serving.

### Current State

- `plant/cmd/plant/main.go`: Loads config, blocks on signals, has placeholder for subsystem startup
- `plant/internal/config/`: `ResolvedEnvironment` with `Placements`, `Devices`, `Networks` fully resolved
- `plant/internal/protocol/modbus/`: `doc.go` stub only
- `plant/internal/device/`: `doc.go` stub only
- `plant/go.mod`: Only `gopkg.in/yaml.v3`; needs `simonvetter/modbus` added

### Architecture

```
main.go -> config.LoadEnvironment() -> modbus.NewServerManager(resolved, logger)
                                          |-- server on :5020 (wt-plc-01, water-intake)
                                          |-- server on :5021 (wt-plc-02, water-treatment)
                                          |-- server on :5022 (wt-plc-03, water-distribution)
                                          +-- server on :5030 (mfg-gateway-01, routes unit IDs)
                                                |-- unit 247 -> gateway status registers (serial-gateway)
                                                |-- unit 1   -> slc-500 registers (mfg-line-a)
                                                +-- unit 2   -> modicon-984 registers (mfg-cooling)
```

**Gateway unit ID assignments**: [OT-REVIEW] Unit ID 0 is the Modbus broadcast address (Modbus Application Protocol V1.1b3, Section 4.1) and must never be used as a device address. Unit ID 247 is the highest valid Modbus slave address (0xF7) and is a recognized convention for gateway management addresses on devices like Digi TransPort, Lantronix, and ProSoft modules. This choice is intentional and must be documented in startup logging so learners understand what they are observing when scanning port 5030.

## 4. Requirements

### Functional Requirements

1. **FR-1**: On startup, the plant binary spawns one Modbus TCP listener per Ethernet placement in the environment template. For greenfield-water-mfg, this is 4 listeners: ports 5020, 5021, 5022, 5030.
2. **FR-2**: Each listener serves holding registers and coils from the placement's `register_map_variant` in the device atom. Register values are initialized per the tiered InitValue rules (see Section 5). If a placement references a `register_map_variant` that does not exist in the device atom, the plant binary logs an error and exits at startup.
3. **FR-3**: The gateway listener (port 5030) routes requests by Modbus unit ID:
   - Unit ID 247: gateway's own status registers (serial-gateway variant). [OT-REVIEW] This is a simulated educational abstraction -- a real Moxa NPort 5150 exposes no Modbus registers.
   - Unit ID matching `serial_address` of downstream devices: that device's registers with additive delay (gateway processing + serial request frame + slave turnaround + serial response frame)
   - Unknown unit ID: `ErrGWPathUnavailable` (exception 0x0A)
   - Startup logging must explicitly identify each unit ID binding and its role, e.g.: "port 5030: unit 247 = gateway status (simulated), unit 1 = SLC-500 (serial device), unit 2 = Modicon 984 (serial device)"
4. **FR-4**: Direct Ethernet PLC listeners (ports 5020-5022) respond to any unit ID, matching the observed behavior of Allen-Bradley CompactLogix devices in Modbus TCP mode. This is common but not universal across vendors -- see TD-007.
5. **FR-5**: Response delays are applied per-device from the device atom's `connectivity.response_delay_ms`. Jitter is added as a random value in `[-jitter, +jitter]` range from `connectivity.response_jitter_ms`. Gateway-proxied devices use additive delay (see Section 5 delay formula).
6. **FR-6**: Write operations to registers/coils marked `writable: false` in the device atom are rejected with `ErrIllegalFunction` (exception 0x01). [OT-REVIEW] This is distinct from `ErrIllegalDataAddress` (0x02) which indicates the address does not exist. The distinction teaches learners: "this register exists and you can read it, but writing is not permitted" vs "this address is not valid."
7. **FR-7**: Address validation respects the device atom's `registers.addressing` field. One-based devices (Modicon 984, SLC-500) reject reads/writes to address 0 with `ErrIllegalDataAddress`. Zero-based devices (CompactLogix) accept address 0.
8. **FR-8**: Out-of-range addresses beyond the register map return `ErrIllegalDataAddress`.
9. **FR-9**: All four Modbus object types are served:
   - FC01/FC05/FC0F: Coils (from register_map_variants coils section)
   - FC02: Discrete inputs (return zeros within device capacity -- TD-005)
   - FC03/FC06/FC10: Holding registers (from register_map_variants holding section)
   - FC04: Input registers (return zeros within device capacity -- TD-005)
10. **FR-10**: Graceful shutdown: all Modbus TCP listeners stop accepting connections and drain in-flight requests when the plant binary receives SIGINT/SIGTERM.
11. **FR-11**: Structured logging at these levels:
    - INFO: Server start with port, device ID, variant name, and unit ID bindings (critical for gateway)
    - DEBUG: Client connections and disconnections
    - WARN: Exception responses (illegal address, illegal function, gateway path unavailable)
    - ERROR: Server bind failures, missing variant references

### Non-Functional Requirements

1. **NFR-1**: Server startup completes within 2 seconds for the 4-listener greenfield-water-mfg environment
2. **NFR-2**: Each listener supports concurrent client connections up to the device atom's `concurrent_connections` value. For the gateway listener (port 5030), this is 4 per the NPort 5150 specification. For CompactLogix listeners, this is 32. Connection limit enforcement is deferred (see TD-010), but the underlying library supports the required concurrency.
3. **NFR-3**: No goroutine leaks on shutdown (verify with `runtime.NumGoroutine()` check)
4. **NFR-4**: All new code has unit tests with table-driven patterns

### Key Types

```go
// device/profile.go

// DeviceProfile extracts the runtime characteristics needed by the Modbus server
// from a resolved device atom and placement.
type DeviceProfile struct {
    PlacementID        string
    DeviceID           string
    Role               string
    ResponseDelay      time.Duration
    ResponseJitter     time.Duration
    Addressing         string            // "zero-based" or "one-based"
    FloatByteOrder     string            // "big-endian" or "little-endian"
    MaxRegistersPerRead int
    HoldingRegisters   []RegisterEntry   // from selected variant
    Coils              []CoilEntry       // from selected variant
    MaxHolding         int               // device capacity
    MaxCoils           int
    MaxInputRegisters  int
    MaxDiscreteInputs  int
}

// RegisterEntry is a runtime-ready register definition.
type RegisterEntry struct {
    Address   int
    Name      string
    Unit      string
    ScaleMin  float64
    ScaleMax  float64
    Writable  bool
    InitValue uint16   // computed per tiered InitValue rules
}

// CoilEntry is a runtime-ready coil definition.
type CoilEntry struct {
    Address  int
    Name     string
    Writable bool
    InitValue bool  // false for both command coils (safe state) and alarm coils (no-alarm idle)
}
```

```go
// protocol/modbus/store.go

// RegisterStore provides thread-safe read/write access to a device's register state.
// It is the single source of truth for register values during simulation.
// SOW-003.0 will add methods for process simulation to update values.
//
// Scaling convention: raw value 0 = scale_min, raw value 32767 = scale_max.
// This is the standard Modbus holding register convention (4xxxx address space).
// SOW-003.0 depends on this convention for process value computation.
type RegisterStore struct {
    holding   []uint16        // indexed by address (adjusted for base)
    coils     []bool
    dinputs   []bool          // discrete inputs (static zeros for now)
    iregs     []uint16        // input registers (static zeros for now)
    writable  map[int]bool    // holding register address -> writable flag
    coilWrite map[int]bool    // coil address -> writable flag
    baseAddr  int             // 0 for zero-based, 1 for one-based
    mu        sync.RWMutex
}
```

```go
// protocol/modbus/handler.go

// Handler implements modbus.RequestHandler for a single device placement.
// For direct Ethernet PLCs, one Handler serves one listener.
// For the gateway, one Handler routes to multiple RegisterStores by unit ID.
type Handler struct {
    stores     map[uint8]*RegisterStore  // unitID -> store (for gateway: multiple entries)
    delays     map[uint8]time.Duration   // unitID -> total response delay
    jitters    map[uint8]time.Duration   // unitID -> jitter range
    isGateway  bool
    logger     logging.Logger
}
```

```go
// protocol/modbus/server.go

// ServerManager creates and manages Modbus TCP listeners for all Ethernet placements.
type ServerManager struct {
    servers []*modbus.ModbusServer
    logger  logging.Logger
}

// NewServerManager builds servers from the resolved environment.
func NewServerManager(resolved *config.ResolvedEnvironment, logger logging.Logger) (*ServerManager, error)

// Start starts all listeners. Returns an error if any listener fails to bind.
func (sm *ServerManager) Start() error

// Stop gracefully shuts down all listeners.
func (sm *ServerManager) Stop()
```

### Server Creation Flow

`NewServerManager` iterates over placements and groups them:

1. **Ethernet placements with `modbus_port`** (wt-plc-01, wt-plc-02, wt-plc-03, mfg-gateway-01): Create a Modbus TCP listener.

2. **Serial placements with `gateway` reference** (mfg-plc-01, mfg-plc-02): Attach to the gateway's handler as an additional unit ID route.

For each Ethernet non-gateway placement:
- Build a `DeviceProfile` from the placement + device atom
- Create a `RegisterStore` initialized per tiered InitValue rules
- Create a `Handler` with `isGateway=false` (responds to any unit ID per FR-4)
- Create `modbus.NewServer` with `ServerConfiguration{URL: "tcp://0.0.0.0:PORT"}`

For the gateway placement:
- Build a `DeviceProfile` for the gateway itself (unit ID 247)
- For each serial placement referencing this gateway:
  - Build a `DeviceProfile` for the serial device
  - Create a `RegisterStore`
  - Add to the gateway handler at the serial device's `serial_address` unit ID
  - Compute additive delay per the formula below
- Create `modbus.NewServer` for the gateway port

### Additive Delay Formula for Gateway-Proxied Devices

[OT-REVIEW] Serial gateway latency is additive. Each component represents a real step in the serial-to-Ethernet conversion:

```
total_delay = gateway_processing_overhead
            + serial_request_frame_time
            + slave_turnaround_delay
            + serial_response_frame_time
```

At 9600 baud with 8N1 framing (1.042ms per character):

| Component | Calculation | Value |
|-----------|-------------|-------|
| Gateway processing | device atom `response_delay_ms` for gateway | 15ms |
| Request frame TX | 8 chars (addr + FC + start + qty + CRC) * 1.042ms + 3.5 char gap | ~12ms |
| Slave turnaround | device atom `response_delay_ms` for serial device | 80ms (Modicon) / 50ms (SLC-500) |
| Response frame TX | ~19 chars (addr + FC + bytecount + data + CRC) * 1.042ms + 3.5 char gap | ~24ms |

Total for Modicon 984 via gateway: 15 + 12 + 80 + 24 = **131ms**
Total for SLC-500 via gateway: 15 + 12 + 50 + 24 = **101ms**
Gateway own registers (no serial): **15ms**

Jitter is applied on top: the gateway's jitter (10ms) and the serial device's jitter are combined as the sum of both ranges.

### Initial Register Values (Tiered)

[OT-REVIEW] Register initialization uses a four-tier system based on the register's `unit` field:

**Tier 1 -- Analog process values** (units: L/s, %, pH, NTU, degC, degF, GPM, PSI, ft/min, A, s, and any other physical measurement unit):
- Initialize to midpoint of scale range: `initValue = uint16(32767 / 2) = 16383`
- Scaling convention: raw 0 = scale_min, raw 32767 = scale_max. This is the standard Modbus holding register linear scaling convention. SOW-003.0 depends on this.

**Tier 2 -- Enum and bitmask registers** (units: enum, bitmask):
- Initialize to 0 (first enum value, no bits set)

**Tier 3 -- Event counters** (units: count, msgs, units):
- Initialize to 0 (no events have occurred since simulation start)

**Tier 4 -- Runtime/uptime registers** (unit: hours, where the register name contains "uptime"):
- Initialize to 8760 (one year of continuous operation, representing an installed device that has been running)
- Exception: if the register description indicates "resets on power cycle" (e.g., pump_runtime_hours in Modicon 984), initialize to a plausible non-zero value like 2400 (~100 days since last power cycle)
- All other "hours" registers (equipment runtime counters without "uptime" in the name): initialize to 0

**Coil initialization**:
- Command coils (writable: true, e.g., pump_run, conveyor_run): initialize to `false` (de-energized = safe startup state, no equipment running until explicitly commanded)
- Status/alarm coils (writable: false, e.g., low_well_level_alarm, e_stop_active): initialize to `false` (no-alarm idle state). Note: the initialization value is the same as command coils, but the reason is different. SOW-003.0's process simulation will transition alarm coils to `true` when thresholds are exceeded, independent of the "start safe" philosophy for command coils.

### Addressing Offset

For one-based devices (addressing: "one-based"), the register store's internal array is zero-indexed but the Modbus address space starts at 1. The handler subtracts `baseAddr` (1) from the incoming request address before indexing. A request for address 0 on a one-based device returns `ErrIllegalDataAddress`.

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Binding to 0.0.0.0 causes port conflicts on developer machines | Medium | Medium | Log clear error messages with port number. Add a --port-offset flag if needed (not in scope but easy to add). |
| Register store mutex contention under load | Low | Low | RWMutex allows concurrent reads. Write contention is minimal (only client writes and future process simulation). |
| Device atoms missing register entries for some variants | High | Low | FR-2 requires fail-loudly at startup: log error and exit if a placement references a variant that does not exist in the device atom. |
| Serial frame time approximation does not match real-world conditions | Low | Medium | The 12ms/24ms estimates are based on 9600 baud 8N1 with inter-frame gaps. Actual serial bus timing depends on baud rate and frame size. Close enough for educational purposes; exact fidelity requires SOW-003.0's process simulation to model variable query sizes. |
