# SOW-003.0 - Process Simulation Engine

**Status**: Approved
**Created**: 2026-03-01
**Priority**: High

## 1. Objective

Make registers change over time so the OT environment feels alive. A 1-second simulation ticker reads coil states, computes process physics (linear models, not differential equations), applies sensor noise, enforces actuator ramp delays, and writes updated values back to RegisterStores. After this SOW, an IT engineer can write a pump coil to ON, watch flow registers climb over 5 seconds, and see correlated changes across temperature, pressure, and level registers.

This is Beta 0.1 Definition of Done items #4-6: observe different response times, see register drift after 30 seconds, write a coil and see related registers respond.

## 2. Background

SOW-002.0 established static register values in RegisterStores. The stores expose thread-safe `ReadHolding`, `WriteHolding`, `ReadCoils`, `WriteCoils` methods. The plant binary's `run()` function blocks on signals after starting Modbus servers -- SOW-003.0 adds a background simulation goroutine between server start and signal wait.

### ADR-001 Minimum Physics Requirements

1. Plausible cause-and-effect (pump on -> flow increases)
2. Values stay within engineering ranges (scale_min/scale_max)
3. Sensor noise (small random fluctuations)
4. Time-dependent behavior (chlorine decays, filters clog, temperatures drift)
5. Actuator response delay (pumps ramp, not step)

### Scaling Convention

Raw 0 = scale_min, raw 32767 = scale_max (established in SOW-002.0). The simulation engine works in engineering units internally and converts to raw 16-bit values when writing to stores.

## 3. Scope

### In Scope

- `plant/internal/process/` package: simulation engine with 1-second tick
- Process models for all three water treatment variants (intake, treatment, distribution)
- Process models for both manufacturing variants (mfg-line-a, mfg-cooling)
- Gateway counter updates (serial_tx_count, serial_rx_count, uptime_hours)
- Sensor noise on all analog registers
- Actuator ramp (gradual change over 3-5 ticks, not instant)
- Alarm coil state management (hysteresis thresholds)
- Integration into main.go (start engine, shutdown coordination via context)
- Unit tests for each process model

### Out of Scope

- Cross-variant register coupling (e.g., intake flow feeding treatment inlet pressure). Variants simulate independently for Beta 0.1. Cross-coupling is a future enhancement.
- PID controller simulation (setpoint convergence uses simple linear convergence, not PID)
- Physics-accurate models (linear approximations only, per ADR-001)
- Configurable tick rate (hardcoded 1s for Beta 0.1)
- Persisting simulation state across restarts

## 4. Requirements

### Functional Requirements

1. **FR-1**: A background goroutine ticks every 1 second, updating all RegisterStores with new process values.
2. **FR-2**: Each tick reads coil states, computes new register values based on process models, adds sensor noise, clamps to scale ranges, converts to raw 16-bit, and writes back.
3. **FR-3**: Actuator ramp: when a pump coil transitions ON, the associated flow/speed register increases by 20% of target per tick over 5 ticks (5-second ramp). When OFF, decreases by 20% per tick.
4. **FR-4**: Sensor noise: all analog registers (units: L/s, %, pH, NTU, degC, degF, GPM, PSI, ft/min, A, s, mW/cm2, mL/min, mg/L, kPa) have random noise applied each tick. Noise magnitude: 1% of scale range (±0.5%). [OT-REVIEW] Noise MUST NOT be applied to: unit "enum" registers, unit "bitmask" registers, unit "units" (counters), unit "hours" (accumulators), unit "msgs" (counters), or unit "count" registers. The `addNoise()` helper must check register unit type before applying noise.
5. **FR-5**: Alarm coils transition to true when their trigger condition is met and to false when a reset condition is met (hysteresis prevents oscillation).
6. **FR-6**: Counter registers (product_count, reject_count, serial_tx_count, serial_rx_count, pump_runtime_hours, uptime_hours) increment at defined rates when their trigger conditions are active. [OT-REVIEW] Counter registers wrap to 0 on 16-bit overflow (raw value exceeds 32767). This mirrors real PLC counter behavior. Saturation at scale_max is incorrect for counter registers. Product_count and reject_count reset to 0 every 28800 ticks (8-hour shift cycle) to model shift-change reset via PLC ladder logic.
7. **FR-7**: Enum registers (serial_port_status, serial_baud_rate, serial_data_format, serial_mode) remain static at their initialized values.
8. **FR-8**: The engine uses `context.Context` for shutdown. When the context is cancelled, the ticker stops and the engine returns.
9. **FR-9**: Bitmask registers (status_word) update individual bits based on simulation state (bit0=running tracks conveyor_run coil, bit2=jammed is random, etc.).

### Non-Functional Requirements

1. **NFR-1**: Simulation tick completes in under 10ms (well within the 1s budget)
2. **NFR-2**: No goroutine leaks on shutdown
3. **NFR-3**: All process models have unit tests

```go
// SimulationEngine runs process models on a 1-second tick.
type SimulationEngine struct {
    models []ProcessModel
    logger logging.Logger
}

func NewSimulationEngine(logger logging.Logger) *SimulationEngine
func (e *SimulationEngine) AddModel(m ProcessModel)
func (e *SimulationEngine) Run(ctx context.Context)  // blocks until ctx cancelled
```

```go
// model.go

// ProcessModel updates registers for one placement on each tick.
type ProcessModel interface {
    Name() string
    Tick()  // called every 1 second
}

// Helper functions (not exported, shared by all models):
// toRaw(value, scaleMin, scaleMax float64) uint16  -- engineering units to raw 16-bit
// toEng(raw uint16, scaleMin, scaleMax float64) float64  -- raw to engineering units
// addNoise(value, scaleRange float64) float64  -- ±0.5% of scale range
// ramp(current, target, step float64) float64  -- move current toward target by step
// clamp(value, min, max float64) float64  -- constrain to range
```

### Simulation Tick vs PLC Scan Time

[OT-REVIEW] The 1-second simulation tick is a simulator throughput constraint, not a model of PLC scan time. Real PLCs scan far faster: CompactLogix 5-20ms, SLC 500 10-50ms, Modicon 984 20-100ms. The tick rate affects register update frequency in the simulator only. A comment in `process/engine.go` must document this distinction to prevent trainees from learning incorrect PLC timing.

### Process Models

Each process model holds a reference to its RegisterStore and maintains internal state (ramp positions, decay accumulators). On each `Tick()`, it:

1. Reads coil states via `store.ReadCoils()`
2. Updates internal state (ramp positions, accumulator values)
3. Computes new engineering-unit values
4. Adds sensor noise
5. Clamps to scale ranges
6. Converts to raw 16-bit and writes via `store.WriteHolding()` using a special internal write path that bypasses the writable check (the simulation engine must be able to update read-only sensor registers)

**RegisterStore enhancement needed**: Add `WriteHoldingInternal(addr, values)` that bypasses the writable flag check. This is used only by the simulation engine, not by Modbus client writes. The existing `WriteHolding()` continues to enforce writable flags for external Modbus writes.

### Water Intake Model

**State**: pump1_ramp (0-1), pump2_ramp (0-1)

| Register | Update Rule |
|----------|-------------|
| intake_flow_rate | `flow = (pump_speed / 100) * (pump1_ramp + pump2_ramp) * 50 L/s` [OT-REVIEW] Each pump contributes 50 L/s at full ramp and 100% speed. Two pumps at full = 100 L/s (scale_max). Parallel centrifugal pumps add flow at the same head -- averaging would show no change when second pump starts. |
| intake_pump_speed | Read from store (writable setpoint, set by Modbus client) |
| raw_water_ph | `drift(current, center=7.2, rate=0.01) + noise` |
| raw_water_turbidity | `drift(current, center=35, rate=0.1) + noise` |
| intake_water_temp | `drift(current, center=18, rate=0.02) + noise` |
| intake_pump_01_run | Read coil, drive pump1_ramp toward 1.0 or 0.0 |
| intake_pump_02_run | Read coil, drive pump2_ramp toward 1.0 or 0.0 |
| screen_wash_active | Random: 5% chance true per tick, auto-clear after 3 ticks |
| low_well_level_alarm | True when simulated well level < 30% (well level is inverse of flow) |

### Water Treatment Model

**State**: filter_fouling (0-1, increases slowly), chlorine_level, uv_on

| Register | Update Rule |
|----------|-------------|
| filter_inlet_pressure | `50 + (intake_flow_rate_est * 0.3) kPa + noise` (estimated from prior tick) |
| filter_outlet_pressure | `inlet - (fouling * 50) kPa` |
| filter_differential_pressure | `inlet - outlet` (directly computed) |
| uv_intensity | `base 85 mW/cm2 when active. Apply slow aging: -0.0001/tick. Noise ±2%. If uv_system_active reads false: decay toward 0 at 0.5/tick.` [OT-REVIEW] Models lamp at ~80% life. Slow aging teaches that UV intensity is a monitored parameter. |
| chemical_feed_rate | Read from store (writable setpoint) |
| chlorine_residual | `if feed_pump_on: converge toward feed_rate * 0.005; decay 0.001/tick` |
| turbidity_post_filter | `base_turbidity * (1 - filter_efficiency) + noise` where efficiency decreases with fouling |
| filter_backwash_command | Read coil. When true: begin fouling decay at 0.05/tick (full reset takes ~20 ticks / ~20 seconds). turbidity_post_filter rises during first 10 ticks (disturbed bed), then falls. Auto-clear coil after 20 ticks. [OT-REVIEW] Real backwash takes 3-15 minutes; 20 seconds is compressed for training but preserves the concept that backwash takes time and affects water quality during the cycle. |
| chemical_feed_pump_run | Read coil. Gates chemical_feed_rate |
| uv_system_active | Static true initially. [OT-REVIEW] Coil is writable: false per device atom, so external writes are blocked. Future scenario can inject lamp fault via test injection. |
| high_dp_alarm | True when DP > 25 kPa, false when DP < 15 kPa (hysteresis) |

### Water Distribution Model

**State**: pump1_ramp, pump2_ramp, well_level

| Register | Update Rule |
|----------|-------------|
| clear_well_level | `level += inflow_rate - outflow_rate` per tick. Inflow from treatment (estimated). Outflow from distribution pumps. |
| distribution_flow_rate | `(pump1_ramp + pump2_ramp) * 75 L/s + noise` |
| distribution_pressure | `500 - flow * 1.0 kPa + noise` [OT-REVIEW] Centrifugal pump curve: pressure DECREASES as flow increases. 500 kPa at deadhead (no flow), ~350 kPa at 150 L/s. Inverse relationship is fundamental to water distribution hydraulics. |
| residual_chlorine | `mirror treatment chlorine with -0.0005/tick decay` |
| distribution_water_temp | `mirror intake temp ± 1 degC + noise` |
| distribution_pump_01_run | Drive pump1_ramp |
| distribution_pump_02_run | Drive pump2_ramp |
| booster_pump_run | When true: pressure += 50 kPa |

### Manufacturing Line-A Model

**State**: conveyor_ramp, jam_timer, product_timer

| Register | Update Rule |
|----------|-------------|
| conveyor_speed | Read from store (writable setpoint) |
| motor_current | `if running: 5 + speed/40 + (jam ? 15 : 0) A + noise` |
| product_count | `if running and not jammed: count += 1 every 3 ticks`. Reset to 0 every 28800 ticks (8-hour shift cycle). [OT-REVIEW] Shift-change reset is via PLC ladder logic timer, not Modbus write. Register stays writable: false per device atom. |
| reject_count | `2% of product_count increments`. Reset to 0 synchronized with product_count shift reset. |
| line_temperature | `drift(72, rate=0.1) + (running ? 0.5 : -0.2) + noise` |
| cycle_time | `if running: 45 + (jam ? 10 : 0) s + noise; else 0` |
| status_word | `bit0=conveyor_run, bit2=jam_detected, bit3=e_stop (compute per tick)` |
| conveyor_run | Read coil, drive conveyor_ramp |
| e_stop_active | Random 0.05% chance per tick. Latches true. Auto-clear after 30 ticks (time for operator to reset). [OT-REVIEW] 0.5% fires every ~200s -- far too frequent for e-stop. 0.05% fires ~every 33 minutes, realistic for training. |
| jam_detected | Random 0.1% chance per tick when running. Latches true. Auto-clear after 20 ticks. [OT-REVIEW] 3% fires every ~33s -- trains engineers that jams are constant noise. 0.1% fires ~every 17 minutes, making each jam educationally observable. |

### Manufacturing Cooling Model

**State**: pump1_ramp, pump2_ramp, supply_temp_actual, tank_level

| Register | Update Rule |
|----------|-------------|
| supply_temp | `converge toward setpoint at rate 0.5 degF/tick + noise` |
| return_temp | `supply_temp + delta_T` where delta_T = 5-15 degF based on flow |
| flow_rate | `(pump1_ramp * 150 + pump2_ramp * 150) GPM + noise` |
| pump_pressure | `10 + flow/10 PSI + noise` |
| tank_level | `level -= 0.02/tick when pump running; periodic refill to 95%` |
| setpoint_temp | Read from store (writable setpoint, attack surface) |
| pump_runtime_hours | `if pump1_run: hours += 1/3600 per tick` |
| pump_1_run | Drive pump1_ramp |
| pump_2_run | Drive pump2_ramp |
| low_coolant_alarm | True when tank_level < 20%, false when > 30% (hysteresis) |
| high_temp_alarm | True when return_temp > 85 degF, false when < 75 degF (hysteresis). [OT-REVIEW] At 100 degF threshold, setpoint_temp max (80 degF) + max delta_T (15) = 95 degF, never triggers alarm. Lowered to 85 degF so setpoint attack (write 80) actually causes alarm (80+15=95 > 85). Training scenario works. |

### Gateway Model

**State**: tx_count, rx_count, uptime_ticks

| Register | Update Rule |
|----------|-------------|
| serial_port_status | Static: 1 (online) |
| serial_baud_rate | Static: 3 (9600) |
| serial_data_format | Static: 0 (8N1) |
| serial_mode | Static: 2 (RS-485 2-wire) |
| active_tcp_connections | Updated by counting actual connections (or simulated: random 0-2) |
| serial_tx_count | `tx += 2 per tick` (one poll per serial device: SLC-500 unit 1 + Modicon 984 unit 2). [OT-REVIEW] 1 tx/tick was 2-4x too slow; serial bus at 9600 baud is a contended medium with active polling. |
| serial_rx_count | `rx += 2 per tick` (matching responses from both serial devices) |
| serial_error_count | `+1 every ~500 ticks` (0.1% error rate at 2 tx/tick = 1 error per ~250 seconds, realistic for 9600 baud RS-485) |
| uptime_hours | `uptime_ticks++; hours = 8760 + uptime_ticks/3600` |

## 8. Success Criteria

1. `cd plant && go build ./cmd/plant/` compiles
2. `cd plant && go test ./...` passes all tests (existing + new)
3. Plant binary starts. After 30 seconds, FC03 reads show register values have changed from their initial midpoint values (drift + noise).
4. Write pump coil (FC05, port 5020, coil 0 = true). Over the next 5 seconds, flow_rate register (addr 0) climbs from 0 to a steady value. Not instant -- gradual ramp.
5. Write pump coil back to false. Flow_rate decreases over 5 seconds to 0.
6. On port 5030 unit 2 (Modicon 984), write setpoint_temp (wire addr 5, device atom addr 6) to a different value. supply_temp (wire addr 0, device atom addr 1) converges toward the new setpoint over several ticks. [OT-REVIEW] Modicon 984 uses one-based addressing; wire address = device atom address - 1. All success criteria addresses are wire-level (zero-based) Modbus protocol addresses.
7. Alarm coils transition: on port 5030 unit 2, when return_temp exceeds 85 degF, high_temp_alarm (wire coil addr 3, device atom addr 4) reads as true. When it drops below 75 degF, alarm clears. [OT-REVIEW] Threshold lowered from 100/90 to 85/75 so setpoint attack can actually trigger alarm.
8. Counter registers (product_count, serial_tx_count, uptime_hours) increment over time when their trigger conditions are met.
9. Sensor noise: consecutive reads of the same register return slightly different values (±1% of range).
10. Plant shuts down cleanly on SIGINT with no goroutine leaks.

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| WriteHoldingInternal bypasses writable check, could mask bugs | Medium | Low | Only the simulation engine calls it. Unit tests verify that external writes still enforce writable flags. |
| Process models produce unrealistic value correlations | Medium | Medium | Test each correlation independently. Keep models simple (linear, no PID). |
| 1-second tick is too coarse for smooth ramp visualization | Low | Medium | 5-tick ramp (5 seconds) is visible but not jarring. If too coarse, future SOW can reduce tick interval. |
| Sensor noise makes register values unpredictable for scenario validation | Low | Low | Noise is ±0.5% of range -- small enough that values are clearly correlated. Scenario success criteria use ranges, not exact values. |
