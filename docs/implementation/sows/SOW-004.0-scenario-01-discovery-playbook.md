**Status**: Approved

## 1. Objective

Write the complete Scenario 01 (Network Discovery) content so that an IT engineer can follow step-by-step instructions to discover, enumerate, and document all OT devices in the greenfield-water-mfg environment. This is Beta 0.1 Definition of Done item #8: "Follow Scenario 01 (Discovery) step-by-step and complete it successfully."

Per ADR-007 D4, scenario content is documentation, not code. This SOW produces markdown files only -- no Go source, no scripts, no automation.

## 2. Background

The existing `scenarios/01-discovery/` directory contains a placeholder README and an `environment.ref` pointing to `greenfield-water-mfg`. ADR-007 defines the scenario file structure: README.md (instructions), success-criteria.md, hints/, and solutions/.

The environment has 6 placements across two distinct network architectures:
- **Water treatment (Purdue model)**: 3 CompactLogix PLCs on ports 5020-5022, modern Ethernet-native
- **Manufacturing (flat network)**: 1 Moxa NPort gateway on port 5030, bridging 2 serial PLCs (SLC-500 at unit ID 1, Modicon 984 at unit ID 2)

The key educational challenge: serial devices behind the gateway are invisible to standard network scans. The engineer must understand Modbus unit ID routing to discover them.

### Prerequisites Available After SOW-003.0

- Registers change over time (drift, noise) -- proves devices are "alive"
- Pump coils respond to writes -- proves interactive control
- Response times differ per device (5ms CompactLogix, ~65ms SLC-500 via gateway, ~95ms Modicon via gateway). [OT-REVIEW] Two serial devices have DIFFERENT response times through the same gateway -- this is itself a discovery signal.
- Counter registers increment -- proves ongoing operation

### Non-Functional Requirements

1. **NFR-1**: A competent IT engineer with no OT experience should complete the scenario in 60-90 minutes (including hint reading time). [OT-REVIEW] Gateway discovery requires absorbing Modbus unit ID routing, which is a new concept for IT engineers.
2. **NFR-2**: All commands use standard, freely available tools. Primary: `mbpoll` (stable C binary, unchanged CLI since 2014). Secondary: `modpoll` or Python `pymodbus`. Also: `nmap` for network scanning. [OT-REVIEW] pymodbus had breaking API changes between v2 and v3; mbpoll is more stable for scenario documentation.

## 5. Design

### Scenario Narrative

The engineer has been brought in as a security consultant for Greenfield Water and Manufacturing. The facility manager knows there are "some PLCs" on the network but has no asset inventory. The engineer's job is to discover everything on the OT networks and produce a comprehensive asset inventory.

### Four Discovery Phases

**Phase A: Network Reconnaissance**
- [OT-REVIEW] Define starting position: "You have a workstation on the water treatment Level 1 network (10.10.10.0/24) with direct IP access to the Modbus TCP ports via localhost port mappings."
- Port scan localhost:5020-5039 to find open Modbus TCP ports
- Expected: 4 ports open (5020, 5021, 5022, 5030)
- Teaching point: **In production, Modbus TCP runs on port 502, not 5020.** These non-standard ports are specific to this simulator. When you enter a real OT facility, scan for port 502. Prominent callout required -- engineers must not carry 5020 into production.

**Phase B: Modbus Enumeration**
- For each open port, read holding registers (FC03) and attempt coils (FC01)
- [OT-REVIEW] Port 5030 (gateway) has no coils (max_coils: 0). FC01 returns exception 02 (Illegal Data Address). This differentiates the gateway from PLCs -- teach that a device speaking Modbus but having no coils is likely a gateway or sensor, not a PLC.
- Compare response times: 5ms for CompactLogix (ports 5020-5022), ~65ms for SLC-500 via gateway (port 5030 unit 1, includes ±20ms jitter), ~95ms for Modicon 984 via gateway (port 5030 unit 2, includes ±50ms jitter). [OT-REVIEW] The Modicon's high jitter (45ms-145ms observed range) is normal scan-time variability on a 38-year-old processor, not a network problem.
- Identify register maps by polling in batches of 10-20 registers, not bulk-polling all addresses. [OT-REVIEW] In a real facility, rapid sequential reads can cause scan time overrun and watchdog trips on legacy PLCs. Practice the discipline even in the simulator.
- Teaching point: different response times and jitter patterns reveal device types, ages, and network topology

**Phase C: Gateway Discovery (the "aha" moment)**
- Port 5030 exposes the gateway's simulated status registers when accessed with a unit ID that does not match a downstream serial device (e.g., unit ID 0 or 255). [OT-REVIEW] The Moxa NPort in TCP Server mode is transparent: it passes the Modbus unit ID to the serial bus unchanged. There is no "gateway unit ID 247" -- that was incorrect. The gateway does not have its own Modbus unit ID in production; its simulated registers are an educational abstraction.
- Engineer must try unit IDs 1 and 2 to discover the serial devices behind the gateway
- SLC-500 at unit ID 1: one-based addressing, 7 holding registers (addr 1-7), 4 coils (addr 1-4)
- Modicon 984 at unit ID 2: one-based addressing, 7 holding registers (addr 1-7), 4 coils (addr 1-4)
- [OT-REVIEW] When polling address 0 on these one-based devices, the engineer receives Modbus exception (Illegal Data Address). First valid address is 1. This is a critical teaching moment about one-based vs zero-based addressing in legacy devices.
- [OT-REVIEW] The Modicon 984 uses little-endian (CDAB) word order while the SLC-500 uses big-endian. If the engineer applies big-endian decoding from the SLC-500 to the Modicon, values will appear wrong. This byte-order difference is one of the most confusing OT protocol details.
- Unit ID 3 returns exception 0x0B (target failed to respond) -- device offline
- Teaching point: serial devices are invisible to IP-based scans; Modbus unit ID routing is the only way to reach them

**Phase D: Asset Documentation**
- Fill in the asset inventory template with all discovered information
- Compare against the expected inventory to validate completeness

## 8. Success Criteria

1. An IT engineer can follow the README.md instructions from start to finish without external documentation.
2. The scenario correctly identifies all 6 devices: 4 IP-addressable (3 CompactLogix PLCs + 1 Moxa gateway) and 2 non-IP serial devices (SLC-500 + Modicon 984, reachable only via gateway IP + unit ID). [OT-REVIEW] Asset inventory uses different column structure for serial devices: no IP column, but "Gateway IP + Unit ID" instead.
3. All port numbers in the scenario (5020, 5021, 5022, 5030) match the environment.yaml placements.
4. All register addresses, names, and counts match the device atom YAML files exactly. [OT-REVIEW] Note: water-intake has 4 coils, water-treatment has 4 coils, but water-distribution has only 3 coils. Do not assume uniform coil counts across variants.
5. The gateway discovery section correctly documents that the NPort gateway is transparent (no own unit ID), and serial devices are reached via unit ID 1 (SLC-500) and unit ID 2 (Modicon 984). [OT-REVIEW] Removed incorrect "247=gateway" -- NPort 5150 has no documented Modbus unit ID for its own registers.
6. Response time differences are documented per-device: 5ms CompactLogix, ~65ms SLC-500 via gateway (±20ms jitter), ~95ms Modicon via gateway (±50ms jitter). [OT-REVIEW] Two serial devices have different times through the same gateway -- this is a discovery signal.
7. The scenario teaches that serial devices are invisible to network scans -- this is the key learning objective.
8. The expected asset inventory contains every register from every device atom variant.
9. Hints are progressive -- reading hint-1 alone should not give away the full answer.
10. The solution walkthrough produces output consistent with SOW-002.0 (static) and SOW-003.0 (dynamic) register behavior.

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Modbus client tool availability varies across platforms | Medium | Medium | Document multiple client options (modpoll, mbpoll, pymodbus CLI). Scenario focuses on protocol concepts, not specific tool syntax. |
| Register values with noise make exact output matching impossible | Low | High | All expected outputs use ranges and patterns, not exact values. E.g., "you should see a value between 14000 and 18000" not "you should see 16383". |
| Engineer may not understand raw-to-engineering-unit scaling | Medium | Medium | Include a sidebar explaining the 0-32767 scaling convention. Not the main learning objective but needed for comprehension. |
| nmap version differences may produce different output formatting | Low | Medium | Show the essential information to look for, not exact nmap output format. |
