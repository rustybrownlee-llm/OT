# SOW-005.0 - Docker Compose Integration

**Status**: Approved
**Created**: 2026-03-01
**Priority**: Medium

## 1. Objective

Ensure `docker compose up` brings up a fully functional OT simulation environment where all four Modbus TCP listeners are serving registers, health checks confirm readiness, and `docker compose down` shuts down cleanly. This is Beta 0.1 Definition of Done item #1: "Run `docker compose up` and get a working OT environment."

## 2. Background

The Docker Compose infrastructure is approximately 70% complete from SOW-000.0. The plant Dockerfile and docker-compose.yml network topology are functional. However, the monitoring service is broken (missing config file), neither service has health checks, and the `depends_on` directive uses start ordering only (no readiness gating).

### Current State

- `docker-compose.yml`: 5 networks with correct Purdue topology, 2 services (plant, monitor), port mappings for 5020-5030 and 8090-8091. No health checks.
- `plant/Dockerfile`: Functional multi-stage build. Design layer copied into image. Entrypoint uses absolute path `/design/environments/greenfield-water-mfg`.
- `monitoring/Dockerfile`: Builds but runtime is broken -- entrypoint expects `/config/monitor.yaml` which does not exist in the image or as a mounted volume.
- `monitoring/config/`: Directory does not exist. No config template.

### What Needs to Change

1. Create `monitoring/config/monitor.yaml` with basic plant connection settings
2. Mount monitor config volume in docker-compose.yml
3. Add health checks to both services
4. Upgrade `depends_on` to use `service_healthy` condition
5. Verify end-to-end: `docker compose up` -> Modbus ports accessible -> clean shutdown

## 4. Requirements

### Functional Requirements

1. **FR-1**: `docker compose up --build` builds both services and starts them with no errors.
2. **FR-2**: Plant service health check verifies that all four Modbus TCP listeners are accepting connections (ports 5020, 5021, 5022, 5030). The check uses a TCP connection test via the binary's own `--health` flag with 2s timeout per port. [OT-REVIEW] Checking only port 5020 would silently pass if the manufacturing floor (port 5030) failed to bind. The water-treatment-vs-manufacturing contrast is the core educational lesson; health check must verify both sides.
3. **FR-3**: Monitor service health check verifies the process is running. Since the monitor binary is a stub with no real functionality, the health check confirms the process has not crashed. [OT-REVIEW] The monitor's own health check is independent from the startup ordering gate. It only affects `docker compose ps` status reporting.
4. **FR-4**: Monitor service starts only after plant service health check passes (`depends_on: plant: condition: service_healthy`). [OT-REVIEW] The `depends_on: condition: service_healthy` gates monitor container STARTUP on the plant container's health check. The monitor's own health check (FR-3) is unrelated to this gate.
5. **FR-5**: `monitoring/config/monitor.yaml` exists with plant connection settings (address, log level). The file is mounted read-only into the monitor container.
6. **FR-6**: `docker compose down` stops both services cleanly with no error messages. Plant binary logs graceful shutdown.
7. **FR-7**: Modbus TCP ports 5020-5022 and 5030 are accessible from the host machine after startup.
8. **FR-8**: HMI port mappings (8080, 8081) are removed from docker-compose.yml since no HMI exists in Beta 0.1. They remain in the plant Dockerfile EXPOSE for future use.

### Non-Functional Requirements

1. **NFR-1**: After `docker compose up` (with pre-built images), both services reach healthy status within 30 seconds. [OT-REVIEW] Build time (`--build`) is unbounded and not included in this NFR.
2. **NFR-2**: `docker compose down` completes within 10 seconds

## 5. Design

### Health Check Strategy

The plant binary runs in a `scratch` container (no shell, no curl, no wget). Standard Docker health checks use shell commands, which are unavailable. Two approaches:

**Option A: Add a `--health` flag to the plant binary** that performs a self-check (attempt TCP connection to its own first Modbus port). If healthy, exit 0. If not, exit 1. The Dockerfile uses: `HEALTHCHECK CMD ["/plant", "--health"]`

**Option B: Use Docker's native TCP health check** via a small health check binary copied into the scratch image. This is overengineered for Beta 0.1.

**Chosen: Option A.** The plant binary already has flag parsing. Adding `--health` is minimal code: connect to all four registered Modbus ports (`localhost:5020`, `5021`, `5022`, `5030`) with 2s timeout each. If all succeed, exit 0. If any fail, exit 1. This keeps the scratch image minimal and avoids adding external tools.

For the monitoring service: same pattern. Add `--health` flag that exits 0 if the process can read its config file successfully.

### Monitor Config Template

```yaml
# monitor.yaml - Monitoring module configuration
# Plant connection settings for the monitoring module to observe the OT environment.
log_level: "info"

plant:
  # [OT-REVIEW] In this simulator, the plant binary hosts all virtual PLCs and is reachable
  # at multiple IPs. The monitor polls via the wt-level3 interface (10.10.30.10) because both
  # containers share that Docker network.
  # In a real facility, a SCADA monitor would poll Level 1 PLC addresses directly
  # (10.10.10.10-12) or via a DMZ historian/proxy -- NOT via the Level 3 firewall address.
  address: "10.10.30.10"    # simulator abstraction: plant binary is multi-homed
  modbus_endpoints:
    - port: 5020
      unit_id: 1             # wt-plc-01 (water intake) -- real facility Level 1 addr: 10.10.10.10
    - port: 5021
      unit_id: 1             # wt-plc-02 (water treatment) -- real facility Level 1 addr: 10.10.10.11
    - port: 5022
      unit_id: 1             # wt-plc-03 (water distribution) -- real facility Level 1 addr: 10.10.10.12
    - port: 5030
      # [OT-REVIEW] NPort 5150 does NOT serve registers -- it relays Modbus RTU to serial devices.
      # Port 5030 reaches serial devices via unit ID routing, not the gateway itself.
      unit_ids: [1, 2]       # mfg-serial-bus: unit 1 = SLC-500, unit 2 = Modicon 984
```

### Docker Compose Changes

```yaml
services:
  plant:
    healthcheck:
      test: ["/plant", "--health"]
      interval: 5s
      timeout: 3s
      retries: 3
      start_period: 10s
    # Remove ports 8080, 8081 (no HMI in Beta 0.1)

  monitor:
    depends_on:
      plant:
        condition: service_healthy
    volumes:
      - ./monitoring/config:/config:ro
    healthcheck:
      test: ["/monitor", "--health"]
      interval: 5s
      timeout: 3s
      retries: 3
      start_period: 5s
```

## 6. Implementation Plan

### Phase 1: Plant Health Check

1. Add `--health` flag to `plant/cmd/plant/main.go`
2. When `--health` is passed: attempt TCP connection to `localhost:5020`, `5021`, `5022`, and `5030` with 2s timeout each. Exit 0 if all succeed, exit 1 if any fail. Do not start the server.
3. Add `healthcheck` section to plant service in `docker-compose.yml`

11. Run `docker compose up --build` and verify both services reach healthy status
12. Test Modbus port accessibility from host: connect to `localhost:5020` and read a register
13. Run `docker compose down` and verify clean shutdown
14. Run `docker compose up -d && docker compose ps` and verify both services show "healthy"

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Scratch image cannot execute health check binary | High | Low | The plant binary IS the health check binary (--health flag). No external tools needed. |
| Port 5020 may conflict with other services on developer machines | Medium | Medium | Document in README. Ports 5020-5030 are non-standard and unlikely to conflict. |
| Docker Compose v1 vs v2 syntax differences | Low | Low | Use v2 syntax (no `version` key). Docker Compose v2 is the current standard. |
