# SOW-006.0 - Design Layer Validation CLI

**Status**: Complete
**Created**: 2026-03-01
**Priority**: High
**OT Domain Review**: Complete (5 critical corrections, 7 new rules added)

## 1. Objective

Create a standalone CLI tool (`ot-design`) in the `design/` layer that validates design layer YAML files against the v0.1 schema and scaffolds new device, network, and environment skeletons. When an engineer runs `ot-design validate design/environments/pipeline-monitoring/`, they get a clear pass/fail with actionable error messages pinpointing exactly what is wrong and where. When they run `ot-design scaffold --device`, they get a ready-to-edit YAML file with every required field and explanatory comments.

After this SOW, the design layer is no longer "validated at runtime by the plant binary." It has its own standalone validation tool that catches errors before anyone attempts to run a simulation, closing the feedback loop from "plant crashes at startup" to "validation fails with a helpful message."

## 2. Background

ADR-009 D1 explicitly anticipated a future Go module in `design/`: "A future Go module in design/ may provide guided editing of YAML files, but this is deferred." This SOW delivers on that commitment.

Currently, design layer validation happens implicitly when the plant binary parses YAML at startup. This has several problems:

- **Late feedback**: Errors surface only when running the plant binary, not during YAML authoring.
- **Unclear messages**: Plant startup errors are logged as Go unmarshalling errors, not as domain-specific guidance ("register address 5 exceeds max_holding of 4" vs "yaml: line 47: cannot unmarshal !!int into string").
- **No cross-reference checking**: The plant binary validates individual files but does not check cross-references (e.g., does the device referenced by a placement actually exist in `design/devices/`?).
- **No authoring assistance**: Engineers creating new environments start from copy-paste of existing files, which propagates errors and stale comments.

The milestone-b0.2.md specification defines the validation CLI as "the cornerstone of this milestone" and lists specific validation rules for device atoms, network atoms, and environments. This SOW implements those rules exactly.

### Current State

- `design/` contains only YAML files: 4 device atoms, 6 network atoms, 1 environment
- No `go.mod` exists in `design/` -- this SOW creates it
- The plant binary in `plant/internal/config/` has YAML parsing structs but no standalone validation logic
- No scaffold tooling exists anywhere in the project

## 3. Scope

### In Scope

- Create a standalone Go module at `design/` with `go.mod` (depends only on `gopkg.in/yaml.v3`)
- Implement `ot-design validate <path>` command that validates:
  - Individual device atom YAML files
  - Individual network atom YAML files
  - Entire environment directories (environment.yaml + cross-references to devices and networks)
- Implement all validation rules from milestone-b0.2.md "Design Layer Validation Tool" section (see Section 5 for full list)
- Implement `ot-design scaffold --device|--network|--environment` command
- Human-readable output with file paths, line context, and actionable error descriptions
- Exit code 0 for pass, 1 for validation failures, 2 for usage errors (bad arguments, missing files)
- Unit tests with table-driven patterns for all validation rules
- Validate the tool against the existing design layer files (all 4 devices, 6 networks, 1 environment must pass)

### Out of Scope

- JSON output format (future enhancement, not in this SOW)
- Schema migration between versions (deferred to future SOW)
- Integration with the plant binary (plant continues to use its own config parsing)
- Watch mode / file system monitoring for live validation
- GUI or web-based validation interface
- Validation of v0.2+ schema features (security_profile, communication_patterns, monitoring overlays)
- CI/CD integration (can be done with the CLI, but pipeline setup is not in scope)

## 4. Requirements

### Functional Requirements

1. **FR-1**: The CLI binary is built from `design/cmd/ot-design/main.go` and produces an `ot-design` binary. The command `cd design && go build ./cmd/ot-design/` compiles without errors.

2. **FR-2**: `ot-design validate <path>` accepts three types of targets:
   - A path to a device atom YAML file (e.g., `design/devices/compactlogix-l33er.yaml`)
   - A path to a network atom YAML file (e.g., `design/networks/wt-level1.yaml`)
   - A path to an environment directory (e.g., `design/environments/greenfield-water-mfg/`)
   The tool auto-detects the type by parsing the YAML and checking for top-level keys: `device:` indicates a device atom, `network:` indicates a network atom, `environment:` indicates an environment (loaded from `environment.yaml` within the directory).

3. **FR-3**: When validating an environment directory, the tool resolves cross-references by walking up to find the design root directory (the directory containing `devices/`, `networks/`, and `environments/` subdirectories). If the design root cannot be found, the tool prints an error and exits with code 2.

4. **FR-4**: Validation output lists all errors found (not just the first one). Each error includes:
   - File path (absolute or relative to the working directory)
   - Field path within the YAML (e.g., `register_map_variants.water-intake.holding[2].scale_min`)
   - A human-readable description of what is wrong and how to fix it
   - Example: `design/devices/compactlogix-l33er.yaml: register_map_variants.water-intake.holding[2].scale_min: scale_min (100) must be less than scale_max (50)`

5. **FR-5**: Exit codes:
   - 0: All validations pass
   - 1: One or more validation errors found
   - 2: Usage error (bad arguments, file not found, unparseable YAML, design root not found)

6. **FR-6**: `ot-design scaffold --device` writes a skeleton device atom YAML to stdout. The skeleton includes all required fields with placeholder values and YAML comments explaining each field, its type, valid values, and purpose. The skeleton is valid YAML that would pass structural validation (though placeholder values like `"your-device-id"` would fail semantic validation).

7. **FR-7**: `ot-design scaffold --network` writes a skeleton network atom YAML to stdout with the same comment-documentation approach.

8. **FR-8**: `ot-design scaffold --environment` writes a skeleton environment directory structure:
   - Creates an `environment.yaml` file to stdout with placeholder placements and comments
   - Prints instructions to stderr explaining that the file should be saved to `design/environments/<your-env-id>/environment.yaml`

9. **FR-9**: `ot-design` with no arguments or `ot-design --help` prints usage information showing both subcommands and examples.

10. **FR-10**: `ot-design validate` with no path argument prints an error message and usage, then exits with code 2.

### Non-Functional Requirements

1. **NFR-1**: Validation of the existing greenfield-water-mfg environment (4 devices, 6 networks, 6 placements) completes in under 500ms.
2. **NFR-2**: The tool has zero external dependencies beyond Go stdlib and `gopkg.in/yaml.v3`. No CLI frameworks (Cobra, urfave/cli, etc.).
3. **NFR-3**: All new code has unit tests with table-driven patterns. Minimum 80% line coverage on validation logic.
4. **NFR-4**: The `design/` Go module does not import any packages from `plant/` or `monitoring/`. It is fully independent.

## 5. Design

### CLI Argument Parsing

The CLI uses `os.Args` directly -- no CLI framework. The argument structure is minimal:

```
ot-design validate <path>           # Validate a file or directory
ot-design scaffold --device         # Scaffold device atom
ot-design scaffold --network        # Scaffold network atom
ot-design scaffold --environment    # Scaffold environment
ot-design --help                    # Usage information
```

`main.go` dispatches based on `os.Args[1]`:
- `"validate"` -> `runValidate(os.Args[2:])`
- `"scaffold"` -> `runScaffold(os.Args[2:])`
- `"--help"` or no args -> `printUsage()`

### Design Root Resolution

The validation tool needs to find the design root to resolve cross-references. The resolution algorithm:

1. Start from the validated path
2. Walk up the directory tree looking for a directory that contains `devices/`, `networks/`, and `environments/` subdirectories
3. If found, that is the design root
4. If the filesystem root is reached without finding it, return an error

For individual file validation (device or network), the design root is not required (no cross-references to resolve). For environment validation, the design root is mandatory.

### Auto-Detection of File Type

When `ot-design validate <path>` is invoked:

1. If `<path>` is a directory, look for `environment.yaml` inside it. If found, validate as environment. If not found, exit with error.
2. If `<path>` is a file, parse the YAML and check for top-level keys:
   - `device:` key present -> validate as device atom
   - `network:` key present -> validate as network atom
   - `environment:` key present -> validate as environment (use the file's parent directory as context)
3. If none of the expected keys are found, print an error: "Unrecognized YAML file: expected top-level 'device:', 'network:', or 'environment:' key" and exit with code 2.

### YAML Types for Parsing

The validation package defines its own YAML structs (not shared with `plant/`). These structs are intentionally permissive -- they parse all fields as their natural types without strict Go struct validation, so the validation logic can produce domain-specific error messages instead of Go unmarshalling errors.

```go
// yaml.go

// RawDocument holds the top-level YAML for type detection.
type RawDocument struct {
    SchemaVersion string          `yaml:"schema_version"`
    Device        *DeviceDoc      `yaml:"device"`
    Network       *NetworkDoc     `yaml:"network"`
    Environment   *EnvironmentDoc `yaml:"environment"`
}

// DeviceDoc represents a device atom YAML for validation.
type DeviceDoc struct {
    ID           string `yaml:"id"`
    Vendor       string `yaml:"vendor"`
    Model        string `yaml:"model"`
    Category     string `yaml:"category"`
    Vintage      int    `yaml:"vintage"`
    Description  string `yaml:"description"`
}

// ConnectivityDoc represents the connectivity section.
type ConnectivityDoc struct {
    Ports                []PortDoc `yaml:"ports"`
    ResponseDelayMs      *int      `yaml:"response_delay_ms"`
    ResponseJitterMs     *int      `yaml:"response_jitter_ms"`
    ConcurrentConnections *int     `yaml:"concurrent_connections"`
}

// PortDoc represents a connectivity port entry.
type PortDoc struct {
    Type      string   `yaml:"type"`
    Protocols []string `yaml:"protocols"`
}

// RegistersDoc represents the register capabilities section.
type RegistersDoc struct {
    MaxHolding         *int    `yaml:"max_holding"`
    MaxCoils           *int    `yaml:"max_coils"`
    MaxInputRegisters  *int    `yaml:"max_input_registers"`
    MaxDiscreteInputs  *int    `yaml:"max_discrete_inputs"`
    Addressing         string  `yaml:"addressing"`
    FloatByteOrder     string  `yaml:"float_byte_order"`
    MaxRegistersPerRead *int   `yaml:"max_registers_per_read"`
}

// HoldingRegisterDoc represents a single holding register entry in a variant.
type HoldingRegisterDoc struct {
    Address     *int     `yaml:"address"`
    Name        string   `yaml:"name"`
    Unit        string   `yaml:"unit"`
    ScaleMin    *float64 `yaml:"scale_min"`
    ScaleMax    *float64 `yaml:"scale_max"`
    Writable    *bool    `yaml:"writable"`
    Description string   `yaml:"description"`
}

// CoilDoc represents a single coil entry in a variant.
type CoilDoc struct {
    Address     *int   `yaml:"address"`
    Name        string `yaml:"name"`
    Writable    *bool  `yaml:"writable"`
    Description string `yaml:"description"`
}

// VariantDoc represents a register map variant.
type VariantDoc struct {
    Holding []HoldingRegisterDoc `yaml:"holding"`
    Coils   []CoilDoc            `yaml:"coils"`
}

// NetworkDoc represents a network atom YAML for validation.
type NetworkDoc struct {
    ID          string `yaml:"id"`
    Name        string `yaml:"name"`
    Type        string `yaml:"type"`
    Description string `yaml:"description"`
}

// NetworkPropertiesDoc represents the network properties section.
type NetworkPropertiesDoc struct {
    Subnet       string `yaml:"subnet"`
    VLAN         *int   `yaml:"vlan"`
    ManagedSwitch *bool `yaml:"managed_switch"`
    SpanCapable   *bool `yaml:"span_capable"`
}

// EnvironmentDoc represents an environment YAML for validation.
type EnvironmentDoc struct {
    ID          string `yaml:"id"`
    Name        string `yaml:"name"`
    Description string `yaml:"description"`
}

// NetworkRefDoc represents a network reference in an environment.
type NetworkRefDoc struct {
    Ref string `yaml:"ref"`
}

// PlacementDoc represents a device placement in an environment.
type PlacementDoc struct {
    ID                  string              `yaml:"id"`
    Device              string              `yaml:"device"`
    Network             string              `yaml:"network"`
    IP                  string              `yaml:"ip"`
    ModbusPort          *int                `yaml:"modbus_port"`
    SerialAddress       *int                `yaml:"serial_address"`
    Gateway             string              `yaml:"gateway"`
    Role                string              `yaml:"role"`
    RegisterMapVariant  string              `yaml:"register_map_variant"`
    Bridges             []BridgeDoc         `yaml:"bridges"`
    AdditionalNetworks  []AdditionalNetDoc  `yaml:"additional_networks"`
}

// BridgeDoc represents a gateway bridge entry.
type BridgeDoc struct {
    FromNetwork string `yaml:"from_network"`
    ToNetwork   string `yaml:"to_network"`
}

// AdditionalNetDoc represents an additional network attachment.
type AdditionalNetDoc struct {
    Network string `yaml:"network"`
    IP      string `yaml:"ip"`
}
```

### Validation Rules (from milestone-b0.2.md)

#### Device Atom Validation (`validate/device.go`)

| Rule ID | Check | Error Message Pattern |
|---------|-------|----------------------|
| DEV-001 | `schema_version` is present and is `"0.1"` | `schema_version: missing or unsupported (got "X", expected "0.1")` |
| DEV-002 | `device.id` is present and non-empty | `device.id: required field is missing or empty` |
| DEV-003 | `device.vendor` is present and non-empty | `device.vendor: required field is missing or empty` |
| DEV-004 | `device.model` is present and non-empty | `device.model: required field is missing or empty` |
| DEV-005 | `device.category` is one of: `plc`, `gateway`, `hmi`, `sensor`, `relay`, `safety-controller` | `device.category: invalid value "X" (must be one of: plc, gateway, hmi, sensor, relay, safety-controller)` |
| DEV-006 | `connectivity.protocols` on at least one port includes a protocol | `connectivity.ports: at least one port with a protocol is required` |
| DEV-007 | `registers.addressing` is `"zero-based"` or `"one-based"` | `registers.addressing: invalid value "X" (must be "zero-based" or "one-based")` |
| DEV-008 | `registers.max_holding` is present and >= 0 | `registers.max_holding: required field, must be >= 0` |
| DEV-009 | `registers.max_coils` is present and >= 0 | `registers.max_coils: required field, must be >= 0` |
| DEV-010 | [OT-REVIEW] For each variant, all holding register addresses must be within the device's max_holding capacity. Gaps between addresses are expected and normal for real PLC register maps. For addressing bounds, see DEV-016. | `register_map_variants.X.holding[N].address: address Y exceeds max_holding (Z)` |
| DEV-011 | For each variant, coil addresses are within `max_coils` | `register_map_variants.X.coils[N].address: address Y exceeds max_coils (Z)` |
| DEV-012 | For each holding register, `scale_min < scale_max` | `register_map_variants.X.holding[N]: scale_min (A) must be less than scale_max (B)` |
| DEV-013 | For each holding register, `unit` is non-empty | `register_map_variants.X.holding[N].unit: required field for holding registers` |
| DEV-014 | Variant register names are unique within a variant (no duplicate names in `holding`, no duplicate names in `coils`) | `register_map_variants.X.holding: duplicate register name "Y"` |
| DEV-015 | Variant register addresses are unique within a variant (no duplicate addresses in `holding`, no duplicate addresses in `coils`) | `register_map_variants.X.holding: duplicate register address Y` |
| DEV-016 | [OT-REVIEW] For one-based devices with max_holding N, valid addresses are 1 through N. For zero-based devices with max_holding N, valid addresses are 0 through N-1. DEV-010 must use this distinction for upper-bound checks. | `register_map_variants.X.holding[N].address: address 0 is invalid for one-based addressing` / `register_map_variants.X.holding[N].address: address Y exceeds max address N for [zero|one]-based addressing` |
| DEV-017 | [OT-REVIEW] (Warning) If `connectivity.response_delay_ms` is present, warn if outside 2-2000ms for devices with an ethernet port, or outside 10-2000ms for devices with only serial ports. Serial devices have higher minimum latency due to bus arbitration. | `connectivity.response_delay_ms: value X is outside expected range [min]-2000ms for [ethernet\|serial] devices` |
| DEV-018 | [OT-REVIEW] `registers.float_byte_order` must be one of: `big-endian`, `little-endian`, `big-endian-byte-swap`, `little-endian-byte-swap`. (Warning) If vendor contains "Schneider" or "Modicon" and float_byte_order is `big-endian`, warn (Schneider/Modicon typically uses `big-endian-byte-swap`). (Warning) If vendor contains "Allen-Bradley" and float_byte_order is not `big-endian`, warn (Allen-Bradley typically uses `big-endian`). | `registers.float_byte_order: invalid value "X" (must be one of: big-endian, little-endian, big-endian-byte-swap, little-endian-byte-swap)` / `registers.float_byte_order: vendor "X" typically uses "Y", got "Z"` |
| DEV-019 | [OT-REVIEW] (Warning) If a holding register name contains measurement-pattern substrings (`_flow`, `_pressure`, `_temp`, `_level`, `_ph`, `_turbidity`, `_current`, `_alarm`, `_status`) AND `writable: true`, emit warning. Sensor measurement registers are typically read-only; writable measurement registers may indicate a misidentified register. | `register_map_variants.X.holding[N]: register "Y" appears to be a sensor measurement but is marked writable` |

#### Network Atom Validation (`validate/network.go`)

| Rule ID | Check | Error Message Pattern |
|---------|-------|----------------------|
| NET-001 | `schema_version` is present and is `"0.1"` | `schema_version: missing or unsupported (got "X", expected "0.1")` |
| NET-002 | `network.id` is present and non-empty | `network.id: required field is missing or empty` |
| NET-003 | [OT-REVIEW] `network.type` is one of: `ethernet`, `serial-rs485`, `serial-rs232`, `serial-rs422`. | NET-005 | For `serial-rs485`, `serial-rs232`, and `serial-rs422` types, `properties.subnet` is absent or empty | `properties.subnet: serial networks must not have a subnet field` |

#### Environment Validation (`validate/environment.go`)

| Rule ID | Check | Error Message Pattern |
|---------|-------|----------------------|
| ENV-001 | `schema_version` is present and is `"0.1"` | `schema_version: missing or unsupported` |
| ENV-002 | `environment.id` is present and non-empty | `environment.id: required field is missing or empty` |
| ENV-003 | All `networks[].ref` values reference a YAML file in `design/networks/` that exists and passes network validation | `networks[N].ref: network "X" not found (expected design/networks/X.yaml)` |
| ENV-004 | All `placements[].device` values reference a YAML file in `design/devices/` that exists and passes device validation | `placements[N].device: device "X" not found (expected design/devices/X.yaml)` |
| ENV-005 | All `placements[].register_map_variant` values exist in the referenced device's `register_map_variants` | `placements[N].register_map_variant: variant "X" not found in device "Y" (available: [A, B, C])` |
| ENV-006 | All `placements[].id` values are unique within the environment | `placements[N].id: duplicate placement ID "X" (first seen at placements[M])` |
| ENV-007 | All `placements[].network` values reference a network listed in the environment's `networks[]` section | `placements[N].network: network "X" not listed in environment networks` |
| ENV-008 | For placements on Ethernet networks, `ip` is present and is a valid IPv4 address within the network's subnet CIDR | `placements[N].ip: address "X" is not within subnet "Y" of network "Z"` |
| ENV-009 | No two placements on the same network have the same IP address | `placements[N].ip: duplicate IP "X" on network "Y" (first seen at placements[M])` |
| ENV-010 | [OT-REVIEW] No two placements have the same `modbus_port` value (port collision). This check only applies to placements where `modbus_port` is present and non-zero. Serial device placements accessed via gateway do not have a `modbus_port` field and must be excluded from this check. | `placements[N].modbus_port: port X already used by placement "Y"` |
| ENV-011 | For placements with `gateway` field, the referenced gateway placement ID exists in the environment | `placements[N].gateway: gateway placement "X" not found in environment` |
| ENV-012 | [OT-REVIEW] For placements with `gateway` field, the referenced gateway placement has a `bridges` entry where the serial device's network appears specifically in `to_network` (the serial side) and `from_network` is an Ethernet network. The directionality matters: `from_network` is always the Ethernet side, `to_network` is always the serial side. | `placements[N].gateway: gateway "X" does not bridge to network "Y" (serial network must appear in to_network, not from_network)` |
| ENV-013 | Serial addresses are unique per gateway (no two serial devices on the same gateway share a `serial_address`) | `placements[N].serial_address: address X already used by placement "Y" on gateway "Z"` |
| ENV-014 | For placements with `additional_networks`, each referenced network is listed in the environment's `networks[]` section | `placements[N].additional_networks[M].network: network "X" not listed in environment networks` |
| ENV-015 | For placements with `additional_networks` on Ethernet networks, the IP is valid and within the network's subnet | `placements[N].additional_networks[M].ip: address "X" is not within subnet "Y" of network "Z"` |
| ENV-016 | Gateway placements (device category = "gateway") must have a `bridges` field | `placements[N]: device "X" is a gateway but has no bridges field` |
| ENV-017 | [OT-REVIEW] If a placement has a `gateway` field, it must NOT have an `ip` or `modbus_port` field. Serial devices accessed via gateway are IP-invisible and do not listen on a Modbus TCP port. | `placements[N]: serial device accessed via gateway must not have an ip or modbus_port (serial devices are IP-invisible)` |
| ENV-018 | [OT-REVIEW] For each placement, verify the referenced device's connectivity ports contain a port type matching the placement's network type. Ethernet network requires an `ethernet` port, `serial-rs485` network requires an `rs485` port, `serial-rs232` requires an `rs232` port, `serial-rs422` requires an `rs422` port. | `placements[N]: device "X" has no port matching network type "Y"` |
| ENV-019 | [OT-REVIEW] If `serial_address` is present, it must be in range 1-247. Address 0 is Modbus broadcast (not a valid device address), addresses 248-255 are reserved by the Modbus specification. | `placements[N].serial_address: address X is outside valid range 1-247 (0 is broadcast, 248-255 are reserved)` |

### Validation Result Types

```go
// result.go

// Severity indicates how serious a validation finding is.
type Severity int

const (
    SeverityError   Severity = iota  // Must fix: file will not work
    SeverityWarning                  // Should fix: may cause confusion
)

// ValidationError represents a single validation finding.
type ValidationError struct {
    File     string   // File path where the error was found
    Field    string   // YAML field path (e.g., "register_map_variants.water-intake.holding[2].scale_min")
    Message  string   // Human-readable error description
    Severity Severity // Error or Warning
    RuleID   string   // e.g., "DEV-012" for traceability
}

// ValidationResult collects all findings from a validation run.
type ValidationResult struct {
    Errors []ValidationError
}

// HasErrors returns true if any findings have SeverityError.
func (r *ValidationResult) HasErrors() bool

// Add appends a validation error.
func (r *ValidationResult) Add(err ValidationError)

// String formats the result for human-readable output.
// Format per error: "<file>: <field>: <message> [<rule-id>]"
// Summary line at end: "Validation complete: N error(s), M warning(s)"
// If no errors: "Validation passed: <file-or-directory>"
func (r *ValidationResult) String() string
```

### Scaffold Templates

Each scaffold command writes a complete, commented YAML skeleton to stdout. The output is designed to be redirected to a file:

```
ot-design scaffold --device > design/devices/my-new-device.yaml
ot-design scaffold --network > design/networks/my-new-network.yaml
ot-design scaffold --environment > design/environments/my-new-env/environment.yaml
```

Scaffold output includes:
- Every required field with a placeholder value
- YAML comments on each field explaining its purpose, type, and valid values
- Example values that illustrate the expected format
- Comments pointing to existing files as references (e.g., "# See design/devices/compactlogix-l33er.yaml for a complete example")
- [OT-REVIEW] Device scaffold register map examples MUST show non-sequential register addresses with gaps (e.g., addresses 1, 2, 3 then 10, 11, 12) and include both zero-based and one-based addressing examples with comments explaining the difference. Scaffold MUST NOT show dense sequential addresses starting at 0 -- this teaches an IT-centric misconception that PLC register maps are contiguous like memory arrays

The scaffold templates are embedded in Go source as string constants -- not loaded from external template files.

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Design root resolution fails in unusual directory structures (e.g., running from `/tmp` with a symlinked design path) | Medium | Low | Use `filepath.Abs()` to canonicalize paths before walking. Document that the tool expects the standard `design/` directory structure from ADR-009 D7. |
| Validation rules are too strict and reject valid YAML that the plant binary accepts | High | Medium | Run all validators against all existing design layer files as part of Phase 2/3/4. Any false positive is a bug to fix before the SOW is complete. |
| Validation rules are too lenient and miss errors that the plant binary catches at startup | Medium | Low | Cross-reference the plant binary's `config.LoadEnvironment()` logic during implementation. The validation tool should catch a superset of what the plant binary checks. |
| CIDR subnet containment check has edge cases (broadcast addresses, network addresses) | Low | Low | Use Go stdlib `net.ParseCIDR()` and `net.IP.Mask()` for all CIDR operations. Do not implement custom subnet math. The stdlib handles all edge cases. |
| Scaffold templates become stale as the schema evolves | Low | High | Scaffold templates are string constants in Go source, not external files. Updating them requires a code change, which makes staleness visible. When schema v0.2 is defined, the scaffold templates must be updated in the same SOW. |
| Go module path `github.com/[owner]/ot-simulator/design` does not match existing module conventions | Low | Low | Check `plant/go.mod` for the actual module path prefix during implementation and use the same convention. |
