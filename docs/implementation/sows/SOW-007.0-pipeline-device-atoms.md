**Status**: Complete
**Created**: 2026-03-01
**Completed**: 2026-03-01

## 1. Objective

Create two new device atoms (Emerson ROC800, ABB TotalFlow G5) and add two new register map variants to existing device atoms (CompactLogix `compressor-control`, Moxa NPort `pipeline-serial`) for the pipeline monitoring station environment defined in the Beta 0.2 milestone. This is a design layer SOW -- all deliverables are YAML files in `design/devices/`. No Go code changes.

The pipeline station introduces device types not present in the water/manufacturing facility: a Remote Terminal Unit (RTU) designed for remote SCADA sites and a flow computer for custody-transfer gas measurement. These devices demonstrate that OT is not just PLCs -- pipeline SCADA operates with specialized instruments that have different register structures, measurement domains, and operational profiles than the PLCs IT engineers may be familiar with.

## 2. Background

Beta 0.2 proves design layer composability by authoring a second OT environment entirely in YAML. The pipeline monitoring station (a natural gas compressor station) was chosen because it is maximally different from the water/manufacturing facility: remote SCADA site, flat network topology with WAN backhaul, RTUs and flow computers instead of traditional PLCs, and measurement-focused registers instead of process-control registers.

The milestone spec (`docs/specs/milestone-b0.2.md`) defines five device placements:

### Device Research Summary

**Emerson ROC800 Series** (Remote Operations Controller):
- Emerson (formerly Fisher-Rosemount / Daniel) ROC800 is the dominant RTU/flow computer platform in North American pipeline SCADA. The ROC827 model with expansion chassis supports up to 27 I/O modules. Used for custody-transfer gas measurement per AGA-3 (orifice), AGA-7 (turbine), and AGA-9 (ultrasonic) standards.
- PowerPC processor (PowerQUICC), 65MHz, 16MB DRAM, 16MB flash (Series 2 CPU). Supports up to 6 meter runs per AGA calculation block.
- Communication: Ethernet 10BASE-T, two RS-232 serial ports, LOI (Local Operator Interface) port. Modbus TCP and Modbus RTU (slave mode) supported natively on all ports. No adapter module needed (unlike the SLC-500).
- Modbus registers are configurable via ROCLink 800 software. Point Type 118 defines Modbus register mappings. Most I/O values are 32-bit floating point (occupying two 16-bit registers), but for this simulator's v0.1 schema we use 16-bit scaled integers per existing convention.
- ROC Protocol (proprietary Emerson protocol) is the primary configuration protocol; Modbus is the SCADA polling protocol. The ROC800 acts as Modbus slave, polled by the SCADA master.
- One-based Modbus addressing is the Emerson convention for ROC800 register maps.
- Response time: 15-30ms typical for Modbus TCP (faster than legacy PLCs, slower than modern PLCs; the ROC800 is an embedded controller, not a high-speed PLC).

**ABB TotalFlow XFC G5** (Flow Computer):
- Fifth-generation TotalFlow flow computer from ABB (formerly Totalflow). Designed for gas measurement and custody transfer at pipeline measurement stations.
- Integrated Multivariable Transducer (XIMV) measures differential pressure, static pressure, and temperature. Supports up to 20 AGA3 measurement tubes.
- Communication: Ethernet 10/100BASE-T, RS-232 (COMM1), RS-485 (COMM2), Wi-Fi access point (up to 10 clients). Modbus TCP on port 502. User-configurable Modbus register mapping (G5 generation allows full register customization, unlike earlier fixed-map models).
- Interfaces with ABB NGC 8206/8209 Natural Gas Chromatographs for real-time gas composition analysis: methane, ethane, propane, butane, pentane, CO2, N2, and calculated values (BTU heating value, specific gravity, moisture content).
- One-based Modbus addressing (ABB convention for TotalFlow devices).
- Response time: 20-40ms typical (measurement-focused device, not optimized for high-speed control loop response).

**Gas composition reference values** (for scale_min/scale_max calibration):
- Methane: 70-96% (typical pipeline: 85-95%)
- Ethane: 1-14% (typical: 2-6%)
- Propane: 0-4% (typical: 0.5-2%)
- CO2: 0-2% (typical: 0.2-1%)
- N2: 0.4-17% (typical: 0.5-3%)
- BTU heating value: 950-1150 BTU/SCF (typical: 1000-1050)
- Specific gravity: 0.55-0.75 (methane = 0.554, heavier hydrocarbons increase it)
- Moisture content: 0-7 lb/MMSCF (pipeline spec typically < 7 lb/MMSCF)

**Compressor station reference values** (for CompactLogix variant calibration):
- Gas turbine-driven centrifugal compressors: 3000-5500 RPM typical operating speed
- Suction pressure: 400-800 PSIG typical for transmission pipeline
- Discharge pressure: 800-1400 PSIG typical (compression ratio 1.2-2.0)
- Bearing temperature: 150-250 degF (alarm at 275 degF, trip at 300 degF)
- Vibration: 0-5 mils peak-to-peak (alarm at 3 mils, trip at 5 mils)
- Lube oil pressure: 15-45 PSI (alarm at 10 PSI)
- Lube oil temperature: 110-160 degF (target 120-140 degF at bearing inlet)

### Key Differences from Water/Mfg Device Atoms

| Aspect | Water/Mfg Devices | Pipeline Devices |
|--------|-------------------|------------------|
| Device types | PLCs, gateway | RTU, flow computer, PLC, gateway |
| Measurement focus | Process control (pumps, valves, chemistry) | Custody transfer (flow rates, gas composition, pressure) |
| Units | Metric (L/s, kPa, degC) + Imperial (ft/min, degF) | Imperial throughout (SCFH, PSI, degF, BTU) |
| Register density | 5-7 holding + 2-4 coils per variant | 7-9 holding + 2-4 coils per variant (more measurement points) |
| Addressing | Mix (CompactLogix zero, SLC/Modicon one) | Mix (CompactLogix zero, ROC800/TotalFlow one) |
| Protocol path | Direct Ethernet or via serial gateway | All Modbus TCP native (pipeline stations are Ethernet-equipped) |

### Educational Value

Pipeline device atoms teach IT engineers several concepts absent from the water/manufacturing environment:

1. **RTUs are not PLCs**: The ROC800 is designed for remote, unmanned sites with environmental extremes. It has different operational constraints (solar power, satellite backhaul, no local operator).
2. **Flow computers are specialized instruments**: The TotalFlow G5 calculates gas volume using AGA standards -- it is a measurement device, not a general-purpose controller.
3. **Custody transfer is a security-critical measurement**: Gas volumes measured at custody transfer points determine who pays whom. Manipulating flow registers has direct financial impact.
4. **Pipeline SCADA has different threat models**: Remote sites with WAN exposure face different attacks than local plant networks.

## 4. Requirements

### Functional Requirements

1. `emerson-roc800.yaml` defines a device atom with category `rtu`, Ethernet and RS-232 ports, Modbus TCP + Modbus RTU protocols, one-based addressing, and response_delay_ms of 20
2. `emerson-roc800.yaml` defines a `pipeline-metering` register map variant with holding registers for meter run 1 (flow rate, volume accumulator, static pressure, temperature, differential pressure) plus station-level totals, and coils for meter run enable/disable. [OT-REVIEW] DP register replaces meter runs 2-4 flow rates -- AGA-3 requires DP, SP, and T for flow calculation, and DP is the financial manipulation attack surface.
3. `emerson-roc800.yaml` defines a `pipeline-monitoring` register map variant with holding registers for station inlet/outlet pressures and temperatures, valve positions, compressor discharge pressure, and status/alarm registers, plus coils for ESD valve and block valve commands
4. `abb-totalflow-g5.yaml` defines a device atom with category `flow-computer`, Ethernet and RS-485 ports, Modbus TCP protocol, one-based addressing, and response_delay_ms of 25
5. `abb-totalflow-g5.yaml` defines a `gas-analysis` register map variant with holding registers for gas composition (methane, ethane, propane, CO2, nitrogen percentages), calculated values (BTU heating value, specific gravity), and moisture content
6. `compactlogix-l33er.yaml` adds a `compressor-control` register map variant with holding registers for compressor speed, suction/discharge pressure, bearing temperatures, vibration, lube oil pressure/temperature, and a status/alarm word, plus coils for compressor run command, emergency stop status, surge protection status, and tripped status. [OT-REVIEW] Single run/stop coil matches water/mfg pattern; e-stop is read-only hardwired safety.
7. `moxa-nport-5150.yaml` adds a `pipeline-serial` register map variant similar to `serial-gateway` but with pipeline-context descriptions and labels for the serial connections (e.g., legacy gas analyzers behind the gateway)
8. All register addresses in ROC800 and TotalFlow G5 variants use one-based addressing (starting at address 1)
9. All register addresses in the new CompactLogix variant use zero-based addressing (starting at address 0)
10. The new Moxa variant uses zero-based addressing (matching the existing serial-gateway convention)
11. All register addresses fall within the device's declared register capacity limits
12. Each register includes unit, scale_min, scale_max, writable flag, and description
13. Each coil includes writable flag and description
14. Safety-critical coils (ESD valve, alarms) are read-only with comments explaining why
15. New device atoms include all four schema layers: identity, connectivity, registers, diagnostics
16. All YAML files include self-documenting comments with OT context

### Non-Functional Requirements

1. Each register map variant contains 7-9 holding registers and 2-4 coils (educational focus: comprehensible, not exhaustive)
2. New device atom YAML files remain under 500 lines each
3. Comments explain the OT significance of each register (what it measures, why it matters, what an attacker could do)
4. Pipeline registers use imperial units (PSI, degF, SCFH, BTU/SCF) -- consistent with US natural gas pipeline industry convention
5. Register scale ranges are realistic for the physical quantities measured (researched against real device specifications)

## 5. Design

### New Device Atom: Emerson ROC800

The ROC800 is Emerson's flagship RTU for pipeline SCADA. Unlike PLCs which execute control logic at high speed, the ROC800 is designed for remote data acquisition and flow calculation. It runs AGA gas measurement calculations, logs historical data, and responds to SCADA master polls. The ROC800 is the first `rtu` category device in the simulator.

```yaml
# emerson-roc800.yaml - Emerson ROC800 Series Remote Operations Controller
schema_version: "0.1"

device:
  id: "emerson-roc800"
  vendor: "Emerson"
  model: "ROC800 Series"
  category: "rtu"               # First RTU in the device library (not a PLC)
  vintage: 2018
  description: "Pipeline SCADA RTU / flow computer for custody-transfer gas measurement. Native Modbus TCP/RTU. Runs AGA-3/7/9 flow calculations for up to 6 meter runs."

connectivity:
  ports:
    # Ethernet: 10BASE-T on the ROC800 base unit. Modbus TCP slave on port 502.
    # The SCADA master polls this device; it does not initiate connections.
    - type: "ethernet"
      protocols: ["modbus-tcp"]
    # RS-232 COMM2 port: Modbus RTU slave for local serial connections or
    # legacy SCADA masters. RS-232 (not RS-485) -- point-to-point only.
    - type: "rs232"
      protocols: ["modbus-rtu"]
  response_delay_ms: 20          # 15-30ms typical: embedded PowerPC, not a high-speed PLC
  response_jitter_ms: 30         # [OT-REVIEW] Changed from 10 to 30: models burst-latency during AGA calculation cycles
  concurrent_connections: 4      # [OT-REVIEW] ROC800 Ethernet supports max 4 TCP connections; 1 typically reserved for ROCLINK 800 configuration. RS-232 serial port is always single-connection (point-to-point).

registers:
  max_holding: 2000
  max_coils: 500
  max_input_registers: 2000
  max_discrete_inputs: 500
  # One-based addressing: Emerson ROC800 convention for Modbus register maps.
  # ROCLink 800 configuration software uses one-based register numbering.
  # Point Type 118 defines the user-configurable Modbus register mapping.
  addressing: "one-based"
  # ROC800 uses big-endian (network byte order) for Modbus by default.
  # 32-bit floats use ABCD word order (high word first), but v0.1 schema
  # only uses 16-bit scaled integers.
  float_byte_order: "big-endian"
  max_registers_per_read: 125    # Full Modbus spec
```

#### ROC800 Variant: `pipeline-metering`

Models a custody-transfer metering station focused on meter run 1 with full AGA-3 input visibility. [OT-REVIEW] AGA-3 orifice flow calculation requires three inputs: differential pressure (DP) across the orifice plate, static pressure (SP), and flowing temperature (T). DP is the primary financial manipulation attack surface -- spoofing DP directly alters calculated volume and billing. Volume accumulators track total gas volume for billing purposes.

The register layout reflects how a SCADA master would poll a metering RTU: meter run values first (the primary data), then station-level totals. This is the data that determines custody transfer -- how much gas flows from one pipeline operator to another.

```yaml
register_map_variants:
  pipeline-metering:
    # Custody-transfer gas metering station: meter run 1 with full AGA-3 inputs.
    # [OT-REVIEW] AGA-3 requires DP + SP + T for flow calculation. DP is the
    # financial manipulation attack surface. Replaced meter runs 2-4 flow rates
    # with differential pressure to show the actual measurement inputs.
    # These registers are the financial backbone of gas pipeline operations --
    # the measured volumes determine who pays whom at custody transfer points.
    # Addresses are one-based per Emerson ROC800 convention.
    holding:
      - address: 1
        name: "meter_run_1_flow_rate"
        description: "Meter run 1 instantaneous flow rate (AGA-3 calculated from DP, pressure, temp)"
        unit: "MSCFH"
        scale_min: 0
        scale_max: 5000           # Thousand standard cubic feet per hour; typical 8-inch orifice run
        writable: false
      - address: 2
        name: "meter_run_1_volume_today"
        description: "Meter run 1 accumulated volume since contract day rollover (midnight or 9:00 AM per contract). [OT-REVIEW] Accumulator resets to zero at contract hour; scale_max is physical range, not raw register limit."
        unit: "MCF"
        scale_min: 0
        scale_max: 100000         # [OT-REVIEW] Changed from 65535: physical daily volume range, not raw register limit. Resets at contract hour.
        writable: false
      - address: 3
        name: "meter_run_1_pressure"
        description: "Meter run 1 static line pressure upstream of orifice plate"
        unit: "PSIG"
        scale_min: 0
        scale_max: 1500           # Transmission pipeline operating pressure range
        writable: false
      - address: 4
        name: "meter_run_1_temperature"
        description: "Meter run 1 flowing gas temperature (RTD sensor in thermowell)"
        unit: "degF"
        scale_min: -20
        scale_max: 150            # Outdoor pipeline: winter lows to summer highs plus compression heat
        writable: false
      - address: 5
        name: "meter_run_1_differential_pressure"
        description: "[OT-REVIEW] Replaced meter_run_2/3/4_flow_rate with DP for meter run 1. AGA-3 orifice flow calculation requires three inputs: differential pressure (DP), static pressure (SP, register 3), and temperature (T, register 4). DP across the orifice plate is the primary flow measurement and the financial manipulation attack surface -- spoofing DP directly alters calculated volume."
        unit: "inH2O"
        scale_min: 0
        scale_max: 200            # Inches of water column. Typical orifice DP: 10-100 inH2O. Range covers full transmitter span.
        writable: false
      - address: 6
        name: "station_total_flow"
        description: "Sum of all active meter run flow rates (station aggregate, calculated by RTU)"
        unit: "MSCFH"
        scale_min: 0
        scale_max: 20000          # 4 runs at 5000 MSCFH max each
        writable: false
      - address: 7
        name: "station_total_volume_today"
        description: "Station accumulated total volume since contract day rollover (sum of all meter runs). [OT-REVIEW] Resets to zero at contract hour; scale_max is physical range, not raw register limit."
        unit: "MCF"
        scale_min: 0
        scale_max: 100000         # [OT-REVIEW] Changed from 65535: physical daily volume range, not raw register limit. Resets at contract hour.
        writable: false
    coils:
      - address: 1
        name: "meter_run_1_enabled"
        description: "Meter run 1 AGA calculation enabled (operators disable runs during maintenance/calibration)"
        writable: true
      - address: 2
        name: "meter_run_2_enabled"
        description: "Meter run 2 AGA calculation enabled"
        writable: true
      - address: 3
        name: "meter_run_3_enabled"
        description: "Meter run 3 AGA calculation enabled"
        writable: true
      - address: 4
        name: "meter_run_4_enabled"
        description: "Meter run 4 AGA calculation enabled"
        writable: true
```

#### ROC800 Variant: `pipeline-monitoring`

Models station-level monitoring: inlet/outlet pressures and temperatures, valve positions, and compressor discharge monitoring. This is the "station health" view that a SCADA operator watches. The ESD (Emergency Shutdown) valve is the most critical coil -- closing it stops all gas flow through the station.

```yaml
  pipeline-monitoring:
    # Station inlet/outlet monitoring and valve control.
    # This variant monitors the station as a whole, not individual meter runs.
    # Inlet = gas arriving from upstream pipeline; outlet = gas departing downstream.
    # Addresses are one-based per Emerson ROC800 convention.
    holding:
      - address: 1
        name: "station_inlet_pressure"
        description: "Pipeline pressure at station inlet (upstream of compressor)"
        unit: "PSIG"
        scale_min: 0
        scale_max: 1500           # Transmission pipeline: 200-1200 PSIG typical
        writable: false
      - address: 2
        name: "station_outlet_pressure"
        description: "Pipeline pressure at station outlet (downstream of compressor)"
        unit: "PSIG"
        scale_min: 0
        scale_max: 1500
        writable: false
      - address: 3
        name: "station_inlet_temperature"
        description: "Gas temperature at station inlet"
        unit: "degF"
        scale_min: -20
        scale_max: 150
        writable: false
      - address: 4
        name: "station_outlet_temperature"
        description: "[OT-REVIEW] Gas temperature at station outlet, downstream of aftercooler (compression heat removed). Post-aftercooler is the more common RTU measurement point."
        unit: "degF"
        scale_min: -20
        scale_max: 150            # [OT-REVIEW] Changed from 200: post-aftercooler measurement point, compression heat removed
        writable: false
      - address: 5
        name: "inlet_block_valve_position"
        description: "Station inlet block valve position (0=closed, 100=full open)"

        scale_max: 100
        writable: false           # Valve position is read from limit switches; commands via coils
      - address: 6
        name: "outlet_block_valve_position"

        unit: "%"

      - address: 7
        name: "esd_valve_position"
        description: "Emergency shutdown valve position (0=closed/safe, 100=open/flowing). ESD valve fails closed on loss of instrument air or power."
        unit: "%"
        scale_min: 0
        scale_max: 100
        writable: false
      - address: 8
        name: "station_status_word"
        description: "Station status bitmask: bit0=running, bit1=ESD-active, bit2=high-pressure-alarm, bit3=low-pressure-alarm, bit4=high-temp-alarm, bit5=comm-failure, bit6=power-failure, bit7=f-and-g-system-alarm [OT-REVIEW] Changed from fire-detected: F&G (Fire and Gas) is a separate safety system; RTU receives consolidated alarm input, not raw fire detection"

        writable: false
    coils:
      - address: 1
        name: "esd_activate"

        writable: false           # [OT-REVIEW] ESD is hardwired safety, not a Modbus-writable coil. Teaching ESD as writable is fundamentally wrong about pipeline safety architecture.
      - address: 2
        name: "inlet_block_valve_cmd"
        description: "Inlet block valve open/close command (0=close, 1=open)"
        writable: true
      - address: 3
        name: "outlet_block_valve_cmd"
        description: "Outlet block valve open/close command (0=close, 1=open)"
        writable: true
      - address: 4
        name: "esd_active_status"

        writable: false
```

### New Device Atom: ABB TotalFlow G5

The TotalFlow G5 is a specialized flow computer, not a general-purpose RTU. Its primary function is receiving gas composition data from a Natural Gas Chromatograph (NGC) and calculating heating value (BTU), specific gravity, and moisture content. These values feed into the AGA flow calculations on the ROC800 for custody-transfer billing accuracy.

This is the first `flow-computer` category device in the simulator. The distinction from `rtu` is important: flow computers are measurement instruments, not control devices. They calculate, they do not command.

```yaml
# abb-totalflow-g5.yaml - ABB TotalFlow XFC G5 Flow Computer
schema_version: "0.1"

device:
  id: "abb-totalflow-g5"
  vendor: "ABB"
  model: "TotalFlow XFC G5"
  category: "flow-computer"      # Specialized measurement device, not a general-purpose controller
  vintage: 2020
  description: "Gas chromatograph interface and flow computer. Receives NGC analysis data, calculates BTU heating value and specific gravity for custody-transfer billing."

connectivity:
  ports:
    # Ethernet: 10/100BASE-T. Modbus TCP slave on port 502.
    # G5 generation supports user-configurable Modbus register mapping.
    - type: "ethernet"
      protocols: ["modbus-tcp"]
    # RS-485 COMM2: connects to ABB NGC 8206/8209 Natural Gas Chromatograph.
    # The chromatograph sends analysis results over this serial link.
    - type: "rs485"
      protocols: ["modbus-rtu"]
  response_delay_ms: 25          # 20-40ms: measurement device, not optimized for fast polling
  response_jitter_ms: 15         # Analysis calculation cycles introduce timing variability
  concurrent_connections: 4      # Ethernet supports multiple TCP clients

registers:
  max_holding: 1000
  max_coils: 100
  max_input_registers: 1000
  max_discrete_inputs: 100
  # One-based addressing: ABB TotalFlow convention for Modbus register maps.
  # G5 generation allows user-configurable register assignments.
  addressing: "one-based"
  float_byte_order: "big-endian"
  max_registers_per_read: 125
```

#### TotalFlow G5 Variant: `gas-analysis`

Gas composition from chromatograph analysis, plus calculated heating value and specific gravity. These values are updated each time the chromatograph completes an analysis cycle (typically every 5-10 minutes). Between cycles, the values remain static -- this is normal behavior, not a communication failure.

```yaml
register_map_variants:
  gas-analysis:
    # Gas composition analysis from NGC (Natural Gas Chromatograph).
    # Values update every 5-10 minutes when chromatograph completes a cycle.
    # Stale values between cycles are normal -- not a communication failure.
    # These composition values feed into AGA flow calculations for billing accuracy.
    # Addresses are one-based per ABB TotalFlow convention.
    holding:
      - address: 1
        name: "methane_pct"
        description: "Methane (CH4) concentration in gas stream (primary component of natural gas)"
        unit: "%"
        scale_min: 0
        scale_max: 100            # Typical pipeline: 85-95%. <70% indicates non-spec gas.
        writable: false
      - address: 2
        name: "ethane_pct"
        description: "Ethane (C2H6) concentration (second-largest component; increases heating value)"
        unit: "%"
        scale_min: 0
        scale_max: 20             # Typical: 2-6%. Rich gas streams from processing plants may reach 14%.
        writable: false
      - address: 3
        name: "propane_pct"
        description: "Propane (C3H8) concentration (valuable NGL component; affects dew point)"
        unit: "%"
        scale_min: 0
        scale_max: 10             # Typical: 0.5-2%. Higher values indicate rich gas.
        writable: false
      - address: 4
        name: "co2_pct"
        description: "Carbon dioxide (CO2) concentration (contaminant; pipeline tariff spec max 2-3%)"
        unit: "%"
        scale_min: 0
        scale_max: 5              # Pipeline spec typically <2%. Exceedance triggers gas rejection.
        writable: false
      - address: 5
        name: "nitrogen_pct"
        description: "Nitrogen (N2) concentration (inert diluent; reduces heating value)"
        unit: "%"
        scale_min: 0
        scale_max: 10             # Typical: 0.5-3%. High N2 reduces BTU value.
        writable: false
      - address: 6
        name: "btu_heating_value"
        description: "Calculated gross heating value (higher heating value, saturated). Primary billing determinant."
        unit: "BTU/SCF"
        scale_min: 900            # [OT-REVIEW] Non-zero scale_min is intentional: a raw register value of 0 maps to 900 BTU/SCF. No pipeline-quality gas has heating value below 900.
        scale_max: 1200           # Pipeline quality: 950-1150. Below 950 = off-spec gas.
        writable: false
      - address: 7
        name: "specific_gravity"
        description: "Gas specific gravity relative to air (air=1.000). Pure methane=0.554. Heavier hydrocarbons increase SG."
        unit: "SG"
        scale_min: 0.50            # [OT-REVIEW] Changed from 0 to 0.50: brackets real pipeline gas (0.554 pure methane to 0.75 rich gas). No physical gas has SG near zero.
        scale_max: 1.00            # Pipeline gas: 0.57-0.72 typical. >0.75 indicates wet/rich gas.
        writable: false
      - address: 8
```

### Register Capacity Update: Moxa NPort 5150

The existing `max_holding: 9` accommodates the `serial-gateway` variant (addresses 0-8 = 9 registers). The `pipeline-serial` variant also uses addresses 0-8, so no capacity increase is needed. Both variants have 9 holding registers and 0 coils. No changes to `max_holding` required.
