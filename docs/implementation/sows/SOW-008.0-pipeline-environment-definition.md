# SOW-008.0: Pipeline Environment Definition

**Status**: Approved
**Created**: 2026-03-01
**Priority**: High

## 1. Objective

Create the pipeline monitoring station environment definition: 3 new network atoms (`ps-station-lan`, `ps-serial-bus`, `ps-wan-link`) and 1 environment template (`design/environments/pipeline-monitoring/environment.yaml`) with 5 device placements. This is the second environment in the simulator, proving that the design layer is composable -- a new OT facility can be authored entirely in YAML without plant code changes.

This is a **design layer SOW** -- all deliverables are YAML files in `design/`. No Go code changes.

## 2. Background

SOW-007.0 created the device atoms and register map variants needed for the pipeline station. This SOW wires those devices onto networks and assigns them to a deployable environment. Once complete, the pipeline monitoring station can be validated with `ot-design validate` (SOW-006.0) and instantiated by the plant runtime alongside the existing greenfield water/manufacturing facility.

### Why This Environment Matters

The pipeline monitoring station is **maximally different** from the greenfield water/manufacturing facility:

| Aspect | Greenfield Water/Mfg | Pipeline Station |
|--------|----------------------|------------------|
| Network model | Purdue (3 levels) + flat legacy | Flat station LAN + WAN backhaul |
| Device types | PLCs + serial gateway | PLCs + RTUs + flow computer + serial gateway |
| Geography | Single building, local | Remote site, wide-area |
| Addressing | Mix of zero-based and one-based | Mix (CompactLogix zero, ROC800 one-based) |
| Security posture | Internal threats, lateral movement | Remote access threats, WAN exposure |
| Register focus | Process control (pumps, valves, chemistry) | Measurement/custody transfer (flow rates, gas composition, pressure) |
| Network count | 6 networks (3 Purdue + flat + serial + cross-plant) | 3 networks (station LAN + serial + WAN) |

This contrast is the core educational value of Beta 0.2. IT engineers learn that "OT network" does not mean one topology -- pipeline SCADA operates with fundamentally different assumptions than a local plant facility.

### Facility Description

A single compressor station on a natural gas transmission pipeline. The station boosts pipeline pressure using gas turbine-driven centrifugal compressors. A remote SCADA master (not simulated) polls the station over a WAN link (satellite or cellular backhaul). The station is unmanned -- operators monitor from a central control room tens or hundreds of miles away.

Key characteristics of a remote pipeline station:
- **Flat network**: All devices on a single Ethernet subnet. No Purdue levels -- a compressor station does not have the scale or organizational complexity to justify segmentation.
- **WAN backhaul**: The station communicates with the SCADA master over a satellite or cellular link with high latency (200ms+ RTT) and limited bandwidth.
- **Legacy serial analyzers**: Gas chromatographs and other analyzers often predate Ethernet by decades. They connect via RS-485 serial bus through a gateway.
- **Unmanned operation**: No local HMI operator. All monitoring and control is remote. This changes the threat model -- remote access is the primary attack surface.

### Device-Network Topology

```
                     WAN Link (ps-wan-link)
                         |
                    [ps-plc-01]  CompactLogix (station PLC / SCADA gateway)
                         |
              Station LAN (ps-station-lan)
              /          |          \
    [ps-rtu-01]    [ps-rtu-02]    [ps-gw-01]  Moxa Gateway
     ROC800          ROC800           |
     metering       monitoring    Serial Bus (ps-serial-bus)
                                      |
                                  [ps-fc-01]  TotalFlow G5
                                  gas analysis
```

The CompactLogix (ps-plc-01) serves dual roles: it controls the compressor and acts as the station's SCADA gateway on the WAN link. This is realistic -- at small remote stations, the station PLC often aggregates data from other devices and presents a unified Modbus interface to the SCADA master over the WAN.

## 5. Design

### Network Atom: ps-station-lan

The station LAN is a flat Ethernet network serving all IP-capable devices at the compressor station. Unlike the greenfield facility's Purdue-segmented water treatment network, the pipeline station has no VLANs, no firewalls, and a single unmanaged switch. This is realistic for remote pipeline stations:

- **Why flat?** Remote compressor stations typically have 3-8 IP devices. The cost and complexity of managed switches and VLANs is not justified at this scale. This is a security-relevant teaching point: flat networks give attackers lateral movement with no barriers.
- **Why no SPAN?** Unmanaged switches have no port mirroring capability. This means passive network monitoring tools (Dragos, Zeek) cannot observe traffic at the station unless a network TAP is physically installed. Another key teaching point: remote sites often have monitoring blind spots.
- **Subnet choice**: 10.20.1.0/24 uses the 10.x.x.x private range. The "20" in the second octet distinguishes this from the water treatment network (10.10.x.x). The /24 is oversized for 4 devices but reflects real deployments where operators leave room for future expansion.

```yaml
# ps-station-lan.yaml
schema_version: "0.1"

network:
  id: "ps-station-lan"
  name: "Pipeline Station LAN"
  type: "ethernet"
  description: "Flat Ethernet for remote compressor station -- single subnet, no segmentation"

properties:
  subnet: "10.20.1.0/24"
  managed_switch: false
  span_capable: false
```

### Network Atom: ps-serial-bus

The serial bus connects legacy gas analyzers (TotalFlow G5 in this environment) to the station network through the Moxa gateway. This follows the identical pattern established by `mfg-serial-bus.yaml` in the water/manufacturing facility.

- **Why RS-485?** Gas chromatographs, H2S analyzers, and moisture analyzers at pipeline stations often predate Ethernet by decades. RS-485 is the standard serial interface for these instruments.
- **[OT-REVIEW] Single serial link, not multidrop bus**: In this environment, the Moxa NPort 5150 has a single RS-485 serial port, and only one device (TotalFlow G5) is connected. This is a point-to-point serial link, not a multidrop bus. The NPort 5150 operates in TCP Server (transparent) mode -- it passes Modbus RTU frames between its Ethernet and serial ports without bus-master arbitration. A station with multiple serial analyzers would require a multi-port gateway (e.g., NPort 5450) or multiple single-port gateways.
- **[OT-REVIEW] Why TotalFlow G5 uses serial despite having Ethernet**: The ABB TotalFlow G5 has both Ethernet and RS-232/RS-485 ports. In this environment, it connects via serial because: (1) the station was built when the analyzers were serial-only, and the gateway infrastructure predates the G5's Ethernet capability; (2) keeping the G5 off the IP network reduces the IP attack surface; (3) this is a common real-world pattern -- new devices are often cabled into existing serial infrastructure rather than migrated to Ethernet. This is an important teaching point: OT brownfield sites rarely upgrade working serial infrastructure just because new devices support Ethernet.
- **Same educational lesson as mfg-serial-bus**: Devices on the serial bus are invisible to IP-based network scanning. Only the gateway IP is visible. This is a critical gap in OT asset inventory tools.

```yaml
# ps-serial-bus.yaml
schema_version: "0.1"

network:
  id: "ps-serial-bus"
  name: "Pipeline Station RS-485 Serial Link"
  type: "serial-rs485"
  description: "RS-485 serial link from Moxa gateway to TotalFlow G5 gas analyzer"

properties:
  managed_switch: false
  span_capable: false
```

### Network Atom: ps-wan-link

The WAN link is the most distinctive network in the pipeline environment -- it has no equivalent in the water/manufacturing facility. It represents the satellite or cellular backhaul connection from the remote station to the central SCADA master.

- **Why a separate network atom?** The WAN link has fundamentally different characteristics than the station LAN: point-to-point topology, high latency (200ms+ RTT over satellite), limited bandwidth (9.6 kbps to 256 kbps typical for VSAT), and exposure to WAN-level threats. Even though both are Ethernet type, they operate at different scales and threat levels.
- **[OT-REVIEW] Type "ethernet" is a logical designation**: The physical layer of the WAN link is satellite (VSAT) or cellular (4G/LTE), not Ethernet. The design layer schema uses `type: "ethernet"` because the IP/TCP framing is Ethernet-encapsulated at both endpoints (the CompactLogix Ethernet port connects to a satellite modem or cellular router). The schema models logical connectivity, not physical media. A future schema version could add a `media` field to distinguish Ethernet-over-copper from Ethernet-over-satellite.
- **Why /30 subnet?** A /30 subnet provides exactly 2 usable host addresses -- one for each end of the point-to-point link. This is standard practice for WAN circuits in pipeline SCADA. The SCADA master would use 10.20.0.1; the station PLC uses 10.20.0.2.
- **SCADA master not simulated**: Only the station side of the WAN link is instantiated. The 10.20.0.1 address exists in the subnet definition but has no corresponding placement. This models the boundary of the simulation -- the station is the training target, not the master.
- **No SPAN**: WAN circuits do not have local switch infrastructure to mirror. Monitoring WAN traffic requires a TAP at the station or at the SCADA master -- both physically difficult at remote sites.

```yaml
# ps-wan-link.yaml
schema_version: "0.1"

network:
  id: "ps-wan-link"
  name: "Pipeline Station WAN Link"
  type: "ethernet"
  description: "Satellite/cellular WAN backhaul to remote SCADA master (point-to-point)"

properties:
  subnet: "10.20.0.0/30"
  managed_switch: false
  span_capable: false
```

### Environment Template: pipeline-monitoring

The environment template wires devices onto networks with specific IP addresses and port assignments. Key design decisions:

**IP Address Assignment (10.20.1.0/24):**

| IP | Placement | Rationale |
|----|-----------|-----------|
| 10.20.1.10 | ps-plc-01 (CompactLogix) | .10 for the station PLC -- consistent with water treatment convention (10.10.10.10) |
| 10.20.1.20 | ps-rtu-01 (ROC800 metering) | .20 block for RTUs |
| 10.20.1.21 | ps-rtu-02 (ROC800 monitoring) | .21 continues RTU block |
| 10.20.1.30 | ps-gw-01 (Moxa gateway) | .30 for infrastructure devices |

**WAN Address Assignment (10.20.0.0/30):**

| IP | Role | Notes |
|----|------|-------|
| 10.20.0.1 | SCADA master (not simulated) | Remote end of the WAN link |
| 10.20.0.2 | ps-plc-01 WAN interface | Station PLC acts as SCADA gateway |

**Port Assignment (5040-5049):**

| Port | Placement | Notes |
|------|-----------|-------|
| 5040 | ps-plc-01 | CompactLogix -- station PLC and SCADA gateway |
| 5041 | ps-rtu-01 | ROC800 -- pipeline metering |
| 5042 | ps-rtu-02 | ROC800 -- station monitoring |
| 5043 | ps-gw-01 | Moxa gateway -- serial analyzer access |

Note: ps-fc-01 has no Modbus port -- it is a serial device accessed through ps-gw-01 at address 5043 with serial_address 1.

**[OT-REVIEW] Port 502 vs 5040-5043 convention**: Real Modbus TCP devices listen on port 502 (the IANA-assigned Modbus port). The simulator uses host-mapped ports (5040-5043) so multiple simulated devices can run on a single host without port collisions. In a Docker deployment, each container listens on port 502 internally, with host port mapping (e.g., 5040:502). The `modbus_port` field in the environment YAML specifies the host-side port for direct-to-host testing. This matches the convention established in the water/mfg environment (ports 5020-5030).

**Dual-homed CompactLogix (ps-plc-01):**

The CompactLogix is connected to both the station LAN (10.20.1.10) and the WAN link (10.20.0.2). This models the real architecture where the station PLC aggregates data from local devices and presents a unified Modbus interface to the remote SCADA master. The CompactLogix L33ER has dual Ethernet ports (EtherNet/IP), making this physically realistic.

**[OT-REVIEW] Dual-homing means listening, not routing**: The CompactLogix listens for Modbus TCP connections on BOTH IP addresses (10.20.1.10 and 10.20.0.2) and serves the same register map on both. However, it does NOT route IP packets between the station LAN and the WAN link. The CompactLogix is not an IP router -- traffic arriving on the WAN interface cannot reach the ROC800s or Moxa gateway through the CompactLogix. The only way to reach station LAN devices from the WAN is to compromise the PLC's application layer (e.g., exploit a vulnerability that allows arbitrary Modbus forwarding). This is a critical distinction for security training: dual-homed does not mean routed.

This dual-homing is a significant security teaching point: the station PLC presents data from both networks but does not bridge them at the IP layer. Compromising the PLC gives an attacker access to compressor control registers from the WAN, but not direct IP access to station LAN devices. In the water/manufacturing facility, the Moxa gateway serves a true bridging role (serial-to-Ethernet protocol translation).

**Gateway and Serial Device:**

The Moxa gateway (ps-gw-01) bridges the station LAN to the serial bus, making the TotalFlow G5 (ps-fc-01) accessible over Modbus TCP. This is the same pattern used in the water/manufacturing facility (mfg-gateway-01 bridges mfg-flat to mfg-serial-bus for the SLC-500 and Modicon 984).
