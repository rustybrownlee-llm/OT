# SOW-009.0: Pipeline Process Models

**Status**: Complete
**Created**: 2026-03-01
**Completed**: 2026-03-01
**Priority**: High

## 1. Objective

Make the pipeline monitoring station registers come alive. Process models for 4 pipeline device variants (compressor-control, pipeline-metering, station-monitoring, gas-analysis) plus the pipeline-serial gateway will drive register values through realistic drift, noise, and write response. After this SOW, an IT engineer can start the compressor (write coil), watch suction/discharge pressures respond, observe AGA-3 flow calculations change when meter run DP drifts, and see gas composition values cycle every 5-10 minutes from the chromatograph.

This is the pipeline counterpart to SOW-003.0, which built process models for the water/manufacturing environment. The same engine, helpers (toRaw, toEng, addNoise, ramp, clamp, drift), and ProcessModel interface are reused. Only new model types and main.go variant registration are added.

## 2. Background

SOW-008.0 defines the pipeline monitoring station environment with 5 device placements: CompactLogix (compressor control), two ROC800s (pipeline metering and station monitoring), TotalFlow G5 (gas analysis), and Moxa NPort 5150 (pipeline serial gateway). SOW-003.0 established the simulation engine and process model patterns for the water/manufacturing environment. The engine ticks every 1 second, dispatching `Tick()` to all registered `ProcessModel` implementations.

### What Makes Pipeline Process Models Different

Pipeline process models differ from water/manufacturing in several key ways:

| Aspect | Water/Manufacturing | Pipeline |
|--------|---------------------|----------|
| Process coupling | Each variant independent (TD-013) | AGA-3 flow calculation couples DP, pressure, and temperature |
| Update cadence | All registers update every tick | Gas analysis updates every 300-600 ticks (5-10 minute NGC cycle) |
| Writable registers | Pumps, setpoints, valves | Meter run enable/disable, valve commands, compressor run/stop |
| Safety systems | Read-only e-stop coil | ESD valve is hardwired -- not writable via Modbus (DOT 49 CFR 192) |
| Addressing | CompactLogix zero-based, SLC/Modicon one-based | CompactLogix zero-based, ROC800 one-based, TotalFlow G5 one-based |
| Counter behavior | Product count, pump runtime hours | Volume accumulators with contract-day rollover |
| Physical process | Fluid mechanics (flow, pressure, level) | Gas thermodynamics (compression, AGA-3 orifice calculation) |

### AGA-3 Orifice Flow Calculation

The AGA-3 (American Gas Association Report No. 3) standard calculates gas flow through an orifice plate from three primary inputs:

1. **Differential Pressure (DP)**: Pressure drop across the orifice plate, measured in inches of water column (inH2O). This is the primary measurement -- the financial manipulation attack surface.
2. **Static Pressure (SP)**: Line pressure upstream of the orifice plate, measured in PSIG. Higher pressure means denser gas, which affects the flow calculation.
3. **Flowing Temperature (T)**: Gas temperature at the orifice, measured in degF. Higher temperature means lower density, reducing calculated flow.

The simplified AGA-3 relationship used in this simulator:

```
flow_rate = K * sqrt(DP * SP_abs / T_abs)
```

Where:
- K = orifice coefficient (determined by orifice bore, pipe diameter, gas composition). [OT-REVIEW] K is a dimensioned constant with units of MSCFH / sqrt(inH2O * PSIA / Rankine), not a dimensionless coefficient. Its value absorbs unit conversions and orifice geometry.
- SP_abs = static pressure + atmospheric (14.696 PSIA)
- T_abs = flowing temperature + 459.67 (Rankine)

This is a simplified approximation. Real AGA-3 calculations include the orifice expansion factor, supercompressibility, and isentropic exponent. The simplified version produces plausible values and correctly models the directional sensitivity: increasing DP increases flow, increasing temperature decreases flow, increasing pressure increases flow.

### Educational Value of Pipeline Process Models

- **Financial attack surface**: Manipulating DP, pressure, or temperature registers directly alters calculated flow volumes used for custody-transfer billing. Unlike water treatment (where attacks affect safety), pipeline metering attacks affect money.
- **AGA-3 coupling**: Unlike the independent water/mfg variants (TD-013), the metering model's flow rate is computed from DP, pressure, and temperature -- demonstrating that OT process values are interdependent.
- **Slow-cycle instruments**: The gas chromatograph updates every 5-10 minutes. Stale values between cycles are normal -- trainees must learn to distinguish "stale but valid" from "stale and stale because of a communication failure."
- **ESD as a safety boundary**: The ESD coil is not writable. Writing to it over Modbus has no effect. This teaches the critical concept that hardwired safety systems are designed to be immune to cyber manipulation.
- **Compressor thermodynamics**: Discharge pressure and bearing temperatures respond to compressor speed, teaching cause-and-effect relationships in rotating equipment.

## 4. Requirements

### Functional Requirements

**CompressorModel (compressor-control variant, ps-plc-01):**

1. **FR-1**: When `compressor_run_cmd` coil is written to true, initiate compressor startup sequence: lube oil pressure ramps up (30 ticks pre-lube), then compressor speed ramps from 0 to operating speed (10 ticks, 450 RPM/tick to 4500 RPM). [OT-REVIEW] Pre-lube extended from 5 to 30 ticks. Real compressor pre-lube cycles run 30-60 seconds to ensure bearing oil film is established before shaft rotation. 5 seconds was dangerously short. During the speed ramp phase, force `surge_protection_active` coil to true (anti-surge recycle valve held open during acceleration). Revert to probabilistic surge control only after speed exceeds 70% of setpoint (3150 RPM). [OT-REVIEW] Centrifugal compressors operate in the surge zone during acceleration. The anti-surge controller holds the recycle valve open until the compressor reaches a stable operating point on the characteristic curve. This models the real startup sequence: prime lube oil, ignite gas turbine, ramp to speed with anti-surge protection.
2. **FR-2**: Suction pressure drifts around a center value with noise. When compressor is stopped, drift center = 600 PSIG. When compressor is running, drift center = 600 - (load_factor * 15) PSIG, where load_factor = compressor_speed / 4500. [OT-REVIEW] A running compressor draws down suction pressure -- the faster it spins, the more gas it pulls from the upstream pipeline. This is the upstream pipeline pressure response to compressor demand.
3. **FR-3**: Discharge pressure is computed as `suction_pressure * compression_ratio` where compression ratio ramps from 1.0 (stopped) to 1.5 (running) as compressor speed increases. The 1.5 ratio is a typical single-stage centrifugal compressor.
4. **FR-4**: Bearing temperatures (drive-end and non-drive-end) increase when running and decrease when stopped. Drive-end is 10-20 degF hotter than non-drive-end (proximity to gas turbine). Normal operating range: 150-250 degF. Drift toward 180 degF (DE) and 165 degF (NDE) at steady state.
5. **FR-5**: Vibration level (peak-to-peak, mils, per API 670 machinery protection standard) drifts around a center value (1.5 mils pk-pk) when running, with 0.01% chance per tick of a vibration spike to 4 mils pk-pk (decays back over 20 ticks). When stopped, vibration is 0. [OT-REVIEW] API 670 is the industry standard for machinery protection systems on compressors. Vibration is measured as shaft displacement in mils peak-to-peak (pk-pk) by proximity probes mounted near each bearing.
6. **FR-6**: Lube oil pressure is 35 PSIG when compressor is running (drift + noise), 0 when stopped but ramping during the 30-tick pre-lube phase before compressor speed ramp begins. [OT-REVIEW] Pre-lube ramp: pressure increases linearly from 0 to ~25 PSIG over 30 ticks, reaching the 20 PSIG ready threshold at approximately tick 24. Speed ramp does not begin until pre-lube is complete. Lube oil temperature drifts toward 130 degF when running, ambient (80 degF) when stopped.
7. **FR-7**: `compressor_status_word` bitmask updates per tick: bit0=running (speed > 0), bit1=ready (lube oil pressure > 20 PSIG and not tripped), bit2=tripped (latched on safety interlock violation), bit3=surge-detected (0.05% random chance when running, 10-tick latch), bit4=high-vibration (vibration > 3 mils), bit5=high-bearing-temp (DE bearing > 275 degF), bit6=low-lube-oil (oil pressure < 15 PSIG when running), bit7=ESD-active (mirrors e-stop coil).
8. **FR-8**: `compressor_e_stop` coil is read-only (writable: false in device atom). The process model reads it but never writes it. If e-stop is active: compressor speed is set to 0 in the current tick (no ramp -- emergency stop trips the fuel valve and applies the shaft brake simultaneously). [OT-REVIEW] E-stop is not a ramp-down. It is a simultaneous fuel cutoff and mechanical brake application. Speed drops to zero within the same control cycle. Bearing temps decay slowly, lube oil pressure drops. The model sets `compressor_tripped` coil to true when e-stop activates.
9. **FR-9**: `surge_protection_active` coil: 0.05% chance per tick when compressor is running. Latches true for 10 ticks (models anti-surge controller opening recycle valve). During surge protection, discharge pressure drops slightly (5% reduction).
10. **FR-10**: `compressor_tripped` coil: latches true on any safety interlock violation (high vibration > 5 mils trip threshold, high bearing temp > 300 degF, low lube oil < 10 PSIG when running, or ESD). Once tripped, compressor speed ramps to 0. Tripped status clears only after 60 ticks (models local operator reset requirement). While tripped, `compressor_run_cmd` writes are accepted but have no effect (permissives not satisfied).

**MeteringModel (pipeline-metering variant, ps-rtu-01):**

11. **FR-11**: `meter_run_1_flow_rate` is computed using the simplified AGA-3 formula: `flow = K * sqrt(DP * SP_abs / T_abs)` where K is a constant calibrated so that at nominal DP=50 inH2O, SP=600 PSIG, T=60 degF, flow = 2500 MSCFH. DP, SP, and T are the model's internal state variables. When `meter_run_1_enabled` coil is false, flow rate is forced to 0 regardless of calculation.
12. **FR-12**: `meter_run_1_differential_pressure` (DP) drifts around a center value (50 inH2O) with noise. This models normal flow variation through the orifice plate. Noise magnitude: 1% of scale range.
13. **FR-13**: `meter_run_1_pressure` (static pressure, SP) drifts around 600 PSIG center with noise. Independent drift models upstream pipeline pressure variation.
14. **FR-14**: `meter_run_1_temperature` drifts around 60 degF center with noise. Slow drift rate (0.005 degF/tick) models ambient temperature changes over the course of a day.
15. **FR-15**: `meter_run_1_volume_today` accumulates: `volume += flow_rate / 3600.0` per tick (MSCFH to MCF conversion: 1 tick = 1 second, so 1 hour = 3600 ticks at the current fixed tick rate per TD-014). [OT-REVIEW] This formula is only correct for a 1-second tick rate. If tick rate ever changes, the divisor must change accordingly. Resets to 0 every 86400 ticks (24-hour contract day rollover at midnight). [OT-REVIEW] Contract day rollover is configurable on real ROC800s (midnight or 9:00 AM per contract). This simulator uses midnight for simplicity.
16. **FR-16**: `station_total_flow` is the same as `meter_run_1_flow_rate` (only 1 meter run has full AGA-3 detail per TD-007). `station_total_volume_today` mirrors `meter_run_1_volume_today`.
17. **FR-17**: Meter run enable coils (1-4) are writable. Disabling `meter_run_1_enabled` sets flow rate and volume accumulation to 0. Meter runs 2-4 have no associated flow calculation (they exist for attack-surface demonstration only -- an attacker could disable all runs to zero out reported flow while gas continues to flow).

**StationMonitorModel (station-monitoring variant, ps-rtu-02):**

18. **FR-18**: `station_inlet_pressure` drifts around 600 PSIG center with noise. This represents the pipeline pressure arriving at the station.
19. **FR-19**: `station_outlet_pressure` is `inlet_pressure * compression_ratio_estimate`. The station monitor does not directly know the compressor state (cross-device coupling is out of scope), so it uses a fixed estimate of 1.4 compression ratio. This gives outlet ~840 PSIG when inlet is 600 PSIG.
20. **FR-20**: `station_inlet_temperature` drifts around 60 degF with noise (ambient pipeline temperature). `station_outlet_temperature` is inlet temperature + 20 degF (post-aftercooler residual heat from compression). [OT-REVIEW] Pre-cooler (compressor discharge) temperature = outlet + 80-120 degF, meaning the gas exits the compressor at 160-200 degF. The aftercooler exists because downstream pipeline coatings and valves have temperature limits (~140 degF). Without the aftercooler, the hot discharge gas would damage pipeline infrastructure. The 20 degF residual rise is what remains after the aftercooler removes the bulk of the compression heat.
21. **FR-21**: `inlet_block_valve_position` and `outlet_block_valve_position` respond to their corresponding command coils (`inlet_block_valve_cmd`, `outlet_block_valve_cmd`). Valve position ramps at 10%/tick (10 seconds full stroke, which is realistic for motor-operated pipeline valves). When ESD is active, both valves ramp to 0% (closed) regardless of commands.
22. **FR-22**: `esd_valve_position` is 100% (open/flowing) during normal operation. When `esd_active_status` is true, ESD valve snaps to 0% at 100%/tick (1 tick, near-instant closure). [OT-REVIEW] Spring-return ESD valves slam closed in under 1 second -- they use stored mechanical energy (spring or hydraulic accumulator) and do not modulate. Block valves at 10%/tick are motor-operated and slow by comparison. The `esd_activate` coil is NOT writable (writable: false in device atom) -- the process model never writes to it. ESD activation is simulated via a random 0.01% per tick chance, modeling communication failure or external trip signal. Once ESD activates, it latches for 120 ticks (2 minutes, requires local operator to reset per DOT 49 CFR 192).
23. **FR-23**: `station_status_word` bitmask updates per tick: bit0=running (ESD not active and valves open), bit1=ESD-active, bit2=high-pressure-alarm (outlet > 1200 PSIG), bit3=low-pressure-alarm (inlet < 200 PSIG), bit4=high-temp-alarm (outlet temp > 120 degF), bit5=comm-failure (0.02% per tick, 15-tick latch), bit6=power-failure (0.005% per tick, 30-tick latch), bit7=F&G-alarm (0.01% per tick, 20-tick latch).

**GasAnalysisModel (gas-analysis variant, ps-fc-01):**

24. **FR-24**: Gas composition registers (methane, ethane, propane, CO2, nitrogen) are NOT updated every tick. They update only during a simulated chromatograph analysis cycle (every 300-600 ticks, randomly chosen per cycle). Between cycles, values are static (held at their last analyzed value). This models the real NGC operating pattern where analysis takes 5-10 minutes.
25. **FR-25**: During an analysis cycle (lasting ~30 ticks to simulate peak detection time): `analysis_cycle_status` transitions from 0 (idle) to 1 (analyzing) to 2 (complete). On completion, composition values are updated with small drift from their previous values. [OT-REVIEW] After analysis completes, `completeTimer` is set to 5. Each tick decrements `completeTimer`. When it reaches 0, `analysis_cycle_status` returns to 0 (idle). This ensures polling clients can observe the completion event before the status resets.
26. **FR-26**: Gas composition values maintain realistic proportions. Methane: 90-94% (center 92%), ethane: 3-5% (center 4%), propane: 0.5-1.5% (center 1%), CO2: 0.2-0.8% (center 0.5%), nitrogen: 1-3% (center 2%). The sum of components need not be exactly 100% -- real NGC analysis has ±0.5% total error, and minor components (C4+, H2S, water) are not modeled.
27. **FR-27**: `btu_heating_value` is computed from gas composition using a simplified formula: `BTU = 1010 * methane_frac + 1770 * ethane_frac + 2516 * propane_frac - 0 * CO2_frac - 0 * N2_frac`. This uses the gross heating values of each component in BTU/SCF. The BTU value updates whenever composition updates (end of analysis cycle). [OT-REVIEW] These are the GPSA Engineering Data Book heating values for pure components. Real BTU calculation also includes C4+ fractions (not modeled per TD-008).
28. **FR-28**: `specific_gravity` is computed from composition: `SG = 0.5539 * methane_frac + 1.0382 * ethane_frac + 1.5226 * propane_frac + 1.5196 * CO2_frac + 0.9672 * N2_frac`. These are the molecular weight ratios relative to air (MW_component / MW_air where MW_air = 28.964). Updates with composition.
29. **FR-29**: `moisture_content` drifts independently around 4 lb/MMSCF with noise (moisture is measured by a separate sensor, not the chromatograph, but reported through the TotalFlow). If moisture exceeds 7 lb/MMSCF, the `moisture_alarm_active` coil (address 3) latches true (pipeline tariff exceedance). [OT-REVIEW] Moisture tariff exceedance is a process/quality alarm, separate from `gc_alarm_active` (coil 2) which covers only GC hardware faults. This separation lets trainees distinguish between instrument failure and gas quality issues.
30. **FR-30**: `analysis_in_progress` coil is NOT writable (writable: false in device atom). [OT-REVIEW] Renamed from `analysis_request` to clarify this is a status indicator, not a command. The process model sets it to true when an analysis cycle begins and false when idle. It reflects cycle status, not a command.
31. **FR-31**: `gc_alarm_active` coil (address 2) latches true on chromatograph hardware fault conditions only: analysis_cycle_status stays at 3 (error) for more than 30 ticks. Clears after 60 ticks. `moisture_alarm_active` coil (address 3) latches true when moisture exceeds 7 lb/MMSCF tariff limit. Clears after 60 ticks. [OT-REVIEW] These are independent alarms: GC fault = instrument problem, moisture alarm = gas quality problem. Both are read-only coils set by the process model.

**PipelineGatewayModel (pipeline-serial variant, ps-gw-01):**

32. **FR-32**: The pipeline serial gateway model follows the same pattern as the existing `GatewayModel` from SOW-003.0. Counter rates differ: 1 tx/tick and 1 rx/tick (only one serial device downstream: TotalFlow G5, versus 2 in the water/mfg gateway). Error rate: 0.3% per tick (higher than water/mfg's 0.2% -- [OT-REVIEW] high-EMI compressor station environment: large electric motors, VFDs, and gas turbine ignition systems produce significant electromagnetic interference on serial bus cabling). All other behavior (enum registers static, uptime accumulation, active TCP connections random 0-2) is identical.

### Non-Functional Requirements

33. **NFR-1**: All process model ticks complete in under 10ms (well within the 1s tick budget, per SOW-003.0 NFR-1).
34. **NFR-2**: No goroutine leaks on shutdown (same requirement as SOW-003.0 NFR-2).
35. **NFR-3**: All process models have unit tests with assertions on key behaviors (ramp, drift, alarm hysteresis, AGA-3 calculation accuracy, composition proportions).
36. **NFR-4**: Pipeline-specific units (MSCFH, PSIG, inH2O, RPM, mils, BTU/SCF, SG, lb/MMSCF) are added to the `noisyUnits` map in `model.go` so that `addNoise` applies sensor fluctuation to these analog measurements. [OT-REVIEW] MCF removed -- volume accumulators are counter-like and should not have addNoise applied directly.

**AGA-3 K Constant Calibration:**

At nominal conditions (DP=50, SP=600, T=60):
```
K = 2500 / sqrt(50 * (600 + 14.696) / (60 + 459.67))
K = 2500 / sqrt(50 * 614.696 / 519.67)
K = 2500 / sqrt(59.13)
K = 2500 / 7.69
K ~= 325.1
```

**Analysis Cycle State Machine:**

```
cycleTimer counts ticks since last cycle start.
cycleLength is random 300-600 ticks per cycle (5-10 minutes).
completeTimer counts down from 5 after analysis completes. [OT-REVIEW]

Each tick:
  if cycleTimer >= cycleLength:
    begin analysis cycle:
      set analyzeTimer = 30  (analysis takes ~30 seconds)
      set analysis_cycle_status = 1 (analyzing)
      set analysis_in_progress coil = true

  if analyzeTimer > 0:
    analyzeTimer--
    if analyzeTimer == 0:
      update composition values (drift from previous)
      recalculate BTU and SG
      set analysis_cycle_status = 2 (complete)
      set completeTimer = 5  [OT-REVIEW] Hold status=2 for 5 ticks so polling clients can observe the completion event.
      reset cycleTimer = 0
      choose new cycleLength = 300 + rand(300)
      set analysis_in_progress coil = false

  if completeTimer > 0:  [OT-REVIEW] Explicit timer for status=2 -> status=0 transition.
    completeTimer--
    if completeTimer == 0:
      set analysis_cycle_status = 0 (idle)

  rare fault (0.01% per tick):
    set analysis_cycle_status = 3 (error)
    set gcAlarmTimer = 60
```

4. Write `compressor_run_cmd` coil (FC05, port 5040, coil 0 = true). Over the next 40 seconds: lube oil pressure ramps up during pre-lube (first 30 ticks), then compressor speed ramps to operating speed (next 10 ticks) with `surge_protection_active` forced true until speed exceeds 3150 RPM. 5. Write `compressor_run_cmd` coil to false. Compressor speed ramps down over 10 ticks (simulator-compressed normal shutdown; real gas turbine coast-down is 2-5 minutes). Bearing temperatures decay slowly over tens of ticks.
6. On port 5041 (ROC800 metering): read `meter_run_1_flow_rate` (wire addr 0, device atom addr 1). Value is approximately 2500 MSCFH ± noise and drift. ## 10. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| AGA-3 simplified formula produces implausible flow rates at edge cases (very low DP or very high temperature) | Medium | Low | Clamp flow rate to 0-5000 MSCFH range. K constant calibrated at nominal conditions. Edge cases produce physically directionally correct results even if magnitude is off. |
| Gas analysis 5-10 minute cycle is too slow for interactive training | Medium | Low | Cycle length is 300-600 ticks. For testing, the trainee can observe the cycle start/complete transition. The static-between-cycles behavior is itself a learning point. |
