# SOW-010.0: Multi-Environment Docker Compose

**Status**: Approved
**Created**: 2026-03-01
**Priority**: High

## 1. Objective

Enable Docker Compose profile-based selection of OT environments so that `docker compose --profile water up` runs only the greenfield water/manufacturing facility, `docker compose --profile pipeline up` runs only the pipeline monitoring station, and both profiles can run simultaneously on non-conflicting port ranges. This is Beta 0.2 Definition of Done item #2: "Run `docker compose --profile pipeline up` and get a working pipeline SCADA environment on different ports than water/mfg" and item #5: "Switch between environments using Docker Compose profiles -- both work independently."

## 2. Background

SOW-005.0 established the Docker Compose infrastructure for a single environment (greenfield water/manufacturing). The current `docker-compose.yml` defines one `plant` service with a hardcoded `--environment` flag pointing to `greenfield-water-mfg`, five Docker networks matching the Purdue/legacy topology, and a `monitor` service that observes the plant from the Level 3 network.

With SOW-008.0 complete, a second environment definition exists at `design/environments/pipeline-monitoring/`. The plant binary already supports arbitrary environments via `--environment <path>` -- no Go code changes are needed to run the pipeline environment. The challenge is purely in Docker Compose orchestration: each environment needs its own service, networks, port mappings, and health checks, all selectable via profiles.

### Current Architecture

```
docker-compose.yml
  |
  +-- plant (single service)
  |     --environment /design/environments/greenfield-water-mfg
  |     ports: 5020-5022, 5030
  |     networks: wt-level3, wt-level2, wt-level1, mfg-flat, cross-plant
  |
  +-- monitor (single service)
        depends_on: plant (service_healthy)
        networks: wt-level3
```

### Target Architecture

```
docker-compose.yml
  |
  +-- plant-water (profile: water)
  |     --environment /design/environments/greenfield-water-mfg
  |     ports: 5020-5022, 5030
  |     networks: wt-level3, wt-level2, wt-level1, mfg-flat, cross-plant
  |
  +-- plant-pipeline (profile: pipeline)
  |     --environment /design/environments/pipeline-monitoring
  |     ports: 5040-5043
  |     networks: ps-station-lan, ps-wan-link
  |
  +-- monitor (profile: water)
        depends_on: plant-water (service_healthy)
        networks: wt-level3
```

### Key Constraint: Hardcoded Health Check Ports

The plant binary's `--health` flag currently checks a hardcoded list of ports: `["5020", "5021", "5022", "5030"]`. This is defined in `plant/cmd/plant/main.go` as the `modbusPorts` variable. The pipeline environment listens on ports 5040-5043, so the health check will fail unless it is made environment-aware.

**Two approaches:**

**Option A: Environment variable for health check ports.** Add a `OTS_HEALTH_PORTS` environment variable that overrides the hardcoded port list. The `--health` flag reads this variable and checks those ports instead. In Docker Compose, each plant service sets `OTS_HEALTH_PORTS` to its own port list.

**Option B: Discovery-based health check.** The `--health` flag loads the environment YAML, extracts all `modbus_port` values from placements, and checks those. The health check uses the same `--environment` flag as the main startup path. No new environment variables.

**Chosen: Option A.** Option B requires loading and parsing the design layer YAML during health check, which adds complexity and a dependency on the design volume mount being available and parseable just for health checks. Option A is simpler: a single environment variable with a comma-separated port list. The health check fallback (when the variable is not set) remains the current hardcoded list for backward compatibility.

### Why Separate Services, Not One Multi-Environment Service

Docker Compose profiles work at the service level. A single `plant` service cannot serve two different environments simultaneously -- each plant binary instance loads one environment definition. Running both environments requires two container instances of the same image with different `--environment` flags, different port mappings, and different network attachments. This is the standard Docker Compose pattern for running the same image with different configurations.

### Monitor Service Scope

The monitor service currently observes only the water treatment environment (attached to `wt-level3`, polls known water/mfg Modbus endpoints). Beta 0.2 explicitly excludes monitoring module work (per milestone spec: "No monitoring module -- plant composability comes first"). The monitor service remains attached to the `water` profile only. Future SOWs will address multi-environment monitoring awareness.

### Out of Scope

- Pipeline process models (SOW-009.0 -- pipeline devices will serve zero-initialized registers)
- Monitor configuration for pipeline environment (deferred to post-Beta 0.2)

- WAN latency simulation for pipeline (deferred)

- Shared Docker network between environments (each environment is fully isolated)
- RS-485 serial bus as a Docker network (same constraint as water/mfg: serial is not modeled as a Docker network)

## 4. Requirements

### Functional Requirements

**Profile Selection:**

1. **FR-1**: `docker compose --profile water up` starts `plant-water` and `monitor` services. Pipeline services do not start.
2. **FR-2**: `docker compose --profile pipeline up` starts `plant-pipeline` service. Water services and monitor do not start.
3. **FR-3**: `docker compose --profile water --profile pipeline up` starts all three services (`plant-water`, `plant-pipeline`, `monitor`). Both environments run simultaneously with no port conflicts.
4. **FR-4**: `docker compose up` (no profile) starts no services. All services require a profile. This prevents accidental resource consumption and forces explicit environment selection.

**Plant-Water Service (profile: water):**

5. **FR-5**: The `plant-water` service uses `--environment /design/environments/greenfield-water-mfg` and serves Modbus TCP on ports 5020-5022 and 5030, identical to the current `plant` service behavior.
6. **FR-6**: The `plant-water` service is attached to networks `wt-level3`, `wt-level2`, `wt-level1`, `mfg-flat`, and `cross-plant` with the same IP assignments as the current `plant` service.
7. **FR-7**: The `plant-water` health check verifies ports 5020, 5021, 5022, and 5030 via `OTS_HEALTH_PORTS` environment variable.

**Plant-Pipeline Service (profile: pipeline):**

8. **FR-8**: The `plant-pipeline` service uses `--environment /design/environments/pipeline-monitoring` and serves Modbus TCP on ports 5040-5043.
9. **FR-9**: The `plant-pipeline` service is attached to Docker networks `ps-station-lan` and `ps-wan-link` with IP assignments matching the pipeline environment definition (10.20.1.10 on station LAN, 10.20.0.2 on WAN link).
10. **FR-10**: The `plant-pipeline` health check verifies ports 5040, 5041, 5042, and 5043 via `OTS_HEALTH_PORTS` environment variable.
11. **FR-11**: The `plant-pipeline` service exposes host ports 5040-5043 mapped to container ports 5040-5043.
12. **FR-12**: [OT-REVIEW CANDIDATE] The pipeline station's `ps-serial-bus` (RS-485) is NOT modeled as a Docker network. This is the same constraint documented in TD-019 for the manufacturing serial bus: RS-485 is a physical serial medium that cannot be represented as a Docker bridge. Port 5043 carries Modbus TCP from the Moxa gateway to host clients; serial device access is via unit ID routing through the gateway port.

**Monitor Service (profile: water):**

13. **FR-13**: The `monitor` service is assigned to the `water` profile and depends on `plant-water` with `condition: service_healthy`.
14. **FR-14**: The `monitor` service configuration and behavior are unchanged from SOW-005.0. No pipeline monitoring capability is added in this SOW.
15. **FR-14a**: [OT-REVIEW] When the monitor detects that a pipeline profile is active but pipeline monitoring is not configured, it emits a startup log line at WARN level: `"Pipeline environment detected but not monitored -- see TD-020."` This prevents silent observability gaps when both profiles are running simultaneously.

**Health Check Port Configuration:**

16. **FR-15**: The plant binary reads `OTS_HEALTH_PORTS` environment variable when `--health` is passed.
17. **FR-16**: `OTS_HEALTH_PORTS` is a comma-separated list of port numbers (e.g., `5040,5041,5042,5043`). Leading/trailing whitespace around each port is trimmed.
18. **FR-17**: When `OTS_HEALTH_PORTS` is not set or empty, the health check falls back to the current hardcoded list (`5020,5021,5022,5030`) for backward compatibility.
19. **FR-18**: Each port in the list is checked with a TCP connect and 2-second timeout, identical to the current behavior.

**Docker Networks (pipeline):**

20. **FR-19**: `ps-station-lan` Docker network uses subnet `10.20.1.0/24` with appropriate labels (`ot.environment: "pipeline-monitoring"`).
21. **FR-20**: `ps-wan-link` Docker network uses subnet `10.20.0.0/30` with appropriate labels. [OT-REVIEW] The `gateway:` field is omitted from the `ps-wan-link` IPAM config. On a /30 subnet the only usable host addresses are 10.20.0.1 and 10.20.0.2. Docker assigns the gateway IP to the bridge interface, which would steal 10.20.0.1 -- the SCADA master address. Omitting `gateway:` lets Docker auto-assign outside the usable range. The `ps-wan-link` network also carries latency-disclaimer labels (see C2).
22. **FR-21**: Pipeline Docker networks use the `bridge` driver and are defined at the top level of docker-compose.yml alongside existing water/mfg networks.

### Non-Functional Requirements

23. **NFR-1**: `docker compose --profile water up` reaches healthy status within 30 seconds (same as current single-environment baseline per SOW-005.0 NFR-1).
24. **NFR-2**: `docker compose --profile pipeline up` reaches healthy status within 30 seconds.
25. **NFR-3**: `docker compose --profile water --profile pipeline up` reaches healthy status for both plant services within 45 seconds (parallel startup).
26. **NFR-4**: `docker compose down` completes within 10 seconds regardless of which profiles are active.
27. **NFR-5**: The Dockerfile remains a single multi-stage build producing one image. Both plant services use the same image with different runtime arguments.

## 5. Design

### Docker Compose Structure

The key structural change is splitting the monolithic `plant` service into two profile-gated services that share the same Docker image but differ in:
- `command:` (different `--environment` path)
- `ports:` (different host port ranges)
- `networks:` (different Docker network attachments)
- `environment:` (different `OTS_HEALTH_PORTS` values)
- `healthcheck:` (same binary, different ports via environment variable)

```yaml
# Simplified view of the target structure
services:
  plant-water:
    profiles: ["water"]
    restart: unless-stopped  # [OT-REVIEW] Carried forward from existing plant service
    build:
      context: .
      dockerfile: plant/Dockerfile
    command: ["--environment", "/design/environments/greenfield-water-mfg", "--log-level", "info"]
    container_name: ot-plant-water
    networks:
      wt-level3: { ipv4_address: 10.10.30.10 }
      wt-level2: { ipv4_address: 10.10.20.10 }
      wt-level1: { ipv4_address: 10.10.10.10 }
      mfg-flat:  { ipv4_address: 192.168.1.100 }
      cross-plant: { ipv4_address: 172.16.0.2 }
    ports:
      - "5020:5020"
      - "5021:5021"
      - "5022:5022"
      - "5030:5030"
    environment:
      - OTS_LOG_LEVEL=info
      - OTS_HEALTH_PORTS=5020,5021,5022,5030
    healthcheck:
      test: ["CMD", "/plant", "--health"]
      interval: 5s
      timeout: 3s
      retries: 3
      start_period: 10s
    volumes:
      - ./design:/design:ro

  plant-pipeline:
    profiles: ["pipeline"]
    restart: unless-stopped  # [OT-REVIEW] Carried forward from existing plant service
    build:
      context: .
      dockerfile: plant/Dockerfile
    command: ["--environment", "/design/environments/pipeline-monitoring", "--log-level", "info"]
    container_name: ot-plant-pipeline
    networks:
      ps-station-lan: { ipv4_address: 10.20.1.10 }
      ps-wan-link:    { ipv4_address: 10.20.0.2 }
    ports:
      - "5040:5040"
      - "5041:5041"
      - "5042:5042"
      - "5043:5043"
    environment:
      - OTS_LOG_LEVEL=info
      - OTS_HEALTH_PORTS=5040,5041,5042,5043
    healthcheck:
      test: ["CMD", "/plant", "--health"]
      interval: 5s
      timeout: 3s
      retries: 3
      start_period: 10s
    volumes:

    profiles: ["water"]

      plant-water:
        condition: service_healthy
```

### Health Check Port Configuration

The plant binary's `healthCheck()` function currently reads from a hardcoded `modbusPorts` slice. The change adds a single environment variable check at the top of the function:

```go
// [OT-REVIEW] Collect ALL port failures before returning so that the health
// check error message provides per-port status. Failing on the first
// unreachable port hides downstream failures and makes multi-device
// environments harder to debug (e.g., "ports 5041,5043 not reachable"
// is far more useful than "port 5041 not reachable" when 5043 is also down).
func healthCheck() error {
    ports := modbusPorts // default: ["5020", "5021", "5022", "5030"]

    if envPorts := os.Getenv("OTS_HEALTH_PORTS"); envPorts != "" {
        ports = parsePortList(envPorts) // split on comma, trim whitespace
    }

    var failed []string
    for _, port := range ports {
        addr := net.JoinHostPort("localhost", port)
        conn, err := net.DialTimeout("tcp", addr, healthCheckTimeout)
        if err != nil {
            failed = append(failed, port)
            continue
        }
        conn.Close()
    }
    if len(failed) > 0 {
        return fmt.Errorf("health check failed: ports not reachable: %s (%d/%d healthy)",
            strings.Join(failed, ", "), len(ports)-len(failed), len(ports))
    }
    return nil
}

func parsePortList(s string) []string {
    parts := strings.Split(s, ",")
    var ports []string
    for _, p := range parts {
        p = strings.TrimSpace(p)
        if p != "" {
            ports = append(ports, p)
        }
    }
    return ports
}
```

### Dockerfile CMD Change

The current Dockerfile CMD hardcodes the greenfield environment path:

```dockerfile
CMD ["--environment", "/design/environments/greenfield-water-mfg", "--log-level", "info"]
```

With multi-environment support, each Docker Compose service provides its own `command:` override. The Dockerfile CMD becomes a sensible default that works for standalone `docker run`:

```dockerfile
# Default: greenfield environment for standalone docker run.
# Docker Compose services override this via the command: key.
CMD ["--environment", "/design/environments/greenfield-water-mfg", "--log-level", "info"]
```

The Dockerfile CMD is unchanged -- it keeps greenfield as the default for backward compatibility. Docker Compose `command:` overrides it per service.

### Pipeline Docker Networks

Two new top-level Docker networks are added. The `ps-serial-bus` is NOT added as a Docker network (same rationale as `mfg-serial-bus` in the existing config -- RS-485 cannot be modeled as a Docker bridge).

```yaml
networks:
  # ... existing water/mfg networks ...

  # Pipeline monitoring station -- flat station LAN
  ps-station-lan:
    name: ot-ps-station-lan
    driver: bridge
    ipam:
      config:
        - subnet: 10.20.1.0/24
          gateway: 10.20.1.1
    labels:
      ot.environment: "pipeline-monitoring"
      ot.description: "Pipeline station LAN - flat Ethernet, unmanaged switch, single subnet"

  # Pipeline monitoring station -- WAN backhaul to SCADA master
  ps-wan-link:
    name: ot-ps-wan-link
    driver: bridge
    ipam:
      config:
        - subnet: 10.20.0.0/30
          # [OT-REVIEW] Gateway omitted intentionally. Docker assigns 10.20.0.1 as
          # the bridge gateway, which collides with the SCADA master address on this
          # /30 subnet. Omitting gateway lets Docker pick a non-conflicting address
          # outside the usable host range, avoiding IP theft from the SCADA master.
    labels:
      ot.environment: "pipeline-monitoring"
      ot.description: "Satellite/cellular WAN backhaul - point-to-point link to remote SCADA master"
      # [OT-REVIEW] Latency is NOT simulated on this Docker bridge network.
      ot.latency-simulated: "false"
      ot.latency-note: "Real VSAT: 500-700ms RTT. Docker bridge: sub-1ms. See TD-022."
```

### Profile Assignment Strategy

All services are assigned to profiles. This means `docker compose up` (no profile) starts nothing. This is intentional:

- **Prevents accidental resource consumption**: Running both environments uses significant memory and CPU. Explicit profile selection forces the user to choose.
- **Consistent mental model**: The user always specifies which environment(s) to run. There is no "default" behavior that silently changes when new environments are added.
- **Educational clarity**: Forces trainees to understand which environment they are working with, preventing confusion between water/mfg and pipeline endpoints.

An alternative (keeping `plant-water` as the no-profile default) was considered and rejected because it would break the expectation that adding `--profile pipeline` adds pipeline services without removing water services. Docker Compose profiles are additive, not selective.

### Container Naming

- `plant-water` service -> container name `ot-plant-water`
- `plant-pipeline` service -> container name `ot-plant-pipeline`
- `monitor` service -> container name `ot-monitor` (unchanged)

The `ot-` prefix is retained per the existing convention. The environment suffix distinguishes containers in `docker ps` output.

| TD-022 | `ps-wan-link` Docker network has no latency simulation. Real satellite/cellular WAN links have 200ms+ RTT. The Docker bridge network delivers packets with LAN latency. | Add `tc` (traffic control) based latency injection in a future SOW, or model latency in the plant runtime response_delay mechanism. |
| TD-023 | The `plant-pipeline` container is attached to `ps-station-lan` at 10.20.1.10 only. The pipeline environment also defines devices at 10.20.1.20, 10.20.1.21, and 10.20.1.30 which the plant binary simulates internally, but Docker cannot assign multiple IPs to one container on the same network. Same class of limitation as TD-018 (cross-plant multiple IPs). | When multi-container architecture is considered, each device could run in its own container. For the simulator, this single-container model is acceptable. |
| TD-024 | `docker compose up` (no profile) starts nothing, which may confuse users who expect default behavior. | Document profile usage in README and environment authoring guide (SOW-011.0). |
