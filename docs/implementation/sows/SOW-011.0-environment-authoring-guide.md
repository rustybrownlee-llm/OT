# SOW-011.0: Environment Authoring Guide

**Status**: Complete
**Created**: 2026-03-01
**Priority**: Medium
**OT Domain Review**: Complete -- 2 critical, 4 major, 4 minor corrections applied (2026-03-01)

## 1. Objective

Create a step-by-step guide that teaches an engineer how to author a new OT environment from scratch using the design layer's YAML schema, scaffold tooling, and validation CLI. The guide uses the pipeline monitoring station as a worked example to illustrate each step, then includes a "Quick Start" section where the reader creates a minimal environment (3 devices, 1 network) that passes `ot-design validate` and runs in the simulator.

This is Beta 0.2 Definition of Done item #6: "Create a minimal custom environment YAML by hand (3 devices, 1 network) and have it pass validation and run."

After this SOW, the simulator's design layer is not just composable in theory -- it has a documented, repeatable process that an engineer can follow to create their own OT facility without reading source code or reverse-engineering existing YAML files.

This is a **documentation SOW** -- all deliverables are markdown files. No Go code changes.

## 2. Background

Beta 0.2 proved that the design layer is composable: the pipeline monitoring station was authored entirely in YAML and runs alongside the greenfield water/manufacturing facility without plant code changes. The `ot-design` CLI (SOW-006.0) provides scaffold templates and validation. SOW-007.0 and SOW-008.0 created the device atoms, network atoms, and environment definition for the pipeline station.

However, the knowledge of *how* to author a new environment is currently spread across:

- ADR-009 (schema definition and layer separation principles)
- Scaffold template comments (field-level documentation embedded in Go string constants)
- Existing YAML files (implicit patterns that must be reverse-engineered)
- SOW-006.0 (44 validation rules that define what constitutes valid YAML, but no guidance on *what to write*)
- The milestone spec (facility descriptions and device selection rationale)

No single document walks an engineer through the process end-to-end. The authoring guide consolidates this knowledge into a structured tutorial.

### Target Audience

IT engineers learning OT security who have:

- Basic YAML literacy (can edit key-value pairs, lists, nested structures)
- Access to a running OT simulator instance (Docker Compose)
- The `ot-design` binary built from `design/cmd/ot-design/`
- No prior OT domain knowledge (the guide teaches OT concepts as they arise)

### Worked Example: Pipeline Monitoring Station

The guide uses the pipeline monitoring station (SOW-007.0 through SOW-008.0) as its primary worked example because:

- It was the **second** environment authored, so the process was deliberate (not the initial ad-hoc creation)
- It exercises **all placement types**: Ethernet devices, serial devices, gateways, multi-homed devices, and `additional_networks`
- It introduces device types that are **different from water/mfg** (RTUs, flow computers), demonstrating that new device atoms can be created for any industry
- The network topology (flat station LAN + WAN + serial bus) is **simple enough to explain** but **different enough from Purdue** to teach that OT networks are not one-size-fits-all

## 3. Scope

### In Scope

**Primary deliverable: Environment Authoring Guide** (`docs/environment-authoring-guide.md`)

A markdown document containing:

1. **Prerequisites** -- tools needed, how to build `ot-design`, directory structure overview
2. **Concepts** -- brief explanation of devices, networks, environments, placements, and variants (with cross-references to ADRs)
3. **Step 1: Plan Your Facility** -- choosing devices, networks, and topology before writing any YAML
4. **Step 2: Create Device Atoms** -- using `ot-design scaffold --device` and editing the skeleton, with the ROC800 and TotalFlow G5 as worked examples. **[OT-REVIEW]** The guide must use correct terminology: the TotalFlow G5 is a flow computer that interfaces with an external gas chromatograph (GC), not a GC itself. The guide must not refer to the TotalFlow G5 as a chromatograph or GC.
5. **Step 3: Create Network Atoms** -- using `ot-design scaffold --network`, with `ps-station-lan`, `ps-serial-bus`, and `ps-wan-link` as worked examples
6. **Step 4: Create the Environment** -- using `ot-design scaffold --environment`, wiring devices to networks with placements, addressing, and port assignments
7. **Step 5: Validate** -- running `ot-design validate` and interpreting error messages; common mistakes and how to fix them
8. **Step 6: Run It** -- adding a Docker Compose profile and starting the environment
9. **Quick Start: Minimal Environment** -- a condensed walkthrough creating a 3-device, 1-network environment from scratch that passes validation and runs
10. **Reference** -- quick-reference tables for schema fields, validation rules, port ranges, and addressing modes

**Secondary deliverable: Quick Start environment files** (`design/environments/quickstart-example/environment.yaml`)

A minimal, working environment YAML (3 devices, 1 network) that reuses existing device atoms (no new device atoms needed). This environment is validated by `ot-design validate` and can be run with Docker Compose. It serves as both:
- The output of the Quick Start tutorial section
- A copy-paste starting point for engineers creating their own environments

### Out of Scope

- Go code changes (no modifications to `ot-design`, plant binary, or monitoring)
- New device atoms or network atoms (the Quick Start reuses existing atoms)
- New Docker Compose profiles for the quickstart environment (the guide shows *how* to add one, but does not add it to the committed `docker-compose.yml`)
- Video or interactive tutorials
- Schema v0.2 features (security profiles, monitoring overlays)
- HMI configuration or scenario authoring
- Process model configuration (the guide focuses on the design layer, not runtime behavior)

## 4. Requirements

### Functional Requirements

**Environment Authoring Guide:**

1. **FR-1**: The guide is a single markdown file at `docs/environment-authoring-guide.md`, readable without any specialized tooling.

2. **FR-2**: The guide begins with a Prerequisites section listing: Go toolchain (for building `ot-design`), Docker and Docker Compose (for running the environment), a text editor with YAML support, and a Modbus TCP client (e.g., `mbpoll`) for verification. **[OT-REVIEW]** pymodbus removed from prerequisites -- it is on the project's Forbidden list and introduces an unnecessary Python dependency.

3. **FR-3**: The guide includes a Concepts section that explains the three-layer architecture (`design/ -> plant/ -> monitoring/`), the relationship between device atoms, network atoms, and environments, and the role of register map variants. Explanations are 2-3 sentences each, not exhaustive -- they provide enough context to follow the guide and point to ADRs for deeper detail. **[OT-REVIEW]** The Concepts section must also explain the Modbus master/slave (client/server) polling model: "Modbus is a request/response protocol. Devices (slaves/servers) do not transmit data on their own -- they wait for a master (client) to poll them. In a real OT environment, the SCADA system or HMI acts as the master, continuously polling PLCs and RTUs for register values. In the simulator, the trainee's Modbus client (e.g., `mbpoll`) plays the SCADA master role. The simulator will not produce any Modbus traffic until you poll it." Without this, IT-background trainees will start the environment, see no network traffic, and assume it is broken.

4. **FR-4**: Step 1 (Plan Your Facility) walks through the planning decisions made for the pipeline monitoring station: what industry, what devices exist at the site, what network topology connects them, what Modbus port range to use. It includes the topology diagram from the pipeline environment YAML header comments. **[OT-REVIEW]** The network topology section must explain the engineering rationale behind topology choices -- specifically that a flat network is appropriate for small, remote OT sites (like an unmanned pipeline station) where the cost and complexity of Purdue-segmented architecture is not justified. The guide must state that this is a known and accepted security tradeoff, not an oversight: "A flat network at a remote station means any device compromise gives lateral access to all other devices. This is accepted because the station is air-gapped or VPN-connected to SCADA, has few devices, and the cost of managed switches and firewalls exceeds the risk reduction. The greenfield-water-mfg environment shows what proper Purdue segmentation looks like when the facility size and risk warrant it." This teaches trainees that OT network design is a risk-informed engineering decision, not a one-size-fits-all template.

5. **FR-5**: Step 2 (Create Device Atoms) shows the `ot-design scaffold --device` command and its output, then walks through editing the skeleton to create a device atom. Uses the Emerson ROC800 as the worked example, showing how register addresses are chosen (one-based, non-contiguous with gaps), how scale ranges map to physical measurements, and which registers should be writable vs read-only. Emphasizes that register maps model the PLC program, not contiguous memory arrays. **[OT-REVIEW]** The register addressing walkthrough must include a concrete worked example of the zero-based vs one-based addressing mismatch -- the single most common IT-to-OT mistake. Show explicitly: "If a zero-based Modbus client (e.g., `mbpoll`) sends FC3 (Read Holding Registers) with `start_address=0` to a one-based device (e.g., the ROC800 which documents its first register as 1), the client either receives an Illegal Data Address exception (0x02) or silently reads the wrong register. The fix: subtract 1 from every documented register address when using a zero-based client, or configure the client for one-based addressing if supported." Include a before/after showing the same register read with correct and incorrect addressing.

6. **FR-6**: Step 3 (Create Network Atoms) shows the `ot-design scaffold --network` command and walks through creating both an Ethernet network (`ps-station-lan`) and a serial network (`ps-serial-bus`). Highlights that serial networks have no subnet and are not directly accessible from IP-based tools. **[OT-REVIEW]** The serial network section must include an explicit "Security Implication" callout box with this text: "OT asset inventory tools that rely on IP discovery (e.g., Nmap, Shodan, Dragos asset identification) will not find devices on the serial bus. Only the gateway IP is visible to network scanners. This is a systematic blind spot in OT security tooling -- the serial bus may contain the majority of field devices at a site, yet none appear in IP-based asset inventories. Physical site surveys or gateway configuration audits are the only reliable discovery methods for serial devices."

7. **FR-7**: Step 4 (Create the Environment) shows the `ot-design scaffold --environment` command and walks through wiring devices to networks. Covers all placement patterns:
   - Ethernet device (direct Modbus TCP): `ps-rtu-01` on `ps-station-lan`
   - Gateway device with `bridges` field: `ps-gw-01` bridging `ps-station-lan` to `ps-serial-bus`
   - Serial device with `gateway` and `serial_address` (no IP, no modbus_port): `ps-fc-01` behind `ps-gw-01`
   - Multi-homed device with `additional_networks`: `ps-plc-01` on both `ps-station-lan` and `ps-wan-link`

8. **FR-8**: Step 5 (Validate) shows running `ot-design validate` against the pipeline environment, demonstrates what a passing result looks like, then shows common validation errors (missing device reference, IP outside subnet, port collision, serial device with IP) and how to fix each one. Shows at least 4 distinct error examples.

9. **FR-9**: Step 6 (Run It) shows how to add a Docker Compose profile for the new environment, including service definition, network definitions, port mappings, health check configuration with `OTS_HEALTH_PORTS`, and volume mount for the design directory. Shows how to start the environment with `docker compose --profile <name> up` and verify it is running by reading registers with a Modbus client. **[OT-REVIEW]** The Docker Compose section must include a requirement that all host-side port mappings are globally unique across all compose profiles, since multiple profiles may run simultaneously. The guide must reference the port range table (FR-12) as the coordination mechanism and state: "Before assigning host ports, check the Port Range Assignments table to avoid collisions. Docker maps container port 502 to unique host ports -- if two services map to the same host port, only the first to start will succeed. Port collisions are a runtime error with no validation-time warning."

10. **FR-10**: The Quick Start section is a self-contained walkthrough that creates a minimal environment with:
    - 3 device placements (reusing existing device atoms: e.g., 2 CompactLogix PLCs + 1 Moxa gateway, or 3 CompactLogix PLCs)
    - 1 Ethernet network (reusing an existing network atom or creating a minimal new one)
    - Unique placement IDs, valid IPs within the network subnet, non-colliding Modbus ports
    - The resulting YAML passes `ot-design validate`
    - **[OT-REVIEW]** An explicit Purdue context statement at the beginning of the Quick Start that explains: this 3-PLC, single-network topology is a deliberate simplification -- a single Purdue Level 1 segment -- intended only for learning the authoring workflow. A production water treatment facility would isolate this network behind Purdue Level 2 (site operations) and Level 3 (site business) boundaries with firewalls and DMZs. The statement must cross-reference `greenfield-water-mfg` as the complete Purdue model example. Without this warning, trainees will internalize "3 PLCs on one flat network" as a valid generic OT network template.

11. **FR-11**: The Quick Start includes the exact `ot-design validate` command and expected output, so the reader can verify their work matches.

12. **FR-12**: The Reference section includes:
    - Table of all device atom required fields with types and descriptions
    - Table of all network atom required fields with types and descriptions
    - Table of all environment/placement required fields with types and descriptions
    - Port range assignments (5020-5039 water/mfg, 5040-5049 pipeline, 5050-5059 quickstart/examples [OT-REVIEW], 5060+ available)
    - Addressing modes (zero-based vs one-based) with examples
    - List of common validation errors and their fixes

**Quick Start environment:**

13. **FR-13**: A working `environment.yaml` file at `design/environments/quickstart-example/environment.yaml` that:
    - Uses `schema_version: "0.1"`
    - References only existing device atoms and network atoms (no new atoms created)
    - Contains exactly 3 placements on 1 Ethernet network
    - Uses Modbus ports in the 5050-5059 range (not colliding with water/mfg or pipeline)
    - Passes `ot-design validate design/environments/quickstart-example/`
    - Includes inline comments explaining each field (this file is a teaching artifact)

14. **FR-14**: The Quick Start environment reuses existing device atoms with existing register map variants. The simplest approach is 3 CompactLogix PLCs with the `water-intake`, `water-treatment`, and `water-distribution` variants on a single network. This proves the design layer's composability -- the same device atoms can be redeployed in different topologies. **[OT-REVIEW]** The Quick Start environment YAML file itself must include a header comment stating that this single-segment topology is a Purdue Level 1 simplification, not a production reference architecture. Cross-reference `greenfield-water-mfg` for the complete Purdue layering.

### Non-Functional Requirements

1. **NFR-1**: The guide uses no jargon without defining it first. OT-specific terms (RTU, PLC, Modbus, register, coil, gateway, Purdue model) are explained in-context on first use.
2. **NFR-2**: All YAML examples in the guide are syntactically valid and match the v0.1 schema. No pseudocode or abbreviated YAML.
3. **NFR-3**: The guide is under 2000 lines of markdown. Long enough to be thorough, short enough to read in one sitting.
4. **NFR-4**: Every CLI command shown in the guide includes the expected output (or a representative sample of the output). The reader should never have to guess what success looks like.
5. **NFR-5**: The guide does not duplicate ADR content. It cross-references ADRs for architectural rationale and focuses on the *how*, not the *why*.

### Quick Start Environment Design

The Quick Start environment creates a minimal facility that reuses existing assets:

```
quickstart-example/
  environment.yaml
```

**Facility**: "Training Lab" -- a simplified 3-PLC Ethernet network for hands-on Modbus practice.

**Network**: Reuses an existing Ethernet network atom or creates a minimal new one. To avoid coupling the Quick Start to a specific existing network's subnet, this SOW creates one new network atom: `quickstart-lab.yaml` with subnet `10.30.0.0/24`.

**Wait -- Scope said "no new network atoms."** Revised approach: The Quick Start reuses `wt-level1` (subnet `10.10.10.0/24`) as its single network. The 3 placements use IPs within this subnet that do not collide with the greenfield water/mfg placements (which use `.10`, `.11`, `.12`). Using `.50`, `.51`, `.52` avoids collision. This demonstrates reuse -- the same network atom serves different environments with different placements. **[OT-REVIEW]** The Quick Start environment YAML and guide text must include inline comments explaining WHY the IPs start at `.50` instead of following sequentially from `.12` -- the gap exists to avoid address collisions with the greenfield water/mfg environment that uses the same network atom with `.10-.12`. This teaches the addressing coordination discipline required when multiple environments share network atoms.

**Devices**: 3 CompactLogix L33ER placements using existing register map variants:

| Placement ID | Device | Variant | IP | Port |
|-------------|--------|---------|-----|------|
| qs-plc-01 | compactlogix-l33er | water-intake | 10.10.10.50 | 5050 |
| qs-plc-02 | compactlogix-l33er | water-treatment | 10.10.10.51 | 5051 |
| qs-plc-03 | compactlogix-l33er | water-distribution | 10.10.10.52 | 5052 |

This is intentionally simple. The point is not to build a realistic facility -- it is to prove the authoring workflow works end-to-end with minimal effort.

### Worked Example YAML Excerpts

The guide includes full YAML excerpts from the pipeline monitoring station, not abbreviated snippets. Each excerpt is followed by a field-by-field explanation. For example, the ROC800 device atom excerpt shows the complete `device:`, `connectivity:`, `registers:`, and one `register_map_variants:` entry with explanation of one-based addressing, non-contiguous register addresses, and the scale range for pipeline pressure measurements.

### Error Examples

The guide manufactures 4+ deliberate validation errors and shows the exact error output:

1. **Missing device reference**: A placement references `device: "nonexistent-plc"` -- shows ENV-004 error
2. **IP outside subnet**: A placement has `ip: "192.168.1.10"` on a `10.20.1.0/24` network -- shows ENV-008 error
3. **Port collision**: Two placements both use `modbus_port: 5040` -- shows ENV-010 error
4. **Serial device with IP**: A serial placement (accessed via gateway) erroneously includes an `ip` field -- shows ENV-017 error
5. **Missing variant**: A placement references `register_map_variant: "nonexistent"` -- shows ENV-005 error

### Docker Compose Section

The guide shows a complete Docker Compose service definition for a hypothetical new environment, modeled on the `plant-pipeline` service pattern from `docker-compose.yml`. It includes:

- `profiles` for environment selection
- `command` with `--environment` flag pointing to the design directory
- `networks` with static IP assignment
- `ports` mapping Modbus TCP ports
- `environment` with `OTS_HEALTH_PORTS`
- `healthcheck` configuration
- `volumes` mounting the design directory read-only

The guide does NOT modify the committed `docker-compose.yml`. It shows the reader what to add and explains each field.

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| YAML excerpts in the guide diverge from actual files as future SOWs modify device/network atoms | Medium | High | Excerpts reference specific files and versions. Include a note at the top of the guide: "Examples based on schema v0.1. If examples do not match your files, check the schema version." Future SOWs that modify referenced YAML must update the guide. |
| Quick Start environment uses IPs that collide with a future third environment's placements | Low | Medium | Quick Start uses `.50-.52` addresses on `wt-level1`, well away from the `.10-.12` range used by water/mfg. Port range 5050-5059 is documented as "quickstart/examples" in the reference section. |
| The guide becomes the de facto schema reference, leading engineers to skip ADR-009 | Low | Medium | The guide explicitly defers "why" questions to ADRs and does not duplicate schema rationale. The Concepts section states: "This guide shows you how to create environments. For why the schema is designed this way, see ADR-009." |
| Engineers attempt to run the quickstart environment with Docker Compose and it fails because no profile is configured | Low | High | The guide's Step 6 section clearly states that adding a Docker Compose profile is required to run the environment. The quickstart YAML validates but is not wired into docker-compose.yml. The guide shows what to add. |
| Scaffold output changes in a future SOW, making the guide's scaffold examples stale | Medium | Medium | Scaffold examples in the guide show the key fields, not the full scaffold output verbatim. The guide directs readers to run `ot-design scaffold --device` themselves rather than relying on the printed example. |
