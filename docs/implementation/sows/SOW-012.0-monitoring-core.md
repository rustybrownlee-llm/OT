# SOW-012.0: Monitoring Core: Device Discovery and Register Polling

**Status**: Approved
**Created**: 2026-03-02
**Priority**: High

## 1. Objective

Build the monitoring module's core runtime: parse the monitoring config YAML, connect to Modbus TCP endpoints across both OT environments, discover devices and their register maps by probing, build an in-memory asset inventory, and enter a continuous polling loop that reads all registers every 2 seconds. Expose the discovered assets and live register values on an HTTP API (port 8091) that SOW-013.0 (Anomaly Detection) and SOW-014.0 (Dashboard) will consume. Resolve TD-011 (stub config), TD-012 (stub health check), and TD-020 (pipeline unmonitored).

After this SOW, an IT engineer can start both plant profiles, wait 30 seconds, and `curl http://localhost:8091/api/assets` to receive a JSON inventory of all 12 device endpoints across both environments -- each with device category, addressing mode, register count, and mean response time -- all discovered by network observation alone.

## 2. Background

### Current State

The monitoring module (`monitoring/`) is a skeleton. The binary parses flags, reads the config path, and blocks on SIGINT. The config file (`monitoring/config/monitor.yaml`) lists water/mfg endpoints only. The health check verifies the config file is readable (TD-012). The Docker Compose service connects to `wt-level3` only (TD-020). The `go.mod` has no dependencies beyond the Go standard library.

### Architecture Constraints

ADR-005 D4 mandates that monitoring MUST NOT import plant packages. All data comes from network observation -- Modbus TCP polling, HTTP responses, ICMP. The config tells the monitor WHERE to look (IP addresses and ports), but NOT WHAT to expect (register maps, device types, normal values). The monitor must discover device characteristics the same way a real security tool would: by probing.

### What This SOW Builds


### Discovery vs. Configuration

The milestone spec's open question -- "Should the monitor discover devices by scanning configured port ranges, or should the config explicitly list every endpoint?" -- is resolved in favor of a **hybrid approach**:

- **Config-driven endpoints**: The config lists IP:port pairs and, for gateway ports, a unit ID scan range. This is realistic -- a security tool must be told which network segments to observe.
- **Discovery-driven characterization**: For each configured endpoint, the monitor probes to discover register count, addressing mode (zero-based vs one-based), and response timing. This is how a real Modbus scanner works.

### simonvetter/modbus Client API

The monitoring module uses the same `simonvetter/modbus` library as the plant (validated in POC-001.0), but as a **client** rather than a server:

```go
c, err := modbus.NewClient(&modbus.ClientConfiguration{
    URL:     "tcp://10.10.30.10:5020",
    Timeout: 2 * time.Second,
})
err = c.Open()
c.SetUnitId(1)
vals, err := c.ReadRegisters(0, 10, modbus.HOLDING_REGISTER)
coils, err := c.ReadCoils(0, 8)
```

Key behavior: `ReadRegisters` returns `ErrIllegalDataAddress` (Modbus exception 0x02) when reading past the end of the register space. This exception is how the register enumeration algorithm determines register count.

- Graceful shutdown: close all Modbus connections on SIGINT/SIGTERM

**HTTP API (port 8091):**
- `GET /api/health` -- per-endpoint connectivity status (resolve TD-012)
- `GET /api/assets` -- discovered device inventory as JSON
- `GET /api/assets/{id}/registers` -- latest register values for a device

**Docker/infrastructure changes:**
- Monitor container joins `ps-station-lan` network (resolve TD-020)

- Volume mount for `design/` directory (read-only, for SOW-014.0 dashboard use)
- Updated `monitor.yaml` config with pipeline endpoints
- Remove TD-020 warning from monitor startup (the gap is resolved)

**Gateway discovery:**
- For gateway ports, poll unit IDs 1-10 and 247
- Unit ID 247 = gateway diagnostics register (Moxa convention)
- Devices that respond on a gateway port are recorded with a `via_gateway` relationship

**Register enumeration:**
- Binary-search holding register count per responding device/unit
- Detect addressing mode (zero-based vs one-based) by testing which start address succeeds

- Respect Modbus spec maximum: 125 registers per read request

## 4. Requirements

### Functional Requirements

**Config Parsing (TD-011):**

1. **FR-1**: The monitor binary parses `monitor.yaml` into typed Go structs at startup. Parse errors cause immediate exit with a descriptive error message.
2. **FR-2**: The config schema supports multiple environments, each with a name, base address, and list of Modbus endpoints. Each endpoint has a port and either a single `unit_id` or a `unit_ids` list (for gateway ports with multiple downstream devices).
3. **FR-3**: Gateway endpoints are identified by having `gateway: true` in the config. Gateway endpoints trigger unit ID scanning (FR-12).
4. **FR-4**: The polling interval is configurable in the config file (`poll_interval_seconds`, default: 2). Minimum allowed value is 1 second.
5. **FR-5**: The API bind address is configurable (`api_addr`, default: `:8091`). The `--addr` CLI flag overrides the config value.

**Device Discovery:**

6. **FR-6**: On startup, the monitor attempts a Modbus TCP connection to every configured endpoint. Endpoints that refuse connection or time out (2-second timeout) are marked `offline` in the asset inventory. Offline endpoints are retried every polling cycle.
7. **FR-7**: For each responding endpoint + unit ID combination, the monitor performs register enumeration (FR-10, FR-11) and response time measurement.
8. **FR-8**: Response time measurement: the monitor records the RTT of at least 3 Modbus reads per device during discovery. The asset inventory stores mean response time and jitter (standard deviation).
9. **FR-9**: Discovery runs at startup and is repeated for offline endpoints on every polling cycle. Already-discovered online endpoints are not re-enumerated (register count is stable).

**Register Enumeration:**

10. **FR-10**: For each responding device/unit, the monitor determines the holding register count by binary search. Algorithm: try reading register 0 (quantity 1). If `ErrIllegalDataAddress`, try register 1. Whichever starting address succeeds determines the addressing mode (0 = zero-based, 1 = one-based). Then expand the read range (doubling) until `ErrIllegalDataAddress` to find the upper bound. Bisect to find the exact register count.
11. **FR-11**: Coil enumeration uses the same binary search approach as holding registers. Start address detection determines coil addressing mode (should match register addressing mode for the same device).
12. **FR-12**: For gateway ports (`gateway: true` in config), the monitor scans unit IDs 1 through 10 and unit ID 247. [OT-REVIEW] Unit ID 247 is used in this simulator as the convention for gateway diagnostic registers (educational abstraction -- real Moxa NPort 5150 devices are managed via HTTP port 80 and SNMP port 161, not Modbus). Each responding unit ID becomes a separate asset in the inventory, linked to the gateway endpoint via a `via_gateway` field.
13. **FR-13**: The register enumeration must respect the Modbus spec maximum of 125 registers per read request. Reads larger than 125 are split into multiple requests.

**Asset Inventory:**

14. **FR-14**: Each discovered device is assigned a unique asset ID derived from the endpoint address and unit ID (e.g., `10.10.30.10:5020:1`). The ID is deterministic across restarts.
15. **FR-15**: The asset inventory stores per device: asset ID, endpoint address (host:port), unit ID, addressing mode (zero-based or one-based), holding register count, coil count, mean response time (ms), response jitter (ms), status (online/offline), first seen timestamp, last seen timestamp, last good poll time (timestamp of last error-free poll cycle), `via_gateway` (gateway asset ID, if accessed through a gateway), `protocol` (always "modbus-tcp"), `category` (parsed from config description keyword or "unknown"), and `purdue_level` (derived from environment name). [OT-REVIEW] The `last_good_poll_time` field enables detection of "stuck" registers -- a value unchanged across many poll cycles may indicate a frozen sensor, not a stable reading. The `category` and `purdue_level` fields are enriched during discovery and start with default values.
16. **FR-16**: The asset inventory is safe for concurrent access (polling goroutine writes, HTTP handler goroutine reads).

**Polling Loop:**

17. **FR-17**: After discovery completes, the monitor enters a continuous polling loop. Each cycle reads all holding registers and all coils from every online device.
18. **FR-18**: The polling interval is measured from the START of one cycle to the START of the next. If a cycle takes longer than the interval (due to slow or offline devices), the next cycle starts immediately with no negative drift.
19. **FR-19**: Each poll cycle updates the latest register values in the asset inventory. SOW-013.0 will add time-series storage; this SOW stores only the most recent snapshot.
20. **FR-20**: Devices that fail to respond during polling are marked `offline` and rediscovered on subsequent cycles.
21. **FR-21**: The polling loop logs a summary at INFO level once per cycle: number of devices polled, number of failures, cycle duration.

**HTTP API (port 8091):**

22. **FR-22**: `GET /api/health` returns JSON with overall status (`healthy`/`degraded`/`unhealthy`), per-endpoint connectivity (online/offline), and the last successful poll timestamp. `healthy` = all endpoints reachable. `degraded` = some endpoints reachable. `unhealthy` = no endpoints reachable.
23. **FR-23**: `GET /api/assets` returns a JSON array of all discovered assets. Each asset includes: id, endpoint, unit_id, addressing, holding_register_count, coil_count, response_time_ms, response_jitter_ms, status, first_seen, last_seen, via_gateway.
24. **FR-24**: `GET /api/assets/{id}/registers` returns the latest holding register values and coil values for the specified asset. Register values are returned as an array of uint16. Coil values are returned as an array of booleans. Returns 404 if the asset ID is not found.
25. **FR-25**: All API responses use `Content-Type: application/json` and include CORS headers (`Access-Control-Allow-Origin: *`) for dashboard consumption.
26. **FR-26**: The API server starts on the configured bind address (default `:8091`) before discovery begins, so that `/api/health` is available immediately (returning `unhealthy` until devices are discovered).

**Health Check (TD-012):**

27. **FR-27**: The `--health` flag performs a real health check: HTTP GET to `http://localhost:8091/api/health` and verifies the response status is 200 and the `status` field is not `unhealthy`. This replaces the current file-existence check.
28. **FR-28**: The health check timeout is 3 seconds (matching the Docker healthcheck timeout).

**Docker/Infrastructure (TD-020):**

29. **FR-29**: The monitor Docker Compose service is attached to both `wt-level3` (10.10.30.20) and `ps-station-lan` (10.20.1.20) networks.
30. **FR-30**: The monitor Docker Compose service is assigned to BOTH `water` and `pipeline` profiles, so it starts whenever either or both plant profiles are active.
31. **FR-31**: The monitor `depends_on` references both `plant-water` (condition: service_healthy) and `plant-pipeline` (condition: service_healthy), but only when both profiles are active. When a single profile is active, the monitor depends only on that profile's plant service. This is achieved by splitting monitor into two profile-specific service definitions or using Docker Compose `depends_on` with `condition: service_started` and `required: false` for the optional dependency.
32. **FR-32**: The `design/` directory is volume-mounted into the monitor container at `/design:ro` for future use by SOW-014.0's Design Library pages. The monitor binary does NOT read this mount in SOW-012.0.
33. **FR-33**: The startup WARNING about pipeline not being monitored (FR-14a from SOW-010.0) is removed. TD-020 is resolved.

### Non-Functional Requirements

34. **NFR-1**: The monitor binary starts and completes initial discovery within 30 seconds when all plant endpoints are reachable.
35. **NFR-2**: The polling loop completes a full cycle (all 12 endpoints) within the 2-second interval under normal conditions (all devices online, local Docker networking).
36. **NFR-3**: Memory usage remains below 50MB under continuous polling (no time-series storage in this SOW).
37. **NFR-4**: The monitor binary shuts down gracefully within 5 seconds on SIGINT/SIGTERM, closing all Modbus TCP connections.
38. **NFR-5**: The HTTP API responds to requests within 100ms under normal conditions.
39. **NFR-6**: `cd monitoring && go test -race ./...` passes with no failures.
40. **NFR-7**: `cd monitoring && go build ./cmd/monitor/` compiles with no errors.

**Key design decisions:**
- The config tells the monitor WHERE to look but NOT what registers exist or what values are normal. The monitor discovers these by probing.
- Gateway ports have `gateway: true` and a `scan_unit_ids` list. Non-gateway ports have a single `unit_id`.
- `description` is for logging and human readability -- the monitor does not use it for any logic.
- Each environment has a base `address` (the Docker network IP of the plant container). Endpoints specify only the port.

### Register Enumeration Algorithm

The register enumeration uses binary search to determine register count efficiently, minimizing the number of Modbus requests:

```
function enumerateRegisters(client, startAddr):
    // Step 1: Determine addressing mode
    _, err0 := client.ReadRegisters(0, 1, HOLDING_REGISTER)
    _, err1 := client.ReadRegisters(1, 1, HOLDING_REGISTER)

    if err0 == nil:
        baseAddr = 0  // zero-based
    else if err1 == nil:
        baseAddr = 1  // one-based
    else:
        return 0, unknown  // no registers

    // Step 2: Exponential expansion to find upper bound
    // [OT-REVIEW] Cap at 65535: Modbus register address is a 16-bit field (max 65535).
    probe = 1
    while probe <= 65535:
        _, err := client.ReadRegisters(baseAddr + probe, 1, HOLDING_REGISTER)
        if err (illegal data address):
            break
        probe *= 2

    // Step 3: Binary search between probe/2 and probe
    low = probe / 2
    high = probe
    while low < high:
        mid = (low + high + 1) / 2
        _, err := client.ReadRegisters(baseAddr + mid, 1, HOLDING_REGISTER)
        if err:
            high = mid - 1
        else:
            low = mid

    // [OT-REVIEW] Count calculation differs by addressing mode:
    // Zero-based: address 4 means 5 registers (0,1,2,3,4) -> count = low + 1
    // One-based: address 7 means 7 registers (1,2,3,4,5,6,7) -> count = low
    if baseAddr == 0:
        return low + 1, "zero-based"  // highest valid offset + 1
    else:
        return low, "one-based"       // highest valid address == count
```

This algorithm requires O(log N) Modbus reads to determine register count, where N is the maximum possible register count. For a 4096-register device: ~24 reads. For a 9-register gateway: ~8 reads.

[OT-REVIEW] Discovery for all endpoints uses a minimum 100ms delay between enumeration reads to avoid competing with the plant's simulated SCADA polling for connection slots. Devices like the ROC800 support only 4 concurrent TCP connections, and aggressive scanning during active SCADA polling is a known cause of production incidents in real OT environments. The training documentation should note: OT security tools are typically run during maintenance windows, not during active production.

### Asset Inventory Model

```go
// Asset represents a discovered Modbus device.

    LastSeen           time.Time `json:"last_seen"`

    EnvironmentName    string    `json:"environment"`           // from config
    ConfigDescription  string    `json:"config_description"`    // from config endpoint description

    PurdueLevel        string    `json:"purdue_level"`          // [OT-REVIEW] derived from environment name (e.g., "level-3", "flat-legacy", "station-lan")

// Inventory provides concurrent-safe access to discovered assets.
type Inventory struct {
    mu     sync.RWMutex
    assets map[string]*Asset  // keyed by asset ID
}
```

### Polling Architecture

The polling loop runs in a single goroutine, polling devices sequentially. Sequential polling is acceptable because:

1. Docker networking delivers sub-1ms latency. A 2-second interval is ample for 12 devices.
2. Sequential polling avoids overloading devices with limited concurrent connections (ROC800: 4 max, Moxa: 4 max).
3. Sequential polling makes the log output deterministic and easier to follow for training purposes.

```
                  ┌───────────────────────────────┐
                  │         Polling Loop           │
                  │                                │
                  │  for each online asset:        │
                  │    1. Read all holding regs     │
                  │    2. Read all coils            │
                  │    3. Update inventory          │
                  │    4. Record response time      │
                  │                                │
                  │  for each offline asset:        │
                  │    1. Attempt reconnect         │
                  │    2. If OK, run discovery      │
                  │    3. Promote to online         │
                  │                                │
                  │  sleep until next interval      │
                  └───────────────────────────────┘
```

Register reads are split into chunks of 125 (Modbus spec maximum per request). A device with 2000 holding registers requires 16 read requests. At sub-1ms per request on Docker networking, this takes approximately 20ms including response delay simulation.

### Connection Management

Each configured endpoint gets one `modbus.ModbusClient` instance that is reused across polling cycles. The client is created during discovery and stored alongside the endpoint metadata. Connection failures during polling mark the device offline; the next cycle attempts reconnection.

For gateway ports, a single TCP connection handles all unit IDs (by calling `SetUnitId()` before each read). This matches real Modbus behavior: the gateway multiplexes unit IDs over one TCP socket.

[OT-REVIEW] For gateway endpoints with multiple unit IDs, the poller introduces a minimum 10ms inter-request delay between successive unit IDs on the same gateway port. This models the RS-485 inter-frame gap requirement (3.5 character times at 9600 baud = ~3.6ms) and prevents the gateway's internal serial queue from filling. The delay is configurable via `gateway_request_delay_ms` in the config schema (default: 10). On real serial buses, aggressive back-to-back polling causes unpredictable gateway behavior and is a common source of SCADA communication alarms.

## 10. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Register enumeration binary search hits edge cases (device returns unexpected exception codes) | Medium | Medium | Handle all Modbus exception codes (not just 0x02). If an unexpected exception is returned, fall back to conservative sequential probing. Log the exception for debugging. |
| simonvetter/modbus client connection pool exhaustion on gateway ports with many unit IDs | Low | Medium | One client per TCP endpoint, reuse via SetUnitId(). Monitor opens at most 8 TCP connections (one per configured endpoint). Well within device connection limits (4-32 concurrent). |
| Docker Compose `required: false` on depends_on not available in older Docker Compose versions | Medium | High | Require Docker Compose v2.20+ in documentation. Alternative: remove depends_on entirely and let the monitor handle plant-not-reachable gracefully (already handled by offline status). |
| Discovery takes longer than start_period (15s), causing Docker healthcheck to fail | Low | Medium | The health check queries the HTTP API, which starts before discovery. Even during discovery, /api/health returns "unhealthy" with a 200 status code, so the health check passes. start_period gives 15 seconds before the first health check attempt. |
| Polling 12 devices every 2 seconds exceeds interval when plant response delays are simulated (Modicon 984: 111ms delay) | Low | Medium | Worst case: 12 devices * 20 reads per device * 120ms = 28.8 seconds. This exceeds 2 seconds! Mitigation: poll registers in bulk (125 per read) to reduce round trips. Actual worst case with bulk reads: 12 devices * 2 register types * 1-16 reads * 120ms = ~46 seconds for highest-latency device. For the simulated environment, most devices have 5-20ms delay. Calculate actual worst case and adjust interval or parallelize if needed. |
| Concurrent access to Modbus client from polling loop (if parallelized) | Low | High | Sequential polling in v1 (single goroutine). No concurrent client access. Document as a future optimization point if polling exceeds interval. |
| Monitor on ps-station-lan can reach plant-pipeline but address is 10.20.1.10 (plant container), not individual device IPs (10.20.1.20, 10.20.1.21, 10.20.1.30) | Medium | Low | All pipeline Modbus ports are served by the single plant-pipeline container at 10.20.1.10. The monitor config uses the container IP. This is a known Docker limitation (TD-023 from SOW-010.0). The monitor sees one IP serving multiple ports, which is different from production but acceptable for training. |

### Polling Time Budget Analysis

Per-device poll time depends on register count and simulated response delay:

| Device | Holding Regs | Coils | Delay (ms) | Reads | Est. Time (ms) |
|--------|-------------|-------|-----------|-------|----------------|
| wt-plc-01..03 | ~5 | ~4 | 5 | 2 | 14 |
| mfg-gateway (uid 247) | ~9 | 0 | 15 | 2 | 34 |
| mfg-plc-01 (SLC-500) | ~7 | ~4 | 50+20 jitter | 2 | 140 (worst) |
| mfg-plc-02 (Modicon) | ~7 | ~4 | 80+50 jitter | 2 | 260 (worst) |
| ps-plc-01 | varies | varies | 5 | 2 | 14 |
| ps-rtu-01 (ROC800 metering) | 7 | 4 | 20+30 jitter | 2 | 100 (worst) |
| ps-rtu-02 (ROC800 monitoring) | 8 | 4 | 20+30 jitter | 2 | 100 (worst) |
| ps-gw-01 (uid 247) | ~9 | 0 | 15 | 2 | 34 |
| ps-fc-01 TotalFlow (via gw) | 9 | 3 | 15+10 jitter | 2 | 54 (worst) |

[OT-REVIEW] Estimated total: ~700ms per cycle (worst case with jitter). This fits within the 2-second interval under normal conditions. Worst-case jitter simultaneously peaking on all serial devices is unlikely but the ceiling is higher than typical. The Modicon 984's simulated 80ms base + 50ms jitter = 130ms worst-case per request is the bottleneck.

Note: In a real facility, legacy devices with slow response times are a genuine operational constraint that affects monitoring tool design and SCADA polling strategy -- this is an intentional teaching point.
