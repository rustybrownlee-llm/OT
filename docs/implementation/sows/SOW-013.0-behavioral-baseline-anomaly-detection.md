# SOW-013.0: Behavioral Baseline and Anomaly Detection

**Status**: Approved
**Created**: 2026-03-02
**Priority**: High

## 1. Objective

Add behavioral baseline learning and anomaly detection to the monitoring module. After SOW-012.0 establishes device discovery and register polling, this SOW records register values over time in per-device ring buffers, learns what "normal" looks like through statistical baselines (mean, standard deviation, min, max), and generates alerts when observed values deviate from baseline. Alerts are surfaced on the existing HTTP API (port 8091) so that SOW-014.0's dashboard can display them and SOW-015.0's scenarios can reference them.

After this SOW, an IT engineer can start both plant profiles, wait 5 minutes for the baseline to establish, write a rogue value to a holding register via `mbpoll`, and within one polling cycle see an anomaly alert appear at `curl http://localhost:8091/api/alerts` -- with severity, rule ID, expected vs actual value, and a human-readable description. The engineer can also query `curl http://localhost:8091/api/baselines` to see which devices have transitioned from "learning" to "established."

This is Beta 0.3 Definition of Done items #4 (baseline learning transition), #5 (rogue write triggers anomaly alert), and #6 (alert API returns anomaly events).

## 2. Background

### Current State (after SOW-012.0)

SOW-012.0 delivers:
- **Asset Inventory** (`monitoring/internal/inventory/inventory.go`): thread-safe store of discovered devices with latest register snapshots (`latestHolding []uint16`, `latestCoils []bool`).
- **Polling Loop** (`monitoring/internal/poller/poller.go`): reads all holding registers and coils from every online device every 2 seconds. Updates the inventory with each cycle's results.
- **HTTP API** (`monitoring/internal/api/api.go`): chi v5 router on port 8091 with `GET /api/health`, `GET /api/assets`, `GET /api/assets/{id}/registers`.
- **Config** (`monitoring/internal/config/config.go`): parsed `monitor.yaml` with environments, endpoints, poll interval.

SOW-012.0 stores only the **latest** register snapshot per device -- no history, no baselines, no alerts. The polling loop calls `inventory.UpdateRegisters()` and moves on. This SOW intercepts the polling results to build time-series history and evaluate anomaly rules.

### Why Simple Statistics

The anomaly detection uses mean +/- 3 standard deviations (the "3-sigma rule"). This is not a novel technique -- it is the same approach used by commercial OT monitoring tools like Dragos and Claroty for initial behavioral profiling. For an educational simulator, simple statistics have three advantages over machine learning:

1. **Transparent**: A trainee can understand why an alert fired ("value was 28,000; baseline mean is 16,000 +/- 2,000").
2. **Deterministic**: Given the same input sequence, the same baselines and alerts are produced.
3. **Fast**: No training phase beyond accumulating samples. No model serialization.

### Anomaly Detection in Real OT Environments

[OT-REVIEW] In production OT environments, behavioral baselines must account for operational context: shift patterns (day vs night staffing), seasonal changes (summer cooling loads), batch cycles (start-up, steady-state, shutdown), and maintenance windows. The simulator's 5-minute learning period captures a single operational mode. The alert API's `description` field should note this limitation when presenting alerts, and scenario documentation (SOW-015.0) should teach trainees that real baseline establishment requires days to weeks of observation across all operational modes.

### Architecture Constraints

Per ADR-005 D4, all anomaly detection is based on network-observed data (Modbus register values from the polling loop). The monitor has no knowledge of device atom YAML, register names, scaling factors, or engineering units. Anomaly rules operate on raw uint16 register values and boolean coil states. This is realistic: a real security monitoring tool deployed onto an unfamiliar OT network operates on raw protocol data until it is enriched with asset context.

### TD-021: No Persistence

Per milestone spec, baselines and alerts are in-memory only. On restart, the monitor re-enters the learning period. This is acceptable for a training tool and documented as TD-021. Persistent storage (SQLite, time-series DB) is deferred to a future milestone.

## 3. Scope

### In Scope

**Time-series storage:**
- In-memory ring buffer per register per device for holding registers
- In-memory ring buffer per coil per device for coil states
- Configurable buffer size (default: 300 samples = 10 minutes at 2-second polling)
- Each sample stores a timestamped value (uint16 for registers, bool for coils)

**Baseline learning:**
- Configurable learning period (default: 150 polling cycles = 5 minutes at 2-second polling)
- During learning: compute running mean, standard deviation, min, max for each holding register
- After learning period: baseline state transitions from "learning" to "established"
- Baseline is per-device, per-register
- Coil baselines track the "expected" state (most frequent value during learning)

**Anomaly detection rules:**
- Value out of range (holding register exceeds baseline mean +/- 3 standard deviations)
- Unexpected write (coil or holding register changes unexpectedly)
- New device (previously unknown unit ID responds during polling)
- Device offline (previously responding device stops responding)
- Response time anomaly (response time exceeds baseline mean + 3 standard deviations)

**Alert model and store:**
- Alert struct with id, timestamp, severity, rule_id, device_id, description, register_address, expected_value, actual_value, acknowledged flag
- In-memory alert store (ring buffer, max 1000 alerts)
- Alerts auto-clear when values return to normal (but remain in history)

**Alert API (extends SOW-012.0's HTTP API on port 8091):**
- `GET /api/alerts` -- list active alerts with severity and device filters
- `GET /api/alerts/{id}` -- single alert detail
- `POST /api/alerts/{id}/acknowledge` -- acknowledge an alert
- `GET /api/baselines` -- baseline status per device (learning/established, sample count)

**Config extensions:**
- `baseline_learning_cycles` in `monitor.yaml` (default: 150)
- `ring_buffer_size` in `monitor.yaml` (default: 300)
- `max_alerts` in `monitor.yaml` (default: 1000)

**Integration with SOW-012.0 polling loop:**
- Hook into the polling loop's post-cycle callback
- After each cycle, feed register snapshots to the baseline engine
- After each cycle, evaluate all anomaly rules against the latest snapshot

### Out of Scope

- Dashboard UI for alerts or baselines (SOW-014.0)
- Persistent storage for baselines or alerts (TD-021)
- Machine learning, clustering, or adaptive thresholds
- Cross-device correlation (e.g., "pump on in device A should cause flow increase in device B")
- Alert escalation, notification, or external integration (email, syslog, SIEM)
- Custom threshold configuration per register (all registers use 3-sigma)
- Alert suppression during learning period (alerts are generated only after baseline is established)
- Input register or discrete input baselines (SOW-012.0 polls holding registers and coils only)

## 4. Requirements

### Functional Requirements

**Time-Series Ring Buffer:**

1. **FR-1**: Each discovered device has a per-register ring buffer storing timestamped holding register samples. Buffer capacity is configurable (`ring_buffer_size`, default: 300).
2. **FR-2**: Each ring buffer entry stores: timestamp (`time.Time`), value (`uint16` for holding registers, `bool` for coils).
3. **FR-3**: When the ring buffer is full, the oldest entry is overwritten (standard ring buffer behavior). No memory allocation after initialization.
4. **FR-4**: Each discovered device has a per-coil ring buffer storing timestamped coil state samples, with the same capacity as holding register buffers.
5. **FR-5**: Ring buffers are created lazily when the first poll result is recorded for a device. Buffer size is determined by the device's register count (from SOW-012.0's discovery).

**Baseline Learning:**

6. **FR-6**: The baseline engine tracks a configurable learning period measured in polling cycles (`baseline_learning_cycles`, default: 150 cycles = 5 minutes at 2-second polling). [OT-REVIEW] The learning period is measured in polling cycles, not wall-clock time, because polling may be delayed by network timeouts or device recovery. Cycle-based measurement ensures the baseline always has the configured number of samples regardless of actual elapsed time. [OT-REVIEW] **Known limitation -- long operational cycles**: The default 150-cycle (5-minute) learning period may be insufficient for devices with long operational cycles. For example, the TotalFlow G5's NGC (Natural Gas Chromatograph) runs analysis cycles every 5-10 minutes. With a 5-minute learning window, the baseline may not capture a full analysis cycle. Registers like chromatograph_c1_mol_pct through chromatograph_c6_mol_pct will show one stable value during learning, then jump to a new value when the next analysis completes -- triggering false alarms. A future SOW should add per-environment or per-device learning period overrides in the monitor.yaml config to address this. See Risk table and TD-029.
7. **FR-7**: During the learning period, the baseline engine computes a running mean, running standard deviation (Welford's online algorithm), min, and max for each holding register's raw uint16 values.
8. **FR-8**: After `baseline_learning_cycles` samples have been collected for a device, that device's baseline transitions from "learning" to "established." This transition is logged at INFO level.
9. **FR-9**: Baseline statistics are per-device, per-register. A device with 5 holding registers has 5 independent baselines.
10. **FR-10**: For coils, the baseline records the most frequent state during the learning period (mode). A coil whose "expected" state is `false` that transitions to `true` without a preceding write command is flagged by the "unexpected write" rule.
11. **FR-11**: Baselines are NOT updated after the learning period ends. The baseline is a fixed reference snapshot of "normal" behavior. [OT-REVIEW] In real OT monitoring tools, baselines are periodically refreshed or use sliding windows. The simulator's fixed baseline is simpler and creates more dramatic anomaly detection -- values that drift naturally after the learning period will eventually trigger alerts, which is itself a teaching point about baseline staleness. [OT-REVIEW] In a training session, if the process state has changed significantly (e.g., after a scenario exercise that modified setpoints), the user can restart the monitor process to re-enter the learning period and re-learn baselines from scratch. Since baselines are in-memory only (TD-021), a restart is a clean slate.
12. **FR-12**: If a device goes offline during the learning period and comes back online, the learning period continues from where it left off (sample count is preserved, not reset).

**Anomaly Detection Rules:**

13. **FR-13**: **Value Out of Range** (rule ID: `value_out_of_range`, severity: Warning): After baseline is established, if a holding register value exceeds `mean + 3 * stddev` or falls below `mean - 3 * stddev`, an alert is generated. If `stddev` is 0 (constant register), any change from the mean triggers the alert. [OT-REVIEW] **Register type classification**: During the learning period, the baseline engine classifies each register into one of three behavioral types based on observed value patterns:
    - **Constant**: All values identical (stddev = 0). Alerts on any change.
    - **Analog**: Values fluctuate around a mean (typical of sensors, process variables). Standard 3-sigma detection applies at Warning severity. [OT-REVIEW] **Severity escalation**: If the deviation exceeds 6 * stddev or the value exceeds the observed learning-period min/max, severity escalates to High. This catches cases where a register representing a safety-critical parameter (e.g., lube_oil_pressure, chlorine_residual) moves far outside its operational envelope. Since the monitor lacks register semantics, deviation magnitude serves as a proxy for severity.
    - **Counter**: Values are monotonically increasing (>=90% of consecutive sample deltas are non-negative and the overall trend is upward). Counter registers are EXCLUDED from value_out_of_range magnitude detection because they will always grow beyond the learned range. Instead, counters are monitored via a `counter_direction_change` sub-rule under value_out_of_range: if a counter-type register decreases between consecutive polls, a High-severity alert is generated (a decreasing counter may indicate register tampering or device reset).

    Classification is automatic during learning: if all samples are identical, it's constant. If >=90% of consecutive deltas are non-negative and the final value exceeds the first value, it's a counter. Everything else is analog. This avoids false positives on registers like pump_runtime_hours, product_count, serial_tx_count, and meter_run_1_volume_today, which are counter registers present in the water treatment, manufacturing, and pipeline device atoms. The `RegisterBaseline` struct stores a `Type` field ("constant", "analog", or "counter") set at the end of the learning period.
14. **FR-14**: **Unexpected Write** (rule ID: `unexpected_write`, severity: High): For holding registers classified as "constant" during learning, any value change between consecutive polls generates an alert. For "analog" registers, a step change exceeding 6 * stddev between consecutive polls generates an alert (6-sigma rather than 3-sigma to distinguish genuine attacks from aggressive process transients). Counter registers are exempt from this rule (they change by design). [OT-REVIEW] For coils, the rule uses **transition frequency** rather than simple mode comparison: during learning, the engine counts how many times each coil toggled. Coils that NEVER toggled during learning ("locked coils" -- e.g., emergency shutdown, safety interlocks) generate a High-severity alert on ANY state change. Coils that toggled at least once during learning ("operational coils" -- e.g., pump start/stop, valve open/close) generate a Warning-severity alert only if they toggle more than 3x the maximum observed toggle frequency per polling window. This prevents false positives from normal operator commands while still detecting rapid automated toggling (e.g., a script cycling a pump). The severity distinction reflects OT reality: a coil that was locked for 5 minutes and suddenly changes is more concerning than one that operators toggle routinely.
15. **FR-15**: **New Device** (rule ID: `new_device`, severity: Critical): If the polling loop discovers a unit ID that was not present in the asset inventory at the end of the initial discovery phase, an alert is generated. The alert includes the endpoint, unit ID, and the timestamp of first response. [OT-REVIEW] In real OT networks, a new device appearing is a strong indicator of unauthorized equipment -- a rogue PLC, a penetration testing tool, or an attacker's man-in-the-middle device. Severity is Critical because asset integrity is foundational to OT security. The simulator triggers this by hot-connecting a new unit ID (future scenario feature).
16. **FR-16**: **Device Offline** (rule ID: `device_offline`, severity: High): If a device that was discovered during the initial discovery phase transitions from online to offline (stops responding), an alert is generated regardless of whether the baseline is still in the learning period or already established. The alert is cleared when the device comes back online. [OT-REVIEW] A device going offline is important to detect immediately -- it does not require a behavioral baseline to identify. Waiting until after baseline establishment would miss critical events during the first 5 minutes of monitoring. In OT environments, an unplanned device going offline may indicate a hardware failure, a denial-of-service attack, or a network segmentation event (e.g., an attacker isolating a PLC from SCADA). Severity is High rather than Critical because brief communication interruptions are common in legacy OT networks, especially over serial gateways.
17. **FR-17**: **Response Time Anomaly** (rule ID: `response_time_anomaly`, severity: Warning): If a device's response time during a polling cycle exceeds `baseline_mean_response_time + 3 * baseline_response_time_stddev`, an alert is generated. Response time baselines use the same learning period as register baselines. [OT-REVIEW] Response time anomalies can indicate network congestion, man-in-the-middle proxying (which adds latency), or a device under computational stress (e.g., a PLC with a logic fault consuming CPU). In the simulator, Docker networking has near-zero jitter, so response time anomalies will only be triggered by simulated response delays (from device atom `response_delay_ms` configurations). This is a teaching point: in real OT networks, response time is far more variable.

**Alert Model:**

18. **FR-18**: The `Alert` struct contains: `ID` (string, UUID v4), `Timestamp` (time.Time, when alert was generated), `Severity` (string: "critical", "high", "warning", "info"), `RuleID` (string: one of the 5 rule IDs), `DeviceID` (string: asset ID from inventory), `Description` (string: human-readable explanation), `RegisterAddress` (int, -1 if not register-specific), `ExpectedValue` (string: baseline value or range), `ActualValue` (string: observed value), `Acknowledged` (bool, default false), `Active` (bool, true when condition persists), `ClearedAt` (*time.Time, nil while active).
19. **FR-19**: Alert IDs are deterministic UUIDs generated from a hash of (rule_id + device_id + register_address). This means the same anomaly condition on the same device and register produces the same alert ID, preventing duplicate alerts. When an existing alert is re-triggered (condition still active), the timestamp is NOT updated -- the original detection time is preserved. [OT-REVIEW] The 16-char hex truncation of the SHA-256 hash yields 64 bits of entropy, which is collision-safe for the expected alert population (fewer than 1000 concurrent alerts across all devices and rules). Birthday-paradox collision probability at 1000 alerts is approximately 2^-45, effectively zero.
20. **FR-20**: The in-memory alert store is a ring buffer with a configurable maximum capacity (`max_alerts`, default: 1000). When full, the oldest cleared (inactive) alert is evicted. If no cleared alerts exist, the oldest active alert is evicted.
21. **FR-21**: An alert transitions from active to cleared when the condition that triggered it no longer holds (e.g., register value returns to within 3-sigma, offline device comes back online). The `ClearedAt` timestamp is set and `Active` is set to false. Cleared alerts remain in the ring buffer as history until evicted.

**Alert API (port 8091, extending SOW-012.0's chi router):**

22. **FR-22**: `GET /api/alerts` returns a JSON array of alerts. Supports query parameters: `severity` (filter by severity level, e.g., `?severity=high`), `device` (filter by device ID, e.g., `?device=10.10.30.10:5020:1`), `active` (filter by active status, e.g., `?active=true`). Multiple filters are combined with AND. No filter returns all alerts. Results are ordered by timestamp descending (newest first).
23. **FR-23**: `GET /api/alerts/{id}` returns a single alert by ID. Returns 404 if the alert ID is not found.
24. **FR-24**: `POST /api/alerts/{id}/acknowledge` sets the alert's `Acknowledged` field to true. Returns the updated alert. Returns 404 if the alert ID is not found. Acknowledging an alert does NOT clear it -- the alert remains active until the underlying condition resolves. [OT-REVIEW] In real OT security operations, acknowledgment means "an operator has seen this and is investigating." The alert continues to be visible and active. This models the workflow used by tools like Dragos and Claroty where alerts are triaged, not dismissed.
25. **FR-25**: `GET /api/baselines` returns a JSON object with per-device baseline status. Each entry includes: device ID, status ("learning" or "established"), sample count, total required samples, and per-register statistics (mean, stddev, min, max) for established baselines. [OT-REVIEW] Exposing per-register statistics on the API enables the dashboard (SOW-014.0) to show a trainee exactly what the monitor considers "normal" -- making the 3-sigma detection rule transparent and educational.
26. **FR-26**: All new API endpoints use the same CORS middleware and JSON content type as SOW-012.0's existing endpoints.

**Integration with Polling Loop:**

27. **FR-27**: The polling loop in `poller.go` is extended with a post-cycle hook. After each cycle updates the inventory, it calls the baseline engine's `RecordCycle()` method with the latest register snapshots for all online devices.
28. **FR-28**: The baseline engine's `RecordCycle()` method: (1) stores values in ring buffers, (2) updates running statistics if in learning mode, (3) evaluates all anomaly rules if baseline is established, (4) generates or clears alerts as needed. This runs synchronously in the polling goroutine -- no additional goroutines.
29. **FR-29**: Alert evaluation must complete within 50ms for all 12 devices. This is well within the 2-second polling interval budget (SOW-012.0 estimates ~700ms worst-case for the polling reads themselves, leaving ~1.3 seconds for alert evaluation and sleep).

**Config Extensions:**

30. **FR-30**: The `monitor.yaml` config schema is extended with three new fields at the top level:
```yaml
baseline_learning_cycles: 150  # Number of polling cycles for learning (default: 150)
ring_buffer_size: 300           # Samples per register per device (default: 300)
max_alerts: 1000                # Maximum alerts in memory (default: 1000)
```
All three fields are optional with the stated defaults.

### Non-Functional Requirements

31. **NFR-1**: `cd monitoring && go build ./cmd/monitor/` compiles with no errors.
32. **NFR-2**: `cd monitoring && go test -race ./...` passes all tests (existing SOW-012.0 tests + new tests).
33. **NFR-3**: Memory usage for time-series storage: 300 samples * (8 bytes timestamp + 2 bytes value) * ~50 registers across 12 devices = ~180 KB. Total memory overhead for ring buffers and baselines stays below 1 MB.
34. **NFR-4**: Alert evaluation for all 12 devices completes in under 50ms (measured, not estimated).
35. **NFR-5**: No goroutine leaks from baseline or alert processing. All processing is synchronous in the polling loop goroutine.
36. **NFR-6**: Ring buffer operations are O(1) for insert and O(n) for iteration (when computing statistics during learning).

### Ring Buffer

The ring buffer is a fixed-size circular array that overwrites the oldest entry when full. It is generic enough to store different sample types.

```go
// timeseries/ringbuffer.go

// Sample represents a timestamped register value.
type RegisterSample struct {
    Timestamp time.Time
    Value     uint16
}

// CoilSample represents a timestamped coil state.
type CoilSample struct {
    Timestamp time.Time
    Value     bool
}

// RingBuffer is a fixed-size circular buffer of RegisterSamples.
type RingBuffer struct {
    data  []RegisterSample
    size  int  // capacity
    head  int  // next write position
    count int  // number of valid entries (0..size)
}

func NewRingBuffer(size int) *RingBuffer
func (rb *RingBuffer) Push(s RegisterSample)        // O(1)
func (rb *RingBuffer) Len() int                      // number of valid entries
func (rb *RingBuffer) Latest() (RegisterSample, bool) // most recent entry
func (rb *RingBuffer) Iterate(fn func(RegisterSample)) // oldest to newest, O(n)

// CoilRingBuffer is the same structure for coil samples.
type CoilRingBuffer struct {
    data  []CoilSample
    size  int
    head  int
    count int
}

func NewCoilRingBuffer(size int) *CoilRingBuffer
func (rb *CoilRingBuffer) Push(s CoilSample)
func (rb *CoilRingBuffer) Len() int
func (rb *CoilRingBuffer) Latest() (CoilSample, bool)
```

### Baseline Engine

The baseline engine is the central coordinator. It maintains per-device state and orchestrates learning, rule evaluation, and alert generation.

```go
// baseline/baseline.go

// DeviceBaseline holds the baseline state for one device.
type DeviceBaseline struct {
    DeviceID          string
    Status            string // "learning" or "established"
    SampleCount       int
    RequiredSamples   int
    RegisterStats     []RegisterBaseline  // indexed by register address
    CoilExpected      []bool              // most frequent coil state during learning
    CoilTrueCounts    []int               // count of true samples during learning
    ResponseTimeMean  float64             // mean response time during learning (ms)
    ResponseTimeStd   float64             // stddev of response time during learning
    ResponseTimeSamples int               // number of response time samples
}

// RegisterBaseline holds running statistics for one register.
type RegisterBaseline struct {
    Mean    float64
    StdDev  float64  // computed via Welford's algorithm
    Min     uint16
    Max     uint16
    Type    string   // "constant", "analog", or "counter" -- classified at end of learning
    // Welford's internal state
    m2      float64  // sum of squared differences from mean
    count   int
}

// Engine orchestrates baseline learning and anomaly detection.
type Engine struct {
    baselines      map[string]*DeviceBaseline  // keyed by device ID
    alertStore     *alert.Store
    learningCycles int
    bufferSize     int
    // Ring buffers: map[deviceID][]RingBuffer (one per register)
    registerBuffers map[string][]*timeseries.RingBuffer
    coilBuffers     map[string][]*timeseries.CoilRingBuffer
    // Set of device IDs present at end of initial discovery (for "new device" rule)
    knownDeviceIDs  map[string]bool
    mu             sync.RWMutex
}

func NewEngine(learningCycles, bufferSize, maxAlerts int) *Engine
func (e *Engine) SetKnownDevices(deviceIDs []string)  // called after initial discovery
func (e *Engine) RecordCycle(snapshots []DeviceSnapshot) // called after each poll cycle
func (e *Engine) GetBaselines() map[string]*DeviceBaseline  // for API
func (e *Engine) GetDeviceBaseline(deviceID string) (*DeviceBaseline, bool)
```

### Welford's Online Algorithm

[OT-REVIEW] Welford's online algorithm computes running mean and variance in a single pass with O(1) memory per register. This is important because the monitor cannot store all historical values for statistics computation (ring buffer overwrites old values). The algorithm maintains three values per register: count, mean, and M2 (sum of squared differences). On each new sample:

```
count += 1
delta = new_value - mean
mean += delta / count
delta2 = new_value - mean
M2 += delta * delta2
variance = M2 / count  (population variance)
stddev = sqrt(variance)
```

This algorithm is numerically stable for the sample sizes involved (150 samples). For comparison, the naive algorithm (sum of squares minus square of sum) suffers from catastrophic cancellation with large values -- a risk with uint16 register values that can reach 65535.

### Anomaly Rules

Each rule is a function that evaluates one aspect of the current polling snapshot against the baseline:

```go
// baseline/rules.go

// Rule represents an anomaly detection rule.
type Rule struct {
    ID       string
    Severity string
    Evaluate func(snapshot DeviceSnapshot, baseline *DeviceBaseline, store *alert.Store)
}

// DeviceSnapshot represents one polling cycle's results for one device.
type DeviceSnapshot struct {
    DeviceID     string
    Timestamp    time.Time
    Holding      []uint16
    Coils        []bool
    ResponseMs   float64
    Online       bool
    PrevHolding  []uint16   // previous cycle's values (for change detection)
    PrevCoils    []bool
}

var Rules = []Rule{
    {ID: "value_out_of_range", Severity: "warning", Evaluate: evaluateValueOutOfRange},
    {ID: "unexpected_write", Severity: "high", Evaluate: evaluateUnexpectedWrite},
    {ID: "new_device", Severity: "critical", Evaluate: evaluateNewDevice},
    {ID: "device_offline", Severity: "high", Evaluate: evaluateDeviceOffline},
    {ID: "response_time_anomaly", Severity: "warning", Evaluate: evaluateResponseTimeAnomaly},
}
```

**Rule: Value Out of Range**

```
for each register addr in device:
    type = baseline.RegisterStats[addr].Type
    if type == "constant":
        if value != baseline.RegisterStats[addr].Mean:
            -> generate alert (any deviation from constant value)
    else if type == "analog":
        if |value - mean| > 3 * stddev:
            severity = "warning"
            if |value - mean| > 6 * stddev OR value > max OR value < min:
                severity = "high"  // severity escalation per Major-3
            -> generate alert at severity
        else if alert exists and was active:
            -> clear alert (value returned to normal)
    else if type == "counter":
        // Counters excluded from out-of-range; monitor direction only
        if counter_direction_changed (value decreased from previous):
            -> generate alert (counter_direction_change sub-rule, severity: high)
        else if alert exists and was active:
            -> clear alert
```

**Rule: Unexpected Write**

```
for each register addr in device:
    type = baseline.RegisterStats[addr].Type
    if type == "constant":
        if current != previous:
            -> generate alert (any change to a constant register is unexpected)
    else if type == "analog":
        if |current - previous| > 6 * stddev:
            -> generate alert (step change exceeds 6-sigma between consecutive polls)
        else if alert exists and was active:
            -> clear alert
    // type == "counter": exempt (counters change by design)

for each coil addr in device:
    if coil is "locked" (never toggled during learning):
        if coil_state changed:
            -> generate alert (High severity: locked coil changed)
    else:  // "operational" coil (toggled at least once during learning)
        if toggle_frequency > 3 * max_observed_toggle_frequency:
            -> generate alert (Warning severity: excessive toggle rate)
    else if alert exists and was active:
        -> clear alert
```

**Rule: New Device**

```
if device.ID not in knownDeviceIDs:
    -> generate alert (device was not present at end of initial discovery)
    -> add to knownDeviceIDs (don't re-alert on next cycle)
```

**Rule: Device Offline**

```
// Runs regardless of baseline status (learning or established)
if device.Online == false:
    -> generate alert
else if device.Online == true and alert was active:
    -> clear alert
```

**Rule: Response Time Anomaly**

```
if baseline.ResponseTimeMean > 0 and baseline.ResponseTimeStd > 0:
    threshold = baseline.ResponseTimeMean + 3 * baseline.ResponseTimeStd
    if device.ResponseMs > threshold:
        -> generate alert
    else if alert exists and was active:
        -> clear alert
```

### Alert Store

```go
// alert/alert.go

// Alert represents a detected anomaly.
type Alert struct {
    ID              string     `json:"id"`
    Timestamp       time.Time  `json:"timestamp"`
    Severity        string     `json:"severity"`         // "critical", "high", "warning", "info"
    RuleID          string     `json:"rule_id"`
    DeviceID        string     `json:"device_id"`
    Description     string     `json:"description"`
    RegisterAddress int        `json:"register_address"` // -1 if not register-specific
    ExpectedValue   string     `json:"expected_value"`
    ActualValue     string     `json:"actual_value"`
    Acknowledged    bool       `json:"acknowledged"`
    Active          bool       `json:"active"`
    ClearedAt       *time.Time `json:"cleared_at,omitempty"`
}

// Store is a thread-safe ring buffer of alerts.
type Store struct {
    mu       sync.RWMutex
    alerts   []*Alert       // ring buffer backing array
    index    map[string]int // alert ID -> position in ring buffer
    size     int            // capacity
    count    int            // number of valid entries
}

func NewStore(maxAlerts int) *Store
func (s *Store) GetOrCreate(id string, create func() *Alert) *Alert  // idempotent alert creation
func (s *Store) Get(id string) (*Alert, bool)
func (s *Store) List(filters ...FilterFunc) []*Alert    // with filtering
func (s *Store) Acknowledge(id string) (*Alert, bool)
func (s *Store) Clear(id string, clearedAt time.Time)   // mark alert as inactive
func (s *Store) Count() int

// FilterFunc filters alerts.
type FilterFunc func(*Alert) bool

func BySeverity(severity string) FilterFunc
func ByDevice(deviceID string) FilterFunc
func ByActive(active bool) FilterFunc
```

### Alert ID Generation

Alert IDs are deterministic to prevent duplicate alerts for the same condition. The ID is generated by hashing the combination of rule ID, device ID, and register address:

```go
func AlertID(ruleID, deviceID string, registerAddr int) string {
    h := sha256.New()
    fmt.Fprintf(h, "%s:%s:%d", ruleID, deviceID, registerAddr)
    return hex.EncodeToString(h.Sum(nil))[:16]  // 16-char hex string
}
```

For device-level rules (new_device, device_offline, response_time_anomaly), `registerAddr` is -1.

### Polling Loop Integration

SOW-012.0's `Poller` is extended with a hook for the baseline engine:

```go
// poller/poller.go (extension)

// CycleHook is called after each polling cycle with the latest snapshots.
type CycleHook func(snapshots []baseline.DeviceSnapshot)

// Poller.SetCycleHook(hook CycleHook) -- register the baseline engine's RecordCycle
```

The poller builds `DeviceSnapshot` structs from the inventory after each cycle and calls the hook. The hook is called synchronously -- the next cycle does not start until the hook returns. This ensures alerts are generated in sequence with polling data.

The poller retains the previous cycle's register values to populate `PrevHolding` and `PrevCoils` in the snapshot (needed by the "unexpected write" rule).

```

    v
[Build DeviceSnapshots]  -- includes current values, previous values, response time
    |
    v
[Baseline Engine: RecordCycle()]
    |
    +-- Store values in ring buffers
    |
    +-- If learning: update running statistics (Welford's)
    |       |

    v
```

## 10. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Counter register classification heuristic may misclassify registers | Low | Medium | [OT-REVIEW] The automatic counter classification heuristic (>=90% non-negative deltas + upward trend = counter) mitigates the original risk of counter false positives. However, the heuristic may misclassify registers that happen to be monotonically increasing during the learning window (e.g., a temperature rising during startup). Misclassified registers will be excluded from value_out_of_range checks when they should not be. See TD-030. |
| Welford's algorithm numerical precision with uint16 values (0-65535) | Low | Low | Welford's is numerically stable by design. Float64 has 15-16 significant digits. uint16 max (65535) squared (4.29e9) is well within float64 precision. Unit test with max values confirms. |
| Polling loop exceeds 2-second interval when alert evaluation is added | Low | Medium | Alert evaluation is O(registers * devices) with simple arithmetic. For 12 devices with ~100 total registers, this is ~100 comparisons -- sub-microsecond work. Measured in unit test (NFR-4: < 50ms). The polling reads themselves dominate cycle time (~700ms per SOW-012.0 analysis). |
| Ring buffer memory growth if many devices are discovered | Low | Low | Memory budget: 300 samples * 10 bytes * 50 registers * 12 devices = ~180 KB. Even at 100 devices with 100 registers each: 300 * 10 * 100 * 100 = ~30 MB. Well within constraints. |
| Concurrent access to baseline engine from polling goroutine and API handler goroutine | Medium | High | Engine uses sync.RWMutex. RecordCycle (write path) acquires write lock. GetBaselines/GetDeviceBaseline (read path, called by API handlers) acquire read lock. Alert store has its own mutex. Race detector tests validate. |
| "New device" rule has no natural trigger in the current simulator (unit IDs are static) | Medium | Low | The rule's logic is tested in unit tests. Manual triggering requires modifying the plant runtime or config to add a new unit ID, which is a future scenario feature (SOW-015.0 or beyond). The rule is implemented correctly even if it is rarely triggered in current scenarios. |
| Deterministic alert IDs based on SHA-256 hash may collide | Negligible | High | 16-character hex string = 64 bits of entropy. With ~60 possible alert conditions (5 rules * 12 devices), collision probability is ~2^-52. Effectively zero. |
| Coil toggle frequency baseline may be inaccurate during short learning periods if operators do not perform typical operations | Medium | Medium | [OT-REVIEW] If the 5-minute learning period does not capture typical operator activity (e.g., pump start/stop cycles, valve operations), coils that are normally toggled may be classified as "locked" and generate false High-severity alerts on first legitimate toggle. Mitigation: document in scenario instructions that the learning period should include representative operations. A future SOW can add per-device learning overrides. |
| TotalFlow G5 NGC analysis cycle (5-10 min) may not complete during the default 5-minute learning period | Medium | Medium | [OT-REVIEW] Chromatograph registers will show one stable value during learning, then jump when the next analysis completes, triggering false alarms. Mitigation: document as a known limitation (see FR-6, TD-029). Future SOW should add per-environment or per-device learning period overrides. |

## 11. Technical Debt Created

| ID | Description | Resolution |
|----|-------------|------------|
| TD-029 | Baselines are fixed after the learning period. In real OT environments, baselines should be periodically refreshed or use sliding windows to account for operational changes (shift patterns, seasonal variation). Additionally, the default 150-cycle learning period may be insufficient for devices with long operational cycles (e.g., TotalFlow G5 NGC analysis runs every 5-10 minutes). [OT-REVIEW] A future SOW should add per-environment or per-device learning period overrides in monitor.yaml, and optionally a sliding-window baseline mode. | Future SOW: configurable per-device learning periods, baseline refresh interval, or sliding-window mode. |
| TD-030 | [OT-REVIEW] Counter register classification is now implemented via the automatic heuristic during learning (>=90% non-negative consecutive deltas + overall upward trend = counter). Counter registers are excluded from value_out_of_range magnitude checks and instead monitored for direction changes only. However, the heuristic may misclassify registers that happen to be monotonically increasing during the learning window but are actually analog (e.g., a temperature slowly rising during a startup sequence). The heuristic also cannot distinguish between daily counters that reset at midnight vs tampered counters. | Future SOW: improve counter classification with additional heuristics (e.g., check if delta magnitudes are consistent, correlate with known device register maps via an optional metadata file). |
| TD-031 | The "unexpected write" rule cannot distinguish between legitimate SCADA writes (from the HMI or a maintenance laptop) and rogue writes (from an attacker). All writes that exceed the change threshold trigger alerts. | Future SOW: integrate with a write-authorization model or whitelist known SCADA write patterns. Out of scope for a training tool but valuable as a teaching point. |
| TD-032 | Response time baselines on Docker networks have near-zero variance. Response time anomaly alerts are unlikely to fire naturally because Docker bridge networking delivers consistent sub-1ms latency. The rule is fully functional but requires simulated latency injection (via device atom `response_delay_ms`) to demonstrate. | Document as a simulator limitation. Consider adding a `tc` (traffic control) latency injection capability in a future SOW (related to TD-022). |
| TD-033 | The "new device" rule has no natural trigger mechanism in the current simulator. All unit IDs are static and present at startup. Demonstrating this rule requires stopping and restarting the plant with modified configuration. | SOW-015.0 scenarios can describe this rule conceptually. A future scenario SOW can add a "rogue device" injection mechanism. |
