# SOW-014.0: Monitoring Dashboard and Design Library

**Status**: Complete
**Created**: 2026-03-02
**Implemented**: 2026-03-02
**Priority**: High

## 1. Objective

Build the web UI served on port 8090 that gives security analysts a live monitoring dashboard ("Observed" section) and an integrated design-layer reference library ("Reference" section). The dashboard consumes the HTTP API from SOW-012.0 (port 8091) and the alert API from SOW-013.0 to present asset inventory, live register values, baseline status, and anomaly alerts. The design library reads YAML files from the `design/` volume mount to display device atoms, network atoms, and environment definitions with syntax highlighting. Cross-links between observed assets and their design-layer specifications let engineers see "what this device is designed to do" alongside "what the monitor is seeing on the wire."

After this SOW, an IT engineer can open `http://localhost:8090`, see a summary of all discovered OT devices, click into any device for live 2-second register polling, browse the design library for reference documentation, and follow cross-links between live observations and design-layer specifications -- all from a single server-rendered web application with no JavaScript build tools, no authentication, and single-binary deployment via `go:embed`.

## 2. Background

### Current State

The monitoring module (SOW-012.0) exposes a JSON API on port 8091 with three endpoints: `GET /api/health`, `GET /api/assets`, and `GET /api/assets/{id}/registers`. SOW-013.0 adds anomaly detection and an alert API (`GET /api/alerts`, `POST /api/alerts/{id}/acknowledge`). Port 8090 is allocated but serves nothing. The `design/` directory is volume-mounted into the monitor container at `/design:ro` (provisioned in SOW-012.0) but unused.

### Architecture Constraints

**ADR-005 D4 boundary**: The monitoring module MUST NOT import plant packages or use design-layer YAML for polling configuration or anomaly rules. The design library is a documentation feature -- it reads YAML files for display only. The dashboard UI clearly labels the distinction: "Observed" pages show data from network observation (Modbus polling), "Reference" pages show data from design-layer YAML files. This boundary is critical to the educational value: the monitor discovers device characteristics the same way a real security tool would, and the design library provides the "answer key" that engineers can reference.

**Single binary deployment**: All HTML templates, CSS, and static assets are embedded in the monitor binary using `go:embed`. No external file dependencies at runtime beyond the config YAML and the design volume mount.

**No JavaScript framework**: The dashboard uses server-rendered HTML with Bootstrap 5 (CDN) for layout and HTMX (CDN) for dynamic updates. No npm, no webpack, no build step. The only client-side scripting is HTMX attributes on HTML elements.

### What SOW-012.0 and SOW-013.0 Provide

**SOW-012.0 API (port 8091):**
- `GET /api/health` -- overall status, per-endpoint connectivity, poll timing
- `GET /api/assets` -- JSON array of all discovered assets with metadata (id, endpoint, unit_id, addressing, register counts, response time, status, environment, config_description)
- `GET /api/assets/{id}/registers` -- latest holding register values (uint16 array) and coil values (boolean array) for one asset

**SOW-013.0 API (port 8091, expected):**
- `GET /api/alerts` -- JSON array of anomaly alerts with timestamp, severity, device ID, rule, description
- `GET /api/alerts?severity={level}` -- filter alerts by severity
- `GET /api/alerts?device={id}` -- filter alerts by device
- `POST /api/alerts/{id}/acknowledge` -- acknowledge an alert
- `GET /api/baselines` -- baseline status per device (learning/established/stale)

The dashboard makes internal HTTP requests to these endpoints from the Go server (localhost loopback) rather than having the browser call the API directly. This simplifies CORS handling and keeps the browser pointed at a single origin (port 8090).

## 3. Scope

### In Scope

**Dashboard web server (port 8090):**
- chi v5 router serving HTML pages and HTMX partial responses
- `go:embed` for all HTML templates and static assets
- Internal HTTP client calling SOW-012.0/SOW-013.0 API on port 8091 (localhost loopback)
- Responsive layout using Bootstrap 5 grid (CDN)
- HTMX-driven live updates (CDN)
- Configurable dashboard bind address (`dashboard_addr` in monitor.yaml, default `:8090`)

**Section A: Monitoring Dashboard ("Observed" pages):**
- Overview page (`/`): summary cards for devices online/offline, active alerts by severity, baseline status
- Asset inventory page (`/assets`): table of all discovered devices grouped by environment
- Device detail page (`/assets/{id}`): live register values with HTMX polling every 2 seconds
- Alerts page (`/alerts`): timeline of anomaly alerts with severity/device/rule filters and acknowledge button

**Section B: Design Library ("Reference" pages):**
- Device library page (`/design/devices`): list all device atom YAML files with metadata summary
- Device detail page (`/design/devices/{id}`): single device atom with register map variants, syntax-highlighted YAML
- Network library page (`/design/networks`): list all network atom YAML files with metadata
- Network detail page (`/design/networks/{id}`): single network atom, syntax-highlighted YAML
- Environment library page (`/design/environments`): list all environments with placement summaries
- Environment detail page (`/design/environments/{id}`): placement table, port map, cross-links to device and network atoms

**Cross-links:**
- Asset inventory rows link to the corresponding device atom page (observed asset -> design reference)
- Device detail page links to its design-layer device atom
- Environment detail pages link to each device atom and network atom referenced in placements
- Design library device pages link back to live asset inventory entries for that device type

**Docker/infrastructure:**
- Dashboard port 8090 already published in SOW-012.0 Docker Compose changes
- No new Docker Compose changes required (port 8090 is already in the monitor service definition)

### Out of Scope

- Authentication or authorization (TD-023: acceptable for localhost training tool)
- TLS (TD-024)
- Persistent storage of dashboard state (alerts viewed, preferences)
- Historical register charting or time-series graphs (requires TD-021 persistent storage)
- WebSocket or SSE for push-based updates (HTMX polling is sufficient for training)
- Custom CSS themes or dark mode
- Mobile-optimized layout (responsive grid is sufficient, no dedicated mobile design)
- Design library YAML editing (read-only viewer only)
- Search or full-text filtering of design library content
- Packet capture or DPI visualization (TD-022)
- Cross-environment correlation views

## 4. Requirements

### Functional Requirements

**Dashboard Server:**

1. **FR-1**: The monitor binary serves the dashboard UI on a configurable bind address (`dashboard_addr` in monitor.yaml, default `:8090`). The `--dashboard-addr` CLI flag overrides the config value.
2. **FR-2**: All dashboard pages are server-rendered HTML using Go `html/template`. Templates are embedded in the binary via `go:embed`.
3. **FR-3**: The dashboard uses Bootstrap 5.3 CSS and JS from `cdn.jsdelivr.net` and HTMX 2.0 from `unpkg.com`. No local JavaScript files. No npm. No build tools.
4. **FR-4**: The dashboard makes internal HTTP GET/POST requests to the API on port 8091 (localhost loopback) to retrieve monitoring data. The browser never calls port 8091 directly.
5. **FR-5**: All dashboard pages share a common layout template with navigation bar. The nav bar has two sections: "Observed" (dashboard pages) and "Reference" (design library pages). The currently active section and page are visually highlighted.
6. **FR-6**: The dashboard is available immediately on startup, before device discovery completes. Pages that depend on discovered data show "Discovery in progress..." placeholder content until the first poll cycle completes.

**Overview Page (/):**

7. **FR-7**: The overview page displays summary cards: (a) devices online count, (b) devices offline count, (c) total alerts by severity (critical/high/warning), (d) baseline status summary (learning/established counts). Cards use Bootstrap color coding: green for healthy states, yellow for warnings, red for critical/offline.
8. **FR-8**: Summary cards auto-refresh every 5 seconds via HTMX `hx-trigger="every 5s"` on a partial template endpoint.
9. **FR-9**: Each summary card links to its detail page (devices card -> `/assets`, alerts card -> `/alerts`).

**Asset Inventory Page (/assets):**

10. **FR-10**: The asset inventory page displays a table of all discovered devices grouped by environment name. Table columns: device ID (config_description), endpoint (host:port), unit ID, connection type ("Direct Ethernet" for unit_id=255 or single-device endpoints, "Serial via Gateway" for unit_id != 255 on multi-unit-ID endpoints), addressing mode, holding register count, coil count, response time (ms), status (online/offline badge). The table is sorted by environment name, then by endpoint within each environment.
11. **FR-11**: Each row's device ID cell links to the device detail page (`/assets/{id}`).
12. **FR-12**: Each row includes a "View Atom" link to the design-layer device atom page (`/design/devices/{device-type}`). The device type is derived by matching the asset's register count and addressing mode to known device atoms. If no match is found, the link is omitted.
13. **FR-13**: Online devices show a green "Online" badge; offline devices show a red "Offline" badge. Offline rows are visually dimmed.
14. **FR-14**: The asset table auto-refreshes every 5 seconds via HTMX partial update.

**Device Detail Page (/assets/{id}):**

15. **FR-15**: The device detail page displays device metadata (asset ID, endpoint, unit ID, addressing, environment, response time, first seen, last seen, via_gateway if applicable) and a live register value table.
16. **FR-16**: The register value table has columns: address, raw value (labeled "Raw uint16 (unscaled)" for holding registers, "State" for coils), hex value. A note above the holding register table reads: "Values are raw Modbus register contents (unsigned 16-bit integers). See the device atom's register map for scaling formulas to convert to engineering units." Holding registers and coils are displayed in separate sub-tables.
17. **FR-17**: Register values auto-refresh every 2 seconds via HTMX `hx-trigger="every 2s"` on a partial template endpoint that returns only the register table rows. The page header does not refresh.
18. **FR-18**: If the design-layer device atom is available and the asset's device type can be resolved, register rows are enriched with design-layer metadata: register name, unit, scale range, writable flag (orange "Writable" badge for writable registers, gray "Read-only" badge for read-only), description. Writable registers are highlighted with an orange left border to draw attention to the security implications. This enrichment is a display-time join between observed register index and design-layer register address -- it does NOT affect monitoring logic.
19. **FR-19**: A "View Design Atom" link at the top of the page navigates to the design-layer device atom detail page for this device type.

**Alerts Page (/alerts):**

20. **FR-20**: The alerts page displays a timeline of anomaly alerts, most recent first. Each alert shows: timestamp, severity badge (critical=red, high=orange, warning=yellow), device ID (linked to device detail page), rule name, and description. For write-type alerts (unexpected_write rule), the alert also displays source context: the register address written, old value, new value, and a note: "Modbus TCP has no authentication -- any host on this network segment can write to this register." This teaches IT engineers why unauthenticated writes are an OT security concern.
21. **FR-21**: Filter controls allow filtering by severity level (dropdown), device ID (dropdown populated from discovered devices), and rule name (dropdown). Filters update the table via HTMX GET with query parameters.
22. **FR-22**: Each unacknowledged alert has an "Acknowledge" button that sends an HTMX POST to a dashboard handler, which proxies to `POST /api/alerts/{id}/acknowledge` on port 8091. The button is replaced with an "Acknowledged" badge on success.
23. **FR-23**: The alerts table auto-refreshes every 5 seconds via HTMX partial update. New alerts appear at the top without full page reload.
24. **FR-24**: If SOW-013.0 is not yet implemented (alert API returns 404 or connection refused), the alerts page displays a "Baseline and anomaly detection not yet available" message instead of an error.

**Design Library -- Device Library (/design/devices):**

25. **FR-25**: The device library page lists all YAML files in `/design/devices/`. Each entry shows: device ID, vendor, model, category, vintage, description, register capacity (max holding, max coils), addressing mode, and number of register map variants.
26. **FR-26**: Each entry links to the device detail page (`/design/devices/{id}`).
27. **FR-27**: Device entries are sorted by category (plc, rtu, flow-computer, gateway), then alphabetically by device ID within each category.

**Design Library -- Device Detail (/design/devices/{id}):**

28. **FR-28**: The device detail page displays the full parsed content of a device atom YAML: identity section (vendor, model, category, vintage, description), connectivity section (ports, response timing, concurrent connections), register capabilities, diagnostics section (if present -- web interface port, SNMP support, serial diagnostics ports), and all register map variants. The diagnostics section helps IT engineers understand what additional attack surface or management interfaces exist beyond Modbus.
29. **FR-29**: Each register map variant is displayed as a structured table with columns: address, name, unit, scale range (min-max), writable (yes/no badge), description. Holding registers and coils are in separate sub-tables per variant.
30. **FR-30**: Below the structured view, the raw YAML source is displayed in a `<pre><code>` block with CSS-based syntax highlighting (YAML keywords in one color, values in another, comments in a third). No JavaScript highlighting library.
31. **FR-31**: If any live assets match this device type, a "Live Assets" section lists them with links back to the device detail page in the Observed section (`/assets/{id}`).

**Design Library -- Network Library (/design/networks):**

33. **FR-33**: Each entry links to the network detail page (`/design/networks/{id}`).
34. **FR-34**: Entries are sorted by network ID alphabetically.

**Design Library -- Network Detail (/design/networks/{id}):**

Raw YAML source is shown below the structured view with syntax highlighting. The Purdue level is displayed as a badge (e.g., "Level 1 - Basic Control", "Level 3 - Site Operations") to help IT engineers understand where this network sits in the OT architecture hierarchy.
36. **FR-36**: An "Environments Using This Network" section lists all environments that reference this network atom, with links to the environment detail pages.

**Design Library -- Environment Library (/design/environments):**

**Design Library -- Environment Detail (/design/environments/{id}):**

40. **FR-40**: The environment detail page displays the full parsed environment: identity, network references (linked to network detail pages), and a placement table.
41. **FR-41**: The placement table has columns: placement ID, device atom (linked to `/design/devices/{device-id}`), network (linked to `/design/networks/{network-id}`), IP address, Modbus port, serial address (if applicable), gateway (if applicable), role, register map variant.

### Internal API Client

The dashboard Go server makes HTTP requests to `http://localhost:8091/api/...` to retrieve monitoring data. This is an in-process HTTP call (both servers run in the same binary, same container, localhost loopback).

```go
// APIClient wraps HTTP calls to the monitoring API (port 8091).
type APIClient struct {
    baseURL    string        // "http://localhost:8091"
    httpClient *http.Client  // shared client with timeout
}

func NewAPIClient(apiAddr string) *APIClient {
    return &APIClient{
        baseURL: "http://" + apiAddr,
        httpClient: &http.Client{
            Timeout: 2 * time.Second,
        },
    }
}

func (c *APIClient) GetHealth() (*HealthResponse, error)
func (c *APIClient) GetAssets() ([]Asset, error)
func (c *APIClient) GetAssetRegisters(id string) (*RegisterResponse, error)
```

### Asset-to-Device-Atom Mapping

Cross-links between observed assets and design-layer device atoms require mapping a discovered asset (identified by endpoint, unit ID, register count, addressing) to a device atom ID. The dashboard uses a heuristic based on the asset's `config_description` field from SOW-012.0, which contains the endpoint description from `monitor.yaml` (e.g., "wt-plc-01 (water intake)"). The mapping logic:

1. Parse the config_description for known placement IDs (e.g., "wt-plc-01", "ps-rtu-01")
2. Look up the placement ID in the environment definitions to find the device atom ID
3. [OT-REVIEW] If no match is found, display raw values only -- no enrichment. A confidently wrong cross-link (e.g., linking a Modicon 984 to a ROC800 because both have 7 one-based registers) teaches incorrect register semantics. The "View Atom" link is omitted for unresolved assets.

This heuristic is imperfect but sufficient for the training environment where placement IDs in `monitor.yaml` descriptions match the design-layer placement IDs. The register count + addressing mode fallback was removed after OT domain review because it produced silent false matches between devices with similar profiles but different register maps.

### HTMX Update Pattern

The dashboard uses HTMX polling for live updates. The pattern for all auto-refreshing sections:

```html
<!-- Full page loads once, HTMX targets replace inner content -->
<div id="register-values"
     hx-get="/partials/asset-registers/{{.AssetID}}"
     hx-trigger="every 2s"
     hx-swap="innerHTML">
    <!-- Initial content rendered server-side on page load -->
    {{template "asset_registers" .}}
</div>
```

The `/partials/*` routes return only the inner HTML fragment (no layout wrapper), keeping HTMX responses small and fast.

### CSS Syntax Highlighting

The YAML syntax highlighter is a Go function that processes YAML text into HTML spans:

```go
// HighlightYAML converts raw YAML text into HTML with CSS class spans.
// No JavaScript library required.
func HighlightYAML(yamlText string) template.HTML {
    // Process line by line:
    // 1. Lines starting with # -> <span class="yaml-comment">
    // 2. Lines with "key: value" -> <span class="yaml-key">key:</span> <span class="yaml-value">value</span>
    // 3. Lines starting with "- " -> <span class="yaml-list-marker">-</span> rest
    // 4. Inline comments (# after value) -> split at # boundary
    // 5. All other text -> plain
}
```

CSS classes (embedded in the layout template):

```css
.yaml-key     { color: #0d6efd; }  /* Bootstrap primary blue */
.yaml-value   { color: #198754; }  /* Bootstrap success green */
.yaml-comment { color: #6c757d; font-style: italic; }  /* Bootstrap secondary gray */
.yaml-list-marker { color: #dc3545; }  /* Bootstrap danger red */
pre.yaml-source { background: #f8f9fa; padding: 1rem; border-radius: 0.375rem; overflow-x: auto; }
```

### Register Enrichment

When displaying live register values on the device detail page, the dashboard attempts to enrich raw uint16/boolean values with design-layer context. The enrichment logic:

1. Resolve the asset's device atom from the design library (see Asset-to-Device-Atom Mapping above)
2. Determine which register map variant is in use (from the environment placement)
3. For each register address in the live data, look up the corresponding entry in the variant's holding/coil register map
4. If a match is found, display: address, name, raw value, scaled value (computed from raw value, scale_min, scale_max), unit, writable badge, description
5. If no match is found (design library unavailable or device type unresolved), display: address, raw value, hex value only

Scaled value calculation:
```
scaled = scale_min + (raw_value / 65535.0) * (scale_max - scale_min)
```

This uses the convention established in the device atoms where 16-bit scaled integers map 0-65535 to scale_min-scale_max. The dashboard performs this scaling purely for display purposes.

## 10. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| CDN unavailable during training session (offline lab environment) | Medium | High | Pages render without JavaScript or CSS if CDNs are unreachable. Static HTML content is still readable. Document this limitation. Future SOW could vendor CDN assets into go:embed for true offline operation. |
| Asset-to-device-atom mapping heuristic fails for some assets | Medium | Low | Graceful degradation: if mapping fails, register enrichment is omitted and raw values are displayed. The cross-link "View Atom" is omitted. All data is still visible, just less contextualized. |
| HTMX polling every 2 seconds creates noticeable load on the API server | Low | Low | Each HTMX poll is one HTTP GET to localhost. The API server already handles polling-loop reads every 2 seconds. Dashboard adds at most one additional reader. For a single-viewer training tool, this is negligible. |
| Go template parsing errors at build time due to embedded template syntax errors | Medium | Medium | Template parse errors are caught at startup (template.Must pattern). Template unit tests verify all templates parse and render without errors. CI/build step catches syntax errors before deployment. |
| Design library YAML schema changes break the dashboard parser | Low | Medium | The design library uses permissive YAML parsing (unknown fields are silently ignored). Schema version check logs a warning if the YAML declares a version newer than what the parser understands, but continues to display what it can parse. |
| Bootstrap 5 version or HTMX version updates break the UI | Low | Low | Pin specific CDN versions (Bootstrap 5.3.3, HTMX 2.0.4) in the layout template. Do not use "latest" or unversioned CDN URLs. |
| Large number of alerts fills the page and slows rendering | Low | Low | The alerts API returns alerts from an in-memory ring buffer (SOW-013.0). The buffer has a fixed size, so the maximum number of alerts is bounded. Pagination is not needed for v1. |
