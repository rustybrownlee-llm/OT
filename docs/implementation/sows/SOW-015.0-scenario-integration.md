# SOW-015.0 - Scenario Integration: Discovery Dashboard Walkthrough and Assessment Draft

**Status**: Approved
**Created**: 2026-03-02
**Priority**: Medium
**Dependencies**: SOW-012.0 (Monitoring Core), SOW-013.0 (Baseline & Anomaly), SOW-014.0 (Dashboard & Design Library)

## 1. Objective

Expand Scenario 01 (Discovery) with a monitoring dashboard walkthrough that demonstrates how the Beta 0.3 monitoring module accomplishes the same discovery tasks automated. Draft Scenario 02 (Vulnerability Assessment) as a complete guided exercise. Both scenarios reference the existing `greenfield-water-mfg` environment, consistent with ADR-007 D5.

## 2. Background

Scenario 01 currently teaches manual discovery using `nmap` and `mbpoll` against the water-mfg environment. Beta 0.3 introduced automated device discovery, baseline learning, and anomaly detection via the monitoring dashboard (port 8090) and alert API (port 8091). Scenario 01 needs a new section showing engineers how the dashboard performs the same discovery work, reinforcing the concepts they learned manually.

Scenario 02 is a placeholder (`README.md` with one line). The milestone Definition of Done requires engineers to "Read Scenario 02 (Assessment) and understand what vulnerabilities exist based on monitoring observations." This SOW delivers the full scenario content.

Pipeline environment (Beta 0.2) scenarios are out of scope for Beta 0.3 -- they will be authored when pipeline-specific exercises are designed.

## 4. Requirements

### Functional Requirements

1. Scenario 01 Phase E SHALL walk the engineer through opening `http://localhost:8090`, viewing the asset inventory, comparing discovered devices to their manual inventory, and understanding what the monitor found automatically.
2. Scenario 01 Phase E SHALL demonstrate the Design Library cross-link from a discovered asset to its device atom YAML.
3. Scenario 01 Phase E SHALL explain the distinction between "Observed" (live monitoring) and "Reference" (design library) data, per ADR-005 D4 boundary.
4. Scenario 02 SHALL follow the ADR-007 D2 structure: README.md, environment.ref, success-criteria.md, hints/, solutions/.
5. Scenario 02 SHALL cover five vulnerability categories: (a) Modbus protocol authentication gaps, (b) network segmentation weaknesses, (c) default/no credentials on infrastructure devices, (d) unencrypted protocol traffic, (e) legacy device risk (no patching, no firmware updates).
6. Scenario 02 SHALL use the monitoring dashboard observations (asset inventory, register values, baseline data) as the starting point for assessment, building on Scenario 01 skills.
7. Scenario 02 SHALL produce a vulnerability report as the deliverable, with a provided template.
8. Scenario 02 success criteria SHALL have at least 15 checkable items covering all five vulnerability categories.
9. Scenario 02 hints SHALL be progressive: hint-1 (conceptual nudge), hint-2 (technique), hint-3 (specific observation).

### Non-Functional Requirements

- All scenario content is documentation only (ADR-007 D4 -- no code, no scripts)
- Markdown formatting consistent with existing Scenario 01 style
- All mbpoll commands tested against live Docker environment
- No references to pipeline environment devices (those are out of scope)

## 5. Design

### Scenario 01 Phase E: Dashboard-Assisted Discovery

Inserted after Phase D (Asset Documentation), before Hints section. Content:

- Step E1: Start the monitoring module (`docker compose --profile water up` with monitor)
- Step E2: Open `http://localhost:8090` and view the Overview page (11 devices online)
- Step E3: Navigate to Assets page, observe grouped-by-environment inventory
- Step E4: Compare dashboard discovery to manual inventory from Phases A-D
- Step E5: Click "View Atom" to see design-layer device specification
- Step E6: Navigate to Alerts page, observe baseline learning status
- Step E7: Wait for baselines to establish, note the transition
- Teaching points: automated vs manual discovery, monitoring network position, ADR-005 D4 boundary

### Scenario 02: Vulnerability Assessment Structure

```
scenarios/02-vulnerability-assessment/
  README.md                              # Full scenario with Phases A-D
  environment.ref                        # Already exists: greenfield-water-mfg
  success-criteria.md                    # 15+ items across 5 categories
  hints/
    hint-1.md                            # "Think about what Modbus doesn't have"
    hint-2.md                            # "Check what's writable vs read-only"
    hint-3.md                            # "Try writing to a coil without auth"
  solutions/
    solution.md                          # Complete walkthrough
  reference/
    vulnerability-report-template.md     # Blank template for deliverable
    expected-vulnerability-report.md     # Reference answer
```

Assessment phases:

- **Phase A: Protocol Analysis** -- Examine Modbus TCP for authentication, encryption, and access control. Use mbpoll to demonstrate unauthenticated read/write.
- **Phase B: Network Architecture Review** -- Use the monitoring dashboard environment view and network atoms to identify segmentation gaps. Analyze cross-plant link, flat manufacturing network, lack of DMZ.
- **Phase C: Device Posture Assessment** -- Review device ages, firmware update capability, and default configurations. Use design library device atoms for reference specifications. Identify devices that cannot be patched.
- **Phase D: Vulnerability Documentation** -- Fill in the vulnerability report template. Assign risk ratings using a simplified OT risk matrix (impact to physical process x likelihood of exploitation).
