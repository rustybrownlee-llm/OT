# SOW-016.0 - Environment Archetype Schema Extensions

**Status**: Approved
**Created**: 2026-03-02
**Priority**: High
**Dependencies**: None (first SOW in Beta 0.4)
**OT Domain Review**: Complete (4 corrections applied, 5 improvements incorporated)

## 1. Objective

Extend the design layer schema with non-breaking fields that classify environments by archetype, mark device installation eras, and describe network boundary states. Update the `ot-design validate` CLI to accept and validate these new fields. Retroactively tag the three existing environments with archetype metadata. After this SOW, all downstream Beta 0.4 work (device atoms, environment definition, topology visualization) has a validated schema foundation to build on.

## 2. Background

ADR-010 (Environment Archetypes and Adaptive Visualization) introduces three canonical environment archetypes: `modern-segmented`, `legacy-flat`, and `hybrid`. The monitoring dashboard will render topology differently based on archetype (SOW-019.0). The brownfield wastewater environment (SOW-017.0, SOW-018.0) will use era markers and boundary states to tell the facility's 25-year evolution story.

These schema additions must come first because:

- SOW-017.0 (device/network atoms) needs the `installed` field in placements to carry era context
- SOW-018.0 (environment definition) needs `archetype`, `era_span`, and `boundaries` to define the hybrid environment
- SOW-019.0 (topology visualization) needs all of the above to render adaptive layouts

The schema version remains "0.1" because all additions are optional. Per ADR-009 D6, adding optional sections is a non-breaking change. Existing environments validate without modification (though we will add archetype tags for completeness).

### Impact Analysis: Three Modules Parse Environment YAML

| Module | File | Struct | Change Required |
|--------|------|--------|----------------|
| `design/` (validation CLI) | `internal/validate/yaml.go` | `EnvironmentDoc`, `PlacementDoc` | Yes -- add new fields, add validation rules |
| `plant/` (runtime) | `internal/config/config.go` | `EnvMeta`, `Placement` | No -- Go yaml.v3 ignores unknown keys. Plant does not consume archetype/boundary data. |
| `monitoring/` (dashboard) | `internal/dashboard/designlib.go` | `EnvironmentMeta`, `Placement` | Deferred to SOW-019.0 -- dashboard will read these fields for topology rendering. |

The plant binary will continue to work with no code changes. The monitoring dashboard changes are scoped to SOW-019.0 (Adaptive Topology Visualization), not this SOW.

## 4. Requirements

### Functional Requirements

1. FR-001: The `EnvironmentDoc` struct SHALL accept an optional `archetype` field with valid values `modern-segmented`, `legacy-flat`, or `hybrid`.
2. FR-002: The `EnvironmentDoc` struct SHALL accept an optional `era_span` field as a string in format `"YYYY"` or `"YYYY-YYYY"`.
3. FR-003: The `PlacementDoc` struct SHALL accept an optional `installed` field as an integer representing the installation year.
4. FR-004: The `RawDocument` struct SHALL accept an optional `boundaries` list, where each boundary specifies `between` (two network refs), `state` (`enforced`, `intended`, or `absent`), and optional `infrastructure`, `installed`, and `notes` fields.
5. FR-005: `ot-design validate` SHALL produce a warning (not error) if an environment with `archetype: "hybrid"` has no `boundaries` section.
6. FR-006: `ot-design validate` SHALL produce a warning if an environment with `archetype: "modern-segmented"` has any boundary with `state: "absent"`. The warning message SHALL read: `Environment "{id}" is classified "modern-segmented" but has an "absent" boundary between "{A}" and "{B}". Consider reclassifying as "hybrid", or remove the absent boundary entry if it is not intentional.` [OT-REVIEW: CORRECTION-3]
7. FR-007: `ot-design validate` SHALL produce an error if `boundaries[].between` references a network not listed in the environment's `networks` section.
8. FR-008: `ot-design validate` SHALL produce an error if `installed` year on a placement is before 1960 or after `current_year + 2` (computed dynamically via `time.Now().Year() + 2`). [OT-REVIEW: CORRECTION-1, CORRECTION-2 -- PLCs exist from 1968; hardcoded upper bound would go stale; +2 years accommodates staged equipment]
9. FR-009: `ot-design validate` SHALL produce an error if `archetype` is present but not one of the three valid values.
10. FR-010: `ot-design validate` SHALL produce a warning if `era_span` format does not match `"YYYY"` or `"YYYY-YYYY"`.
11. FR-011: `ot-design validate` SHALL produce an error if `era_span` is in `"YYYY-YYYY"` format and the start year is greater than the end year. [OT-REVIEW: IMPROVEMENT-3]
12. FR-012: All existing environments SHALL continue to validate without modification (new fields are optional).
13. FR-013: The `ot-design scaffold --environment` command SHALL include commented-out examples of the new fields with documentation.
14. FR-014: The `infrastructure` field on boundaries SHALL be validated against an enum: `managed-switch`, `firewall`, `ids-sensor`, `vlan-only`, `other`. Warn on `other` to encourage specificity. [OT-REVIEW: IMPROVEMENT-1]
15. FR-015: The `installed` field on placements SHALL be documented as "the year this device was first commissioned in this facility (first run in production)" in both the schema documentation and scaffold comments. [OT-REVIEW: IMPROVEMENT-4]

### Non-Functional Requirements

- NFR-001: No new Go dependencies (all validation uses stdlib + yaml.v3)
- NFR-002: All new validation rules have corresponding unit tests with both passing and failing cases
- NFR-003: Validation error messages include the ADR-010 decision reference where relevant (e.g., "see ADR-010 D1" for archetype values)

## 5. Design

### New/Modified Structs

**`design/internal/validate/yaml.go`** -- EnvironmentDoc extension:

```go
type EnvironmentDoc struct {
    ID          string `yaml:"id"`
    Name        string `yaml:"name"`
    Description string `yaml:"description"`
    Archetype   string `yaml:"archetype"`   // NEW: optional
    EraSpan     string `yaml:"era_span"`    // NEW: optional
}
```

**`design/internal/validate/yaml.go`** -- PlacementDoc extension:

```go
type PlacementDoc struct {
    // ... existing fields unchanged ...
    Installed *int `yaml:"installed"` // NEW: optional, year device was installed
}
```

**`design/internal/validate/yaml.go`** -- New BoundaryDoc struct:

```go
type BoundaryDoc struct {
    Between        []string `yaml:"between"`        // exactly 2 network refs
    State          string   `yaml:"state"`           // enforced | intended | absent
    Infrastructure string   `yaml:"infrastructure"`  // optional enum: managed-switch | firewall | ids-sensor | vlan-only | other
    Installed      *int     `yaml:"installed"`        // optional: year boundary was commissioned
    Notes          string   `yaml:"notes"`            // optional: human context
}
```

**`design/internal/validate/yaml.go`** -- RawDocument extension:

```go
type RawDocument struct {
    // ... existing fields unchanged ...
    Boundaries []BoundaryDoc `yaml:"boundaries"` // NEW: optional
}
```

### New Validation Rules

| Rule ID | Check | Severity | Field |
|---------|-------|----------|-------|
| ENV-020 | `archetype` value must be one of: `modern-segmented`, `legacy-flat`, `hybrid` | Error | `environment.archetype` |
| ENV-021 | `era_span` must match pattern `YYYY` or `YYYY-YYYY` | Warning | `environment.era_span` |
| ENV-021b | `era_span` in range form must have start year <= end year | Error | `environment.era_span` |
| ENV-022 | `installed` must be >= 1960 and <= `time.Now().Year() + 2` | Error | `placements[N].installed` |
| ENV-023 | `boundaries[].between` must have exactly 2 elements | Error | `boundaries[N].between` |
| ENV-024 | `boundaries[].between` elements must reference networks in environment | Error | `boundaries[N].between` |
| ENV-025 | `boundaries[].state` must be `enforced`, `intended`, or `absent` | Error | `boundaries[N].state` |
| ENV-026 | Hybrid archetype without boundaries section | Warning | `boundaries` |
| ENV-027 | Modern-segmented archetype with absent boundary (message guides toward reclassification) | Warning | `boundaries[N].state` |
| ENV-028 | `boundaries[].installed` must be >= 1960 and <= `time.Now().Year() + 2` (if present) | Error | `boundaries[N].installed` |
| ENV-029 | `boundaries[].infrastructure` must be one of: `managed-switch`, `firewall`, `ids-sensor`, `vlan-only`, `other` (if present) | Error | `boundaries[N].infrastructure` |
| ENV-030 | `boundaries[].infrastructure` is `other` | Warning | `boundaries[N].infrastructure` |

### Retroactive Environment Tags

[OT-REVIEW: CORRECTION-5/IMPROVEMENT-5] The `greenfield-water-mfg` environment contains both a Purdue-segmented water treatment side AND a flat 1993-era manufacturing floor connected by a direct cable. This is a `hybrid` environment, not `modern-segmented`. Tagging it `modern-segmented` would cause the topology visualization to render a clean Purdue stack, hiding the manufacturing floor's flat topology from trainees.

| Environment | Archetype | Era Span | Rationale |
|------------|-----------|----------|-----------|
| `greenfield-water-mfg` | `hybrid` | `1993-2024` | Segmented water treatment (2024) + flat manufacturing floor (~1993) + unsegmented cross-plant link |
| `pipeline-monitoring` | `legacy-flat` | `2015` | Intentionally flat remote station topology. The `legacy` label is an imperfect fit for an intentionally small-site design (see TD-036), but acceptable for Beta 0.4. |
| `quickstart-example` | `modern-segmented` | `2024` | Single-network teaching environment. Archetype is a simplification (see TD-037). |

Only `archetype` and `era_span` are added to existing environments. Boundaries and `installed` fields are not retroactively added to existing environments -- they will be authored fresh for the wastewater environment in SOW-018.0 and can be backfilled later if the topology visualization benefits from them.

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| New optional fields cause yaml.v3 parsing issues in plant/monitoring | High | Very Low | Go yaml.v3 ignores unknown keys by default. Verified: plant and monitoring structs use `yaml:` tags; unrecognized fields are silently dropped. |
| Schema version confusion -- users expect "0.2" for new fields | Low | Medium | ADR-009 D6 explicitly states optional additions do not bump the version. Add a comment in the schema docs. |
| Boundary `between` references may conflict with future network renaming | Low | Low | Validation ensures references match current environment networks. Renaming is a manual operation; this is acceptable. |
| `era_span` format is a string, not structured -- harder to validate programmatically | Low | Medium | Warning-only validation for format. The topology visualization (SOW-019.0) will parse the string; if it fails, the UI gracefully falls back to "unknown." |
