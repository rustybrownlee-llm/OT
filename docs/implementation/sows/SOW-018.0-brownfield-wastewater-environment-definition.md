# SOW-018.0 - Brownfield Wastewater Environment Definition

**Status**: Complete
**Created**: 2026-03-02
**Priority**: High
**Dependencies**: SOW-016.0 (Schema Extensions), SOW-017.0 (Device & Network Atoms)
**OT Domain Review**: Complete (7 corrections applied)

## 1. Objective

Compose the brownfield wastewater treatment facility from the atomic device and network YAML created in SOW-017.0. Deliver a complete environment definition with placements, era markers, and boundary states. Add Docker Compose profile and process model implementations so the wastewater environment runs alongside existing water and pipeline environments. Prove that the design layer can model a Frankenstein hybrid environment entirely from atomic YAML without modifying existing plant runtime code. **This is the composability proof for the design layer architecture (ADR-009).**

## 2. Background

SOW-016.0 extended the schema with `archetype`, `era_span`, `installed`, and `boundaries`. SOW-017.0 created the atomic building blocks: 4 network atoms, 1 new device atom (Cradlepoint IBR600), and wastewater-specific register map variants on 3 existing device atoms (SLC-500, CompactLogix, Moxa NPort). This SOW composes those atoms into a running environment.

The composability test: if the `ot-design validate` CLI passes, the Docker Compose profile starts cleanly, the monitoring module discovers all wastewater devices, and anomaly detection fires on rogue writes -- all without modifying the validation CLI, monitoring module, or existing plant environments -- then the atomic design layer architecture is validated.

### What Makes This the Proof

| Composability Dimension | Validation |
|------------------------|------------|
| Atom reuse | SLC-500, CompactLogix, and Moxa atoms used with wastewater-specific variants -- same atoms serve both environments |
| Schema extensibility | `installed`, `boundaries`, `archetype` fields used on wastewater environment without breaking existing environments |
| Plant runtime generality | Plant binary loads wastewater environment.yaml from the same code path as water and pipeline |
| Monitor agnosticism | Monitor discovers wastewater devices identically to water and pipeline -- no environment-specific logic |
| Docker isolation | Wastewater runs as a separate profile with independent networks |

## 4. Requirements

### Functional Requirements

1. FR-001: The environment definition SHALL reference 4 networks: `ww-flat`, `ww-serial-bus`, `ww-level3`, `ww-wan`.
2. FR-002: The environment definition SHALL contain 5 placements:
   - `ww-plc-01`: SLC-500, `ww-influent` variant, serial on `ww-serial-bus`, serial_address 1, gateway `ww-gw-01`, installed 1997
   - `ww-plc-02`: SLC-500, `ww-effluent` variant, serial on `ww-serial-bus`, serial_address 2, gateway `ww-gw-01`, installed 1997
   - `ww-plc-03`: CompactLogix, `ww-aeration` variant, Ethernet on `ww-flat`, IP 192.168.10.30, port 5062, installed 2013
   - `ww-gw-01`: Moxa NPort, `ww-serial-gateway` variant, Ethernet on `ww-flat`, IP 192.168.10.20, port 5063, bridges ww-flat to ww-serial-bus, installed 2008
   - `ww-modem-01`: Cradlepoint IBR600, `vendor-remote-access` variant, Ethernet on `ww-flat`, IP 192.168.10.99, port 5064, additional_networks ww-wan, installed 2022
3. FR-003: The environment definition SHALL set `archetype: "hybrid"` and `era_span: "1997-2022"`.
4. FR-004: The environment definition SHALL include a `boundaries` section with:
   - Between `ww-level3` and `ww-flat`: state `enforced`, infrastructure `managed-switch`, installed 2018, notes about compliance audit
   - A YAML comment explaining why no boundary exists between ww-level3 and ww-serial-bus (serial bus is a physical layer, not a Purdue level)
5. FR-005: The environment definition SHALL pass `ot-design validate` with zero errors. Expected warnings: ENV-026 is NOT expected (boundaries section is present).
6. FR-006: Docker Compose SHALL include a `plant-wastewater` service with profile `wastewater`.
7. FR-007: Docker Compose SHALL define network `ot-ww-flat` with subnet 192.168.10.0/24.
8. FR-008: The `plant-wastewater` container SHALL expose host ports 5062, 5063, 5064 mapped to the same container ports.
9. FR-009: The `monitor` service SHALL join `ot-ww-flat` network when the `wastewater` profile is active.
10. FR-010: Running `docker compose --profile water --profile pipeline --profile wastewater up` SHALL start all three environments with independent monitoring.
11. FR-011: Five process models SHALL be implemented in `plant/internal/process/`:
    - `wastewater_influent.go`: Simulates bar screening, grit removal, primary clarification (1-second tick, 16-bit scaled integers, one-based addressing)
    - `wastewater_effluent.go`: Simulates secondary clarification, RAS/WAS pumping, UV disinfection, effluent discharge (one-based addressing)
    - `wastewater_aeration.go`: Simulates DO control loop with PID-like behavior (blower speed responds to DO setpoint vs measurement), MLSS, SOUR calculation (zero-based addressing)
    - `wastewater_gateway.go`: Simulates Moxa gateway status registers (reuse existing gateway model pattern)
    - `wastewater_cellular.go`: Simulates Cradlepoint management registers (WAN status, signal strength noise, data usage accumulator, uptime counter)
12. FR-012: Process models SHALL use the existing `toRaw()`, `addNoise()`, `ramp()`, `clamp()`, and `drift()` utility functions from `plant/internal/process/`.
13. FR-013: The aeration process model SHALL implement a simplified PID loop: when DO control is enabled and DO setpoint is written, the blower speed setpoint SHALL ramp toward a value that drives measured DO toward the setpoint. Response time SHALL be realistic (minutes, not seconds).
14. FR-014: The monitoring config SHALL include a `wastewater` environment section with endpoints for ports 5062 (ww-plc-03), 5063 (ww-gw-01, gateway mode), and 5064 (ww-modem-01).
15. FR-015: The monitor SHALL discover all wastewater devices and establish baselines without any wastewater-specific code changes to the monitoring module.
16. FR-016: Writing a rogue value to a wastewater PLC register (e.g., coil write via mbpoll to port 5062) SHALL trigger an anomaly alert on the monitoring dashboard, identical in behavior to existing environments.
17. FR-017: Serial devices (ww-plc-01, ww-plc-02) SHALL be accessible via the gateway at port 5063 using unit IDs 1 and 2 respectively.

- NFR-001: All process model ticks SHALL complete in <10ms (consistent with existing NFR-1 in engine.go)
- NFR-002: Process models SHALL follow the exact patterns established in existing models (water.go, manufacturing.go, pipeline_*.go)

  id: "brownfield-wastewater"
  name: "Brownfield Wastewater Treatment Plant"
  description: "Municipal WWTP built 1997 with ad-hoc modernization through 2022. Hybrid architecture: partial Purdue segmentation (Level 3 only), flat Level 1, persistent serial backbone, cellular vendor access."
  archetype: "hybrid"
  era_span: "1997-2022"

networks:
  - ref: "ww-flat"
  - ref: "ww-serial-bus"
  - ref: "ww-level3"
  - ref: "ww-wan"

boundaries:
  - between: ["ww-level3", "ww-flat"]
    state: "enforced"
    infrastructure: "managed-switch"
    installed: 2018

### Process Model Design

All process models follow the `ProcessModel` interface: `Name() string` and `Tick()` called every 1 second.

**InfluentModel** (`wastewater_influent.go`):
- Flow rate: base 2000 GPM with diurnal pattern (±30% over 24-hour cycle, peaks at morning and evening)
- Bar screen dP: accumulates 0.1 in WC per tick while bar_screen_motor coil is off; ramps down toward ~0.3 in WC when motor is on (raking -- mechanical rake clears debris but never reaches zero)
- Grit level: slow accumulation (0.01% per tick), resets when grit_pump coil is on
- Clarifier level: influenced by flow rate, settles toward setpoint when sludge pump cycles
- Sludge blanket: rises 0.001 ft per tick without pumping; drops when primary_sludge_pump is on
- pH: noise around 7.0 with occasional drift (0.1% chance per tick of excursion toward 5.5 or 9.0)
- Temperature: diurnal drift between 50-72 degF (~28800 ticks = 8 hours per half-cycle, representing day/night influent temperature variation)

**EffluentModel** (`wastewater_effluent.go`):
- Effluent flow: tracks influent flow (could be coupled if shared process state exists, or independent with similar diurnal pattern)
- Clarifier level: slow variation around setpoint
- RAS flow: proportional to effluent flow (default ~50%)
- WAS flow: operator setpoint (writable register drives output when WAS pump coil is on)
- TSS: normal ~15 mg/L with noise; rises if WAS is too low (sludge buildup) or too high (biomass washout)
- Ammonia: normal ~5 mg/L; rises if aeration is insufficient (coupled to DO if cross-model state exists, or independent estimate)
- UV transmittance: inverse correlation with TSS (high TSS = low UV transmittance)
- Permit interlock: bitmask computed from UV enable coil + flow + static permit bit

**AerationModel** (`wastewater_aeration.go`):
- **DO PID loop** (simplified): error = do_setpoint - dissolved_oxygen. Blower speed setpoint ramps toward value that minimizes error. Response is slow (blower speed changes by ~1%/tick, DO responds over ~60 ticks = 1 minute).
- DO measurement: influenced by blower speed (more air = more DO transfer) and MLSS (more biomass = more oxygen demand). Steady state at DO setpoint when system is balanced.
- Blower speed feedback: tracks blower speed setpoint with 5-tick lag (VFD response time)
- MLSS: slow drift (0.001%/tick) around 3000 mg/L target; influenced by WAS rate if cross-model state exists
- SOUR: calculated from DO uptake rate / MLSS; ranges 8-15 mg/g/hr in steady state. SOUR SHALL decay toward near-zero when DO < 1.0 mg/L for an extended period (oxygen-limited conditions suppress aerobic uptake). Recovery is slow (~120 ticks) when DO is restored.
- Blower run hours: increments by 1/3600 per tick when blower coil is on (1 hour per 3600 ticks)
- Motor current: proportional to blower speed (0 at 0%, 180A at 100%)
- Basin level: slow variation ±0.5 ft around 12 ft

**WastewaterGatewayModel** (`wastewater_gateway.go`):
- Reuse pattern from existing gateway model (serial port status, message counters, uptime)
- Fixed serial mode RS-485-2wire (enum value 2)
- Active connections: tracks Modbus TCP connections (increment on poll, decrement on timeout)

**CellularModemModel** (`wastewater_cellular.go`):
- WAN status: 2 (connected) steady state, occasional brief drop to 1 (0.01% chance per tick)
- Signal strength: noise around 50 (representing -50 dBm, good LTE) ± 5 units per tick
- Data usage: increments 0.0001 GB per tick (~8.6 GB/day, realistic for periodic SCADA polling + VPN overhead). Clamps at 999 GB (16-bit scale_max). Does not reset automatically (represents monthly billing cycle, would need manual/coil reset).
- Uptime: increments 1 per tick (wraps at 65535)
- Active clients: fixed at 3 (CompactLogix + Moxa + Cradlepoint itself visible on flat network)
- NAT mode: fixed at 1 (enabled)
- VPN tunnel: fixed at 2 (active) -- vendor VPN to cloud

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Process model factory requires changes beyond simple registration | Medium | Low | Examine existing factory pattern before implementation. If factory is variant-keyed, new variants register without code changes to the factory itself. |
| Cross-model process coupling (influent flow → effluent flow, DO → ammonia) adds complexity | Medium | Medium | Implement each model independently first. Cross-coupling is a future enhancement. Independent models with similar statistical patterns are sufficient for monitoring training. |
| Docker network count may approach Docker daemon limits on some systems | Low | Low | Only 1 new Docker network added (ot-ww-flat). Total becomes 7, well within limits. |
| Monitor may need code changes to handle 3+ environments | Medium | Low | Monitor config already supports multiple environments. Verify no hardcoded 2-environment assumptions exist. |
| Wastewater Modbus ports conflict with existing services | High | Very Low | Ports 5062-5064 are explicitly reserved in milestone spec. No conflicts with existing 5020-5043 range. |
