# SOW-019.0 - Adaptive Topology Visualization

**Status**: Approved
**Created**: 2026-03-02
**Priority**: High
**Dependencies**: SOW-016.0 (Schema Extensions), SOW-018.0 (Environment Definition)
**OT Domain Review**: Complete (5 corrections, 5 improvements applied)

## 1. Objective

Add an adaptive topology visualization page to the monitoring dashboard that renders environment architecture based on YAML metadata, not hardcoded layout. The visualization uses environment archetype to select a layout algorithm, boundary states to draw level separators, and era markers to show facility evolution. Engineers can compare the visual shape of all three environments side-by-side, immediately understanding architectural differences without reading text. Per ADR-010 D7, the topology view is server-rendered HTML with CSS-driven styling and HTMX for environment switching -- no JavaScript visualization libraries.

## 2. Background

The monitoring dashboard (port 8090) currently has five page groups: Overview, Assets, Alerts, and Design Library (devices, networks, environments). Per ADR-010 D1-D4 and milestone-b0.4.md, a topology view is the sixth page group that teaches architecture through visual shape.

Three distinct visual shapes correspond to the three archetypes:

| Archetype | Visual Shape | Teaching |
|-----------|-------------|----------|
| `modern-segmented` | Vertical Purdue stack, solid boundaries between levels | What good looks like |
| `legacy-flat` | Single horizontal plane, no boundaries | What flat OT networks look like |
| `hybrid` | Partially collapsed stack, mix of solid/dashed/absent boundaries, era markers | What you'll actually find in the wild |

SOW-016.0 added the schema fields (`archetype`, `era_span`, `installed`, `boundaries`). SOW-018.0 composes the wastewater environment using them. This SOW reads those fields in the monitoring module and renders the topology.

### Technology Constraints

Per CLAUDE.md and ADR-010 D7:
- Server-side rendered HTML (Go templates + chi router)
- CSS-driven visual styling (solid/dashed/absent borders, era marker badges)
- HTMX for switching between environments without full page reload
- Bootstrap 5 for consistent styling (via CDN)
- No JavaScript visualization library (D3, vis.js, etc.)
- No build step

### TD-035 Resolution: Purdue Level Inference (Archetype-Aware)

Network atoms do not carry a `purdue_level` property (TD-035 from SOW-016.0). The topology visualization resolves this by inferring Purdue levels using an archetype-aware heuristic. The key insight is that flat networks in `legacy-flat` and `hybrid` environments have NO Purdue level -- they exist outside the Purdue model. Assigning them to "Level 1" would teach engineers that a flat OT network is just Level 1 without upper levels, when the reality is worse: all Purdue levels coexist on the same wire.

**Inference rules (evaluated in order):**

1. **Archetype pre-filter**: If archetype is `legacy-flat` or `hybrid`, and network ID contains `flat` OR (managed_switch == false AND vlan == 0 AND network type is ethernet), classify as **"Flat (Unclassified)"** -- no Purdue level. Render as a single horizontal plane.
2. **Explicit level**: If network ID contains `level1` → **Level 1** (Process Control). Only valid in `modern-segmented` or `hybrid` environments.
3. **Explicit level**: If network ID contains `level2` → **Level 2** (Area Supervisory).
4. **Explicit level**: If network ID contains `level3` → **Level 3** (Site Operations).
5. **Serial backbone**: If network ID contains `serial` → **"Serial Backbone"** (not a Purdue level). Render attached to its gateway device's level. Serial buses are a physical layer, not an architectural tier.
6. **WAN/External**: If network ID contains `wan` → **"External/WAN"** (not a Purdue level). Render at the top of the diagram as an external connection, distinct from the DMZ concept (Level 3.5 in Purdue). A satellite or cellular WAN link is external connectivity, not a demilitarized zone.
7. **Point-to-point links**: If network has a /30 subnet or connects exactly 2 placements → **"Link"** (render as a connection arrow between the two endpoints, not as a Purdue level containing devices). Applies to networks like `cross-plant`.
8. **Default**: **"Unclassified"** -- render at the bottom of the diagram with a visual flag indicating level assignment is needed.

**Important**: The managed_switch and vlan properties are NOT used for level inference. Managed switches exist at all Purdue levels (wt-level1 has managed_switch=true and vlan=10). These properties describe network capability, not architectural tier.

**Note**: This heuristic is a simulator convention for controlled naming, not a standard. Real environments use function and architecture to determine Purdue levels, not naming conventions. A tooltip on inferred level labels should note: "Level assignments reflect the reference design. Real environments require architectural analysis."

TD-035 remains open for a future `purdue_level` field on network atoms that would make inference unnecessary.

## 4. Requirements

### Functional Requirements

1. FR-001: The dashboard SHALL have a new Topology page accessible at `/topology`.
2. FR-002: The Topology page SHALL display an environment selector listing all environments found in the design library.
3. FR-003: Selecting an environment SHALL load its topology view via HTMX partial update (no full page reload).
4. FR-004: The `/topology/{env-id}` route SHALL render the topology view for a specific environment (direct link support).
5. FR-005: For `modern-segmented` archetype, the topology SHALL render as a vertical stack with one row per Purdue level, solid boundary lines between all levels, and devices grouped within their level.
6. FR-006: For `legacy-flat` archetype, the topology SHALL render as a single horizontal plane with no boundary lines, devices clustered by physical proximity. The layout focal point SHALL be the device with the most network connections (dual-homed device or device with `bridges` defined), not necessarily the serial gateway. For pipeline: `ps-plc-01` (dual-homed on station LAN and WAN) is the focal point. Serial devices cluster around their gateway.
7. FR-007: For `hybrid` archetype, the topology SHALL render as a partially collapsed stack with mixed boundary visualization:
   - `enforced` boundaries: solid line with infrastructure label (e.g., "managed-switch")
   - `intended` boundaries: dashed line
   - `absent` boundaries: wide red dashed line with "NO BOUNDARY" label -- absent boundaries are the dangerous condition and must look more alarming than enforced boundaries, not less
8. FR-008: Each device node SHALL display: device name/role, device atom type (e.g., "SLC-500"), network name, and IP address (for Ethernet devices) or "serial" label (for serial devices).
9. FR-009: Each device node with an `installed` year SHALL display an era marker badge showing the installation year. Era markers SHALL be color-coded by decade (1990s, 2000s, 2010s, 2020s) to make era mixing visually apparent.
10. FR-010: The topology view SHALL show the environment's `archetype` and `era_span` in the header area.
11. FR-011: Gateway devices that bridge networks SHALL show a visual connection (line or arrow) between the bridged networks.
12. FR-012: Devices with `additional_networks` SHALL show secondary network attachment indicators.
13. FR-013: The Purdue level of each network SHALL be inferred using the heuristic described in section 2 (TD-035 resolution).
14. FR-014: The topology view SHALL degrade gracefully if archetype/boundaries/installed fields are missing (environments from before SOW-016.0 still render).
15. FR-015: The navigation bar SHALL include a "Topology" link between "Alerts" and "Design Library".
16. FR-016: For environments with no archetype field, the default rendering SHALL be `legacy-flat` (single plane, no boundaries).

### Non-Functional Requirements

- NFR-001: No new Go dependencies (HTML templates + CSS only)
- NFR-002: Topology rendering SHALL complete in <50ms for environments with up to 20 devices
- NFR-003: All new code SHALL have unit tests
- NFR-004: CSS SHALL use Bootstrap 5 utility classes where possible, with custom CSS only for topology-specific styling (boundary lines, era badges, device nodes)
- NFR-005: Templates SHALL follow the existing pattern: base templates parsed once, page templates cloned per-render

### Topology Data Model

```go
type TopologyData struct {
    EnvID       string
    EnvName     string
    Archetype   string  // "modern-segmented", "legacy-flat", "hybrid"
    EraSpan     string
    Levels      []TopologyLevel
    Boundaries  []TopologyBoundary
    Bridges     []TopologyBridge
    AllEnvs     []EnvSummary // for environment selector
}

type TopologyLevel struct {
    Name     string           // "Level 3", "Level 1 (Flat)", etc.
    PLevel   int              // 0, 1, 2, 3 (for ordering)
    Networks []TopologyNetwork
    Devices  []TopologyDevice
}

type TopologyNetwork struct {
    ID      string
    Name    string
    Type    string // ethernet, serial-rs485
    Subnet  string
    Managed bool
    SPAN    bool
}

type TopologyDevice struct {
    ID          string
    DeviceType  string // "SLC-500", "CompactLogix", etc.
    Role        string
    IP          string // empty for serial
    Port        int
    Serial      bool
    SerialAddr  int
    GatewayID   string
    Installed   *int
    EraClass    string // "era-1990s", "era-2000s", "era-2010s", "era-2020s"
}

type TopologyBoundary struct {
    FromLevel      string
    ToLevel        string
    State          string // enforced, intended, absent
    Infrastructure string
    Installed      *int
    CSSClass       string // "boundary-enforced", "boundary-intended", "boundary-absent"
}

type TopologyBridge struct {
    DeviceID     string
    FromNetwork  string
    ToNetwork    string
}
```

### Layout Algorithm

```
buildTopologyData(envDef, designLib):
  1. Determine archetype (default: "legacy-flat" if missing)
  2. For each network in environment:
     a. Infer Purdue level using heuristic
     b. Create TopologyNetwork
  3. Group networks by level
  4. For each placement:
     a. Create TopologyDevice
     b. Compute era class from installed year
     c. Assign to level based on primary network
  5. For each boundary in environment:
     a. Resolve which levels the boundary spans
     b. Create TopologyBoundary with CSS class
  6. For each bridge in placements:
     a. Create TopologyBridge linking networks
  7. Sort levels: Level 3 → Level 2 → Level 1 → Level 0
  8. Return TopologyData
```

**topology.html** (page):
- Environment selector (Bootstrap dropdown or card list)

**partials/topology_view.html** (HTMX partial):

### CSS Design

```css
/* Boundary states */
.boundary-enforced { border-top: 3px solid #28a745; } /* green solid */
.boundary-intended { border-top: 3px dashed #ffc107; } /* yellow dashed */
.boundary-absent { border-top: 3px dashed #dc3545; } /* red dashed -- dangerous: missing boundary */

/* Era marker badges -- neutral hue scale (light→dark blue).
   Communicates age without implying risk ranking. A 1990s device on an
   isolated serial bus may be safer than a 2020s device with internet exposure.
   Per ADR-010 D3: "A 1998 SLC-500 that has run reliably for 28 years is not
   a failure -- it's a constraint." */
.era-badge { font-size: 0.75rem; padding: 2px 6px; border-radius: 3px; }
.era-1990s { background-color: #b3d9ff; color: #0d3b66; } /* lightest blue: oldest */
.era-2000s { background-color: #66b3ff; color: #0d3b66; } /* light blue */
.era-2010s { background-color: #3399ff; color: white; }   /* medium blue */
.era-2020s { background-color: #0066cc; color: white; }   /* dark blue: newest */

/* Device nodes */
.device-node { border: 1px solid #dee2e6; border-radius: 8px; padding: 12px; }
.device-serial { border-style: dashed; } /* serial devices have dashed border */

```

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| CSS-only topology may not clearly distinguish archetypes | Medium | Medium | Use strong visual contrasts (vertical vs horizontal layout, color-coded boundaries). User testing with prototype. |
| Purdue level heuristic may fail for future environments with non-standard naming | Low | Medium | Heuristic is documented and extensible. TD-035 remains open for proper purdue_level field. |
| Large environments (15+ devices) may overflow the viewport | Low | Low | Current environments have 5-7 devices. Future scrolling/grouping can be added (TD-028 scope). |
| HTMX partial update may cause layout shift | Low | Low | Fixed-height topology container. Loading indicator during fetch. |
