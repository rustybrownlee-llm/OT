# SOW-020.0 - Hybrid Environment Scenarios

**Status**: Approved
**Created**: 2026-03-02
**Priority**: Medium
**Dependencies**: SOW-018.0 (Environment Definition), SOW-019.0 (Topology Visualization)
**OT Domain Review**: Complete (5 corrections, 5 improvements applied)

## 1. Objective

Create scenario content for the brownfield wastewater environment that teaches engineers how to discover, assess, and understand hybrid OT architectures. Extend Scenario 01 (Discovery) with a wastewater variant highlighting hybrid-specific challenges. Create a new section in Scenario 02 (Assessment) focused on partial-segmentation blind spots. All content is documentation-only (ADR-007 D4: no code, no scripts) and references the `brownfield-wastewater` environment definition.

## 2. Background

Scenarios 01 and 02 currently reference the `greenfield-water-mfg` environment (Beta 0.3, SOW-015.0). The wastewater environment introduces teaching challenges that don't exist in the existing environments:

| Challenge | Water/Mfg | Pipeline | Wastewater (NEW) |
|-----------|-----------|----------|-----------------|
| Era mixing | Single build (2024) | Single build (2015) | 5 eras (1997-2022) |
| Partial segmentation | Full Purdue | None | Level 3 only (false confidence) |
| Internet exposure | None | Controlled WAN (satellite backhaul to SCADA master only, no general internet) | Uncontrolled cellular modem on flat network (default creds, NAT, internet-facing) |
| Serial backbone | Modbus RTU via gateway (standard tools work) | Modbus RTU via gateway (standard tools work) | DH-485 proprietary protocol (no standard monitoring tools, ProSoft translation required, cannot retire without plant shutdown) |
| Monitoring blind spots | Minimal (SPAN everywhere) | Flat (no SPAN) | Mixed (SPAN on L3 only) |
| Unauthorized additions | None | None | Cradlepoint "temporary" modem |

These challenges require scenario content that guides engineers through discovering the hybrid architecture, understanding why it looks the way it does, and assessing the security implications of partial modernization.

## 5. Design

### Scenario 01 Phase F: Hybrid Environment Discovery

**Prerequisite**: Phases A-E completed, `docker compose --profile wastewater up` running, monitoring dashboard available.

Steps:
- F1: Start wastewater environment profile
- F2: Scan wastewater simulator ports with `nmap -sV -p 5062-5064 localhost` -- note 3 open Modbus TCP ports found (CompactLogix at 5062, Moxa gateway at 5063, Cradlepoint at 5064). Note: in the real facility, you would scan 192.168.10.0/24 on port 502; the simulator maps each device to a unique host port per the port assignment table.
- F3: Use mbpoll to read CompactLogix aeration registers at port 5062 (zero-based addressing): `mbpoll -a 1 -t 3 -r 0 -c 12 localhost -p 5062`. Note the blower_run_hours register at address 11 -- this is the data the Cradlepoint was installed to read.
- F4: Note the Cradlepoint at port 5064 -- "what is this device doing on an OT network?" moment. Use mbpoll to read its management registers: `mbpoll -a 1 -t 3 -r 0 -c 7 localhost -p 5064`. Discover WAN connected, signal strength, data usage. Note: these Modbus TCP registers are an educational abstraction (TD-038); the real Cradlepoint uses HTTPS/SNMP management.
- F5: Use mbpoll to read through the Moxa gateway (port 5063) -- discover SLC-500s at unit IDs 1 and 2. **Critical**: SLC-500s use one-based addressing (via ProSoft MVI46-MCM module), while the CompactLogix at port 5062 used zero-based. Start registers at address 1, not 0: `mbpoll -a 1 -t 3 -r 1 -c 9 localhost -p 5063` (unit 1: influent), `mbpoll -a 2 -t 3 -r 1 -c 8 localhost -p 5063` (unit 2: effluent). These devices are invisible to nmap -- they exist on a DH-485 serial bus behind the gateway.
- F6: Compare manual discovery to monitoring dashboard asset inventory -- dashboard found all 5 devices
- F7: Open topology view and identify the hybrid architecture: partial Level 3 segmentation, flat Level 1, serial backbone
- F8: Compare wastewater topology to water treatment and pipeline topologies side-by-side
- F9: Note the era markers: devices installed across 25 years (1997-2022)
- F10: Document the discovery in a facility network map

Teaching points:
- IP scan finds 3 open Modbus TCP ports (5062, 5063, 5064); 2 devices are invisible (serial behind gateway on DH-485 bus)
- Three Ethernet endpoints respond: the CompactLogix (process data at 5062), the Moxa gateway (infrastructure status registers at 5063, not process data), and the Cradlepoint (management registers at 5064). The Moxa responds with gateway port state, not process data -- the SLC-500 process data lives behind it at unit IDs 1 and 2.
- The monitoring dashboard found all 5 because it's configured to scan gateway unit IDs (scan_unit_ids in monitor.yaml)
- The Cradlepoint's presence is a surprise -- it wasn't in any facility documentation
- The CompactLogix uses zero-based addressing while the SLC-500s behind the gateway use one-based -- the one-based/zero-based contrast is one of the most valuable teaching moments in the hybrid environment
- The topology view immediately shows the architectural shape: this is not a clean Purdue stack
- Era markers tell the story of why the architecture looks the way it does: the 1997 SLC-500 was designed before commercial internet was widespread; the 2022 Cradlepoint assumes 4G/LTE connectivity. Placing them on the same flat network creates a threat model collision -- the SLC-500 has no defenses against internet-connected adversaries because such adversaries did not exist in its design era.

### Scenario 02 Phase E: Hybrid Architecture Assessment

**Prerequisite**: Phases A-D completed for water/mfg environment. Wastewater environment running and discovered.

Steps:
- E1: Analyze the 2018 compliance audit response -- what was actually segmented? (Only Level 3 SCADA server)
- E2: Map what an attacker on the flat segment can reach: all 5 simulated devices are reachable (3 directly on Ethernet, 2 via gateway). In the real facility, a 6th device -- the SCADA server -- would be protected by the Level 3 boundary, but it is not modeled in this simulator (TD-030). Frame this: the 2018 audit's only achievement (Level 3 boundary) protects a device that isn't even in scope.
- E3: Assess the Cradlepoint: the HTTPS management interface (port 443) uses default credentials (admin/admin) that have never been changed. NAT enabled, internet-facing. **Teaching point**: the simulated Modbus TCP registers at port 5064 require no authentication at all (per Modbus protocol design -- there is no auth mechanism). The real default-credential risk is the web management plane, not the Modbus data plane. Engineers must understand where authentication exists in OT devices and where it does not.
- E4: Assess the cellular WAN as a monitoring blind spot -- no SPAN, no packet capture on cellular traffic
- E5: Perform era-based vulnerability analysis: which devices can receive firmware updates? (CompactLogix yes -- but requires Rockwell Studio 5000, trained technician, planned shutdown, and ladder logic re-validation; SLC-500 no -- firmware is factory-set; Cradlepoint yes -- but no one is managing it, last updated at installation). **Teaching point**: "firmware-updateable" does not mean "easily patched" -- this is a common misconception IT engineers carry from IT infrastructure where automated patching is standard.
- E6: Assess the two-layer serial architecture: The DH-485 bus carries native Allen-Bradley traffic that no IP tool can reach. The ProSoft MVI46-MCM modules in each SLC-500 chassis expose a separate RS-485 port for Modbus RTU, which the Moxa NPort bridges to TCP. Monitoring the Moxa's serial_rx_count and serial_error_count registers shows Modbus traffic is transiting the gateway, but it cannot reveal what is happening on the DH-485 bus itself. Replacing the serial infrastructure requires a plant shutdown because DH-485 wiring is physically part of the SLC-500 chassis backplane.
- E7: Use the topology view to identify the "false confidence" pattern: Level 3 boundary exists but protects only the (unmodeled) SCADA server -- none of the 5 simulated operational devices are protected
- E8: Document findings in the vulnerability report template (extend from Phase D)
