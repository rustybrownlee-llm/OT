# SOW-021.0 - Process Schematic Schema and Validation

**Status**: Approved
**Created**: 2026-03-02
**Priority**: High
**Dependencies**: None (first SOW in Beta 0.5)

## 1. Objective

Define the `process.yaml` schema for physical process schematics and extend the `ot-design validate` CLI to validate process files. This establishes the configuration layer foundation that all downstream Beta 0.5 SOWs depend on -- the SVG rendering engine (SOW-022.0) reads this schema, and the schematic authors (SOW-023.0, SOW-024.0) write to it.

## 2. Background

The design layer currently defines three artifact types: device atoms, network atoms, and environment templates. Beta 0.5 introduces a fourth: **process schematics**. A process schematic maps physical plant equipment (tanks, pumps, valves, sensors) to Modbus register addresses on specific PLC placements, using ISA-5.1 instrument tagging conventions.

The existing validation framework (SOW-006.0, extended by SOW-016.0) follows a consistent pattern: YAML structs in `yaml.go`, type-specific validator functions (`ValidateDevice`, `ValidateNetwork`, `ValidateEnvironment`), auto-detection via `DetectFileType`, and scaffold templates. This SOW adds `ValidateProcess` following the identical pattern.

Process schematics live alongside environment templates: `design/environments/<env-id>/process.yaml`. They cross-reference placement IDs from the companion `environment.yaml` in the same directory, and register addresses from the device atoms those placements reference. This is analogous to how environment validation already cross-references device and network atoms.

Resolved design decisions RD-1 through RD-4 from the milestone spec are codified in this schema.

## 4. Requirements

### Functional Requirements

1. `ot-design validate` SHALL auto-detect files with a top-level `process:` key and route them to `ValidateProcess()`
2. `ot-design validate design/environments/<env-id>/` SHALL validate both `environment.yaml` and `process.yaml` (if present) when validating a directory
3. Process validation SHALL cross-reference placement IDs against the companion `environment.yaml`
4. Process validation SHALL cross-reference register addresses against device atom register capabilities (max_holding, max_coils)
5. `ot-design scaffold --process` SHALL emit a complete skeleton process.yaml with inline documentation
6. All validation findings SHALL use the `PRO-NNN` rule ID namespace
7. ISA-5.1 instrument tag prefixes SHALL be validated against a known set split into two categories: **instrument prefixes** (LT, FT, PT, TT, AT, AIT, PDT, ST, VT, BT, IT, HS, ZS, ZSO, ZSC, FE) and **equipment designators** (P, V, B, C, SV, GC, FM). The project-defined extension `run` (binary coil run-status) SHALL be accepted with a comment in the scaffold template that it is NOT standard ISA-5.1.
8. The `flow_direction` field SHALL accept `horizontal` or `vertical` (default `horizontal` if omitted)
9. The `network_context` section SHALL validate placement references and `type` enum values
10. Instrument tags SHALL be unique within a process schematic

### Non-Functional Requirements

- All new code follows existing validation framework patterns exactly
- Test coverage for every PRO rule (positive and negative cases)
- Existing device, network, and environment validation unchanged
- All existing tests continue to pass

## 5. Design

### YAML Struct Types (additions to yaml.go)

```go
// ProcessDoc represents the process: section of a process schematic YAML.
type ProcessDoc struct {
    ID             string `yaml:"id"`
    Name           string `yaml:"name"`
    Description    string `yaml:"description"`
    FlowDirection  string `yaml:"flow_direction"` // "horizontal" (default) or "vertical"
}

// StageDoc represents a stage in the process flow.
type StageDoc struct {
    ID         string         `yaml:"id"`
    Name       string         `yaml:"name"`
    Controller *ControllerDoc `yaml:"controller"`
    Equipment  []EquipmentDoc `yaml:"equipment"`
}

// ControllerDoc optionally identifies the primary PLC for a stage.
type ControllerDoc struct {
    Placement string `yaml:"placement"`
    Device    string `yaml:"device"`
}

// EquipmentDoc represents a physical piece of equipment.
// Valid types: tank, basin, pump, valve, blower, analyzer_station, chromatograph,
// uv_system, clarifier, screen, compressor, meter, heat_exchanger.
// Note: "gateway" is NOT valid here -- gateways belong in NetworkContext per RD-1.
// "basin" is distinct from "tank" (aeration basins, equalization basins use OT-specific vocabulary).
type EquipmentDoc struct {
    ID          string          `yaml:"id"`
    Type        string          `yaml:"type"`
    Label       string          `yaml:"label"`
    Era         *int            `yaml:"era"`
    Instruments []InstrumentDoc `yaml:"instruments"`
}

// InstrumentDoc represents a sensor or actuator mapped to a Modbus register.
type InstrumentDoc struct {
    Tag       string      `yaml:"tag"`
    Name      string      `yaml:"name"`
    ISAType   string      `yaml:"isa_type"`
    Placement string      `yaml:"placement"`
    Register  RegisterRef `yaml:"register"`
    Unit      string      `yaml:"unit"`
    Range     []float64   `yaml:"range"`
    Scale     []float64   `yaml:"scale"`
    Thresholds *ThresholdDoc `yaml:"thresholds"`
}

// RegisterRef identifies a specific Modbus register on a device.
type RegisterRef struct {
    Type    string `yaml:"type"`    // "holding" or "coil"
    Address int    `yaml:"address"`
}

// ThresholdDoc defines optional alarm thresholds for color coding.
// Supports bidirectional thresholds: high (warning, alarm) and low (warning_low, alarm_low).
// Level instruments (tanks, basins, clarifiers) need low thresholds for pump cavitation
// and overflow protection. All fields are optional.
type ThresholdDoc struct {
    Warning    *float64 `yaml:"warning"`      // High warning threshold
    Alarm      *float64 `yaml:"alarm"`         // High alarm threshold
    WarningLow *float64 `yaml:"warning_low"`   // Low warning threshold
    AlarmLow   *float64 `yaml:"alarm_low"`     // Low alarm threshold
}

// ConnectionDoc represents a physical process connection between equipment.
// Only physical process connections belong here: "pipe" (fluid/gas) and "duct" (air delivery).
// Serial bus and wireless/WAN links belong on the topology page (RD-1) or in NetworkContext (RD-4).
type ConnectionDoc struct {
    From  string `yaml:"from"`
    To    string `yaml:"to"`
    Type  string `yaml:"type"`  // "pipe" or "duct"
    Label string `yaml:"label"`
}

// NetworkContextDoc represents a security-relevant network element overlay (RD-4).
type NetworkContextDoc struct {
    ID        string `yaml:"id"`
    Type      string `yaml:"type"`      // "wan_link", "internet_gateway", "wireless_bridge"
    Label     string `yaml:"label"`
    Era       *int   `yaml:"era"`
    Placement string `yaml:"placement"`
    Warning   string `yaml:"warning"`
    Notes     string `yaml:"notes"`
}
```

### RawDocument Extensions

```go
// Add to RawDocument struct:
Process        *ProcessDoc        `yaml:"process"`
Stages         []StageDoc         `yaml:"stages"`
Connections    []ConnectionDoc    `yaml:"connections"`
NetworkContext []NetworkContextDoc `yaml:"network_context"`
```

### Validation Rules

| Rule ID | Severity | Field | Check |
|---------|----------|-------|-------|
| PRO-001 | Error | schema_version | Must be "0.1" |
| PRO-002 | Error | process.id | Required, non-empty |
| PRO-003 | Error | process.name | Required, non-empty |
| PRO-004 | Error | process.flow_direction | Must be "horizontal", "vertical", or empty (defaults to "horizontal") |
| PRO-005 | Error | stages | At least one stage required |
| PRO-006 | Error | stages[].id | Required, unique across all stages |
| PRO-007 | Error | stages[].equipment[].id | Required, unique across entire process |
| PRO-008 | Error | stages[].equipment[].type | Must be valid enum (tank, basin, pump, valve, blower, analyzer_station, chromatograph, uv_system, clarifier, screen, compressor, meter, heat_exchanger). Note: `gateway` is NOT a valid equipment type in stages -- gateways belong in `network_context` per RD-1. `basin` is distinct from `tank` (aeration basins, equalization basins). |
| PRO-009 | Error | instruments[].tag | Required, unique across entire process |
| PRO-010 | Warning | instruments[].isa_type | Should be a recognized prefix. **Instrument prefixes** (ISA-5.1): LT, FT, PT, TT, AT, AIT, PDT, ST, VT, BT, IT, HS, ZS, ZSO, ZSC, FE. **Equipment designators**: P, V, B, C, SV, GC, FM. **Project extension** (not ISA-5.1): `run` (binary coil run-status). Note: `DPT` is non-standard (use `PDT`); `UV` is an equipment type not an instrument prefix (use `uv_system` in equipment type). |
| PRO-011 | Error | instruments[].placement | Must reference a valid placement ID in companion environment.yaml |
| PRO-012 | Error | instruments[].register.type | Must be "holding" or "coil" |
| PRO-013 | Error | instruments[].register.address | Must be >= 0 and within device atom register capacity for the referenced placement's device |
| PRO-014 | Error | connections[].from, connections[].to | Must reference valid equipment IDs within this process |
| PRO-015 | Error | connections[].type | Must be "pipe" or "duct". Note: `serial` and `wireless` removed per OT domain review -- serial bus topology and WAN links belong on the topology page (RD-1) or in `network_context` (RD-4), not in physical process connections. `duct` added for aeration air delivery (blower to basin diffusers). |
| PRO-016 | Warning | network_context[].placement | If present, must reference a valid placement in companion environment.yaml |
| PRO-017 | Error | network_context[].type | Must be "wan_link", "internet_gateway", or "wireless_bridge" |
| PRO-018 | Warning | equipment[].era | If present, must be >= 1970 and <= current year |
| PRO-019 | Warning | instruments[].tag vs instruments[].isa_type | If tag prefix is a recognized ISA prefix and isa_type is also set, they should match (e.g., tag "FT-101" should have isa_type "FT", not "LT"). Teaches P&ID tag/type alignment discipline. |
| PRO-020 | Warning | instruments[].scale vs device atom scale_min/scale_max | If both process.yaml scale and device atom scale_min/scale_max are defined for the same register, warn if they differ. A mismatched scale produces wrong engineering unit conversions. |

### Directory Validation Update

When `ot-design validate` receives a directory path, it currently checks for `environment.yaml`. The update:

1. Validate `environment.yaml` as before (required)
2. Check for `process.yaml` in the same directory
3. If `process.yaml` exists, validate it with `ValidateProcess()`, passing the already-loaded environment data for cross-reference resolution
4. Merge results from both validations

### Cross-Reference Resolution

`ValidateProcess()` requires access to:

1. **Companion environment.yaml** -- to resolve placement IDs (PRO-011, PRO-016)
2. **Device atoms** -- to check register address bounds (PRO-013). The device atom file is found via the placement's `device` field → `design/devices/<device>.yaml`
3. **Design root** -- found by walking up from the process file's directory (same `FindDesignRoot` pattern)

This mirrors how `ValidateEnvironment()` already resolves device and network atom cross-references.

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Process schema may need revision during SOW-023.0 (actual schematic authoring) | Low | Medium | Schema is deliberately extensible; new optional fields can be added without breaking validation. SOW-021.0 defines the skeleton; SOW-023.0 may surface additional equipment types or instrument fields. |
| Cross-reference resolution performance for large environments | Low | Low | Validation loads all files once and indexes placement IDs in a map; same pattern as existing environment validation. |
| ISA-5.1 prefix list may be incomplete | Low | Medium | Unrecognized prefixes generate a Warning, not an Error (PRO-010). The list can be extended without a schema change. |

|----|-------------|-----------------|
| TD-038 | Process validation does not check register address uniqueness per placement (two instruments mapping to the same register on the same PLC) | May be intentional (read-only instruments sharing a register, or bitmask registers where multiple instruments map to individual bits). Evaluate during SOW-023.0 authoring. |
| TD-039 | No schema migration tooling (process.yaml schema is v0.1 only) | Address if/when schema_version 0.2 is needed |
| TD-040 | No bitmask support in register references | Status word registers (compressor_status_word) pack multiple alarms into individual bits. Current schema maps one instrument to one register address with no bit-mask field. Will surface in pipeline process schematic (SOW-024.0). |
