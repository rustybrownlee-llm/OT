# SOW-022.0 - SVG Symbol Library and Layout Engine

**Status**: Approved
**Created**: 2026-03-02
**Priority**: High
**Dependencies**: SOW-021.0 (Process Schematic Schema and Validation)

## 1. Objective

Build the server-side SVG rendering infrastructure for the process view: a symbol library of reusable equipment templates (tanks, pumps, valves, sensors, etc.), a layout engine that computes SVG coordinates from process.yaml declarations, and the HTMX partial update pattern for live register value display. This SOW delivers the rendering capability; SOW-025.0 wires it into the dashboard.

## 2. Background

The monitoring dashboard renders all content server-side using Go `html/template` and HTMX for dynamic updates. The topology view (SOW-019.0) demonstrated the pattern: a Go layout engine reads YAML metadata, computes display-ready data structures, and passes them to templates that render HTML with CSS classes. The topology view uses CSS flex/grid for layout.

The process view requires a different approach: SVG with computed coordinates. Physical process schematics have connecting pipes, spatial equipment placement, and instrument callout lines that CSS flex cannot express. SVG is native browser rendering, accessible, print-friendly, and can be generated entirely by Go templates without any JavaScript library.

The key HTMX innovation for the process view: instrument values are rendered as `<text>` elements with stable `id` attributes. A polling partial endpoint returns only the updated `<text>` elements using HTMX `hx-swap-oob`, replacing individual values without re-rendering the entire SVG. This minimizes bandwidth and avoids visual flicker.

## 3. Scope

### In Scope

- SVG symbol template partials for all 13 equipment types defined in SOW-021.0
- Go layout engine that reads `ProcessDoc` and computes x,y positions for all elements
- Connection path routing between equipment (SVG `<path>` with flow arrows)
- Instrument value `<text>` elements with stable IDs for HTMX updates
- Network context callout box rendering (RD-4)
- Era marker badge rendering on equipment
- Color-coded status indicators (pump running/stopped, valve open/closed)
- Threshold-based value coloring (green/yellow/red, bidirectional per SOW-021.0)
- `viewBox`-based responsive scaling (minimum readable at 1024px)
- HTMX partial endpoint pattern for live value updates
- Unit tests for layout engine coordinate computation

### Out of Scope

- Authoring actual process.yaml files (SOW-023.0, SOW-024.0)
- Dashboard route registration and page templates (SOW-025.0)
- Vertical flow direction layout (horizontal only in this SOW per RD-2 implementation sequence)
- Interactive features (drag, zoom, pan)
- Animated fluid flow
- Print stylesheet

## 4. Requirements

### Functional Requirements

1. The layout engine SHALL accept a `ProcessDoc` (from SOW-021.0) and produce a `ProcessViewData` struct containing positioned symbols, connections, and instrument value slots
2. Each equipment type SHALL have a corresponding SVG template partial that renders at a given (x, y) position with configurable size
3. Connections between equipment SHALL render as SVG `<path>` elements with arrowhead markers indicating flow direction
4. Instrument values SHALL render as `<text>` elements with `id="inst-{tag}"` attributes for HTMX OOB swap targeting
5. Pump and valve status SHALL render with color-coded fill (green = active/open, grey = inactive/closed, red = alarm/trip)
6. Instrument values SHALL color-code based on threshold ranges: green (normal), yellow (warning), red (alarm), with bidirectional support per ThresholdDoc
7. Era markers SHALL render as small badge elements on equipment (year in colored pill, using decade-based color scale from topology view)
8. Network context elements SHALL render in a visually distinct callout box (dashed border, separate from process flow SVG)
9. The SVG SHALL use a `viewBox` attribute enabling responsive scaling to container width
10. The HTMX partial update response SHALL return only updated `<text>` and `<circle>` elements using `hx-swap-oob="true"` attributes, not the full SVG

### Non-Functional Requirements

- All SVG generation is server-side Go templates; no JavaScript rendering
- SVG output is valid SVG 1.1 that renders in Chrome, Firefox, Safari, and Edge
- Layout engine is deterministic: same input always produces same coordinates
- Symbol library follows existing template embedding pattern (`go:embed`)
- Layout computation is separated from template rendering (testable independently)

### ProcessViewData Structure

```go
// ProcessViewData is the top-level data passed to SVG templates.
type ProcessViewData struct {
    EnvID          string
    EnvName        string
    FlowDirection  string           // "horizontal" (vertical deferred)
    ViewBox        ViewBox          // SVG viewBox dimensions
    Stages         []StageLayout    // Positioned stage groups
    Connections    []ConnectionPath // SVG path data for pipes/ducts
    NetworkContext *NetworkContextLayout // Optional callout box (RD-4)
    Defs           SVGDefs          // Reusable SVG definitions (arrowheads, gradients)
}

// ViewBox defines the SVG coordinate space.
type ViewBox struct {
    X, Y, Width, Height float64
}

// StageLayout is a positioned stage group.
type StageLayout struct {
    ID          string
    Name        string
    Controller  string           // PLC placement label
    X, Y        float64         // Top-left of stage bounding box
    Width       float64
    Height      float64
    Equipment   []EquipmentLayout
}

// EquipmentLayout is a positioned equipment symbol.
type EquipmentLayout struct {
    ID          string
    Type        string           // Symbol type for template dispatch
    Label       string
    X, Y        float64         // Center position
    Width       float64
    Height      float64
    Era         *int             // For era badge
    EraClass    string           // CSS class (era-1990s, era-2000s, etc.)
    Instruments []InstrumentLayout
}

// InstrumentLayout is a positioned instrument value display.
type InstrumentLayout struct {
    Tag         string           // ISA-5.1 tag (unique ID for HTMX targeting)
    Name        string
    ISAType     string
    Placement   string           // PLC placement ID (for API fetch)
    RegisterType string          // "holding" or "coil"
    RegisterAddr int
    Unit        string
    Range       [2]float64       // Engineering unit range
    Scale       [2]float64       // Raw uint16 range
    X, Y        float64         // Position for value text
    // Live data (populated during render, not layout)
    Value       string           // Formatted engineering value
    RawValue    uint16
    CoilState   *bool            // For coil registers
    StatusColor string           // "green", "yellow", "red", "grey"
}

// ConnectionPath is an SVG path between two equipment items.
type ConnectionPath struct {
    FromID    string
    ToID      string
    Type      string             // "pipe" or "duct"
    Label     string
    PathData  string             // SVG path d="" attribute
    ArrowEnd  bool              // Show arrowhead at end
}

// NetworkContextLayout is a positioned callout box for security-relevant network elements.
type NetworkContextLayout struct {
    X, Y        float64
    Width       float64
    Height      float64
    Items       []NetworkContextItem
}

type NetworkContextItem struct {
    ID      string
    Label   string
    Type    string
    Era     *int
    Warning string
    Notes   string
}
```

### Layout Algorithm (Horizontal)

```
Stage spacing: 280px between stage left edges
Equipment spacing: 120px vertical between equipment centers
Instrument offset: 150px right of equipment center (callout line)
Stage padding: 20px internal

For each stage (left to right):
  stageX = stageIndex * 280 + padding
  For each equipment (top to bottom):
    equipX = stageX + stageWidth/2
    equipY = equipIndex * 120 + headerHeight + padding
    For each instrument:
      instX = equipX + equipWidth/2 + 20  (callout offset)
      instY = equipY + instrIndex * 18     (stacked text lines)

Instrument callout lines:
  Leader line from equipment edge to instrument value text
  Small circle (r=3) at equipment connection point
  Horizontal line to text position
  Style: thin grey stroke, non-intrusive

Connection routing:
  Simple horizontal paths between stage boundaries
  From equipment center-right to next stage equipment center-left
  SVG path: M fromX,fromY L toX,toY (straight line with arrowhead)
  Arrowheads at pipe endpoints; for pipes > 300px, add mid-pipe arrow
  Future: orthogonal routing for complex layouts

ViewBox computed from total content bounding box + 40px margin on all sides
```

### SVG Symbol Library

Each symbol is a Go template partial that accepts an `EquipmentLayout` struct:

| Symbol | Template Name | SVG Elements | Size |
|--------|--------------|--------------|------|
| `tank` | `svg_tank` | `<rect>` with rounded corners + wave fill indicator | 80x100 |
| `basin` | `svg_basin` | Wider `<rect>` with wave fill, distinct from tank | 120x80 |
| `pump` | `svg_pump` | `<circle>` with triangle impeller (oriented to show flow direction) + status fill | 50x50 |
| `valve` | `svg_valve` | Two triangles (bowtie) + status fill | 40x30 |
| `blower` | `svg_blower` | `<circle>` with fan blade paths + status fill | 50x50 |
| `compressor` | `svg_compressor` | `<rect>` with turbine icon + status fill | 80x60 |
| `clarifier` | `svg_clarifier` | Circular `<circle>` with horizontal settling layer line at ~75% height | 80x80 |
| `screen` | `svg_screen` | Vertical `<rect>` with grid lines | 40x80 |
| `uv_system` | `svg_uv_system` | `<rect>` with UV lamp icon (zigzag) | 60x40 |
| `meter` | `svg_meter` | `<circle>` with hourglass/orifice shape per ISO 1219 | 50x50 |
| `chromatograph` | `svg_chromatograph` | `<rect>` with "GC" label | 60x40 |
| `analyzer_station` | `svg_analyzer_station` | `<rect>` with 3-4 probe circles indicating multi-parameter sensing | 60x60 |
| `heat_exchanger` | `svg_heat_exchanger` | `<rect>` with parallel horizontal lines (tube bundle per ISO 1219) | 60x50 |

**OT convention note**: Status fill colors (green/grey/red) on pump, valve, blower, and compressor symbols are a **simulator convention** chosen for clarity with IT engineers. Real P&IDs use separate status indicator glyphs or position indicators (ISO 1219), not fill color changes on the symbol itself. Each symbol template SHALL include a comment documenting this pedagogical simplification.

### Threshold Color Logic

```go
func thresholdColor(value float64, t ThresholdDoc) string {
    if t.Alarm != nil && value >= *t.Alarm { return "#dc3545" }      // red
    if t.Warning != nil && value >= *t.Warning { return "#ffc107" }    // yellow
    if t.AlarmLow != nil && value <= *t.AlarmLow { return "#dc3545" }  // red
    if t.WarningLow != nil && value <= *t.WarningLow { return "#ffc107" } // yellow
    return "#28a745"  // green (normal)
}
```

### HTMX Live Value Update Pattern

The process page template includes an HTMX polling container:

```html
<div id="process-view"
     hx-get="/partials/process-values/{{.EnvID}}"
     hx-trigger="every 2s"
     hx-swap="none">
    <!-- Full SVG rendered here on initial page load -->
    {{template "process_svg" .}}
</div>
```

The partial endpoint returns **only** updated elements with OOB swap:

```html
<!-- Each instrument value is a <text> element with a stable ID -->
<text id="inst-LT-101" hx-swap-oob="true" x="230" y="85"
      fill="#28a745" font-size="12">72.4%</text>
<text id="inst-AT-101" hx-swap-oob="true" x="230" y="103"
      fill="#28a745" font-size="12">7.2 pH</text>
<!-- Pump status circles update color -->
<circle id="equip-intake-pump-01-status" hx-swap-oob="true"
        cx="185" cy="220" r="8" fill="#28a745"/>
```

This swaps individual `<text>` and `<circle>` elements by ID without re-rendering the SVG structure, pipes, or labels.

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| SVG complexity in Go templates may be hard to maintain | Medium | Medium | Keep symbols simple; use named template blocks for modularity; each symbol is independently testable |
| HTMX OOB swap may not work with SVG namespaced elements | Medium | Low | Test early with a minimal SVG + HTMX prototype. Fallback: replace entire SVG content (slightly more bandwidth) |
| Layout algorithm may produce overlapping elements for complex processes | Low | Medium | Start with generous spacing; future SOWs can add overlap detection |
