# SOW-024.0 - Pipeline and Wastewater Process Schematics

**Status**: Approved
**Created**: 2026-03-02
**Priority**: High

## 1. Objective

Author `process.yaml` for the `pipeline-monitoring` and `brownfield-wastewater` environments, mapping every physical equipment item and instrument to its Modbus register address on the corresponding PLCs, RTUs, and flow computers. This extends the process schematic coverage from one environment (water treatment, SOW-023.0) to all three environments, enabling the SVG process view and scenario extensions downstream.

## 2. Background

SOW-023.0 established the reference implementation pattern with `greenfield-water-mfg/process.yaml` (564 lines, 28 instruments across 3 stages). SOW-024.0 applies the same schema to two significantly different environments:

- **Pipeline-monitoring**: Unmanned remote gas compressor station. Five devices spanning three categories (PLC, RTU, flow computer) with mixed addressing conventions (zero-based CompactLogix, one-based ROC800 and TotalFlow). Financial/custody-transfer measurement focus. Imperial units throughout.

- **Brownfield-wastewater**: Municipal WWTP built 1997 with ad-hoc modernization through 2022. Three PLCs spanning two generations (SLC-500 from 1997, CompactLogix from 2013) with mixed addressing conventions. First environment with `network_context` overlay (Cradlepoint cellular modem). Era markers on all equipment.

Both environments introduce ISA-5.1 tag prefixes not present in the current validation list, requiring a Phase 0 prefix addition before YAML authoring.

## 3. Scope

### In Scope

- Add four new ISA-5.1 prefixes to `process_checks.go` validation list
- Author `design/environments/pipeline-monitoring/process.yaml` with complete register-to-instrument mapping
- Author `design/environments/brownfield-wastewater/process.yaml` with complete register-to-instrument mapping, era markers, and `network_context`
- Validate both files pass `ot-design validate` with zero PRO-series errors

### Out of Scope

- Modifying device atoms or environment definitions (existing YAML is consumed as-is)
- SVG rendering or dashboard integration (SOW-025.0)
- Scenario extensions (SOW-026.0)
- New equipment type SVG symbols (all types used exist in SOW-022.0 library)
- Mapping infrastructure device registers (gateways, modems) as process instruments

## 4. Requirements

### Functional Requirements

1. **FR-1**: Pipeline process.yaml SHALL map 38 instruments from four devices (ps-plc-01, ps-rtu-01, ps-rtu-02, ps-fc-01) across three stages
2. **FR-2**: Wastewater process.yaml SHALL map 39 instruments from three devices (ww-plc-01, ww-plc-02, ww-plc-03) across three stages
3. **FR-3**: Every instrument's `placement`, `register.type`, `register.address`, `unit`, and `scale` SHALL match the corresponding device atom register map variant exactly
4. **FR-4**: Addressing convention SHALL be preserved: zero-based for CompactLogix, one-based for SLC-500, ROC800, and TotalFlow G5
5. **FR-5**: Writable registers SHALL use controller-suffix ISA tags (SC, FIC, AIC, KIC, HS) per the transmitter-vs-controller convention established in SOW-023.0
6. **FR-6**: Bidirectional thresholds SHALL follow the ordering rule: alarm_low < warning_low < normal < warning < alarm
7. **FR-7**: Wastewater process.yaml SHALL include `network_context` with the Cradlepoint WAN indicator
8. **FR-8**: Pipeline process.yaml SHALL include `network_context` with the satellite/cellular WAN backhaul
9. **FR-9**: All equipment SHALL carry an `era` year matching the facility installation context

### Non-Functional Requirements

1. **NFR-1**: Both YAML files SHALL pass `ot-design validate` with zero PRO-series errors
2. **NFR-2**: Inline YAML comments SHALL document treatment strategy, ISA tag rationale, attack surfaces, and known gaps (matching SOW-023.0 annotation depth)
3. **NFR-3**: The `flow_direction` for both environments SHALL be `horizontal`

## 5. Design

### 5.1 New ISA Prefix Additions (Phase 0)

Four ISA-5.1 prefixes must be added to `isaInstrumentPrefixes` in `design/internal/validate/process_checks.go`:

| Prefix | Meaning | ISA-5.1 Basis | Usage |
|--------|---------|---------------|-------|
| ZT | Position Transmitter | Z=Position, T=Transmitter | Analog valve position feedback (0-100%) |
| FQ | Flow Quantity | F=Flow, Q=Quantity/Integrate | Accumulated volume totalizers (custody transfer, daily NPDES reporting) |
| AIC | Analyzer Indicating Controller | A=Analysis, I=Indicate, C=Controller | Writable analytical setpoints (dissolved oxygen) |
| KIC | Time Indicating Controller | K=Time, I=Indicate, C=Controller | Writable timer setpoints (sludge pump cycle time) |

The PRO-010 warning message must also be updated to include the new prefixes.

### 5.2 Pipeline-Monitoring Process Schematic

**Physical flow**: Station Inlet -> Compressor -> [Aftercooler] -> ESD Valve -> Outlet -> Metering -> Gas Analysis (side-stream sample)

[OT-REVIEW] The aftercooler between compressor discharge and the ESD valve removes compression heat. TT-102/103 measure compressor discharge temperature (up to 300 degF). TT-105 measures post-aftercooler temperature (ambient range). The temperature drop is expected thermodynamic behavior, not sensor error. No aftercooler equipment is modeled (no dedicated registers); documented via YAML inline comment.

[OT-REVIEW] The gas chromatograph physically samples a side stream (~100 cc/min via sample conditioning system), not the main gas flow. It is positioned as the terminal stage for layout simplicity (TD-051). YAML inline comment on the meter-run-01 to gas-chromatograph connection must state this explicitly.
