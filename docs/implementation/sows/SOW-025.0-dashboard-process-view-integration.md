# SOW-025.0 - Dashboard Process View Integration

**Status**: Approved
**Created**: 2026-03-02
**Priority**: High

## 1. Objective

Wire the process view SVG engine (SOW-022.0) and process schematics (SOW-023.0, SOW-024.0) into the monitoring dashboard as a live interactive page. Add `/process` routes with environment selector, navigation links, and HTMX live value polling. This is a thin integration SOW -- all data layer code, templates, and the layout engine already exist.

## 2. Background

SOW-022.0 implemented:
- SVG symbol templates (process_symbols.html, process_instruments.html, process_connections.html, process_network_context.html) -- already loaded in the base template set
- Layout engine (process_layout_engine.go) -- `ComputeProcessLayout()` computes positions from `ProcessSchematic`
- Process page template (process.html) with environment selector sidebar and HTMX polling container
- Process SVG wrapper template (process_svg.html) dispatching to symbol templates
- Data builders (process_data.go): `BuildProcessViewData()`, `BuildProcessValuesPartial()`, `populateLiveValues()`
- Page data type (process_data.go): `ProcessViewPageData` with Title, ActivePage, EnvID, Process, AllEnvs fields

SOW-023.0 and SOW-024.0 authored process.yaml files for all three environments. The `DesignLibrary.Schematics` map is populated at startup by `loadProcessSchematics()`.

What is **missing** (and what this SOW delivers):
1. Route registration in `buildRouter()`
2. Handler functions in `handlers.go`
3. Navigation link in `nav.html`
4. Registration of `process_svg.html` in the base template set (currently only the partials are registered; the SVG wrapper is not)
5. Default environment selection logic for the process view (only environments with schematics should appear)

### Reference Implementations

The topology page follows the identical pattern: `topologyHandler` / `topologyEnvHandler` / `topologyPartialHandler` in handlers.go, with `buildTopologyPageData()` as the assembly function, `defaultEnvID()` for default selection, and `buildEnvSummaries()` for the environment selector list.

## 3. Scope

### In Scope

- Route registration: GET /process, GET /process/{env-id}, GET /partials/process-values/{env-id}
- Handler functions following the topology handler pattern
- Navigation link in the "Observed" section of nav.html
- Template registration of process_svg.html in the base template set
- Process-specific default environment helper (filters to environments with schematics)
- Environment selector filtering (show only environments that have a process.yaml)

### Out of Scope

- Template modifications to process.html, process_svg.html, or any symbol/instrument partials (those were delivered by SOW-022.0 and are frozen)
- Process schematic authoring (covered by SOW-023.0 and SOW-024.0)
- Layout engine changes
- Data layer changes to process_data.go (BuildProcessViewData, BuildProcessValuesPartial, populateLiveValues are complete)
- Cross-links from asset detail or topology pages to process view (deferred -- would require passing placement context between pages)
- Equipment status OOB swap (HTMX partial only updates instrument text values per SOW-022.0 FR-10 design)

## 4. Requirements

### Functional Requirements

**FR-1**: GET /process renders the process view page using the default environment (first environment alphabetically that has a schematic in `lib.Schematics`).

**FR-2**: GET /process/{env-id} renders the process view for a specific environment. Returns 404 if the environment does not exist or has no schematic.

**FR-3**: GET /partials/process-values/{env-id} returns HTMX OOB swap elements (instrument `<text>` elements with `hx-swap-oob="true"`) for live value refresh. Called every 2 seconds by the HTMX polling container in process.html. [OT-REVIEW] The 2-second interval is intentionally longer than a real SCADA poll cycle (500ms-1s) because this is a web monitoring overlay, not the SCADA itself. Trainees should understand that the dashboard is a secondary read of data the PLC is already acting on. The distinction between control loop polling and monitoring polling is a key OT/IT conceptual difference. If the monitoring API is unreachable, the partial returns an empty response (zero OOB elements) rather than a 404 -- stale data on the display is preferable to a broken display, which is the OT availability principle trainees should internalize.

**FR-4**: The "Process" link appears in the nav.html "Observed" section, between "Topology" and the right-side "Reference" section. Active state highlights when `ActivePage == "process"`.

**FR-5**: The environment selector sidebar shows only environments that have a process schematic (key exists in `lib.Schematics`). Environments without a process.yaml are omitted from the list.

**FR-6**: The `process_svg.html` template is added to the base template set so the `{{template "process_svg" .Process}}` call in process.html resolves correctly.

### Non-Functional Requirements

**NFR-1**: All new handler code follows the topology handler pattern exactly (same error handling, same 404 pattern, same data assembly pattern).

**NFR-2**: No new Go dependencies introduced.

**NFR-3**: Existing tests continue to pass (`go test ./...` in monitoring module).

### Handler Functions (handlers.go)

Three handlers plus one assembly function, mirroring the topology pattern:

```go
// processHandler: GET /process -- default environment
func (d *Dashboard) processHandler(w http.ResponseWriter, r *http.Request) {
    envID := defaultProcessEnvID(d.lib)
    data := d.buildProcessPageData(envID)
    d.render(w, "process.html", data)
}

// processEnvHandler: GET /process/{env-id} -- specific environment
func (d *Dashboard) processEnvHandler(w http.ResponseWriter, r *http.Request) {
    envID := chi.URLParam(r, "env-id")
    if _, ok := d.lib.Schematics[envID]; !ok {
        http.NotFound(w, r)
        return
    }
    data := d.buildProcessPageData(envID)
    d.render(w, "process.html", data)
}

// processValuesPartialHandler: GET /partials/process-values/{env-id}
// [OT-REVIEW] Guard on schematic map lookup explicitly before calling data builders.
// Return empty partial (not 404) when API is down or returns no data -- stale data
// on the display is preferable to a broken display (OT availability principle).
// A 404 would cause HTMX to stop polling, freezing the live view permanently.
func (d *Dashboard) processValuesPartialHandler(w http.ResponseWriter, r *http.Request) {
    envID := chi.URLParam(r, "env-id")
    sc, ok := d.lib.Schematics[envID]
    if !ok {
        http.NotFound(w, r)
        return
    }
    pv := d.BuildProcessViewData(envID, sc)
    instruments := d.BuildProcessValuesPartial(pv)
    d.renderPartial(w, "svg_instrument_oob", instruments)
}

// buildProcessPageData: assembles ProcessViewPageData for render
func (d *Dashboard) buildProcessPageData(envID string) ProcessViewPageData {
    sc := d.lib.Schematics[envID]
    pv := d.BuildProcessViewData(envID, sc)
    return ProcessViewPageData{
        Title:      "Process View",
        ActivePage: "process",
        EnvID:      envID,
        Process:    pv,
        AllEnvs:    buildProcessEnvSummaries(d.lib),
    }
}
```

### Default Environment Helper (handlers.go)

```go
// defaultProcessEnvID returns the first environment alphabetically that has a
// process schematic. Prefers "greenfield-water-mfg" if it has a schematic.
func defaultProcessEnvID(lib *DesignLibrary) string {
    if lib == nil {
        return ""
    }
    if _, ok := lib.Schematics["greenfield-water-mfg"]; ok {
        return "greenfield-water-mfg"
    }
    ids := sortedKeys(lib.Schematics)
    if len(ids) == 0 {
        return ""
    }
    return ids[0]
}
```

Note: The topology default prefers "brownfield-wastewater" because it demonstrates the most complex network topology. [OT-REVIEW] The process view default prefers "greenfield-water-mfg" because it shows a modern Purdue-segmented environment with clean ISA-5.1 tagging, era markers on recent hardware, and no legacy complexity. This is the right starting point for a trainee seeing a process view for the first time. Brownfield-wastewater is a better second view because it introduces era mixing, the Cradlepoint cellular backdoor, and legacy SLC-500 addressing conventions -- all of which make more sense after the trainee has a clean reference to compare against.

### Environment Selector Filter (handlers.go)

```go
// buildProcessEnvSummaries returns EnvSummary entries only for environments
// that have a process schematic. Alphabetically sorted by ID.
func buildProcessEnvSummaries(lib *DesignLibrary) []EnvSummary {
    if lib == nil {
        return nil
    }
    ids := sortedKeys(lib.Schematics)
    summaries := make([]EnvSummary, 0, len(ids))
    for _, id := range ids {
        env, ok := lib.Environments[id]
        if !ok {
            // [OT-REVIEW] Intentional silent skip: handles edge case where a schematic was
            // loaded for a directory that has no corresponding environment.yaml. This can
            // happen if a process.yaml exists in a directory without a valid environment def.
            // Do NOT remove this guard -- it is not dead code.
            continue
        }
        summaries = append(summaries, EnvSummary{
            ID:        env.Env.ID,
            Name:      env.Env.Name,
            Archetype: env.Env.Archetype,
        })
    }
    return summaries
}
```
