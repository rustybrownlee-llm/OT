# SOW-026.0 - Process View Scenario Extensions

**Status**: Approved
**Created**: 2026-03-02
**Priority**: High

## 1. Objective

Extend Scenarios 01 (Asset Discovery) and 02 (Vulnerability Assessment) with new terminal phases that use the process view dashboard to bridge the gap between abstract register discovery and physical process understanding. Scenario 01 Phase G teaches register-to-instrument mapping via ISA-5.1 tags. Scenario 02 Phase F teaches vulnerability impact assessment through process context. These phases are the payoff for the entire Beta 0.5 process view work.

## 2. Background

Scenarios 01 and 02 currently end with:
- **Scenario 01 Phase F**: Hybrid environment discovery (brownfield-wastewater via nmap and mbpoll)
- **Scenario 02 Phase E**: Hybrid architecture assessment (wastewater topology, era analysis, V-W01 through V-W04)

Both scenarios teach protocol-level skills (register enumeration, write demonstration, topology analysis) but stop short of connecting register addresses to physical equipment and process consequences. With the process view live at `/process` (SOW-025.0), trainees can now see that register HR[0] on port 5020 is not just "a number" -- it is FT-101, the intake flow transmitter on a raw water pump, measured in GPM.

This is the core educational thesis of Beta 0.5: IT engineers who learn OT through protocol-only exercises will not understand why their findings matter. Process context makes vulnerability assessment meaningful.

### Reference Implementations

Existing scenario phases in Scenarios 01-02 set the structure and tone. Key patterns:
- Each phase has a heading, narrative context, numbered steps with code blocks, "Document for vulnerability report" callouts, and "Teaching point" blocks
- Success criteria are numbered SC-XX with checkbox format and explicit expected observations
- Hints are structured as three tiers: conceptual nudge, technique guidance, specific observation
- Solutions contain complete command outputs and expected dashboard states

## 5. Design

### Scenario 01 Phase G: Process View Context

**Narrative**: "You have completed a thorough asset discovery of three OT environments. You found registers, identified devices, and mapped network topology. But your asset inventory is still abstract -- register addresses and raw values. Now open the process view to understand what these devices actually do."

**Steps**:
1. Navigate to `http://localhost:8090/process` -- greenfield-water-mfg loads as default
2. Identify the three stages (Intake, Treatment, Distribution) and their controller PLCs
3. Map discovered registers from Phase B to ISA-5.1 tags (e.g., port 5020 HR[0] = FT-101 intake_flow_rate)
4. [OT-REVIEW] Observe live instrument values updating. The process view refreshes at the monitoring dashboard's polling interval (approximately 2 seconds). This is intentionally slower than a real SCADA poll cycle (500ms-1s) because the dashboard is a secondary monitoring overlay, not the control system itself
5. Note the era markers on equipment symbols and the legend explaining status colors
6. Switch to brownfield-wastewater environment selector -- note the era mixing (1997-2022) visible on equipment
7. Identify the Cradlepoint WAN network context callout (security overlay on the process diagram)
8. Switch to pipeline-monitoring environment -- observe a completely different process domain (gas compression, custody transfer, chromatography)
9. [OT-REVIEW] Note the ISA-5.1 tag patterns: FT/FQ for flow, PT for pressure, PDT for differential pressure, AT for analysis, ZT for valve position (analog 0-100%), ZS for position switch (discrete boolean). The pipeline environment has both ZT-101 (inlet block valve analog position feedback) and ZS-101 (ESD active status, discrete coil) -- trainees must distinguish these. Also note: ZS-101 is a read-only hardwired safety interlock that CANNOT be overridden via Modbus; not all instruments on the process view are attack surfaces
10. Understand that the same Modbus protocol underlies vastly different physical processes

**Teaching points**:
- "Register addresses are abstract. ISA-5.1 tags encode physical meaning: the first letter is the measured variable (F=flow, P=pressure, T=temperature, A=analysis, Z=position), the suffix indicates the function (T=transmitter, IC=indicating controller, S=switch, Q=quantity)."
- "The process view is a training tool, not a control interface. In a real facility, the SCADA operator sees this in their HMI. As a security assessor, you need this context to understand the impact of your findings."
- "The pipeline environment introduces custody transfer metering. The registers you would discover on ps-rtu-01 (the metering RTU) directly determine financial billing. Spoofing PDT-201 (differential pressure) alters calculated flow volume and therefore pipeline revenue. Note: ps-rtu-01 is station-LAN-only; reaching it from the WAN requires pivoting through the dual-homed ps-plc-01."
- [OT-REVIEW] Hint-4.md for this scenario must include a one-paragraph domain orientation for trainees unfamiliar with gas pipelines: what a compressor station does (boost pipeline pressure for long-distance gas transport), what custody transfer metering measures (the volume of gas delivered at a contractual boundary using AGA-3 orifice calculation from DP, static pressure, and flowing temperature), and what chromatograph analysis determines (gas quality -- BTU heating value and specific gravity -- for billing and tariff compliance). Also include ZT vs ZS distinction with specific pipeline examples, and note that FQ totalizer registers reset to zero at contract hour rollover (often 9:00 AM per NAESB standards, not midnight) -- a zero reading is normal behavior, not a meter attack

### Scenario 02 Phase F: Process Impact Analysis

**Narrative**: "Your vulnerability report documents protocol gaps, network weaknesses, and device posture. But the findings describe register addresses and function codes -- language that means nothing to the facility manager who must decide where to spend the remediation budget. Open the process view to translate your findings into physical consequences."

**Steps**:
1. Navigate to the process view with greenfield-water-mfg selected
2. [OT-REVIEW] Revisit V-02 (unauthenticated write): locate FIC-202 (chemical_feed_rate, the sodium hypochlorite dosing setpoint) on the Treatment stage -- NOT AT-201, which is a read-only turbidity measurement. Write a modified value to HR[4] on wt-plc-02 via mbpoll and watch the process view update. This step remains within Rules of Engagement (holding register write, not coil write; original value restored immediately)
3. Revisit V-09 (cross-plant dependency): trace the connection from Distribution stage (wt-plc-03) to the manufacturing cooling loop. Understand that coil[2] on wt-plc-03 controls booster_pump_run, and stopping it starves the manufacturing process
4. [OT-REVIEW] Switch to brownfield-wastewater: locate the dissolved oxygen setpoint (AIC-201, HR[2] on ww-plc-03). AIC-201 is what you write (the setpoint); AT-201 (HR[3], dissolved oxygen probe) is what you watch fall. Writing 0.0 mg/L to AIC-201 causes the PID loop to reduce blower speed, which then causes the actual DO reading (AT-201) to decline. The AT-201 value will begin to decline within one polling cycle as the PID reduces blower output. A full anoxic condition (AT-201 < 0.5 mg/L) would develop over 10-30 minutes in a real facility; the simulator may compress this timeline. This setpoint-vs-feedback distinction is a core OT lesson
5. Locate the Cradlepoint WAN callout on the wastewater process view. Connect V-W02 (Cradlepoint default credentials) to the process context: cellular WAN access to the flat OT network gives an internet-based attacker a path to write AIC-201
6. Switch to pipeline-monitoring: introduce three new pipeline-specific findings
   - [OT-REVIEW] V-P01: Custody transfer manipulation -- PDT-201 (differential pressure, HR[5] on ps-rtu-01) is the dominant AGA-3 flow calculation input because flow rate is proportional to sqrt(DP). PT-201 (static pressure, HR[3]) and TT-201 (flowing temperature, HR[4]) are also AGA-3 inputs and equally writable. All three are on ps-rtu-01 (ROC800, station LAN only at 10.20.1.20). These registers are NOT WAN-reachable; attack requires station LAN access or pivoting through the dual-homed ps-plc-01
   - V-P02: Compressor control via dual-homed WAN -- run-C-101 (coil 0 on ps-plc-01, CompactLogix) IS directly WAN-reachable because ps-plc-01 is dual-homed (station LAN 10.20.1.10 and WAN 10.20.0.2). This is the only WAN-writable register in the pipeline environment. Contrast with V-P01: the compressor PLC is WAN-exposed, the metering RTU is not
   - [OT-REVIEW] V-P03: Chromatograph data integrity -- AT-306 BTU (HR[6]) and AT-307 SG (HR[7]) on ps-fc-01 (TotalFlow G5, serial device) feed into AGA billing calculations. ps-fc-01 is accessible only via ps-gw-01 (Moxa gateway at 10.20.1.30:5043, station LAN). Attack chain requires: WAN -> ps-plc-01 (dual-homed pivot) -> station LAN -> ps-gw-01 -> ps-fc-01. This is a multi-hop covert financial attack -- not directly WAN-reachable
7. Update vulnerability report with "Process Impact" descriptions for all findings
8. Re-assess risk ratings with process context (some findings may warrant upgrading)

**Teaching points**:
- "A vulnerability finding rated Medium based on register address alone may become Critical when you understand it controls chlorine dosing for drinking water."
- "The process view reveals attack chains that register enumeration alone cannot show. The Cradlepoint (V-W02) -> flat network -> AIC-201 (dissolved oxygen setpoint) chain is a three-hop path from the internet to an environmental compliance violation."
- [OT-REVIEW] "Pipeline custody transfer is a financial attack surface. Unlike water treatment (safety impact) or manufacturing (production impact), pipeline metering has direct revenue impact. This is why TSA pipeline security directives (SD-02C series, issued after the Colonial Pipeline attack) and DOT 49 CFR 192 exist. Note: NERC CIP applies to the bulk electric system, not pipelines -- conflating the two is a common IT-engineer mistake."
