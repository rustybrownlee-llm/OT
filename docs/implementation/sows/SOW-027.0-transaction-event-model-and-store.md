# SOW-027.0 - Transaction Event Model and Store

**Status**: Complete
**Created**: 2026-03-02
**Completed**: 2026-03-02
**Priority**: High

## 1. Objective

Define the TransactionEvent Go type and implement a persistent SQLite event store package for the monitoring module. This is the foundation for all Beta 0.6 telemetry work -- every subsequent SOW (poller logging, dashboard pages, syslog forwarding, baseline extensions) depends on this event model and store. The store provides insert, query, and prune operations with WAL mode for concurrent read/write access.

## 2. Background

The monitoring module currently captures device state as in-memory register snapshots (poller → baseline → alerts). There is no persistent record of individual Modbus transactions -- when the monitor restarts, all history is lost (td-alert-021). ADR-011 identifies Protocol Telemetry (Layer 1) as the foundation of the five-layer monitoring stack, and the Beta 0.6 milestone spec defines the transaction event model and SQLite schema.

### Current Architecture

```
Poller ──(DeviceSnapshot)──> Baseline Engine ──(Alert)──> Alert Store (in-memory)
```

### Target Architecture (after full Beta 0.6)

```
Poller ──(DeviceSnapshot)──> Baseline Engine ──(Alert)──> Alert Store (in-memory)
   │
   └──(TransactionEvent)──> Event Store (SQLite) ──> Dashboard, Syslog, Baseline Extensions
```

This SOW delivers the `TransactionEvent` type and `Event Store (SQLite)` box. SOW-028.0 connects the poller to it.

### Reference Implementations

- **Alert Store** (`monitoring/internal/alert/alert.go`): Thread-safe in-memory store with insert/query/filter pattern. The event store follows a similar API shape but uses SQLite instead of in-memory slices.
- **Config package** (`monitoring/internal/config/config.go`): Parsing pattern with raw struct, defaults, and validation. The new config fields follow this pattern exactly.
- **Ring Buffer** (`monitoring/internal/timeseries/ringbuffer.go`): Fixed-capacity storage with eviction. The event store uses time-based retention (pruning) instead of capacity-based eviction.

### New Dependency: SQLite Driver

This SOW introduces a new Go dependency: `modernc.org/sqlite` (pure-Go SQLite implementation). This is flagged per the shared development standards requirement to explicitly identify new dependencies.

## 4. Requirements

### Functional Requirements

**FR-1**: The `TransactionEvent` type contains all fields specified in the milestone spec: ID (UUID v4), Timestamp, SrcAddr, DstAddr, UnitID, FunctionCode, FunctionName, AddressStart, AddressCount, IsWrite, Success, ExceptionCode, ResponseTimeUs, DeviceID, DeviceName, EnvID. [OT-REVIEW] `AddressStart` and `AddressCount` use protocol-neutral naming because Modbus has four separate address spaces (coils 0x, discrete inputs 1x, input registers 3x, holding registers 4x). Using "register" terminology for coil operations would teach incorrect address space semantics.

**FR-2**: The `WriteDetail` type holds register values ([]uint16) and coil values ([]bool) for write operations. It is embedded in TransactionEvent as a pointer field (nil for read operations).

**FR-3**: A `FunctionCodeInfo` lookup function maps Modbus function codes (uint8) to their canonical name, category ("read", "write", "diagnostic"), and IsWrite boolean. Covers FC 1-4 (reads), FC 5/6/15/16 (writes), and FC 43 (MEI Transport). Unknown function codes return "Unknown (FC {n})" with category "unknown". [OT-REVIEW] FC 43 is named "MEI Transport (Device Identification)" to indicate this captures sub-function 14 specifically. FC 43 has multiple sub-functions including MEI Type 0x0D (CANopen) which can carry write payloads. The comment in `funccode.go` must document that FC 43 was used for reconnaissance in the TRITON/TRISIS attack against Schneider Tricon safety controllers, so trainees understand it is not inherently benign.

**FR-3a**: [OT-REVIEW] In Beta 0.6, the poller (SOW-028.0) emits only FC 1 (Read Coils) and FC 3 (Read Holding Registers) events from normal polling. FC 5/6/15/16 events will appear only when a write is injected by an attacker or scenario exercise. This is expected behavior, not a bug -- the function code filter in the dashboard is useful precisely for detecting the write FCs that should never appear in a read-only monitoring environment.

**FR-3b**: A `LookupExceptionCode` function maps standard Modbus exception codes (1-8) to their canonical name and a brief educational description. [OT-REVIEW] Exception codes have specific operational meanings: 01 (Illegal Function) = device does not support this FC, 02 (Illegal Data Address) = address beyond register map (most common in OT; often indicates misconfigured SCADA poll or attacker probing), 03 (Illegal Data Value) = value outside valid range, 04 (Slave Device Failure) = hardware fault (critical for uptime monitoring). Trainees need human-readable exception names in the dashboard, not raw numbers.

**FR-4**: The event store opens (or creates) a SQLite database at the configured path. The schema is auto-created on first open using `CREATE TABLE IF NOT EXISTS`. [OT-REVIEW] No migration framework -- this is a prototype event store for an educational tool. The schema is append-only; structural changes in future milestones can drop and recreate the table without data loss concerns (7-day retention means old data is already pruned).

**FR-5**: The store inserts events using `database/sql` prepared statements. A batch insert method accepts a slice of events and wraps them in a single transaction for performance.

**FR-6**: The store queries events with composable filter options: time range (after/before), device ID (exact match), function code (exact match), is_write (boolean), success (boolean), limit, and offset. Filters are combined with AND logic. Results are ordered by timestamp descending (newest first).

**FR-7**: The store counts events matching the same filter set as FR-6 (for pagination support in the dashboard).

**FR-8**: The store prunes events older than the configured retention period. Pruning is a DELETE operation, not a table drop. The `Prune` method returns the number of rows deleted.

**FR-9**: WAL mode is enabled on database open via `PRAGMA journal_mode=WAL`. This allows the poller goroutine to insert events while the dashboard goroutine queries them concurrently.

**FR-10**: Write detail values are stored as JSON text in the `write_values` column. `encoding/json` is used for serialization; `WriteDetail` implements custom marshal/unmarshal. Null for read operations.

**FR-11**: The config package accepts two new fields: `event_db_path` (string, default `"data/events.db"`) and `event_retention_days` (int, default `7`, minimum `1`). Validation rejects `event_retention_days < 1`.

**FR-12**: The store constructor (`New`) creates the parent directory for the database file if it does not exist (using `os.MkdirAll`). This handles the common case where `data/` does not yet exist on first run.

### Non-Functional Requirements

**NFR-1**: No new Go dependencies beyond `modernc.org/sqlite`. All SQL is written directly with `database/sql` (no ORM, no query builder).

**NFR-2**: The event store package has zero imports from other monitoring internal packages. It defines its own types (`TransactionEvent`, `WriteDetail`, `FilterOptions`). This keeps it reusable and testable in isolation.

**NFR-3**: All store methods are safe for concurrent use by multiple goroutines. The `database/sql` connection pool handles SQLite's single-writer constraint; WAL mode handles concurrent readers.

**NFR-4**: Unit tests use an in-memory SQLite database (`:memory:`) for speed and isolation. No test fixtures on disk.

**NFR-5**: Insert latency for a single event is under 1ms on localhost (SQLite WAL mode typical). Batch insert of 100 events is under 10ms.

**NFR-6**: Existing tests continue to pass (`go test ./...` in monitoring module). No changes to existing packages.

### TransactionEvent Type

```go
// TransactionEvent represents a single Modbus transaction captured by the
// monitoring poller. This is the foundational event type for ADR-011 Layer 1
// (Protocol Telemetry). Each polling read or write produces one event.
type TransactionEvent struct {
    ID             string       // UUID v4, generated on creation
    Timestamp      time.Time    // when the transaction completed
    SrcAddr        string       // source IP:port (monitor's address)
    DstAddr        string       // destination IP:port (device endpoint)
    UnitID         uint8        // Modbus unit/slave ID
    FunctionCode   uint8        // Modbus function code (1-127)
    FunctionName   string       // human-readable ("Read Holding Registers")
    AddressStart  uint16       // starting address (coil address for FC 1/2/5/15; register address for FC 3/4/6/16)
    AddressCount  uint16       // number of addresses in request (coils or registers depending on FC)
    IsWrite        bool         // true for FC 5, 6, 15, 16
    Success        bool         // completed without Modbus exception
    ExceptionCode  uint8        // Modbus exception code if !Success
    ResponseTimeUs int64        // round-trip time in microseconds
    DeviceID       string       // resolved asset ID (ip:port:unit_id)
    DeviceName     string       // human-readable name from design library (populated by SOW-028.0; empty until then)
    EnvID          string       // environment ID from config
    WriteDetail    *WriteDetail // non-nil for write operations only
    // [OT-REVIEW] SrcAddr in Beta 0.6 is always the monitor's ephemeral client address
    // because the poller is the only active Modbus client. This field becomes meaningful
    // in Beta 0.7 (passive capture) when third-party clients (including attackers) are observed.
}
```

### WriteDetail Type

```go
// WriteDetail holds the values written in a Modbus write operation.
```

### Store API

```go
// Store provides persistent storage for Modbus transaction events.
// It wraps a SQLite database with WAL mode for concurrent access.
type Store struct {
    db        *sql.DB
    insertStmt *sql.Stmt // prepared INSERT statement
}

// New opens (or creates) the SQLite database at dbPath, enables WAL mode,
// creates the schema if needed, and prepares statements.
func New(dbPath string) (*Store, error) { ... }

// Close releases the prepared statements and closes the database connection.
func (s *Store) Close() error { ... }

// Insert stores a single TransactionEvent. Returns an error if the insert fails.
func (s *Store) Insert(ctx context.Context, e *TransactionEvent) error { ... }

// InsertBatch stores multiple events in a single SQLite transaction.
// More efficient than calling Insert in a loop.
func (s *Store) InsertBatch(ctx context.Context, events []*TransactionEvent) error { ... }

// FilterOptions specifies query filters for event retrieval.
type FilterOptions struct {
    After      *time.Time // events after this timestamp (inclusive)
    Before     *time.Time // events before this timestamp (exclusive)
    DeviceID   *string    // exact device ID match
    FuncCode   *uint8     // exact function code match
    IsWrite    *bool      // filter by write operations
    Success    *bool      // filter by success/failure
    Limit      int        // max results (0 = no limit; capped at 10000)
    Offset     int        // pagination offset
}

// Query returns events matching the filter options, ordered by timestamp desc.
func (s *Store) Query(ctx context.Context, opts FilterOptions) ([]*TransactionEvent, error) { ... }

// Count returns the number of events matching the filter options.
func (s *Store) Count(ctx context.Context, opts FilterOptions) (int64, error) { ... }

// Prune deletes events older than the given cutoff time. Returns rows deleted.
func (s *Store) Prune(ctx context.Context, olderThan time.Time) (int64, error) { ... }
```

### SQL Schema

```sql
CREATE TABLE IF NOT EXISTS events (
    id           TEXT PRIMARY KEY,
    timestamp    TEXT NOT NULL,
    src_addr     TEXT NOT NULL,
    dst_addr     TEXT NOT NULL,
    unit_id      INTEGER NOT NULL,
    func_code    INTEGER NOT NULL,
    func_name    TEXT NOT NULL,
    addr_start    INTEGER,
    addr_count    INTEGER,
    is_write     INTEGER NOT NULL DEFAULT 0,
    success      INTEGER NOT NULL DEFAULT 1,
    exception    INTEGER,
    response_us  INTEGER,
    device_id    TEXT,
    device_name  TEXT,
    env_id       TEXT,
    write_values TEXT
);

CREATE INDEX IF NOT EXISTS idx_events_timestamp ON events(timestamp);
CREATE INDEX IF NOT EXISTS idx_events_device    ON events(device_id);
CREATE INDEX IF NOT EXISTS idx_events_write     ON events(is_write) WHERE is_write = 1;
CREATE INDEX IF NOT EXISTS idx_events_func      ON events(func_code);
```

### UUID Generation

UUIDs are generated using `crypto/rand` and formatted as standard UUID v4 strings. No external UUID library -- the implementation is 10 lines of Go using the standard library.

```go
// newUUID generates a random UUID v4 string.
func newUUID() string {
    var u [16]byte
    _, _ = rand.Read(u[:])
    u[6] = (u[6] & 0x0f) | 0x40 // version 4
    u[8] = (u[8] & 0x3f) | 0x80 // variant 10
    return fmt.Sprintf("%08x-%04x-%04x-%04x-%012x",
        u[0:4], u[4:6], u[6:8], u[8:10], u[10:16])
}
```

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| `modernc.org/sqlite` adds significant binary size | Binary grows by ~15-20MB (pure Go SQLite is large) | High | Acceptable tradeoff for zero-CGo cross-compilation. Document the size increase. |
| `modernc.org/sqlite` compilation is slow on first build | Developer experience degradation | Medium | One-time cost; subsequent builds use Go build cache. Document in SOW work tracking. |
| SQLite WAL mode file locking on NFS/network filesystems | Database corruption | Low | Event store runs locally; Docker volumes are local ext4/APFS. Document that network mounts are unsupported. |
| UUID collisions | Duplicate key errors | Negligible | crypto/rand UUID v4 collision probability is astronomically low; PRIMARY KEY constraint catches any collision with a clear error. |
