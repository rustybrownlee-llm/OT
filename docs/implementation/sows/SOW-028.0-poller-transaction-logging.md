# SOW-028.0 - Poller Transaction Logging

**Status**: Approved
**Created**: 2026-03-02
**Priority**: High

## 1. Objective

Instrument the existing Modbus poller to emit a `TransactionEvent` for every read/write cycle and wire the `eventstore.Store` into monitor startup. After this SOW, every Modbus transaction the poller executes is recorded as a structured, queryable event in the SQLite event store, and old events are automatically pruned on an hourly schedule. This is the critical integration that connects Layer 1 telemetry (SOW-027.0) to the running monitoring infrastructure.

## 2. Background

SOW-027.0 delivered the `TransactionEvent` type, `eventstore.Store` package, and the SQLite schema. The store is a standalone package with no imports from other monitoring internal packages. It has no connection to the running system -- nothing creates events or opens the database yet.

The poller (`monitoring/internal/poller/poller.go`) executes Modbus reads on a configurable interval:
- `readHoldingChunked(client, baseAddr, count)` → FC 3 (Read Holding Registers), chunked at 125 registers max per request
- `readCoilsChunked(client, baseAddr, count)` → FC 1 (Read Coils), chunked at 125 coils max per request (existing code; see FR-3 for protocol-correct limit)

Each chunk is a separate Modbus transaction on the wire and must produce a separate `TransactionEvent`.

### Current Architecture

```
Poller.Run() → runCycle() → pollAsset() → readHoldingChunked / readCoilsChunked
                                                    ↓
                                            inventory.UpdateRegisters()
                                                    ↓
                            hook(snapshots) → Baseline Engine
```

### Target Architecture (after this SOW)

```
Poller.Run() → runCycle() → pollAsset() → readHoldingChunked / readCoilsChunked
                                                    ↓
                                   TransactionEvent per chunk ──→ EventHook
                                                    ↓                  ↓
                                            inventory.UpdateRegisters()  eventstore.Store.InsertBatch()
                                                    ↓
                            hook(snapshots) → Baseline Engine

main.go startup:
  eventstore.New(cfg.EventDBPath) → pruning goroutine (hourly)
                                  → poller.SetEventHook(batchInsertFunc)
```

### Hook Design Rationale

The existing `CycleHook` fires once per cycle with aggregated `DeviceSnapshot` values. The new `EventHook` fires once per cycle with a slice of `TransactionEvent` values collected during that cycle. This is a batch delivery pattern, not per-transaction callback:

1. **Per-transaction callback** would require passing a hook into `readHoldingChunked`/`readCoilsChunked`, which threads through three levels of helper functions and changes their signatures. [OT-REVIEW] It would also create a misleading mental model for trainees: Modbus reads within a poll cycle are not independent events visible on the network -- they are sequential reads from a single TCP connection to a single device. Batching events per cycle reflects the actual polling cadence that trainees should understand.
2. **Batch per-cycle delivery** collects events in a slice during `pollAsset`, delivers them all at once after the cycle. This maps naturally to `InsertBatch` for a single SQLite transaction per cycle. Simpler code, better write performance, more accurate mental model.

### Reference Implementations

- **CycleHook pattern** (`poller.go:38`): `type CycleHook func(snapshots []baseline.DeviceSnapshot)` -- called synchronously at end of `runCycle()`. The `EventHook` follows the same pattern.
- **main.go wiring** (`main.go:226`): `p.SetCycleHook(engine.RecordCycle)` -- the EventHook is wired the same way.
- **Inventory asset fields** (`inventory.go:23-47`): `Asset.ID`, `Asset.Endpoint`, `Asset.UnitID`, `Asset.EnvironmentName`, `Asset.ConfigDesc` -- used to populate TransactionEvent fields.

## 4. Requirements

### Functional Requirements

**FR-1**: The Poller type exposes a new `EventHook` type: `type EventHook func(events []*eventstore.TransactionEvent)`. The hook is called synchronously after each poll cycle with all events collected during that cycle, before the existing `CycleHook` fires. [OT-REVIEW] EventHook fires first so that the event record exists in the store before any derivative processing (baseline engine, alerting) consumes the same cycle's data. If the CycleHook (baseline) fires first and generates an alert, but the subsequent InsertBatch fails, there would be an alert with no corresponding persisted event -- an inconsistency that confuses forensic analysis. Event persistence before derivative processing is standard telemetry pipeline design.

**FR-2**: `SetEventHook(hook EventHook)` registers the event hook on the Poller. Calling `SetEventHook` while the poller is running is safe (same guarantee as `SetCycleHook`).

**FR-3**: Each chunked Modbus read produces one `TransactionEvent`. For an asset with 250 holding registers, `readHoldingWithEvents` issues two FC 3 requests (125 + 125), producing two TransactionEvent records. For an asset with 100 coils, `readCoilsWithEvents` issues one FC 1 request, producing one TransactionEvent record. Total events per asset per cycle = ceil(holdingRegCount/125) + ceil(coilCount/modbusMaxCoilRead). [OT-REVIEW] The Modbus specification defines different maximum quantities per function code: FC 3/4 (registers) max 125 per request (250 bytes = 125 x 2 bytes per register), but FC 1/2 (coils) max 2000 per request (250 bytes = 2000 bits). The existing `readCoilsChunked` conservatively uses 125 as the chunk size, which works but issues more requests than necessary. This SOW introduces `modbusMaxCoilRead = 2000` for FC 1/FC 2 to match the protocol specification. For the simulator's typical coil counts (10-50 per device), the 2000-coil limit is never hit, but using the correct limit teaches trainees accurate protocol constraints.

**FR-4**: TransactionEvent fields are populated as follows:

| Field | Source |
|-------|--------|
| ID | Generated internally by `eventstore.NewEvent()` constructor (UUID v4) |
| Timestamp | `time.Now()` captured immediately after the Modbus response is received (transaction completion boundary) |
| SrcAddr | Monitor's source address placeholder: host portion of the first configured environment address with port `"0"`, e.g., `"10.10.30.100:0"`. [OT-REVIEW] In Beta 0.6, this is always the monitor's address. The `simonvetter/modbus` client does not expose `net.Conn.LocalAddr()`, so the ephemeral TCP source port is unavailable. Using the monitor's known IP with port 0 is less misleading than using the HTTP API port (8091), which would never appear in Modbus traffic. A trainee comparing event logs with Wireshark output would see TCP connections from ephemeral ports (49152-65535), not port 8091. The field becomes fully accurate in Beta 0.7 (passive capture). |
| DstAddr | `asset.Endpoint` (IP:port) |
| UnitID | `asset.UnitID` |
| FunctionCode | 3 for holding registers, 1 for coils |
| FunctionName | Resolved internally by `eventstore.NewEvent()` via `LookupFuncCode(fc)` |
| AddressStart | Base address of this chunk |
| AddressCount | Number of registers/coils in this chunk request |
| IsWrite | `false` (poller only reads in Beta 0.6) |
| Success | `true` if read succeeded, `false` if Modbus exception |
| ExceptionCode | Modbus exception code if `!Success`, 0 otherwise |
| ResponseTimeUs | `time.Since(txStart).Microseconds()` |
| DeviceID | `asset.ID` (format: "ip:port:unit_id") |
| DeviceName | `asset.ConfigDesc` (human-readable from config) |
| EnvID | `asset.EnvironmentName` |

**FR-5**: When a chunked read returns an error, the event is still recorded with `Success=false`. [OT-REVIEW] Recording failed transactions is essential for OT monitoring. Exception codes reveal device health problems (exception 04 = Slave Device Failure) and potential probing activity (exception 01 = Illegal Function, exception 02 = Illegal Data Address). A monitoring system that only records successes would miss critical diagnostic signals. The implementation must attempt to extract the Modbus exception code from the `simonvetter/modbus` error return. The library wraps Modbus exceptions in a typed error; the implementation agent must inspect the error to extract the exception code (e.g., check if the error string contains "exception" or if it implements an interface exposing the code). If the error is a transport-level error (TCP timeout, connection refused) rather than a Modbus exception, set `ExceptionCode=0` and `Success=false`. If the error is a Modbus exception, set `ExceptionCode` to the extracted code and `Success=false`.

**FR-6**: When an asset is offline (reconnect fails), no events are emitted for that asset. [OT-REVIEW] An offline device does not produce Modbus transactions. The absence of events for a device is itself an observable signal (polling gap anomaly in SOW-032.0). Recording synthetic "offline" events would contaminate the transaction log with non-protocol data.

**FR-7**: `main.go` opens the event store at startup using `eventstore.New(cfg.EventDBPath)`. The store is closed during shutdown, after the poller stops.

**FR-8**: `main.go` starts a pruning goroutine that runs every hour. Each run calls `store.Prune(ctx, time.Now().Add(-retention))` where retention is `cfg.EventRetentionDays * 24 * time.Hour`. The goroutine logs the number of deleted events at `slog.Info` level. The goroutine stops when the context is cancelled.

**FR-9**: `main.go` wires the event store into the poller via `p.SetEventHook(...)`. The hook function calls `store.InsertBatch(ctx, events)` and logs errors at `slog.Warn` level. Insert failures must not crash the monitor or block polling. [OT-REVIEW] Availability over completeness: a monitoring system that crashes because the event store is full or corrupt is worse than one that drops a few events. The poller must continue polling regardless of event store health.

**FR-10**: The SrcAddr field is populated with the monitor's known IP and placeholder port 0. The implementation resolves the IP from the first configured environment's address field (e.g., `"10.10.30.100"` → `"10.10.30.100:0"`). [OT-REVIEW] The `simonvetter/modbus` client does not expose `net.Conn.LocalAddr()`, so the actual ephemeral TCP source port is unavailable. Using port 0 as a placeholder is honest -- it clearly indicates the port is unknown. Using the API port (8091) would be actively misleading because that port never appears in Modbus traffic. The IP portion is accurate because the monitor has a stable Docker network IP. This is acceptable for Beta 0.6 where all events originate from the same monitor instance. The communication graph (SOW-030.0) uses the IP for node identity, not the port.

### Non-Functional Requirements

**NFR-1**: The poller `import` list adds only `eventstore` from the monitoring module. No new external dependencies.

**NFR-2**: Event collection adds negligible overhead to the polling loop. UUID generation and `time.Now()` calls are sub-microsecond. The main cost is the `InsertBatch` call in the hook, which runs after the poll cycle completes (does not increase per-device poll latency).

**NFR-3**: All existing tests continue to pass. New tests cover event collection logic.

**NFR-4**: The EventHook is optional. If `SetEventHook` is never called, the poller operates exactly as before (no events collected, no hook fired).

### Event Collection in pollAsset

The `pollAsset` method currently calls `readHoldingChunked` and `readCoilsChunked` which are standalone functions. To collect per-chunk events, the chunked read functions need to return events alongside their data results.

**Approach**: Create new wrapper methods on Poller that call the existing chunked read functions chunk-by-chunk, measuring timing and creating events per chunk. This avoids modifying the existing function signatures.

```go
// pollAssetWithEvents polls a single asset and returns the collected events.
// This replaces the direct call to pollAsset in the event-collecting code path.
func (p *Poller) pollAssetWithEvents(ctx context.Context, a *inventory.Asset) ([]*eventstore.TransactionEvent, error) {
    client, err := p.pool.get(a.Endpoint)
    if err != nil {
        return nil, fmt.Errorf("get client: %w", err)
    }
    if err := client.SetUnitId(a.UnitID); err != nil {
        return nil, fmt.Errorf("set unit ID %d: %w", a.UnitID, err)
    }

    baseAddr := uint16(0)
    if a.Addressing == "one-based" {
        baseAddr = 1
    }

    var events []*eventstore.TransactionEvent

    // Read holding registers chunk by chunk, creating events.
    holdingEvents, holding, err := p.readHoldingWithEvents(client, baseAddr, a.HoldingRegCount, a)
    events = append(events, holdingEvents...)
    if err != nil {
        return events, fmt.Errorf("read holding registers: %w", err)
    }

    // Read coils chunk by chunk, creating events.
    coilEvents, coils, err := p.readCoilsWithEvents(client, baseAddr, a.CoilCount, a)
    events = append(events, coilEvents...)
    if err != nil {
        return events, fmt.Errorf("read coils: %w", err)
    }

    p.inv.UpdateRegisters(a.ID, holding, coils, time.Now())
    p.inv.SetStatus(a.ID, inventory.StatusOnline, time.Now())
    return events, nil
}
```

### Per-Chunk Event Creation

```go
// readHoldingWithEvents reads holding registers in chunks, creating a
// TransactionEvent for each chunk request. Returns events collected and
// the concatenated register values.
func (p *Poller) readHoldingWithEvents(client *modbus.ModbusClient, baseAddr uint16,
    count int, a *inventory.Asset) ([]*eventstore.TransactionEvent, []uint16, error) {

    if count == 0 {
        return nil, nil, nil
    }

    var events []*eventstore.TransactionEvent
    result := make([]uint16, 0, count)
    addr := baseAddr

    for remaining := count; remaining > 0; {
        qty := remaining
        if qty > modbusMaxRead {
            qty = modbusMaxRead
        }

        txStart := time.Now()
        vals, err := client.ReadRegisters(addr, uint16(qty), modbus.HOLDING_REGISTER)
        txEnd := time.Now()
        responseUs := txEnd.Sub(txStart).Microseconds()

        fc := uint8(3)
        excCode := extractExceptionCode(err) // [OT-REVIEW] Extract Modbus exception code from error
        evt := eventstore.NewEvent(
            txEnd,           // timestamp: transaction completion boundary
            p.srcAddr,       // source: monitor placeholder
            a.Endpoint,      // destination: device endpoint
            a.UnitID,        // Modbus unit/slave ID
            fc,              // function code: Read Holding Registers
            addr,            // address start for this chunk
            uint16(qty),     // address count for this chunk
            false,           // isWrite: poller only reads
            err == nil,      // success
            excCode,         // exception code (0 if transport error or success)
            responseUs,      // response time in microseconds
            a.ID,            // device ID
            a.ConfigDesc,    // device name from config
            a.EnvironmentName, // environment ID
            nil,             // write detail: nil for reads
        )
        if err != nil {
            // Record the failed transaction event and return error.
            events = append(events, evt)
            return events, nil, fmt.Errorf("read at addr %d qty %d: %w", addr, qty, err)
        }

        events = append(events, evt)
        result = append(result, vals...)
        addr += uint16(qty)
        remaining -= qty
    }

    return events, result, nil
}
```

A parallel `readCoilsWithEvents` method follows the same pattern for FC 1, using `modbusMaxCoilRead = 2000` as the chunk limit instead of `modbusMaxRead = 125`.

### Pruner Goroutine

```go
// runPruner deletes events older than the retention window on an hourly schedule.
func runPruner(ctx context.Context, store *eventstore.Store, retentionDays int) {
    ticker := time.NewTicker(1 * time.Hour)
    defer ticker.Stop()

    retention := time.Duration(retentionDays) * 24 * time.Hour

    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            cutoff := time.Now().Add(-retention)
            n, err := store.Prune(ctx, cutoff)
            if err != nil {
                slog.Warn("event pruning failed", "error", err)
                continue
            }
            if n > 0 {
                slog.Info("pruned old events", "deleted", n, "retention_days", retentionDays)
            }
        }
    }
}
```
