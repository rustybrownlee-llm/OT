# SOW-029.0 - Transaction Log Dashboard Page

**Status**: Approved
**Created**: 2026-03-02
**Priority**: High

## 1. Objective

Add an `/events` page to the monitoring dashboard that displays a live-updating, filterable table of Modbus transaction events. This page is the primary interface for security analysts to review protocol telemetry -- the foundational skill for OT network monitoring before commercial tools are deployed.

## 2. Background

SOW-027.0 delivered the `eventstore` package with `TransactionEvent`, `Store`, `FilterOptions`, `Query()`, and `Count()`. SOW-028.0 instruments the poller to populate the event store with one event per Modbus chunk read. This SOW adds the dashboard page that surfaces those events to the analyst.

The milestone spec (Beta 0.6 DoD #1-#3) requires:
1. A live-updating table at `/events` with timestamp, source, destination, function code, register address, value, and response time
2. Filtering by device, function code, time range, and write-only toggle
3. Write operations highlighted as a distinct category

The existing alerts page (`alerts.html` / `alerts_table.html` / `alertsHandler` / `buildAlertsData`) is the closest reference implementation: filterable table with dropdowns, HTMX 5-second polling, and a partial handler for the table body.

## 3. Scope

### In Scope
- `/events` route with full page and HTMX partial
- Filterable event table with columns: Time, Device, FC, Register, Write, Value, Status, RTT
- Filters: device dropdown, FC dropdown, write-only toggle
- Pagination: limit/offset with page navigation
- HTMX auto-refresh (5-second polling)
- Write operations visually highlighted with badge
- Navigation link in "Observed" section after "Process"
- Direct `eventstore.Store` injection into Dashboard (no API endpoint)

### Out of Scope
- Event detail view / modal (deferred to future enhancement)
- Time range picker filter (deferred -- adds date picker JS dependency; basic pagination is sufficient)
- CSV/JSON export (td-events-063)
- Communication graph (SOW-030.0)
- API endpoints on port 8091 for events (can be added when external tool access is needed)

### Functional Requirements

**FR-2**: Register two new routes in `buildRouter()`:
- `GET /events` â†’ `eventsHandler` (full page)
- `GET /partials/events-table` â†’ `eventsTableHandler` (HTMX partial)

**FR-3**: The events table displays the following columns (matching milestone spec):

| Column | Source Field | Format | Notes |
|--------|-------------|--------|-------|
| Time | `Timestamp` | `HH:MM:SS.mmm` | Millisecond precision; tooltip with full RFC3339 |
| Device | `DeviceID` + `UnitID` | Code link | Links to `/assets/{id}`; UnitID shown as tooltip |
| FC | `FunctionCode` + `FunctionName` | `FC3 Holding` | FC prefix + data type (see below) |
| Register | `AddressStart` + `AddressCount` | `40001-40125 (125)` | 4x/1x/0x/3x notation with count (see below) |
| Write | `IsWrite` | Badge | Red "WRITE" badge if true, empty if false |
| Value | `WriteDetail` | Comma values | Written values for writes; "--" for reads |
| Status | `Success` + `ExceptionCode` | Badge | Green "OK" or red "FAIL (Illegal Data Address)" with exception name |
| RTT | `ResponseTimeUs` | `N.Nms` | Microseconds converted to milliseconds |

[OT-REVIEW] FC column uses "FC{N} {DataType}" format (e.g., "FC3 Holding", "FC1 Coils", "FC6 WrReg") rather than truncated function names like "Read Holding". The FC number prefix disambiguates visually; the data type word identifies the Modbus address space. This matches how SCADA HMI diagnostics abbreviate function codes and teaches trainees the critical distinction between Holding Registers (4x, read/write) and Input Registers (3x, read-only). FC short names: FC1 Coils, FC2 Disc.In, FC3 Holding, FC4 Input, FC5 WrCoil, FC6 WrReg, FC15 WrCoils, FC16 WrRegs, FC43 MEI.

[OT-REVIEW] Register column displays addresses in Modbus 4x/1x/0x/3x prefix notation (e.g., "40001-40125 (125)" for FC3, "00001-00100 (100)" for FC1) rather than raw PDU offsets (0-124). Every OT device manual and HMI uses one-based 4x notation. A trainee who learns to read raw PDU offsets will be unable to cross-reference event logs against real device data sheets. The prefix is derived from the function code: FC1â†’0x, FC2â†’1x, FC3â†’4x, FC4â†’3x, FC5â†’0x, FC6â†’4x, FC15â†’0x, FC16â†’4x. The `formatAddrRange` helper computes: `prefix + (addressStart + 1)` through `prefix + (addressStart + addressCount)`.

[OT-REVIEW] UnitID is visible as a tooltip on the Device link. In serial OT networks, multiple physical devices share a single IP:port endpoint and are distinguished only by Modbus Unit ID (slave address). A typical RS-485 bus behind a Moxa NPort gateway might serve Unit IDs 1-8. Training engineers to notice Unit ID is critical for distinguishing attacks targeting specific devices on a shared bus.

[OT-REVIEW] Status column displays exception code names from `eventstore.LookupExceptionCode()` (e.g., "FAIL (Illegal Data Address)") rather than raw numbers ("FAIL (exc 2)"). Exception code 02 is the most common in OT environments and distinguishes misconfigured SCADA polls from attacker probing. Showing the name teaches trainees to recognize exception categories without consulting a reference table.

**FR-4**: Filters parsed from URL query parameters:
- `device` (string): Exact match on `DeviceID`. Populated as `<select>` dropdown from known devices.
- `fc` (string, parsed to uint8): Exact match on `FunctionCode`. Populated as `<select>` dropdown. Each option displays `"FC{N} - {FullName}"` (e.g., `"FC3 - Read Holding Registers"`). The value attribute is the FC number (for URL parameter parsing). This distinguishes FC 1 (Read Coils) from FC 2 (Read Discrete Inputs) -- two codes that IT engineers frequently confuse.
- `write_only` (string "1"): When present, sets `FilterOptions.IsWrite = true`.

Filter values are forwarded to the HTMX polling URL so refresh preserves the active filter state.

**FR-5**: Pagination using `FilterOptions.Limit` and `FilterOptions.Offset`:
- Default page size: 100 events
- Display: "Page X of Y" with Previous/Next links
- Total count from `Store.Count()` using the same filter options
- Page links include current filter parameters

**FR-6**: HTMX auto-refresh. The `events_table` div polls `GET /partials/events-table?...` every 5 seconds with `hx-trigger="every 5s"` and `hx-swap="innerHTML"`. The partial URL includes all current filter and pagination parameters.

[OT-REVIEW] Auto-refresh preserves the current page offset. This means an analyst on page 2 reviewing older events will see page 2 refreshed but will NOT see new events arriving on page 1. This is the correct behavior for analyst-controlled navigation -- jumping back to page 1 on every refresh would make it impossible to review historical events. The trade-off is documented here so the implementor does not "fix" it.

**FR-7**: Write events are visually distinct:
- The "Write" column shows a red Bootstrap badge (`bg-danger`) with text "WRITE" for `IsWrite=true` events
- The "Value" column displays `WriteDetail.Values` as comma-separated uint16 values, or `WriteDetail.CoilValues` as "ON"/"OFF" strings, or "--" for read events

**FR-8**: Add "Events" link to the nav bar in the "Observed" section, after "Process" and before the "Reference" section. ActivePage value: `"events"`.

**FR-9**: Add `events_table.html` to the base template set parsed in `NewDashboard` so the `events_table_content` named template is available for `renderPartial()`.

**FR-10**: The `buildEventsData(r)` function constructs `eventsPageData`:
- Parses query parameters for device, fc, write_only, page
- Builds `eventstore.FilterOptions` from parameters
- Calls `Store.Query()` for the current page of events
- Calls `Store.Count()` for total matching events
- Populates device dropdown options from `APIClient.GetAssets()` (reuses existing API pattern for device list)
- Populates FC dropdown options from `eventstore.AllFuncCodes()` (new exported function; see FR-10 prerequisite below)

### Non-Functional Requirements

**NFR-1**: Page load time under 500ms for 100 events (SQLite indexed query on timestamp DESC).

**NFR-2**: No JavaScript dependencies beyond HTMX (already loaded via CDN). No date picker libraries.

**NFR-3**: Template follows existing patterns: `events.html` defines `{{define "content"}}`, `events_table.html` defines `{{define "events_table_content"}}`.

## 5. Design

### Dashboard Struct Change

```go
type Dashboard struct {
    api    *APIClient
    lib    *DesignLibrary
    events *eventstore.Store  // NEW: direct SQLite access for event queries
    router chi.Router
    base   *template.Template
}
```

`NewDashboard` gains a third parameter: `func NewDashboard(api *APIClient, lib *DesignLibrary, events *eventstore.Store) *Dashboard`.

[OT-REVIEW] Direct store injection (rather than HTTP API loopback) is appropriate here because:
1. The event store uses SQLite WAL mode, designed for concurrent readers
2. The dashboard and store run in the same process
3. Avoiding HTTP serialization for local data is both faster and simpler
4. External tool access to events is through syslog forwarding (SOW-031.0), not HTTP API

### Data Types

```go
// eventsPageData is the template data for the /events page.
type eventsPageData struct {
    Title          string
    ActivePage     string
    Events         []eventRow
    EventsAvailable bool
    TotalCount     int64
    Page           int
    PageSize       int
    TotalPages     int
    FilterDevice   string
    FilterFC       string
    FilterWrite    bool
    DeviceOptions  []string
    FCOptions      []fcOption
}

// eventRow is a display-ready event row for the template.
type eventRow struct {
    ID            string
    TimeShort     string  // "14:23:05.123"
    TimeFull      string  // RFC3339 for tooltip
    DeviceID      string
    DeviceName    string
    UnitID        uint8   // Modbus slave address; shown as tooltip on device link
    FC            uint8
    FCName        string  // "Read Holding Registers" (full name for tooltip)
    FCShort       string  // "FC3 Holding" (abbreviated for column display)
    AddrRange     string  // "40001-40125 (125)" (4x/1x/0x/3x notation)
    IsWrite       bool
    WriteValues   string  // "1234, 5678" or "ON, OFF" or "--"
    Success       bool
    ExceptionName string  // "" or "Illegal Data Address" (from LookupExceptionCode)
    RTTms         string  // "1.2ms"
}

// fcOption represents a function code dropdown option.
type fcOption struct {
    Code uint8
    Name string
}
```
