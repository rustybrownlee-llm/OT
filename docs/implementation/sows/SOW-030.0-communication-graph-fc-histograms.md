# SOW-030.0 - Communication Graph and Function Code Histograms

**Status**: Complete
**Created**: 2026-03-02
**Priority**: High

## 1. Objective

Add a `/comms` page displaying a device communication matrix (who-talks-to-whom) and a per-device function code distribution histogram on the asset detail page. These views teach IT engineers to recognize normal communication patterns and identify anomalous protocol behavior -- foundational skills for OT security monitoring.

## 2. Background

SOW-027.0 delivered the event store with Query/Count. SOW-028.0 populates it with per-chunk Modbus events. SOW-029.0 adds the event table page. This SOW provides the analytical views that aggregate event data into communication patterns and protocol usage statistics.

The milestone spec (Beta 0.6 DoD #4-#5) requires:
4. A communication graph (who-talks-to-whom matrix) at `/comms` showing which devices communicate and how frequently
5. A per-device function code distribution histogram on the asset detail page showing the breakdown of read vs. write vs. diagnostic function codes

[OT-REVIEW] In Beta 0.6, all traffic is poller→device (the monitor is the only Modbus client). The communication matrix is therefore a star topology: monitor → each device. This is still educationally valuable -- it establishes the baseline communication pattern that a passive monitoring tool would observe. In Beta 0.7 (passive capture), third-party clients including simulated attackers will produce genuine multi-source traffic, making the matrix significantly more interesting.

## 3. Scope

### In Scope
- `/comms` route with device communication matrix table
- Per-device function code distribution on `/assets/{id}` detail page
- Aggregate SQL queries in eventstore for communication and FC statistics
- "Comms" link in nav bar "Observed" section after "Events"
- HTMX auto-refresh for live updates

### Out of Scope
- Force-directed graph visualization (D3/SVG) -- deferred per milestone spec
- Cross-device correlation rules (Beta 0.7+)
- Source address analysis (meaningful only with passive capture in Beta 0.7)
- Communication graph export (CSV, JSON)

## 4. Requirements

### Functional Requirements

**FR-1**: Add three aggregate query methods to `eventstore.Store`:

```go
// DeviceStats returns per-device aggregate statistics from the event store.
func (s *Store) DeviceStats(ctx context.Context) ([]DeviceStat, error)

// CommEdges returns communication edge statistics (source → destination).
func (s *Store) CommEdges(ctx context.Context) ([]CommEdgeStat, error)

// FCDistribution returns function code distribution for a single device.
func (s *Store) FCDistribution(ctx context.Context, deviceID string) ([]FCCount, error)
```

**FR-2**: Aggregate data types in eventstore:

```go
// DeviceStat holds per-device aggregate statistics.
type DeviceStat struct {
    DeviceID    string
    DeviceName  string
    TotalEvents int64
    WriteEvents int64
    LastSeen    time.Time
    DistinctFCs []uint8  // distinct function codes observed for this device
}

// CommEdgeStat holds communication edge statistics.
// [OT-REVIEW] CommEdges() is implemented for Beta 0.7 forward compatibility.
// In Beta 0.6, all traffic originates from the monitor (single source), so
// the /comms page uses DeviceStats() instead. When passive capture introduces
// multi-source traffic in Beta 0.7, CommEdges() will power the NxN matrix.
type CommEdgeStat struct {
    SrcAddr       string
    DstDeviceID   string
    DstDeviceName string
    EventCount    int64
    WriteCount    int64
    LastEvent     time.Time
}

// FCCount holds the count of events for a single function code.
type FCCount struct {
    FuncCode uint8
    FuncName string
    Count    int64
    IsWrite  bool
}
```

[OT-REVIEW] All aggregate types (`DeviceStat`, `CommEdgeStat`, `FCCount`) are defined in `eventstore_aggregate.go` alongside their query methods, NOT in `event.go`. This keeps the aggregate query layer self-contained and avoids polluting the core event model.

**FR-3**: Register new routes in `buildRouter()`:
- `GET /comms` → `commsHandler` (full page)
- `GET /partials/comms-table` → `commsTableHandler` (HTMX partial)

**FR-4**: Communication matrix page (`/comms`):
- Header: "Communications" with subtitle explaining the matrix
- Matrix rendered as a Bootstrap table with devices as rows
- Columns: Device, Total Events, Write Events, Last Seen, FC Codes
- Write count highlighted with badge when > 0
- [OT-REVIEW] FC codes column shows distinct function codes observed per device (e.g., "FC1, FC3" for a read-only device, "FC3, FC6, FC16" for a read-write device). This immediately reveals which devices receive write operations -- the highest-risk indicator in an OT environment.
- HTMX auto-refresh every 10 seconds (slower than events page -- aggregate queries are heavier)

**FR-5**: Per-device FC distribution on asset detail page (`/assets/{id}`):
- New card below "Device Metadata" in the left column (or a new row below)
- Title: "Function Code Distribution"
- Bootstrap progress bars showing proportion of each function code
- Each bar labeled with FC short name and count (e.g., "FC3 Holding: 1,234 (95%)")
- Write function codes colored red (`bg-danger`); read function codes colored blue (`bg-primary`); diagnostic green (`bg-success`)
- [OT-REVIEW] The histogram visually distinguishes read-heavy devices (expected: sensors, flow computers) from write-capable devices (expected: PLCs with setpoint updates). A device that was read-only during the learning period but now shows write FCs is a critical anomaly indicator.

**FR-6**: Add "Comms" link to nav bar in "Observed" section, after "Events". ActivePage value: `"comms"`.

**FR-7**: SQL queries use GROUP BY aggregation, not full-table scans:

DeviceStats query:
```sql
SELECT device_id, MAX(device_name) as device_name, COUNT(*) as total_events,
       SUM(CASE WHEN is_write = 1 THEN 1 ELSE 0 END) as write_events,
       MAX(timestamp) as last_seen
FROM events
GROUP BY device_id
ORDER BY device_id
```

[OT-REVIEW] `device_name` requires MAX() because it is not in the GROUP BY clause. In practice, device_name is functionally dependent on device_id (same value for all rows), but strict SQL semantics require an aggregate function. MAX() returns the correct value.

DeviceStats distinct FC sub-query (populate `DistinctFCs` field):
```sql
SELECT DISTINCT func_code FROM events WHERE device_id = ? ORDER BY func_code
```

This runs once per device after the main DeviceStats query. For the typical 5-15 device count in Beta 0.6, this is negligible overhead.

CommEdges query:
```sql
SELECT src_addr, device_id, MAX(device_name) as device_name,
       COUNT(*) as event_count,
       SUM(CASE WHEN is_write = 1 THEN 1 ELSE 0 END) as write_count,
       MAX(timestamp) as last_event
FROM events
GROUP BY src_addr, device_id
ORDER BY event_count DESC
```

FCDistribution query:
```sql
SELECT func_code, MAX(func_name) as func_name, COUNT(*) as count,
       MAX(is_write) as is_write
FROM events
WHERE device_id = ?
GROUP BY func_code
ORDER BY count DESC
```

[OT-REVIEW] FCDistribution groups by `func_code` only. `func_name` and `is_write` are functionally dependent on `func_code` (one-to-one mapping from the Modbus specification), so MAX() returns the correct value. Grouping by all three would produce identical results but obscures the intent.

**FR-8**: FC distribution data is passed to the asset detail template via the existing `assetDetailData` struct (add a `FCDistribution []FCCount` field). The `buildAssetDetailData` method calls `Store.FCDistribution()` when the store is non-nil.

**FR-9**: Add `comms_table.html` to the base template set parsed in `NewDashboard` for HTMX partial rendering.

### Non-Functional Requirements

**NFR-1**: Aggregate queries complete in under 2 seconds for 7 days of data (~3.6M rows). SQLite GROUP BY on indexed columns (device_id, is_write) should meet this.

**NFR-2**: No JavaScript dependencies. FC histogram uses Bootstrap progress bars (CSS only).

**NFR-3**: Templates follow existing patterns: `comms.html` defines `{{define "content"}}`, `comms_table.html` defines `{{define "comms_table_content"}}`.

## 5. Design

### Event Store Additions

New methods added to `eventstore.Store` in a new file `eventstore_aggregate.go` to keep aggregate queries separate from the core CRUD operations.

### Dashboard Integration

The Dashboard already has `events *eventstore.Store` (added in SOW-029.0). The comm graph and FC histogram queries use the same store reference. No additional constructor changes needed.

### Comm Matrix Layout

The communication matrix is a simple table (not a cross-reference matrix) because in Beta 0.6 there is only one source (the monitor). Each row represents a device with its aggregate statistics. This layout is directly upgradeable to a full NxN matrix in Beta 0.7 when passive capture introduces multiple sources.

```
| Device                    | Events | Writes | Last Seen           | FC Codes       |
|---------------------------|--------|--------|---------------------|----------------|
| 10.10.10.5:5020:1 (PLC)  | 12,345 | 0      | 2026-03-02 14:23:05 | FC1, FC3       |
| 10.10.10.5:5021:1 (RTU)  | 8,901  | 234    | 2026-03-02 14:23:04 | FC3, FC6, FC16 |
```

### FC Histogram Layout

Bootstrap progress bars stacked vertically in a card on the asset detail page:


For a device with writes:

### Handler Flow

```
GET /comms
  → commsHandler(w, r)
    → buildCommsData()
      → Store.DeviceStats(ctx)
      → enrichDeviceNames (from asset list)
    → render("comms.html", data)

GET /assets/{id}
  → assetDetailHandler(w, r)  [existing]
    → buildAssetDetailData(id) [modified]
      → Store.FCDistribution(ctx, id)  [NEW]
    → render("asset_detail.html", data)
```
