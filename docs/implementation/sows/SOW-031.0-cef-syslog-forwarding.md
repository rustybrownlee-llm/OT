# SOW-031.0 - CEF Syslog Forwarding

**Status**: Approved
**Created**: 2026-03-02
**Priority**: Medium

## 1. Objective

Add CEF (Common Event Format) syslog forwarding so that Modbus transaction events can be sent to external SIEM platforms (Splunk, QRadar, Elastic). This teaches IT engineers how OT monitoring tools integrate with existing security infrastructure -- a critical skill for SOC analysts supporting converged IT/OT environments.

## 2. Background

SOW-027.0 delivered the `TransactionEvent` type and event store. SOW-028.0 populates the store via the poller EventHook. This SOW adds a syslog emitter that receives the same events and forwards them as CEF-formatted syslog messages over UDP or TCP.

The milestone spec (Beta 0.6 DoD #7) requires:
7. Configure syslog forwarding in `monitor.yaml` and see Modbus transaction events appear in CEF format on an external syslog receiver, ready for import into Splunk, QRadar, or Elastic

CEF (Common Event Format) is an industry-standard event format developed by ArcSight (now Micro Focus/OpenText). It is widely supported by SIEM platforms and is the natural choice for OT security event forwarding because it provides structured key-value extensions alongside a standardized header.

[OT-REVIEW] CEF over plaintext UDP/TCP is acceptable for lab environments (td-events-061). Production OT environments would use TLS-encrypted syslog (RFC 5425) or a dedicated log shipper (Filebeat, Fluentd). The educational value is in the CEF format itself, not the transport security.

## 3. Scope

### In Scope
- New `syslog` package with CEF formatter and UDP/TCP sender
- Configuration in `monitor.yaml` (syslog section)
- Config parsing and validation
- Integration via EventHook pipeline in main.go
- Severity mapping: writes=7, read failures=5, diagnostics=3, read successes=1

### Out of Scope
- TLS-encrypted syslog (RFC 5425) -- deferred per milestone spec
- LEEF format (IBM QRadar native -- CEF is also supported)
- JSON syslog format (deferred; CEF is sufficient for Beta 0.6)
- Buffering/retry for dropped UDP packets
- Syslog health monitoring or delivery confirmation

## 4. Requirements

### Functional Requirements

**FR-1**: Add syslog configuration to `config.Config`:

```go
// SyslogConfig holds syslog forwarding configuration.
type SyslogConfig struct {
    Enabled  bool   `yaml:"enabled"`   // default: false
    Target   string `yaml:"target"`    // e.g., "localhost:514" (host:port, NOT a URL)
    Protocol string `yaml:"protocol"`  // "udp" or "tcp"
    Facility string `yaml:"facility"`  // syslog facility name (default: "local0")
    Format   string `yaml:"format"`    // "cef" (only supported format in Beta 0.6)
}
```

YAML structure:
```yaml
syslog:
  enabled: false
  target: "localhost:514"
  protocol: "udp"
  facility: "local0"
  format: "cef"
```

[OT-REVIEW] The `target` field uses plain `host:port` format, NOT a URL. The milestone spec shows `"udp://localhost:514"` which is an IT log-shipper convention (Logstash/Fluentd). Real syslog configuration (rsyslog, syslog-ng) separates transport from address. Validation rejects `://` in the target with a descriptive error. The `format` field is included for forward compatibility (JSON format in a future milestone) and currently only accepts `"cef"`.

**FR-2**: Create `monitoring/internal/syslog/` package with:
- `Emitter` struct managing the network connection
- `New(cfg SyslogConfig) (*Emitter, error)` -- opens UDP/TCP connection
- `Send(events []*eventstore.TransactionEvent) error` -- formats and sends all events
- `Close() error` -- closes the connection

**FR-3**: CEF format per the milestone spec. Each event produces one syslog line:

```
<facility*8+severity>CEF:0|OTSimulator|Monitor|0.6|{func_code}|{func_name}|{cef_severity}|src={src_addr} dst={dst_addr} cs1={func_name} cs1Label=FunctionCode cn1={addr_start} cn1Label=AddressStart cn2={addr_count} cn2Label=AddressCount cs2={device_id} cs2Label=DeviceID cs3={env_id} cs3Label=Environment rt={timestamp_ms} outcome={success|failure}
```

[OT-REVIEW] CEF field names updated from milestone spec: `cn1Label=AddressStart` and `cn2Label=AddressCount` (not RegisterStart/RegisterCount) to match the delivered event model's protocol-neutral naming. SIEM correlation rules referencing these fields must use the correct names.

**FR-4**: CEF severity mapping:
- Write operations (`IsWrite=true`): 7 (High) -- regardless of success/failure
- Diagnostic operations (FC 43, `category=diagnostic`): 3 (Medium-Low)
- Read failures (`Success=false`): 5 (Medium)
- Read successes: 1 (Low/Informational)

[OT-REVIEW] Write operations are always High severity regardless of success/failure because the act of writing to an OT device has physical consequences. A failed write attempt (exception code response) is still a high-severity event -- it indicates someone tried to modify a process control register. Read failures are Medium because they may indicate device probing or misconfiguration.

[OT-REVIEW] Diagnostic function codes (FC 43 MEI Transport / Device Identification) are severity 3 -- above Informational but below read failures. In the TRITON/TRISIS attack context, FC 43 from an unknown source is a high-fidelity indicator of compromise (device enumeration phase). If diagnostic events were severity 1 (Informational), trainees building SIEM alert rules would never flag them. Severity 3 ensures they appear in standard threat-hunting dashboards.

**FR-5**: Syslog facility mapping. The `facility` config field maps to the standard syslog facility code:
- `"local0"` → 16 (default)
- `"local1"` through `"local7"` → 17-23
- Other standard facilities accepted but not recommended for OT events

**FR-6**: Wire the syslog emitter into the EventHook pipeline in `main.go`. When syslog is enabled:
- The EventHook calls `store.InsertBatch()` AND `emitter.Send()` for each batch
- Syslog emission errors are logged at Warn level (non-fatal; event store insertion is the primary persistence)
- The emitter is created after config parse and closed during shutdown

**FR-7**: Validate syslog configuration:
- When `enabled: true`, `target` must be non-empty
- `target` must NOT contain `"://"` -- reject with error: "target must be host:port format, not a URL (e.g., 'localhost:514' not 'udp://localhost:514')"
- `protocol` must be `"udp"` or `"tcp"` (default: `"udp"`)
- `facility` must be a recognized syslog facility name (default: `"local0"`)
- `format` must be `"cef"` (default: `"cef"`; error on any other value)

### Non-Functional Requirements

**NFR-1**: Syslog emission must not block the polling cycle for more than 10ms per batch. UDP is non-blocking by nature. TCP may block; the emitter uses a write timeout.

**NFR-2**: No external syslog libraries. Use standard library `net.Dial` for UDP/TCP.

**NFR-3**: CEF output must be parseable by Splunk's CEF add-on and Elastic's CEF pipeline without configuration changes.

## 5. Design

### Package Structure

```
monitoring/internal/syslog/
    emitter.go     -- Emitter type, New, Send, Close
    cef.go         -- CEF formatting logic
    facility.go    -- syslog facility name → code mapping
    emitter_test.go
    cef_test.go
```

### CEF Message Construction

The CEF format has a fixed header structure:
```
CEF:0|{vendor}|{product}|{version}|{signatureId}|{name}|{severity}|{extensions}
```

For Modbus transaction events:
- vendor: `OTSimulator`
- product: `Monitor`
- version: `0.6`
- signatureId: function code number (e.g., `3`, `6`, `16`)
- name: function code name (e.g., `Read Holding Registers`)
- severity: 1, 3, 5, or 7 per FR-4
- extensions: key=value pairs separated by spaces

### Syslog Framing

UDP: One UDP datagram per CEF message. No framing delimiter needed.
TCP: Each message terminated with newline `\n` (RFC 6587 non-transparent framing).

### Syslog Priority

Priority = facility * 8 + severity_code

Syslog severity mapping (CEF severity → syslog severity):
- CEF 7 (High / Writes) → syslog 2 (Critical)
- CEF 5 (Medium / Read failures) → syslog 4 (Warning)
- CEF 3 (Medium-Low / Diagnostics) → syslog 5 (Notice)
- CEF 1 (Low / Read successes) → syslog 6 (Informational)

[OT-REVIEW] The CEF-to-syslog severity mapping is documented explicitly in `cef.go` comments, not left as magic numbers. Trainees and future maintainers must understand that writes at syslog Critical (2) is intentional for OT monitoring -- writes to process control devices warrant pager-level alerting.

### main.go Integration

```go
// When syslog enabled:
var emitter *syslog.Emitter
if cfg.Syslog.Enabled {
    emitter, err = syslog.New(cfg.Syslog)
    // handle error
    defer emitter.Close()
}

p.SetEventHook(func(events []*eventstore.TransactionEvent) {
    // Primary: persist to event store
    if err := store.InsertBatch(ctx, events); err != nil {
        slog.Warn("event store insert failed", "error", err)
    }
    // Secondary: forward to syslog
    if emitter != nil {
        if err := emitter.Send(events); err != nil {
            slog.Warn("syslog send failed", "error", err)
        }
    }
})
```

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| CEF format rejected by Splunk/Elastic parser | Events not ingested by SIEM | Low | Test against Splunk CEF add-on parsing rules; use standard CEF 0 format |
| TCP connection drops during send | Events lost | Medium | Log at Warn; do not retry (event store is the primary record). Future: add reconnect logic |
| UDP packet size exceeds MTU | Truncated messages | Low | CEF messages for Modbus events are ~300 bytes, well under 1500-byte MTU |
| main.go EventHook grows complex with store + syslog | Hard to maintain | Medium | Extract into a named function: `makeEventHook(store, emitter)` |
