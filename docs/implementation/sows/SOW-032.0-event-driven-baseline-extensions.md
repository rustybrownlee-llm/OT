# SOW-032.0 - Event-Driven Baseline Extensions

**Status**: Approved
**Created**: 2026-03-02
**Priority**: High

## 1. Objective

Extend the baseline engine to consume `TransactionEvent` data and detect four new anomaly categories: write-to-readonly device, unauthorized communication source, unusual function code, and polling interval anomaly. These event-driven rules complement the existing register-value baselines and teach IT engineers how OT monitoring tools build behavioral profiles from protocol metadata -- not just data values.

## 2. Background

The existing baseline engine (SOW-013.0) learns per-device register/coil statistical profiles and detects value anomalies. It operates on `DeviceSnapshot` structs delivered via the CycleHook. SOW-027.0/028.0 introduced `TransactionEvent` data containing protocol-level metadata (function codes, source addresses, response times) that the snapshot-based engine cannot see.

The milestone spec (Beta 0.6 DoD #8-#9) requires:
8. See the baseline engine consume structured transaction events instead of raw register snapshots, with the same anomaly detection quality (zero regression)
9. See a new alert type: "Write to read-only device" -- triggered when any Modbus write operation targets a device that was baselined as read-only during the learning period

[OT-REVIEW] The existing five snapshot-based rules remain unchanged. The four new event-driven rules are additive. This preserves the existing detection capability (zero regression) while adding protocol-metadata awareness. In production OT monitoring, the protocol-metadata layer (who talks to whom, using which function codes) is often more valuable than register-value analysis because it detects attacker reconnaissance and unauthorized commands before process values change.

## 3. Scope

### In Scope
- Four new event-driven anomaly rules in the baseline engine
- Event baseline learning: track observed function codes, source addresses, and polling intervals per device during the learning period
- Integration: Engine receives `TransactionEvent` slices via a new `RecordEvents` method
- Wiring: main.go EventHook calls both `store.InsertBatch()` and `engine.RecordEvents()`
- New alert types: `write_to_readonly`, `new_source`, `fc_anomaly`, `poll_gap`
- Constraint: every device in the event stream must also appear in the snapshot polling cycle (event-only devices will learn but never finalize)

[OT-REVIEW] Rule IDs use underscores (`write_to_readonly`, not `write-to-readonly`) to match the existing five rules (`value_out_of_range`, `unexpected_write`, `new_device`, `device_offline`, `response_time_anomaly`). The milestone spec table uses hyphens; the milestone spec will be updated to match the implementation convention. Consistent rule ID format is critical for SIEM correlation rules and trainee confidence.

### Out of Scope
- Modifying existing snapshot-based rules (value_out_of_range, unexpected_write, etc.)
- Cross-device correlation (e.g., "if device A gets a write, check if device B also changed")
- Sliding window baselines (learning period is fixed, per td-baseline-029)
- Alert persistence (alerts remain in-memory, per td-alert-021)
- Source address enrichment (SrcAddr is always the monitor in Beta 0.6; meaningful in Beta 0.7)

## 4. Requirements

### Functional Requirements

**FR-1**: Add event baseline state to `DeviceBaseline`:

```go
// EventBaseline holds protocol-metadata baselines learned from TransactionEvents.
// Populated during the learning period, frozen at transition to established.
type EventBaseline struct {
    ObservedFCs    map[uint8]bool   // function codes seen during learning
    ObservedSrcs   map[string]bool  // source addresses seen during learning
    IsWriteTarget  bool             // true if any write FC was observed during learning
    PollIntervals  []time.Duration  // intervals between consecutive events (for mean/stddev)
    PollIntervalMean   time.Duration
    PollIntervalStdDev time.Duration
    LastEventTime  time.Time        // timestamp of most recent event (for interval calc)
}
```

**FR-2**: Add `RecordEvents(events []*eventstore.TransactionEvent)` method to `Engine`. This is the event-driven counterpart to `RecordCycle()`. Processing per event:
- Route to the correct device baseline via `e.DeviceID` using a new `ensureEventBaseline(deviceID string) *DeviceBaseline` helper. This helper differs from `getOrCreateBaseline(snap)` because it does not require a `DeviceSnapshot`; it initializes only the `EventBaseline` maps (ObservedFCs, ObservedSrcs) and sets Status to "learning" with the configured RequiredSamples. If the device already has a baseline (created by `RecordCycle`), the helper returns it.
- If learning: update `EventBaseline` (add FC to set, add source to set, compute polling interval)
- If established: evaluate event-driven rules

**FR-3**: Event-driven rule: `write_to_readonly` (Critical severity)
- Trigger: `IsWrite=true` AND `EventBaseline.IsWriteTarget=false`
- Fires regardless of `Success` value -- a failed write attempt (exception code response) is still Critical because it indicates someone tried to modify a process control register
- Alert description includes whether the write succeeded or failed (response procedure differs)
- Meaning: A write operation (FC 5/6/15/16) targeted a device that only received read function codes during the entire learning period
- Alert ID: `alert.AlertID("write_to_readonly", deviceID, -1)`
- [OT-REVIEW] This is the single most important detection rule for OT security. A device that was read-only during normal operations (sensor, flow meter, read-only PLC program) should never receive write commands. This rule catches both accidental SCADA misconfigurations (writing to the wrong device) and deliberate attacks (modifying process control registers). The Critical severity is justified: any write to a read-only device has potential physical consequences. A failed write (device returned exception code 01 "Illegal Function") means the hardware protection worked but someone still tried.
- [OT-REVIEW] `IsWriteTarget` is a coarse filter: it collapses all write types (FC 5 coils, FC 6/16 registers) into a single boolean. A device that received FC 5 (Write Single Coil) during learning but later receives FC 16 (Write Multiple Registers) will NOT trigger `write_to_readonly` because `IsWriteTarget=true`. This case is caught instead by `fc_anomaly` (FR-5), which detects any function code not seen during learning. FC 5 writes to the 0x (coil) address space; FC 16 writes to the 4x (register) address space -- these are physically different operations. The cross-reference between the two rules is deliberate: `write_to_readonly` catches the broad case (any write to a never-written device); `fc_anomaly` catches the narrow case (a new write type on a device that already receives some writes).

**FR-4**: Event-driven rule: `new_source` (High severity)
- Trigger: `e.SrcAddr` not in `EventBaseline.ObservedSrcs`
- Meaning: A transaction originated from a source IP:port not seen during learning
- Alert ID: `alert.AlertID("new_source", deviceID, -1)`
- The new source is added to the observed set after the first alert to prevent re-alerting
- [OT-REVIEW] In Beta 0.6, SrcAddr is always the monitor process (single Modbus client). This rule will only fire if the monitor's ephemeral port changes (e.g., after restart). In Beta 0.7 (passive capture), this becomes a high-value detection: a new IP addressing a PLC that was only ever talked to by the SCADA server is a strong indicator of compromise. The rule is implemented now for forward compatibility and to establish the learning mechanism.

**FR-5**: Event-driven rule: `fc_anomaly` (High severity)
- Trigger: `e.FunctionCode` not in `EventBaseline.ObservedFCs`
- Meaning: A function code not observed during the learning period appeared
- Alert ID: `alert.AlertID("fc_anomaly", deviceID, -1)`
- The new FC is added to the observed set after the first alert to prevent re-alerting
- [OT-REVIEW] FC anomaly detection catches reconnaissance (FC 43 on a device that only received FC 3/4) and unauthorized write attempts (FC 6 on a device that only received FC 3). Unlike write_to_readonly, this rule also catches diagnostic FCs and read FCs not seen during learning (e.g., FC 1 Read Coils on a device that only had FC 3 Read Holding Registers during learning). The alert description includes the FC name for SIEM integration.

**FR-6**: Event-driven rule: `poll_gap` (Warning severity)
- Trigger: Time since `EventBaseline.LastEventTime` exceeds `3 * PollIntervalMean`
- Guard: Skip evaluation entirely if `PollIntervalMean == 0` (device had too few valid intervals during learning). This prevents infinite false positives on devices with very slow polling.
- Meaning: No transaction was seen for this device within the expected polling window
- Alert ID: `alert.AlertID("poll_gap", deviceID, -1)`
- Clear path: When a new event arrives and `interval <= 3 * PollIntervalMean`, call `store.Clear(alertID, e.Timestamp)`. This is analogous to how `evaluateDeviceOffline` clears when `snap.Online=true`. Without explicit clearing, the alert would persist permanently after the first gap even when normal polling resumes.
- [OT-REVIEW] Polling interval anomalies indicate network congestion, device failure, or an attacker disrupting communications (e.g., ARP spoofing to redirect traffic). The 3x threshold is conservative; production tools use tighter thresholds (1.5x-2x) but for an educational tool with variable polling jitter, 3x avoids false positives.

**FR-7**: Compute `PollIntervalMean` and `PollIntervalStdDev` at baseline finalization. Use the same Welford's algorithm pattern as response time statistics. Ignore the first event (no interval yet) and events where the interval exceeds 60 seconds (likely a restart, not a legitimate gap). After finalization, set `eb.PollIntervals = nil` to release memory (same pattern as `sampleAccum` cleanup in `finaliseBaseline`).

**FR-8**: Wire `engine.RecordEvents()` into the EventHook in `main.go`:

```go
p.SetEventHook(func(events []*eventstore.TransactionEvent) {
    // Primary: persist to event store
    if err := store.InsertBatch(ctx, events); err != nil {
        slog.Warn("event store insert failed", "error", err)
    }
    // Secondary: forward to syslog (if enabled)
    if emitter != nil {
        if err := emitter.Send(events); err != nil {
            slog.Warn("syslog send failed", "error", err)
        }
    }
    // Tertiary: feed baseline engine
    engine.RecordEvents(events)
})
```

**FR-9**: Add the four new rules to the `Rules` slice in `rules.go`:

```go
{ID: "write_to_readonly", Severity: "critical"},
{ID: "new_source", Severity: "high"},
{ID: "fc_anomaly", Severity: "high"},
{ID: "poll_gap", Severity: "warning"},
```

**FR-10**: Update `GetBaselines()` and `GetDeviceBaseline()` to include the `EventBaseline` in the returned copy. The API `/api/baselines` endpoint exposes the event baseline state without code changes (the serialization follows the struct).

### Non-Functional Requirements

**NFR-1**: `RecordEvents()` must complete in under 5ms for a batch of 100 events. All operations are in-memory map lookups and simple comparisons.

**NFR-2**: Zero regression on existing rules. All existing tests must continue to pass without modification. The five snapshot-based rules are completely independent of the event-driven rules.

**NFR-3**: Event-driven rules fire only after the baseline transitions to "established". During learning, events are accumulated but no alerts are generated.

## 5. Design

### Engine Extension

The `Engine` struct gains no new fields. The `DeviceBaseline` struct gains an `EventBaseline` sub-struct. Event processing is in a new file `event_rules.go` to maintain separation from snapshot-based rules.

### Learning Phase

During learning, `RecordEvents()` updates the `EventBaseline`:
1. Adds `e.FunctionCode` to `ObservedFCs` set
2. Adds `e.SrcAddr` to `ObservedSrcs` set
3. If `e.IsWrite`, sets `IsWriteTarget = true`
4. Computes interval from `LastEventTime` to `e.Timestamp`, appends to `PollIntervals`
5. Updates `LastEventTime`

### Finalization

When `finaliseBaseline()` runs (triggered by `SampleCount >= RequiredSamples`), it also finalizes the event baseline:
1. Compute `PollIntervalMean` and `PollIntervalStdDev` from accumulated intervals
2. The `ObservedFCs`, `ObservedSrcs`, and `IsWriteTarget` fields are already complete

### Established Phase

After finalization, each call to `RecordEvents()` evaluates the four event-driven rules:

```
For each event in batch:
  interval = e.Timestamp - LastEventTime
  update LastEventTime = e.Timestamp

  1. write_to_readonly: if IsWrite && !EventBaseline.IsWriteTarget → Critical
     (description includes Success/failure status for triage)
  2. fc_anomaly: if FC not in ObservedFCs → High (then add FC to set)
  3. new_source: if SrcAddr not in ObservedSrcs → High (then add SrcAddr to set)
  4. poll_gap:
     if PollIntervalMean == 0 → skip (no valid intervals learned)
     if interval > 3 * PollIntervalMean → Warning (fire/keep alert)
     if interval <= 3 * PollIntervalMean → Clear poll_gap alert
```

### Alert ID Determinism

All four rules use `alert.AlertID(ruleID, deviceID, -1)` for device-level alerts (no register address). This follows the same pattern as `device_offline` and `new_device`.

### main.go EventHook Evolution

The EventHook in main.go now has three consumers:
1. `store.InsertBatch()` -- event persistence
2. `emitter.Send()` -- syslog forwarding (if enabled, from SOW-031.0)
3. `engine.RecordEvents()` -- baseline event learning/evaluation

The hook calls all three sequentially. Since SOW-031.0 may not be implemented before SOW-032.0, the syslog call is conditional (only when emitter is non-nil).

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Circular import: baseline imports eventstore | Build fails | Medium | eventstore has no baseline imports; direction is baseline → eventstore, which is safe |
| EventHook grows to 3 consumers (store, syslog, engine) | Complexity in main.go | Medium | Extract to `makeEventHook(store, emitter, engine)` helper |
| RecordEvents called before any CycleHook (event baseline missing device) | Nil pointer | Low | RecordEvents creates EventBaseline on first event via getOrCreateBaseline |
| poll_gap rule fires during startup (first event has no interval) | False positive | Low | Skip interval computation for first event per device |
| SrcAddr always same in Beta 0.6 (monitor IP) | new_source rule never fires | Expected | Documented; rule is implemented for Beta 0.7 forward compatibility |
