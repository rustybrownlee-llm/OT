# SOW-033.0 - Scenario 04 Phase A: Deploy Passive Monitoring

**Status**: Draft
**Created**: 2026-03-02
**Priority**: High

## 1. Objective

Author Scenario 04 Phase A educational content that guides IT engineers through deploying the monitoring telemetry stack, building a traffic baseline, identifying normal communication patterns, and detecting a simulated unauthorized write. This is the capstone exercise for Beta 0.6, bringing together the transaction event model, communication graph, function code analysis, CEF syslog forwarding, and event-driven baseline alerts into a coherent learning experience.

## 2. Background

Scenarios 01-03 cover discovery, vulnerability assessment, and attack simulation. Scenario 04 ("Defense and Security Overlay Deployment") is a multi-phase exercise spanning multiple milestones:

- **Phase A** (Beta 0.6): Deploy the built-in monitoring telemetry stack (this SOW)
- **Phase B** (Beta 0.7): Deploy Cisco CyberVision passive monitoring
- **Phase C** (Beta 0.8): Deploy Dragos Platform threat detection
- **Phase D** (Beta 0.9): Deploy Blastwave BlastShield microsegmentation

Phase A teaches the foundational monitoring skills that all subsequent phases build upon. The engineer learns to manually observe, baseline, and detect OT protocol anomalies using the built-in tools -- understanding *what* commercial tools automate before deploying them.

The milestone spec (Beta 0.6 DoD #10-#11) requires:
10. Open Scenario 04 Phase A and follow a guided exercise that uses the new telemetry capabilities to build a traffic baseline, identify communication patterns, and detect a simulated unauthorized write
11. Understand from the scenario materials why this manual telemetry layer exists and what commercial tools (Cisco CyberVision, Dragos) would automate or enhance

[OT-REVIEW] Phase A deliberately uses manual/semi-automated tooling (the monitor's built-in dashboard) rather than commercial products. This is the same pedagogical approach used in IT security training: teach Wireshark before deploying a SIEM, teach firewall rules before deploying an NGFW. Engineers who skip the manual layer cannot troubleshoot the automated layer when it fails -- and in OT environments, "when it fails" often means "during the one incident that matters."

**FR-3**: Scenario structure -- five exercise phases:

**Phase A1: Deploy and Configure Monitoring** (Setup)
- Start the monitoring dashboard at `http://localhost:8090`
- Navigate to `/events` and observe the live transaction feed
- Identify the polling pattern: which devices are polled, at what interval, using which function codes
- Navigate to `/comms` and observe the communication matrix
- Identify the star topology (monitor → each device)
- [OT-REVIEW] The trainee should notice that all traffic originates from a single source (the monitor). In a real SOC, seeing a star topology from a single IP to all PLCs is the expected pattern for SCADA polling. Deviations (new source IPs, peer-to-peer PLC traffic) would be anomalous.

**Phase A2: Build a Traffic Baseline** (Learning)
- Wait for the baseline engine to complete its learning period (150 cycles at 2-second intervals = ~5 minutes)
- Navigate to `/api/baselines` and observe the baseline status transition from "learning" to "established"
- Record the per-device function code profile (which FCs does each device receive?)
- Record the per-device communication pattern (frequency, response time range)
- Use the FC distribution on `/assets/{id}` to confirm read-only vs. read-write devices
- [OT-REVIEW] The baseline learning period mirrors how commercial tools like Dragos and CyberVision build their behavioral models. Dragos calls this the "characterization phase" and recommends 30-90 days of observation. Our 5-minute learning period is compressed for training, but the principle is identical: observe first, alert later.

**Phase A3: Analyze Communication Patterns** (Analysis)
- Use `/comms` to identify which devices communicate most frequently
- Use `/events` with the write-only filter to verify which devices receive write operations
- Categorize devices: read-only sensors vs. read-write actuators
- Compare the function code distribution histograms across devices
- Identify the "normal" communication pattern for this environment
- [OT-REVIEW] Device categorization by function code profile is the first step in building an allowlist. Read-only devices (sensors) should NEVER receive write commands. Read-write devices (PLCs with setpoints) should only receive writes from authorized sources. This categorization is the foundation of every OT detection rule.

**Phase A4: Configure SIEM Forwarding** (Integration)
- Enable syslog forwarding in `monitor.yaml`
- Start a local syslog receiver (e.g., `nc -u -l 1514`)
- Observe CEF-formatted events arriving on the receiver
- Identify the CEF severity levels: writes=7, read failures=5, diagnostics=3, reads=1
- Parse the CEF extensions to extract device ID, function code, address range
- [OT-REVIEW] CEF syslog forwarding is the bridge between OT monitoring and IT security operations. In a converged SOC, the SIEM (Splunk, QRadar, Elastic) receives CEF events from OT monitoring tools and correlates them with IT security events. The trainee learns to read CEF format here so they can build SIEM correlation rules in a real engagement.

**Phase A5: Detect Unauthorized Activity** (Detection)
- After the baseline is established, the baseline engine will detect anomalies
- Navigate to `/api/alerts` and observe any active alerts
- Observe the `write_to_readonly` alert if a write operation targets a read-only device
- Observe the `fc_anomaly` alert if an unexpected function code appears
- Review alert details: severity, rule ID, device ID, expected vs. actual values
- Navigate to the Alerts page in the dashboard to see the alert rendered with context
- [OT-REVIEW] The detection phase demonstrates the core value proposition of OT monitoring: behavioral anomaly detection. Unlike IT security where signatures dominate (known bad), OT security relies heavily on behavioral baselines (known good). Any deviation from the learned communication pattern is suspicious because OT environments are deterministic -- the same SCADA server sends the same function codes to the same PLCs on the same schedule, forever.

**FR-4**: The scenario MUST explicitly teach these concepts:
1. Why writes are the highest-risk operations in OT (physical consequences)
2. Why behavioral baselines work better than signatures in OT (deterministic environments)
3. What commercial tools (Dragos, CyberVision) automate vs. what this manual layer teaches
4. The difference between transaction-level monitoring (Beta 0.6) and passive packet capture (Beta 0.7)
5. Why source address analysis requires passive capture (the monitor is the only client in Beta 0.6)

### Teaching Approach

Each phase follows a structured pattern:
1. **Task**: What to do (specific actions with URLs and commands)
2. **Observe**: What to look for (specific values, patterns, columns)
3. **Record**: What to write down (structured template)
4. **Understand**: Why this matters (OT security context)
5. **Compare**: How commercial tools handle this (future phases preview)
