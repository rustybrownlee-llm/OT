# SOW-034.0 - Design Layer JSON Schemas

**Status**: Complete
**Created**: 2026-03-03
**Completed**: 2026-03-03
**Priority**: High

## 1. Objective

Create JSON Schema (Draft 2020-12) definition files for all four design layer YAML types -- device atoms, network atoms, environment definitions, and process definitions. These schemas formalize the design layer contract that currently exists only implicitly in Go struct definitions, enabling schema-driven YAML validation in the admin CLI (SOW-036.0) and real-time editor feedback in the admin web dashboard (SOW-038.0). Partner engineers and platform operators will use these schemas as authoritative documentation of what constitutes a valid design element.

## 2. Background

The design layer (`design/`) contains YAML files that define the OT simulation environment. Currently, the "schema" for these files is implicit in the Go structs in `plant/internal/config/` that parse them. This creates three problems:

1. **Validation only at runtime**: An engineer creating a new device atom discovers errors only when plant fails to start -- no feedback during authoring.
2. **Undocumented contract**: The expected structure is spread across Go struct tags, validation functions, and tribal knowledge. A partner engineer cannot determine what fields are required without reading Go source.
3. **No editor support**: The planned YAML editor (SOW-038.0) needs machine-readable schema definitions to provide autocomplete, validation markers, and documentation tooltips.

JSON Schema (Draft 2020-12) is the industry standard for YAML/JSON structure validation. The approved `santhosh-tekuri/jsonschema` Go library supports this draft for server-side validation. Schemas are pure JSON files with no code dependencies, consumable by any toolchain.

The milestone spec (Beta 0.7 DoD #12) requires:
12. See JSON Schema files in `design/schemas/` that formally document the expected structure of every design layer YAML type

## 3. Scope

### In Scope
- JSON Schema file for device atoms (`device-atom.schema.json`)
- JSON Schema file for network atoms (`network-atom.schema.json`)
- JSON Schema file for environment definitions (`environment.schema.json`)
- JSON Schema file for process definitions (`process.schema.json`)
- All fields observed across all existing YAML files
- Enum constraints for all categorical fields
- Conditional validation (e.g., subnet required for ethernet networks)
- Format validation for IP addresses, port ranges
- Documentation via `description` and `title` annotations in each schema property
- Validation that all existing YAML files pass their respective schemas

### Out of Scope
- Cross-file reference validation (placement.device referencing a device atom file) -- SOW-036.0
- Go code for loading or validating against schemas -- SOW-035.0 and SOW-036.0
- Web editor integration -- SOW-038.0
- Schema versioning migration tooling
- Generating Go structs from schemas (td-admin-070)

## 4. Requirements

### Functional Requirements

**FR-1**: Create `design/schemas/` directory containing four JSON Schema files using Draft 2020-12 (`$schema: "https://json-schema.org/draft/2020-12/schema"`).

**FR-2**: Device atom schema (`device-atom.schema.json`) must validate:
- `schema_version` (string, required, const "0.1")
- `device` object with: `id` (string), `vendor` (string), `model` (string), `category` (string enum: plc, gateway, rtu, flow-computer, hmi, sensor, relay, safety-controller), `vintage` (integer, min 1960, max 2030; description: "Year this device model was introduced by the manufacturer. Minimum 1960 pre-dates earliest commercial PLC era. Modicon 984 (1988), SLC-500 (1992), and similar legacy devices remain in active OT service decades after their vintage year."), `description` (string)
- `connectivity` object with: `ports` array (minItems 1) of port objects with `type` (enum: ethernet, rs485, rs232, dh485) and `protocols` (array of strings from enum: modbus-tcp, modbus-rtu, dh-485, minItems: 0 -- empty arrays are valid, e.g., Cradlepoint WAN port has no OT protocols), plus `response_delay_ms` (integer, min 0), `response_jitter_ms` (integer, min 0), `concurrent_connections` (integer, min 1)
- `registers` object with: `max_holding` (integer, min 0), `max_coils` (integer, min 0), `max_input_registers` (integer, min 0), `max_discrete_inputs` (integer, min 0), `addressing` (enum: zero-based, one-based), `float_byte_order` (enum: big-endian, little-endian; description: "Modbus float byte order. big-endian = ABCD network byte order, used by CompactLogix. little-endian = CDAB word-swap order used by Modicon 984 and many older devices. Note: 'little-endian' here simplifies the CDAB word-swapped encoding common in legacy Modicon devices -- the real OT world has four byte order variants (ABCD, DCBA, CDAB, BADC)."), `max_registers_per_read` (integer, min 1, max 125)
- `register_map_variants` (object, minProperties 1, additionalProperties are RegisterMap objects containing `holding` array and `coils` array with minItems: 0 -- empty coils arrays are valid, e.g., moxa-nport-5150 and cradlepoint-ibr600 have `coils: []`)
- RegisterDef: `address` (integer, min 0), `name` (string), `description` (string), `unit` (string), `scale_min` (number), `scale_max` (number), `writable` (boolean)
- CoilDef: `address` (integer, min 0), `name` (string), `description` (string), `writable` (boolean)
- `diagnostics` object with: `web_server` (boolean), `web_port` (integer, min 0), `snmp` (boolean), `syslog` (boolean), `fault_log` (boolean), `fault_log_depth` (integer, min 0)
- All top-level sections required; all fields within each section required

**FR-3**: Network atom schema (`network-atom.schema.json`) must validate:
- `schema_version` (string, required, const "0.1")
- `network` object with: `id` (string), `name` (string), `type` (string enum: ethernet, serial-rs485, serial-rs232), `description` (string)
- `properties` object with: `managed_switch` (boolean), `span_capable` (boolean)
- Conditional: when `network.type` is "ethernet", `properties.subnet` is required (string, format: CIDR-like pattern via regex `^\d+\.\d+\.\d+\.\d+/\d+$`)
- Conditional: `properties.vlan` (integer, min 1, max 4094) is optional but only valid when `managed_switch` is true
- Use `if/then` or `allOf` with conditional schemas for type-dependent properties

**FR-4**: Environment schema (`environment.schema.json`) must validate:
- `schema_version` (string, required, const "0.1")
- `environment` object with: `id` (string, required), `name` (string, required), `description` (string, required), `archetype` (string, optional, enum: hybrid, modern-segmented, legacy-flat), `era_span` (string, optional)
- `networks` array (minItems 1) of objects with `ref` (string, required)
- `placements` array (minItems 1) of placement objects with: `id` (string, required), `device` (string, required), `network` (string, required), `ip` (string, optional, format: ipv4), `modbus_port` (integer, optional, min 1, max 65535; description: "Modbus TCP port. Standard OT port is 502; this simulator uses 5020-5059 to avoid privileged port requirements. The schema accepts any valid port (1-65535)."), `serial_address` (integer, optional, min 1, max 247), `gateway` (string, optional), `role` (string, optional), `register_map_variant` (string, optional), `installed` (integer, optional), `additional_networks` (array, optional) of objects with `network` (string) and `ip` (string, optional), `bridges` (array, optional) of objects with `from_network` (string) and `to_network` (string)
- `boundaries` array (optional) of objects with: `between` (array of strings, minItems 2, maxItems 2), `state` (string enum: enforced, intended, absent; description: "'intended' represents a boundary in the design but not yet enforced in hardware -- a critical real-world OT pattern where budget approval precedes installation"), `infrastructure` (string enum: managed-switch, firewall, ids-sensor, vlan-only, other; description: "Type of boundary enforcement device per environment.schema.md reference"), `installed` (integer, optional), `notes` (string)

**FR-5**: Process schema (`process.schema.json`) must validate:
- `schema_version` (string, required, const "0.1")
- `process` object with: `id` (string), `name` (string), `description` (string), `flow_direction` (string, optional, enum: horizontal, vertical, radial)
- `stages` array (minItems 1) of stage objects with: `id` (string), `name` (string), `controller` object with `placement` (string) and `device` (string), `equipment` array (minItems 1)
- Equipment items with: `id` (string), `type` (string enum: pump, screen, basin, tank, valve, meter, compressor, chromatograph, uv_system, clarifier, blower), `label` (string), `era` (integer, optional), `instruments` array (minItems 0)
- Instrument items with: `tag` (string), `name` (string), `isa_type` (string; description: "ISA-5.1 instrument type prefix. First letter = measured variable (F=flow, A=analyzer, T=temperature, P=pressure, L=level, Z=position, V=vibration, S=speed/switch), subsequent letters = function (T=transmitter, C=controller, I=indicator, S=switch, Q=quantity/totalizer). Project conventions: 'run' = motor run status coil, 'FIC' = flow indicating controller. See ISA-5.1-2009 for the full standard."), `placement` (string), `register` object with `type` (enum: holding, coil) and `address` (integer, min 0), `unit` (string), `scale` (array of exactly 2 numbers, optional; note: coil-type instruments with unit "bool" always omit this field -- the schema must not require it), `thresholds` (object, optional) with optional fields: `warning` (number), `alarm` (number), `warning_low` (number), `alarm_low` (number)
- `connections` array (optional) of objects with: `from` (string), `to` (string), `type` (string), `label` (string)
- `network_context` array (optional) of objects with: `id` (string), `type` (string), `label` (string), `era` (integer, optional), `placement` (string, optional), `warning` (string, optional)

**FR-6**: Every schema property must include a `description` annotation explaining the field's purpose in OT context. These descriptions serve as inline documentation for the YAML editor tooltips (SOW-038.0).

**FR-7**: All existing YAML files in `design/` must pass validation against their respective schemas. Any field discovered in existing files that the schema rejects indicates a schema bug, not a data bug.

**FR-8**: Schemas must use `additionalProperties: false` at each object level to catch typos and unknown fields. This is the primary value for partner engineers: a misspelled field name produces an immediate error rather than being silently ignored.

[OT-REVIEW] The `additionalProperties: false` decision is deliberate and educational. In IT JSON APIs, `additionalProperties: true` provides forward compatibility. In OT configuration, strictness prevents misconfiguration that could cause physical consequences. A misspelled `writable` field on a chlorine dosing register means the register defaults to read-only, and an operator's emergency shutdown command is silently ignored.

### Non-Functional Requirements

**NFR-1**: Schemas are pure JSON files with no code dependencies. Any JSON Schema Draft 2020-12 compliant validator must be able to consume them.

**NFR-2**: Each schema file must be self-contained (no `$ref` to other schema files). This simplifies loading in the admin CLI and avoids relative path resolution issues. OT justification: In OT environments, configuration tools must operate without network connectivity to schema registries. A self-contained schema file can be loaded from a USB drive on an air-gapped engineering workstation -- a common deployment pattern in pipeline and water treatment facilities.

**NFR-3**: Schema files should be under 500 lines each for readability. Use `$defs` for shared sub-schemas within each file (e.g., RegisterDef, CoilDef).

## 5. Design

### File Structure

```
design/schemas/
    device-atom.schema.json       -- Device atom validation
    network-atom.schema.json      -- Network atom validation
    environment.schema.json       -- Environment definition validation
    process.schema.json           -- Process definition validation
```

### Schema Pattern

Each schema follows this top-level pattern:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://github.com/rustybrownlee/ot-simulator/design/schemas/<name>.schema.json",
  "title": "<Human-readable title>",
  "description": "<What this schema validates>",
  "type": "object",
  "required": ["schema_version", ...],
  "additionalProperties": false,
  "properties": {
    "schema_version": {
      "type": "string",
      "const": "0.1",
      "description": "Schema version for this YAML type"
    },
    ...
  },
  "$defs": {
    "RegisterDef": { ... },
    "CoilDef": { ... }
  }
}
```

### Conditional Validation Strategy

Network atom type-dependent properties use `if/then`. In JSON Schema Draft 2020-12, `required` is a keyword at the same level as `properties`, not nested inside it.

The conditional must be applied at the correct schema nesting level. Since the `subnet` field lives inside the `properties` object (which is a YAML key named "properties", not the JSON Schema keyword), the `if/then` must target the right depth. The implementation agent must verify the conditional actually enforces the subnet requirement by testing rejection of an ethernet network without subnet (Phase 5 checklist).

Conceptual requirement: when `network.type` is `"ethernet"`, the `properties` section must include a `subnet` field. When `network.type` is serial, `subnet` should be absent (serial buses have no IP addressing). The implementation agent should use `if/then/else` to enforce both directions of this constraint.

### Enum Strategy

Device category, port type, protocol, network type, addressing, byte order, equipment type, register type, boundary state, and archetype fields all use `enum` constraints. The enum lists include all values observed in existing files plus values mentioned in code comments as valid but not yet instantiated (e.g., hmi, sensor, relay, safety-controller for device categories).

### Description Annotations

Every property includes a `description` field for editor tooltip support:

```json
"writable": {
  "type": "boolean",
  "description": "Whether this register accepts Modbus write operations. In OT, writable registers on process control devices (dosing pumps, valve actuators) carry physical safety implications."
}
```

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Existing YAML files have inconsistencies that prevent strict schema | Schema must be loosened | Medium | FR-7 mandates schema-conforms-to-data. If a field is optional in some files, schema marks it optional. |
| Draft 2020-12 conditional features not supported by target Go library | Conditional validation fails | Low | santhosh-tekuri/jsonschema v6 supports Draft 2020-12 including if/then/else. Verify in SOW-036.0. |
| Schema becomes stale as new device/network types are added | New YAML rejected by outdated enum | Medium | td-admin-070 tracks this. Enum lists include forward-looking values (hmi, sensor, relay, safety-controller). |
| additionalProperties: false breaks future extensibility | New fields require schema update | Low | This is intentional for an educational tool -- strictness catches mistakes. Schema updates are a one-line enum/property addition. |
