# SOW-035.0 - Admin CLI Module and Core Commands

**Status**: Complete
**Created**: 2026-03-03
**Priority**: High

## 1. Objective

Create the `admin/` Go module as a new independent top-level module and implement the foundational CLI commands for platform health checking, event database management, and monitoring configuration validation. This module provides the command-line interface that non-developer engineers and integration partners use to operate the OT simulator without touching Go source code or writing SQL queries. The admin CLI binary is named `admin` (not `otsim`; see td-admin-075 for future consolidation).

## 2. Background

The OT simulator currently requires engineers to know internal details to operate it:
- Checking platform health requires knowing Modbus TCP port numbers and curling the monitoring API
- Querying the event database requires opening SQLite and writing SQL
- Validating the monitoring configuration requires running the monitor binary and reading error messages
- Checking baseline status requires curling the monitoring API directly

Beta 0.7 introduces platform administration tools. This SOW establishes the Go module, CLI entry point with command routing, and the first set of commands covering health, database, and configuration. Subsequent SOWs layer design validation (SOW-036.0) and the web dashboard (SOW-037.0) on top of this foundation.

### Architectural Constraints (CLAUDE.md, ADR-005, ADR-009)

- `admin/` is an **independent Go module** with its own `go.mod`. It does NOT import `plant/` or `monitoring/` Go packages.
- Admin reads design YAML directly from `design/` (per ADR-009).
- Admin accesses the monitoring SQLite event database directly for read operations (per CLAUDE.md: "accesses the monitoring SQLite DB").
- Admin calls the monitoring HTTP API over the network for health checks and baseline operations (per ADR-005).
- Admin checks plant port reachability via TCP connection probes (no Modbus protocol knowledge required for health checks).

### Reference Implementations

- `monitoring/cmd/monitor/main.go`: CLI pattern using `flag` package, health check via HTTP
- `monitoring/internal/config/config.go`: YAML config parsing with validation, defaults, and error wrapping
- `monitoring/internal/eventstore/`: SQLite operations with WAL mode and parameterized queries
- `monitoring/internal/api/api.go`: HTTP API patterns (health response, baselines response)

## 3. Scope

### In Scope

- New `admin/` Go module with `go.mod`, `go.sum`
- CLI entry point with multi-command routing using standard `flag` package
- `admin health` -- platform health summary (Modbus port probes, monitoring API check, DB check)
- `admin db status` -- event store statistics (row count, DB size, oldest/newest event, retention)
- `admin db validate` -- SQLite schema integrity check
- `admin db prune [--older-than <days>]` -- manual retention enforcement
- `admin db export [--format csv|json] [--output <path>] [--device <id>] [--after <time>] [--before <time>]` -- event export
- `admin config view [<path>]` -- display parsed monitoring configuration
- `admin config validate [<path>]` -- validate monitoring configuration with error reporting
- `admin baseline status` -- per-device baseline status from monitoring API
- `admin baseline reset [--device <id>]` -- trigger baseline re-learning via monitoring API
- `admin web [--addr :8095]` -- placeholder web server (skeleton only; full implementation in SOW-037.0)
- Monitoring API addition: `POST /api/baselines/reset` and `POST /api/baselines/{deviceID}/reset` endpoints (minimal addition to enable the admin CLI's baseline reset command)
- Global flags: `--design-dir`, `--config`, `--db`, `--api-addr`, `--plant-ports`
- Environment variable support: `OTS_DESIGN_DIR`, `OTS_MONITOR_CONFIG`, `OTS_API_ADDR`, `OTS_ADMIN_ADDR`

### Out of Scope

- Design YAML validation commands (`admin design validate`, `admin design list`) -- SOW-036.0
- Full web dashboard implementation -- SOW-037.0
- YAML editor -- SOW-038.0
- CLI reference documentation -- SOW-039.0
- JSON Schema loading or validation logic -- SOW-036.0
- Authentication or authorization
- Plant start/stop/control (admin observes only)
- Modbus protocol operations from admin (health checks use TCP probes, not Modbus reads)

## 4. Requirements

### Functional Requirements

**FR-1**: Create `admin/` Go module with `go.mod` declaring `module github.com/rustybrownlee/ot-simulator/admin` using Go 1.24+. Dependencies: `gopkg.in/yaml.v3` (config parsing), `modernc.org/sqlite` (event DB access), `github.com/go-chi/chi/v5` (web server routing for SOW-037.0 readiness).

**FR-2**: CLI entry point at `admin/cmd/admin/main.go` with command routing using `os.Args` and `flag.NewFlagSet` per command. No Cobra, urfave/cli, or other CLI frameworks. Commands: `health`, `db`, `config`, `baseline`, `web`. Unknown commands print usage and exit 1.

**FR-3**: `admin health` probes platform services and prints a summary table:
- TCP connect to each configured Modbus port (from `--plant-ports` or `OTS_HEALTH_PORTS`, default: `5020,5021,5022,5030,5040,5041,5042,5043,5050,5051,5052,5062,5063,5064` on localhost)
- HTTP GET to monitoring API health endpoint (`/api/health` at `--api-addr`)
- File existence and readability check on the event database (`--db` path)
- Overall status: healthy (all pass), degraded (some fail), unhealthy (all fail)
- TCP connect timeout: 2 seconds per port
- Output format: human-readable table to stdout with port type labels (PLC, Gateway, Modem) derived from a port-to-type mapping. Gateway ports (serial-to-Ethernet bridges like Moxa NPort) and modem ports (Cradlepoint cellular) must be labeled differently from PLC Modbus ports. This teaches the topological difference between reaching a PLC directly versus through a serial gateway.
- [OT-REVIEW] Exit code 0 for all services reachable. Exit code 1 for one or more services unreachable (includes partial failure -- in OT, a degraded plant is NOT a "zero" condition; it may indicate a communications failure on the control network, which can be a precursor to a safety event). The health check follows OT convention, not IT convention (where degraded = exit 0 in Nagios/Icinga).

**FR-4**: `admin db status` opens the event database read-only and reports:
- Total event count (`SELECT COUNT(*) FROM events`)
- Database file size (os.Stat)
- Oldest event timestamp (`SELECT MIN(timestamp) FROM events`)
- Newest event timestamp (`SELECT MAX(timestamp) FROM events`)
- Configured retention window (from monitoring config, default 7 days)
- Events older than retention window (count of pruneable events)
- Output format: human-readable key-value pairs to stdout

**FR-5**: `admin db validate` runs `PRAGMA integrity_check` and `PRAGMA foreign_key_check` against the event database. Reports results to stdout. Exit code 0 for valid, 1 for errors.

**FR-6**: `admin db prune [--older-than <days>]` deletes events older than the specified number of days (default: uses the configured retention period from monitor.yaml). Reports the number of deleted rows. Requires confirmation prompt (user must type "yes") before deleting unless `--force` flag is passed.

**FR-7**: `admin db export` queries events from the database with optional filters and writes to a file or stdout:
- `--format csv|json` (default: csv)
- `--output <path>` (default: stdout)
- `--device <id>` (filter by device_id)
- `--after <time>` (RFC3339 timestamp, inclusive)
- `--before <time>` (RFC3339 timestamp, exclusive)
- CSV columns: id, timestamp, src_addr, dst_addr, unit_id, func_code, func_name, addr_start, addr_count, is_write, success, exception, response_us, device_id, device_name, env_id, write_values. [OT-REVIEW] The write_values column (JSON-encoded WriteDetail) is the most forensically significant column for OT security analysis. It stores the actual register values written during Modbus FC6/FC16/FC5/FC15 operations. Omitting it would mean an analyst cannot see what value an attacker wrote to a control register.
- JSON format: array of event objects with the same fields
- Maximum export: 100,000 rows (warn if limit reached)

**FR-8**: `admin config view [<path>]` parses the monitoring config file and prints a formatted summary showing all configured fields with their values and defaults applied. Path defaults to `--config` flag or `OTS_MONITOR_CONFIG` env var.

**FR-9**: `admin config validate [<path>]` parses and validates the monitoring config file using the same validation logic as the monitoring module (reimplemented in admin, since admin cannot import monitoring packages). Reports validation errors with field names and descriptions. Exit code 0 for valid, 1 for errors. The validation must catch the same errors as `monitoring/internal/config/config.go:Validate()`.

**FR-10**: `admin baseline status` calls `GET /api/baselines` on the monitoring API and displays a table of per-device baseline status: device ID, status (learning/established), sample count, required samples, register count.

**FR-11**: `admin baseline reset [--device <id>]` calls `POST /api/baselines/reset` (all devices) or `POST /api/baselines/{deviceID}/reset` (single device) on the monitoring API. Reports the result. Requires the monitoring service to be running.
- [OT-REVIEW] Full reset (no --device flag) requires `--force` or a confirmation prompt. Resetting all baselines triggers a learning period (~5 min at default poll interval) during which anomaly detection alerts are suppressed. This is operationally equivalent to disabling the IDS. The command must print a warning: "WARNING: Baseline reset triggers a learning period. Anomaly detection alerts are suppressed during this window." Single-device reset (with --device) may proceed without --force since the detection gap is limited to one device.
- When the monitoring API endpoint is called, the Reset/ResetDevice methods must preserve `knownDeviceIDs` to avoid false "new device" alerts after reset. Only the learned baseline statistics, register buffers, coil buffers, and sample accumulators should be cleared.

**FR-12**: Add `POST /api/baselines/reset` and `POST /api/baselines/{deviceID}/reset` endpoints to the monitoring module's API router (`monitoring/internal/api/api.go`). The reset handler calls `engine.Reset()` or `engine.ResetDevice(deviceID)` to trigger baseline re-learning. Returns JSON confirmation. This is the minimum monitoring module change needed for admin CLI baseline reset support.

**FR-13**: `admin web [--addr :8095]` starts a minimal HTTP server that serves a single "Admin Dashboard - Coming Soon" page. This is a skeleton for SOW-037.0 to build upon. The server uses chi v5 and `go:embed` for a minimal HTML template.

**FR-14**: Global flags are parsed before command dispatch. Flag precedence: CLI flag > environment variable > default value. Flags:
- `--design-dir` / `OTS_DESIGN_DIR` (default: `./design`)
- `--config` / `OTS_MONITOR_CONFIG` (default: `./config/monitor.yaml`)
- `--db` / (no env var; derived from config's `event_db_path`, or `--db` flag override)
- `--api-addr` / `OTS_API_ADDR` (default: `localhost:8091`)
- `--plant-ports` / `OTS_HEALTH_PORTS` (default: `5020,5021,5022,5030,5040,5041,5042,5043,5050,5051,5052,5062,5063,5064`)

### Non-Functional Requirements

**NFR-1**: All CLI output is human-readable plain text. No JSON output for CLI commands (JSON is for the web API layer in SOW-037.0). Tables use aligned columns with headers.

**NFR-2**: All database operations open the SQLite file in read-only mode except `db prune` which requires write access. Use `?mode=ro` in the SQLite connection string for read-only access.

**NFR-3**: The admin CLI must handle gracefully the case where services are not running. Health checks report "offline" rather than crashing. DB commands report clear errors when the database file does not exist. Baseline commands report that the monitoring service is unreachable rather than crashing.

**NFR-4**: Config validation must produce error messages that include the field name, the invalid value, and a description of what is expected. Follow the pattern in `monitoring/internal/config/config.go`.

**NFR-5**: Function size limit: 60 lines maximum. File size limit: 500 lines maximum (excluding tests). Interface limit: 5 methods maximum.

**NFR-6**: Every public function and type has a documentation comment. Internal packages use clear, descriptive names.

### CLI Command Routing Pattern

```go
// main.go -- standard library command routing (no Cobra)
func main() {
    globals := parseGlobalFlags()

    if len(os.Args) < 2 {
        printUsage()
        os.Exit(1)
    }

    cmd := os.Args[1]
    args := os.Args[2:]

    switch cmd {
    case "health":
        runHealth(globals, args)
    case "db":
        runDB(globals, args)
    case "config":
        runConfig(globals, args)
    case "baseline":
        runBaseline(globals, args)
    case "web":
        runWeb(globals, args)
    default:
        fmt.Fprintf(os.Stderr, "admin: unknown command %q\n", cmd)
        printUsage()
        os.Exit(1)
    }
}
```

### Database Access Pattern

Admin opens the SQLite database directly (not via the monitoring API) for all `db` commands. This is permitted per CLAUDE.md. The admin module reimplements the minimum SQLite operations needed (count, min/max, prune, export) without importing the monitoring eventstore package.

```go
// dbutil/status.go -- direct SQLite access for stats
func Status(dbPath string) (*DBStatus, error) {
    db, err := sql.Open("sqlite", dbPath+"?mode=ro")
    // ...
}
```

### Config Reimplementation Strategy

The admin module cannot import `monitoring/internal/config`. It reimplements the config struct and validation logic in `admin/internal/configparse/`. The struct must match the monitoring module's YAML tags exactly. Validation rules are copied from `monitoring/internal/config/config.go:Validate()`.

This is tracked as technical debt (td-admin-090): config validation logic is duplicated between monitoring and admin. Future resolution: extract shared config types to a common library package or generate from a schema.

### Monitoring API Additions

Two new endpoints in `monitoring/internal/api/api.go`:

```go
r.Post("/api/baselines/reset", resetAllBaselinesHandler(engine))
r.Post("/api/baselines/{deviceID}/reset", resetDeviceBaselineHandler(engine))
```

The baseline engine already has a `Reset()` method concept -- the admin CLI just needs an HTTP endpoint to trigger it. If `baseline.Engine` does not have a public reset method, a minimal one must be added to `monitoring/internal/baseline/baseline.go`.

## 8. Success Criteria

1. `go build ./cmd/admin/` in the `admin/` directory produces a working binary with no errors
2. `admin health` prints a service summary table with OT-aware port type labels (PLC/Gateway/Modem) and exits 0 when all services are reachable, exits 1 when any service is unreachable
3. `admin health` with no services running reports all services as offline and exits 1, does not crash
4. `admin db status` displays event count, DB size, oldest/newest timestamps, and retention window
5. `admin db status` with a missing database file prints a clear error, does not crash
6. `admin db validate` runs integrity checks and reports results
7. `admin db prune --older-than 1 --force` deletes events older than 1 day and reports count
8. `admin db prune` without `--force` prompts for confirmation
9. `admin db export --format csv` produces a valid CSV file with correct column headers
10. `admin db export --format json` produces valid JSON array
11. `admin db export --device <id>` filters output to a single device
12. `admin config view` displays the parsed monitoring configuration
13. `admin config validate` with a valid config file exits 0
14. `admin config validate` with an invalid config file exits 1 and prints field-level errors
15. `admin baseline status` displays a table of device baseline states
16. `admin baseline reset` triggers baseline re-learning via the monitoring API
17. `admin web` starts an HTTP server on :8095 serving a placeholder page
18. All commands handle missing services gracefully (no panics, clear error messages)
19. No `plant/` or `monitoring/` Go packages are imported in the admin module

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Config validation logic drifts between admin and monitoring | Admin validates differently than monitor | Medium | td-admin-090 tracks this. Config struct and validation copied verbatim. Future: shared config package. |
| Monitoring baseline engine has no public Reset method | Baseline reset command cannot work | Medium | Check baseline.go; add minimal Reset/ResetDevice methods if absent. |
| SQLite file locking conflicts between monitoring (writer) and admin (reader) | DB commands fail when monitor is running | Low | WAL mode supports concurrent read/write. Admin uses read-only mode (?mode=ro) for all queries except prune. |
| Health check default port list becomes stale as environments are added | Health reports incomplete | Medium | Port list is configurable via --plant-ports flag. Default list covers all current environments. |
| go:embed in templates/ requires at least one file | Build fails if template dir is empty | Low | Placeholder HTML file ensures dir is non-empty. |
