# SOW-036.0 - Design Validation CLI

**Status**: Complete
**Created**: 2026-03-03
**Priority**: High

## 1. Objective

Add `admin design validate` and `admin design list` commands to the admin CLI module (SOW-035.0) that validate design layer YAML files against the JSON Schemas (SOW-034.0) and verify cross-file reference integrity. These commands enable non-developer engineers and integration partners to validate and inventory the design layer without reading Go source or running the plant binary. Engineers creating new device atoms, modifying environment definitions, or authoring new environments get immediate, specific feedback on every invalid field -- with line numbers and schema-derived descriptions of what's expected.

## 2. Background

The JSON Schemas created in SOW-034.0 formalize the design layer contract:
- `design/schemas/device-atom.schema.json` -- validates 7 device atom files
- `design/schemas/network-atom.schema.json` -- validates 13 network atom files
- `design/schemas/environment.schema.json` -- validates 4 environment definition files
- `design/schemas/process.schema.json` -- validates 3 process definition files

The admin CLI module created in SOW-035.0 provides the command routing foundation (`admin <command> <subcommand>`) with global flags for `--design-dir` to locate the design directory.

What's missing is the Go code that loads schemas, validates YAML against them, and reports results. JSON Schema validates structure but cannot validate cross-file references (e.g., a placement's `device: "compactlogix-l33er"` references a device atom file that must exist in `design/devices/`). Cross-reference validation is Go logic layered on top of schema validation.

The milestone spec (Beta 0.7 DoD items #5 and #6) requires:
- #5: Run `otsim design validate design/devices/compactlogix-l33er.yaml` and see whether a device atom conforms to the JSON Schema, with line-level error reporting
- #6: Run `otsim design validate design/environments/greenfield-water-mfg/` and validate an entire environment (environment.yaml, process.yaml, all referenced device and network atoms) in one command

### Reference Implementations

- `admin/cmd/admin/main.go`: CLI command routing pattern (SOW-035.0)
- `admin/internal/cli/output.go`: Globals struct, TablePrinter for tabular output
- `admin/internal/cli/db.go`: Subcommand pattern with `flag.NewFlagSet` (SOW-035.0)
- `design/schemas/*.schema.json`: JSON Schema files (SOW-034.0)
- `monitoring/cmd/monitor/main.go`: Design directory resolution pattern

### Architectural Constraints

- Admin reads design YAML files directly (per ADR-009 and CLAUDE.md)
- Admin does NOT import plant/ or monitoring/ Go packages
- Schema validation uses `santhosh-tekuri/jsonschema` v6 (approved in CLAUDE.md tech stack, Draft 2020-12 support)
- YAML-to-JSON conversion required: JSON Schema validates JSON documents; YAML files must be converted to JSON (via `encoding/json` after `gopkg.in/yaml.v3` unmarshaling) before validation
- No Cobra or external CLI frameworks (per shared standards)

## 3. Scope

### In Scope

- `admin design validate <path>` command -- validates a single YAML file or an entire environment directory
- `admin design list` command -- lists all design layer elements in a categorized table
- JSON Schema loading and compilation from `design/schemas/`
- YAML-to-JSON conversion for schema validation
- Line-level error reporting (map YAML parse errors and schema validation paths to source line numbers)
- Schema type inference from file path (devices/ -> device-atom, networks/ -> network-atom, etc.)
- Environment directory validation: validate environment.yaml + process.yaml + all referenced device and network atoms in one invocation
- Cross-reference validation for environment files:
  - placement.device references a file in design/devices/
  - placement.network references a network listed in the environment's networks array
  - networks[].ref references a file in design/networks/
  - placement.register_map_variant references a variant key in the device atom's register_map_variants
  - placement.gateway references another placement ID in the same environment
  - process stage.controller.placement references a placement ID in the environment
  - process stage.controller.device matches the placement's device field
  - process stage.equipment[].instruments[].placement references a placement ID in the environment
  - process network_context[].placement references a placement ID in the environment
  - boundaries[].between network IDs match environment networks[].ref values
  - serial-network placements do NOT have an ip field (serial uses unit addresses)
  - register addresses do not exceed device register_capacity limits
- Integration with main.go command routing (add `design` case to switch statement)

### Out of Scope

- Web-based editor validation UI (SOW-038.0 uses the same schema/validator package via HTTP handlers)
- Modifying JSON Schema files (SOW-034.0 output is consumed as-is)
- Validating monitor.yaml (already handled by `admin config validate` in SOW-035.0)
- Schema versioning or migration
- Validating YAML files outside the design/ directory
- Auto-fixing validation errors

## 4. Requirements

### Functional Requirements

**FR-1**: `admin design validate <path>` accepts a file path or directory path:
- **Single file mode**: When `<path>` is a `.yaml` file, validate it against the appropriate JSON Schema based on its location within the design directory:
  - `design/devices/*.yaml` -> `device-atom.schema.json`
  - `design/networks/*.yaml` -> `network-atom.schema.json`
  - `design/environments/*/environment.yaml` -> `environment.schema.json`
  - `design/environments/*/process.yaml` -> `process.schema.json`
- **Environment directory mode**: When `<path>` is a directory containing `environment.yaml`, validate the entire environment:
  1. Schema-validate `environment.yaml` against `environment.schema.json`
  2. Schema-validate `process.yaml` against `process.schema.json` (if present; process.yaml is optional -- quickstart-example has no process.yaml)
  3. Run cross-reference validation on the environment
  4. For each unique device referenced in placements, schema-validate the device atom file
  5. For each unique network referenced in the networks array, schema-validate the network atom file
- Exit code 0 when all validations pass, exit code 1 when any error is detected.

**FR-2**: Schema type inference resolves the file's location relative to the design directory (`--design-dir`). Paths outside the design directory or unrecognized paths produce a clear error: `"cannot determine schema type for <path>: file is not in design/devices/, design/networks/, or design/environments/"`.

**FR-3**: Line-level error reporting maps JSON Schema validation errors back to YAML source line numbers. The approach:
1. Parse YAML into a `yaml.Node` tree (preserves line/column information)
2. Convert `yaml.Node` to a JSON-compatible Go value for schema validation
3. When schema validation returns a JSON Pointer path (e.g., `/placements/2/modbus_port`), walk the `yaml.Node` tree to find the corresponding YAML node and extract its line number
4. Display errors as: `<file>:<line>: <field-path>: <schema error message>`

When line mapping fails (edge cases), fall back to displaying the JSON Pointer path without a line number.

**FR-4**: Cross-reference validation for environment files checks:
1. **Device existence**: For each placement, verify `design/devices/<placement.device>.yaml` exists
2. **Network existence**: For each `networks[].ref`, verify `design/networks/<ref>.yaml` exists
3. **Placement-network consistency**: For each placement, verify its `network` field matches one of the environment's `networks[].ref` values
4. **Additional network consistency**: For each placement's `additional_networks[].network`, verify it matches one of the environment's `networks[].ref` values
5. **Bridge network consistency**: For each placement's `bridges[].from_network` and `bridges[].to_network`, verify they match environment networks
6. **Gateway existence**: For each placement with a `gateway` field, verify the gateway references another placement ID in the same environment
7. **Register map variant existence**: For each placement with a `register_map_variant`, load the device atom and verify the variant key exists in `register_map_variants`
8. **Process-environment consistency** (when process.yaml is present):
   - Each `stage.controller.placement` references a placement ID in environment.yaml
   - Each `stage.controller.device` matches the device field of the referenced placement
   - Each `stage.equipment[].instruments[].placement` references a placement ID in environment.yaml
9. **Process network_context consistency** (when process.yaml is present):
   - Each `network_context[].placement` references a placement ID in environment.yaml
10. **Boundary network consistency**: For each `boundaries[].between`, verify both network IDs match a `networks[].ref` in the environment
11. **Serial-network IP absence**: For each placement on a serial-type network (serial-rs485 or serial-rs232), verify that the placement does NOT have an `ip` field (serial devices use unit addresses, not IPs)
12. **Register address bounds**: For each placement with a `register_map_variant`, after loading the device atom, verify that register addresses in the variant's register definitions do not exceed the device's `register_capacity.holding_registers` or `register_capacity.input_registers` limits

Cross-reference errors are reported with the same format as schema errors: `<file>:<line>: <description>`.

### Non-Functional Requirements

**NFR-1**: Schema compilation is done once per invocation. When validating an environment directory (which may reference the same device schema multiple times), each JSON Schema file is loaded and compiled exactly once, then reused for all files of that type.

**NFR-2**: Validation of all 27 existing YAML files completes in under 2 seconds. Schema compilation dominates; individual file validation is fast.

**NFR-3**: Error messages include the full file path (relative to the working directory), line number, JSON Pointer path, and human-readable description. No error should require the user to consult the schema file to understand what's wrong.

**NFR-4**: Function size limit: 60 lines maximum. File size limit: 500 lines maximum (excluding tests). If schema/validator.go exceeds 500 lines, split into schema/validator.go (core validation) and schema/linemap.go (YAML node line mapping).

**NFR-5**: All public types and functions have documentation comments.

## 5. Design

### Package Structure

```

    schema/
        loader.go           -- Load and compile JSON Schemas from design/schemas/
        validator.go         -- Validate YAML files against compiled schemas
        linemap.go           -- Map JSON Pointer paths to YAML source line numbers
        crossref.go          -- Cross-reference validation for environments
    cli/
        design.go            -- admin design validate|list command handlers
```

### Schema Loading Architecture

```go
// schema/loader.go

// SchemaSet holds compiled JSON Schemas, one per design layer type.
// Schemas are compiled once and reused across all file validations.
type SchemaSet struct {
    DeviceAtom  *jsonschema.Schema
    NetworkAtom *jsonschema.Schema
    Environment *jsonschema.Schema
    Process     *jsonschema.Schema
}

// Load reads and compiles all JSON Schema files from the schemas directory.
// Returns an error if any schema file is missing or invalid.
func Load(schemasDir string) (*SchemaSet, error)
```

The `schemasDir` is derived from `--design-dir` + `/schemas`. Schemas are loaded eagerly (all four at once) to fail fast if any schema file is missing.

### YAML-to-JSON Validation Flow

```
YAML file
  |
  v
yaml.Unmarshal -> yaml.Node (preserves line info)
  |
  v
nodeToInterface() -> map[string]interface{} (JSON-compatible)
  |
  v
jsonschema.Validate(schema, jsonDoc) -> []ValidationError
  |
  v
For each error: walk yaml.Node tree using JSON Pointer path -> line number
  |
  v
Format: "file:line: /json/pointer: error message"
```

The `nodeToInterface` function converts a `yaml.Node` tree to a Go `interface{}` tree that matches JSON types (maps with string keys, arrays, strings, numbers, booleans, null). This is necessary because `santhosh-tekuri/jsonschema` validates against Go interface values, not raw JSON bytes.

### Line Mapping Strategy

JSON Schema validation errors include a JSON Pointer (RFC 6901) path like `/placements/2/modbus_port`. To map this back to a YAML line number:

1. Parse the JSON Pointer into path segments: `["placements", "2", "modbus_port"]`
2. Walk the `yaml.Node` tree from the root, following each segment:
   - For mapping nodes: find the key matching the segment, then descend into the value
   - For sequence nodes: convert segment to integer index, descend into the indexed child
3. The final `yaml.Node` in the walk has `.Line` and `.Column` fields

Edge cases:
- If the path references a missing field (required but absent), the line number comes from the parent object node
- If the path is empty (root-level error), use line 1
- If a segment doesn't match (shouldn't happen with valid schemas), fall back to no line number

### Cross-Reference Validation Design

Cross-reference validation operates on the parsed YAML data (not the schema). It loads the environment.yaml into memory, then checks each reference against the filesystem and internal consistency.

```go
// schema/crossref.go

// CrossRefResult holds the result of cross-reference validation.
type CrossRefResult struct {
    Errors []CrossRefError
}

// CrossRefError describes a single cross-reference failure.
type CrossRefError struct {
    File    string  // Source file path
    Line    int     // YAML source line number (0 = unknown)
    Message string  // Human-readable error description
}

// ValidateCrossRefs checks all cross-file and intra-file references
// in an environment definition.
func ValidateCrossRefs(envDir string, designDir string) (*CrossRefResult, error)
```

The cross-reference validator:
1. Parses `environment.yaml` into a `yaml.Node` tree (for line numbers) and a typed struct (for field access)
2. For each `networks[].ref`, checks `os.Stat(designDir + "/networks/" + ref + ".yaml")`
3. For each placement, checks `os.Stat(designDir + "/devices/" + placement.device + ".yaml")`
4. For each placement with `register_map_variant`, loads the device atom YAML and checks the variant key exists in `register_map_variants`
5. Builds a set of valid placement IDs and network refs for internal consistency checks
6. For each boundary, checks both network IDs in `between` exist in the environment networks
7. For each placement on a serial network, checks that no `ip` field is set
8. For each placement with a register_map_variant, verifies register addresses against device register_capacity
9. If process.yaml exists, checks controller.placement and controller.device, instrument.placement, and network_context[].placement against the placement ID set

### CLI Integration

The `admin design` command is added to the main.go switch statement (SOW-035.0 established the pattern). The design command has its own subcommand routing:

```go
// cli/design.go
func RunDesign(g Globals, args []string) {
    if len(args) == 0 {
        // print design subcommand usage
        os.Exit(1)
    }
    switch args[0] {
    case "validate":
        runDesignValidate(g, args[1:])
    case "list":
        runDesignList(g, args[1:])
    default:
        // unknown subcommand
        os.Exit(1)
    }
}
```

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| `santhosh-tekuri/jsonschema` v6 error paths don't include JSON Pointer | Line mapping cannot work | Low | Library documents JSON Pointer in validation errors. If format differs, parse the error message to extract field path. |
| YAML-to-JSON conversion loses type fidelity (e.g., YAML `"true"` vs `true`) | Schema validation rejects valid files | Medium | Test all 27 files; `nodeToInterface` must handle yaml.Node scalar tags correctly (!!bool, !!int, !!float, !!str). |
| yaml.Node line numbers are 1-indexed or 0-indexed inconsistently | Off-by-one errors in reported lines | Low | `yaml.Node.Line` is 1-indexed. Verify with a test file. |
| Cross-reference validation needs to parse device atom YAML to check variant keys | Performance overhead from loading many device files | Low | Device atoms are small (< 300 lines). Cache parsed device atoms within a single validation run. |
| Environment directory detection fails for paths ending with `/` or paths with symlinks | Directory mode not triggered | Low | Use `os.Stat` to check if path is a directory; normalize trailing slashes. |
| `if/then/else` conditional validation in network schema not supported by library | Network type-conditional validation fails silently | Very Low | SOW-034.0 tested this; `santhosh-tekuri/jsonschema` v6 supports Draft 2020-12 conditionals. Verify during Phase 2 testing. |
