# SOW-037.0 - Admin Web Dashboard

**Status**: Complete
**Created**: 2026-03-03
**Completed**: 2026-03-03
**Priority**: High

## 1. Objective

Build the admin web dashboard at `http://localhost:8095`, replacing the placeholder page from SOW-035.0 with a fully functional administration interface. The dashboard provides non-developer engineers with browser-based access to platform health, event database management, configuration viewing, design library browsing, and baseline monitoring -- without CLI familiarity. All pages are server-rendered HTML with HTMX for dynamic updates, consistent with the monitoring dashboard at :8090.

## 2. Background

SOW-035.0 established the admin CLI module with commands for health, database, config, and baseline management. It also created a skeleton web server at :8095 serving a "Coming Soon" placeholder. Engineers who prefer browser-based tools over CLI currently have no access to admin functions.

The monitoring dashboard at :8090 provides the reference pattern for this implementation:
- `Dashboard` struct holding an API client, template set, and chi router
- `go:embed` templates with a shared layout, nav, and per-page content blocks
- `render()` and `renderPartial()` methods for full-page and HTMX responses
- Bootstrap 5 via CDN for layout, HTMX via CDN for dynamic updates
- Server-side data assembly in handler functions

The admin dashboard follows this same pattern but operates on different data sources:
- TCP port probes for plant health (same as CLI health command)
- Direct SQLite access via `dbutil` package for event DB stats
- `configparse` package for monitor.yaml display and validation
- `apiclient` package for monitoring API health and baseline queries
- Direct filesystem access for design library browsing

The milestone spec (Beta 0.7 DoD #7) requires:
> Open `http://localhost:8095` and see an admin dashboard with service health, event DB statistics, and navigation to all admin functions

### Reference Implementations

- `monitoring/internal/dashboard/dashboard.go`: Dashboard struct, template compilation, render/renderPartial pattern
- `monitoring/internal/dashboard/handlers.go`: Handler methods calling buildXxxData functions
- `monitoring/internal/templates/layout.html`: Bootstrap 5 + HTMX layout template
- `admin/internal/web/server.go`: Existing skeleton web server (SOW-035.0)
- `admin/internal/cli/health.go`: Port probing, API check, DB check logic
- `admin/internal/dbutil/status.go`: DB statistics queries
- `admin/internal/apiclient/`: Monitoring API client (health, baselines)
- `admin/internal/configparse/config.go`: Config parsing and validation

### Architectural Constraints

- Admin web server is part of the `admin/` Go module -- no imports from plant/ or monitoring/
- Reuse existing internal packages (dbutil, apiclient, configparse) -- web handlers call the same functions the CLI uses
- Bootstrap 5 and HTMX loaded via CDN (no build step, no npm)
- HTML templates embedded via `go:embed` for single-binary deployment
- No CodeMirror in this SOW -- that's SOW-038.0 (YAML Editor)
- No authentication per milestone spec ("No RBAC, authentication, or authorization")

## 4. Requirements

### Functional Requirements

**FR-1**: Dashboard page (`GET /`) displays three sections:
1. **Health** cards in two groups:
   - **Device Availability**: Plant Modbus TCP ports labeled by device ID and category from the DesignCache (e.g., "wt-plc-01 (CompactLogix): online" rather than "PLC 5020: online"). When a port is not in the cache, label it "Device <port>". Shows online/offline count per device category. This teaches trainees the difference between physical device availability and IT service health.
   - **Platform Services**: Monitoring API status and event DB status. Each card has a colored indicator (green=all online, yellow=degraded, red=offline).
   HTMX auto-refreshes every 10 seconds via `GET /partials/health-cards`.
2. **Quick Stats** row showing total events, DB size, newest event age, and baseline status as "X established / Y learning" (not a raw count -- the learning/established split teaches trainees that learning devices are in a detection blind period).
3. **Navigation** cards linking to each admin function (/db, /config, /design, /baseline) with a brief description.

**FR-2**: Event Database page (`GET /db`) displays:
- Statistics panel: event count, DB file size, oldest/newest event timestamps, retention window, pruneable event count. Data from `dbutil.Stats()`.
- Prune form: "Delete events older than N days" input with submit button. HTMX POST to `/db/prune` with confirmation dialog (JavaScript `confirm()`). Response updates the stats panel.
- Export form: format selector (CSV/JSON), optional device ID filter, optional time range (after/before). Submit triggers a file download via `GET /db/export?format=csv&device=...&after=...&before=...`.
- HTMX auto-refresh of stats panel every 30 seconds via `GET /partials/db-stats`.

**FR-3**: Configuration page (`GET /config`) displays:
- Config summary: key settings from monitor.yaml displayed as a key-value table (poll interval, environments, API addr, dashboard addr, retention days, syslog status). Data from `configparse.ParseLenient()`.
- Config source: the raw YAML content displayed with syntax highlighting (using the same CSS classes as the monitoring dashboard: `.yaml-key`, `.yaml-value`, `.yaml-comment`, `.yaml-list-marker`). Read-only display in a `<pre>` block.
- Validate button: HTMX POST to `/config/validate` runs `configparse.Parse()` (strict validation). Response shows "Valid" badge or error list with field names and descriptions.
- Note: editing and saving config is SOW-038.0. This page is read-only.

**FR-4**: Design Library page (`GET /design`) displays a categorized listing:
- **Devices** section: table of device atoms (ID, vendor, model, category, vintage). Each row links to the device atom YAML view. Parsed from `design/devices/*.yaml`.
- **Networks** section: table of network atoms (ID, type, subnet or "-" for serial). Parsed from `design/networks/*.yaml`.
- **Environments** section: table of environments (ID, device count, network count, Process Flow). The "Process Flow" column shows "Defined" (green badge) when process.yaml exists, or "None" (gray badge) when absent. This teaches trainees that process flow definitions describe ISA-5.1 P&ID-style process stages used by the monitoring dashboard's process visualization page -- not just file existence. Parsed from `design/environments/*/environment.yaml` directory scan.
- Each table is sorted alphabetically by ID.
- File counts shown in section headers (e.g., "Devices (7)").
- Note: the file tree sidebar and YAML editor are SOW-038.0. This page shows a structured catalog.

**FR-5**: Baseline Status page (`GET /baseline`) displays:
- Per-device baseline table from `GET /api/baselines` via `apiclient`: device ID, status (learning/established), sample count, required samples, register count. Data from `apiclient.GetBaselines()`.
- Overall status summary: "X of Y devices established, Z still learning".
- Reset button for individual devices: HTMX POST to `/baseline/reset?device=<id>`. Response updates the row.
- Reset All button: HTMX POST to `/baseline/reset-all` with JavaScript `confirm()` dialog. The confirmation text must read: "Resetting all baselines will restart the learning period for all devices. During this period, anomaly detection is suspended -- any Modbus writes, new polling sources, or unexpected function codes observed during learning will be accepted as normal behavior. Proceed?" This teaches trainees that the learning period is a detection blind spot, not just a "suspended" service.
- HTMX auto-refresh of baseline table every 10 seconds via `GET /partials/baseline-table`.
- Graceful degradation: when monitoring API is unreachable, show a warning banner instead of crashing.

**FR-6**: Shared layout template (`layout.html`) provides:
- Dark navbar with "OT Simulator Admin" brand and link to :8090 monitoring dashboard
- Sidebar navigation with links to all pages, active page highlighting
- Content area for page-specific templates
- Footer with "Admin Dashboard" version/build info placeholder
- Bootstrap 5 CSS via CDN, HTMX JS via CDN, Bootstrap JS bundle via CDN

**FR-7**: The web server accepts dependencies via constructor:
```go
type Server struct {
    addr       string
    globals    Globals  // design dir, config path, db path, api addr, plant ports
}

func New(addr string, globals Globals) *Server
```
The `Globals` struct from `cli/output.go` provides all the paths and addresses needed. Handlers use these to call `dbutil.Stats()`, `configparse.Parse()`, `apiclient.New()`, etc.

**FR-8**: Export download endpoint (`GET /db/export`) generates CSV or JSON and returns it as a file download (Content-Disposition: attachment). Uses `dbutil.Export()` for data retrieval. Supports the same query parameters as the CLI: format, device, after, before.

**FR-9**: YAML syntax highlighting for the config viewer reuses the CSS classes from the monitoring dashboard layout. The highlighting function converts raw YAML text to HTML with spans for keys, values, comments, and list markers. This is a Go function in the web package (not imported from monitoring -- admin cannot import monitoring packages).

**FR-10**: Design library data is loaded once at server startup and cached in memory. The design directory is scanned for YAML files, and metadata (device id/vendor/model/category, network id/type/subnet, environment id/counts) is extracted. No live filesystem polling. The data is read-only.

### Non-Functional Requirements

**NFR-1**: All pages load within 500ms on localhost. Template rendering and data assembly are the only operations; no external API calls block the initial page load (HTMX fetches dynamic data asynchronously).

**NFR-2**: Pages degrade gracefully when services are unavailable. Health cards show "offline". DB pages show "database not found" instead of crashing. Baseline page shows "monitoring API unreachable" instead of crashing.

**NFR-3**: Function size limit: 60 lines maximum. File size limit: 500 lines maximum (excluding templates and tests). Split handler files by page section (handlers_health.go, handlers_db.go, etc.).

**NFR-4**: Template files are under 200 lines each. Complex sections use named sub-templates ({{define "..."}}...{{end}}).

**NFR-5**: Every public function and type has a documentation comment.

**NFR-6**: HTMX requests include `HX-Request: true` header. Handlers that serve both full pages and partials check for this header. All partial endpoints are under `/partials/`.

### Server Architecture

The web server follows the monitoring dashboard pattern:

```go
// web/server.go
type Server struct {
    addr    string
    globals Globals
    router  chi.Router
    base    *template.Template
    design  *DesignCache  // cached design library metadata
}

func New(addr string, globals Globals) *Server {
    s := &Server{addr: addr, globals: globals}
    s.design = loadDesignCache(globals.DesignDir)
    s.base = template.Must(template.New("").Funcs(funcMap).ParseFS(
        templates.FS,
        "layout.html",
        "partials/nav.html",
        "partials/health_cards.html",
        "partials/db_stats.html",
        "partials/baseline_table.html",
    ))
    s.router = s.buildRouter()
    return s
}
```

### Template Rendering Pattern

Following the monitoring dashboard exactly:

```go
// render clones the base template set, parses the page template, and executes "layout".
func (s *Server) render(w http.ResponseWriter, name string, data any) {
    t, err := template.Must(s.base.Clone()).ParseFS(templates.FS, name)
    if err != nil { ... }
    w.Header().Set("Content-Type", "text/html; charset=utf-8")
    t.ExecuteTemplate(w, "layout", data)
}

// renderPartial clones the base set and executes a named partial template.
func (s *Server) renderPartial(w http.ResponseWriter, name string, data any) {
    t, err := s.base.Clone()
    if err != nil { ... }
    w.Header().Set("Content-Type", "text/html; charset=utf-8")
    t.ExecuteTemplate(w, name, data)
}
```

### Data Assembly Pattern

Each page handler follows the monitoring dashboard pattern:

```go
func (s *Server) dbHandler(w http.ResponseWriter, r *http.Request) {

}

func (s *Server) buildDBData() map[string]any {
```

### Design Cache

The design library is loaded once at startup into a `DesignCache`:

```go
// web/designcache.go
type DesignCache struct {
    Devices      []DeviceSummary
    Networks     []NetworkSummary
    Environments []EnvSummary
}

type DeviceSummary struct {
    ID, Vendor, Model, Category string
    Vintage                     int
}

// PortDevice maps a Modbus TCP port to the device placement that uses it.
// Used by health cards to label ports by device ID and category instead of
// static port ranges. Populated from environment placements at cache load.
type PortDevice struct {
    Port     int
    DeviceID string  // placement device reference (e.g., "compactlogix-l33er")
    Category string  // device category from device atom (e.g., "plc", "gateway")
    PlaceID  string  // placement ID (e.g., "wt-plc-01")
}
```

This parses just the metadata fields from each YAML file (not the full file). The cache is read-only after initialization.

### YAML Highlighting

```go
// web/highlight.go
func highlightYAML(raw string) template.HTML {
```

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| HTMX CDN unavailable on first load | Dashboard has no dynamic updates | Low | HTMX is a single JS file; pages function without it (just no auto-refresh). Note in docs. td-admin-072 tracks this. |
| Port probing in health cards is slow (14 ports, 2s timeout each) | Dashboard page loads slowly | Medium | Run port probes in parallel goroutines (same as CLI). Cache results with 5s TTL for HTMX refreshes. |
| Template compilation errors crash on startup | Server won't start | Low | template.Must pattern used with go:embed -- compilation errors are caught at build time if all templates are valid. |
| DB export of large datasets blocks the HTTP handler | Export request times out or uses excessive memory | Low | dbutil.Export already has a 100,000 row limit. Stream rows to HTTP response rather than buffering in memory. |
| Design cache becomes stale if files are modified | Design library page shows old data | Medium | Acceptable for Beta 0.7 (single-user training). Note in docs. Restarting admin refreshes the cache. |
| go:embed template FS path changes when files are reorganized | Build fails | Low | Template FS root is admin/internal/templates/ -- stable location. Tests verify template parsing. |
