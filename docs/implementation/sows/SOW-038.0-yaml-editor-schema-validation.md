# SOW-038.0 - YAML Editor with Schema Validation

**Status**: Complete
**Created**: 2026-03-03
**Completed**: 2026-03-03
**Priority**: High

## 1. Objective

Add a browser-based YAML editor to the admin web dashboard (SOW-037.0) using CodeMirror 6 for syntax highlighting, inline schema validation feedback, and safe save-to-disk workflow. Engineers creating or modifying device atoms, network atoms, environment definitions, and monitoring configuration see schema errors highlighted at the exact line in the editor before saving, eliminating the edit-save-run-discover-error cycle. This implements the milestone DoD items #8 (browse and view with syntax highlighting), #9 (validate with inline errors), #10 (save with validation guard), and #11 (edit monitor.yaml with same experience).

## 2. Background

SOW-037.0 delivers the admin web dashboard at :8095 with a read-only design library page (`/design`) showing a categorized catalog of devices, networks, and environments. The `/config` page shows monitor.yaml with CSS-based YAML syntax highlighting in a `<pre>` block. Neither page supports editing or schema validation in the browser.

SOW-036.0 delivers the `schema` package (`admin/internal/schema/`) with:
- `SchemaSet`: compiled JSON Schemas for all four design layer types
- `ValidateFile()`: validates YAML content against a schema, returns errors with line numbers
- `InferSchemaType()`: determines which schema to use based on file path
- `ValidateCrossRefs()`: cross-reference validation for environment directories

SOW-034.0 delivers the JSON Schema files in `design/schemas/` with `description` annotations on every property -- these descriptions power the editor's error messages.

The milestone spec requires:
- #8: Open the YAML editor, browse the design/ directory tree, select a device atom, and see it with syntax highlighting in CodeMirror 6
- #9: Edit a device atom, click "Validate", and see schema validation errors highlighted inline
- #10: Save YAML edits back to disk, with the editor preventing saves that fail validation
- #11: View and edit monitor.yaml with the same schema-validated editor experience

## 3. Scope

### In Scope

- CodeMirror 6 initialization script loaded via esm.sh CDN (YAML mode, lint extension, line numbers, bracket matching)
- Editor page at `/design/edit?file=<path>` with three-panel layout: file tree sidebar, CodeMirror editor, validation results panel
- File tree sidebar showing `design/` directory structure with collapsible folders (devices/, networks/, environments/*/)
- Server endpoint `GET /api/design/file?path=<path>` returning raw YAML file content for editor loading
- Server endpoint `POST /api/design/validate` accepting YAML content + file path, returning JSON array of validation errors with line numbers and descriptions
- Server endpoint `POST /api/design/save` accepting YAML content + file path, validating before save, writing to disk on success
- Inline lint markers in CodeMirror at error lines (red underline/gutter markers)
- Validation results panel below editor showing error list with line number, field path, and schema description
- "Validate" button triggers server-side validation and updates both lint markers and results panel
- "Save" button validates first, then saves; disabled state when validation fails (with "Force Save" override)
- Upgrade `/config` page from read-only `<pre>` to CodeMirror editor with validation and save
- Server endpoint `POST /api/config/validate` accepting YAML content, validating via `configparse.Parse()`, returning errors
- Server endpoint `POST /api/config/save` validating and writing monitor.yaml to disk
- JavaScript module (`editor.js`) embedded via go:embed that initializes CodeMirror and wires validate/save buttons
- Path traversal protection on all file read/write endpoints (reject paths outside design dir or config path)

### Out of Scope

- Client-side JSON Schema validation (validation is always server-side via the schema package)
- CodeMirror autocomplete or intellisense (schema-driven autocomplete is complex; defer to future)
- Live/continuous validation as the user types (validate on button click only; debounced typing validation is future)
- Multi-file editing or tabs (one file open at a time)
- Git integration (no commit, diff, or version history from the editor)
- Creating new files from the editor (edit existing files only; new files are created via CLI or OS file manager)
- Deleting files from the editor
- Cross-reference validation in the editor (SOW-036.0's cross-ref validation is CLI-only; extending to web requires loading multiple files)
- Offline CodeMirror support (CDN required; td-admin-072 tracks this)

## 4. Requirements

### Functional Requirements

**FR-1**: The editor page (`GET /design/edit?file=<path>`) renders a three-panel layout:
1. **File tree** (left sidebar, ~250px width): Directory browser for `design/` showing devices/, networks/, environments/ as collapsible folders. Each YAML file is a clickable link that navigates to the editor for that file. The currently open file is highlighted. The tree is populated server-side in the template (not dynamically fetched).
2. **Editor** (center panel): CodeMirror 6 instance with YAML syntax highlighting, line numbers, bracket matching, and active line highlighting. The editor loads file content from `GET /api/design/file?path=<path>`. The editor panel has a toolbar with "Validate" and "Save" buttons.
3. **Validation panel** (bottom panel, ~200px height): Initially shows "No validation results. Click Validate to check this file." After validation, shows a table of errors (line, field path, description) or a green "Valid" badge. Clicking an error row scrolls the editor to that line.

Folders are expandable/collapsible via JavaScript toggle (no server round-trip). All folders start expanded. Each file link navigates via standard `<a href="/design/edit?file=...">`.

**FR-3**: The `GET /api/design/file?path=<path>` endpoint returns:
- `200 OK` with `Content-Type: text/plain` and raw YAML file content when the file exists and is within the design directory
- `404 Not Found` when the file does not exist
- `403 Forbidden` when the path resolves outside the design directory (path traversal attempt)

Path validation: the endpoint resolves the `path` parameter relative to the design directory using `filepath.Join(designDir, path)`, then calls `filepath.EvalSymlinks()` to resolve both the target path and the base directory, and verifies the resolved target starts with the resolved base directory prefix. This blocks both `..` traversal and symlink traversal. When `EvalSymlinks` returns an error (file does not exist or broken symlink), the handler must distinguish "file not found" (return 404) from "symlink escapes base dir" (return 403) by attempting `os.Stat()` on the pre-resolution path.

**FR-4**: The `POST /api/design/validate` endpoint accepts:
- Request body: JSON `{"path": "<relative-path>", "content": "<yaml-content>"}`
- The `path` is used to determine which JSON Schema to apply (via `schema.InferSchemaType()`)
- Validation uses `schema.ValidateFile()` or equivalent function that accepts YAML content as a string (not a file path)
- Response: JSON `{"valid": true}` or `{"valid": false, "errors": [{"line": 35, "path": "/placements/2/modbus_port", "message": "must be >= 1, got 0", "description": "Modbus TCP port..."}]}`
- The `description` field is the JSON Schema `description` annotation for the failing property (from `--verbose` mode in SOW-036.0)
- If the YAML is syntactically invalid (cannot parse), return `{"valid": false, "errors": [{"line": <parse-error-line>, "path": "", "message": "YAML parse error: <details>"}]}`

**FR-5**: The `POST /api/design/save` endpoint accepts:
- Request body: JSON `{"path": "<relative-path>", "content": "<yaml-content>"}`
- First validates using the same logic as `/api/design/validate`
- If validation fails, returns `{"saved": false, "errors": [...]}` (same error format)
- If validation passes, writes the content to disk at `filepath.Join(designDir, path)`, returns `{"saved": true}`
- If the `force` field is true in the request (`{"path": "...", "content": "...", "force": true}`), saves without validation (for cases where the engineer intentionally saves invalid YAML)
- Path traversal protection identical to FR-3

2. Registers a lint source that displays diagnostics from the last validation response
3. Wires the "Validate" button to POST content to `/api/design/validate`, then updates the lint diagnostics
4. Wires the "Save" button to POST content to `/api/design/save`

- Each error row shows: line number (clickable), field path, error message
- Clicking a line number scrolls the CodeMirror editor to that line and highlights it

### Non-Functional Requirements

**NFR-1**: CodeMirror 6 loads from esm.sh CDN. First page load requires internet access. Subsequent loads use browser cache. No local bundling or npm involvement. The import map approach avoids deep dependency resolution at module load time -- esm.sh resolves transitive dependencies server-side.

**NFR-2**: File read and write operations complete within 100ms for files up to 1000 lines (all existing design YAML files are under 300 lines). Validation completes within 500ms including JSON Schema compilation (schemas are pre-compiled in the Server struct at startup per SOW-036.0's SchemaSet pattern).

**NFR-3**: Function size limit: 60 lines maximum. File size limit: 500 lines maximum (excluding templates and embedded JavaScript). Handler files split by concern.

**NFR-4**: Template files containing CodeMirror JavaScript are under 300 lines each. JavaScript is inline in the template (no separate .js build pipeline) but logically organized.

**NFR-5**: All file write operations use atomic write pattern: write to a temporary file in the same directory, then rename. This prevents partial writes if the process is interrupted.

**NFR-6**: File paths in API requests are validated to prevent path traversal. Paths containing `..`, absolute paths, and paths resolving outside the design directory are rejected with 403 Forbidden.

### CodeMirror Integration Strategy

CodeMirror 6 uses ES modules. The import map in the layout (or editor page) resolves bare specifiers to esm.sh URLs. The editor JavaScript:

1. Fetches file content via `fetch('/api/design/file?path=...')`
2. Creates `EditorView` with extensions: `yaml()`, `lineNumbers()`, `bracketMatching()`, `highlightActiveLine()`, `lintGutter()`
3. Maintains a reactive `diagnostics` state field that the lint extension reads
4. On "Validate" click: extracts editor content via `view.state.doc.toString()`, POSTs to `/api/design/validate`, transforms response errors into CodeMirror diagnostics, dispatches state transaction to update lint markers
5. On "Save" click: similar flow but POSTs to `/api/design/save`

CodeMirror Diagnostic format:
```javascript
{
    from: offsetAtLine(error.line),  // character offset of line start
    to: offsetAtLineEnd(error.line), // character offset of line end
    severity: "error",
    message: error.message
}
```

Line-to-offset conversion uses `view.state.doc.line(lineNumber)` which returns `{from, to, text}`.

### Path Traversal Guard

```go
// web/pathguard.go

// ResolveSafePath resolves a relative path within a base directory.
// Returns the absolute resolved path and true if safe, or empty string and false
// if the path escapes the base directory.
func ResolveSafePath(baseDir, relPath string) (string, bool) {
    if filepath.IsAbs(relPath) {
        return "", false
    }
    if strings.Contains(relPath, "..") {
        return "", false
    }
    abs := filepath.Join(baseDir, relPath)
    absResolved, err := filepath.Abs(abs)
    if err != nil {
        return "", false
    }
    baseResolved, err := filepath.Abs(baseDir)
    if err != nil {
        return "", false
    }
    if !strings.HasPrefix(absResolved, baseResolved+string(filepath.Separator)) {
        return "", false
    }
    return absResolved, true
}
```

### Atomic Write Pattern

```go
func atomicWrite(path string, content []byte) error {
    dir := filepath.Dir(path)
    tmp, err := os.CreateTemp(dir, ".otsim-save-*")
    if err != nil {
        return fmt.Errorf("create temp file: %w", err)
    }
    tmpName := tmp.Name()
    defer os.Remove(tmpName) // cleanup on error
    if _, err := tmp.Write(content); err != nil {
        tmp.Close()
        return fmt.Errorf("write temp file: %w", err)
    }
    if err := tmp.Close(); err != nil {
        return fmt.Errorf("close temp file: %w", err)
    }
    return os.Rename(tmpName, path)
}
```

## 10. Risks and Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| esm.sh CDN unavailable or slow | Editor page loads without CodeMirror; broken editing | Low | esm.sh has high uptime. Browser caches modules after first load. Degrade gracefully: show raw `<textarea>` as fallback with validation-only (no syntax highlighting). td-admin-072 tracks CDN dependency. |
| CodeMirror 6 ESM import map resolution fails in some browsers | Editor JavaScript errors | Low | Import maps supported in Chrome 89+, Firefox 108+, Safari 16.4+. All current browsers support them. Test in target browsers. |
| Schema descriptions missing for some validation errors | Error messages lack context | Medium | SOW-034.0 added descriptions to ALL properties. If a schema error path doesn't have a description (e.g., root-level additionalProperties error), fall back to the raw error message without a description field. |
| Large YAML files cause slow editor response | CodeMirror sluggish on large files | Very Low | All existing design YAML files are under 300 lines. CodeMirror 6 handles files up to 10,000+ lines efficiently. Not a practical concern. |
| Concurrent edits (two browser tabs editing same file) | Last save wins, data loss | Low | Single-user training deployment. No locking or conflict resolution needed. Document the limitation. |
| Atomic write rename fails on Windows | Save operation fails | Low | Project is macOS/Linux focused. `os.Rename` within the same directory is atomic on POSIX. Windows NTFS also supports this pattern. |
| ValidateContent function not in schema package | Need to modify SOW-036.0 deliverable | Medium | SOW-036.0 specifies ValidateFile (file path). This SOW needs ValidateContent (string content). Either extend SOW-036.0 or add the function in this SOW. The function is simple: parse YAML string into yaml.Node, convert, validate against schema. |
