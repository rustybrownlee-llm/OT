# SOW-039.0 - CLI Reference Documentation

**Status**: Complete
**Created**: 2026-03-03
**Priority**: Medium

## 1. Objective

Create comprehensive CLI reference documentation for the admin tool (`admin`) so that non-developer engineers and integration partners (e.g., Blastwave) can operate the OT simulator platform independently -- without reading Go source code, examining `flag.NewFlagSet` calls, or running commands blind to discover their behavior. The documentation covers every command, subcommand, flag, environment variable, exit code, and expected output format, with worked examples showing realistic OT simulator workflows. This implements milestone DoD items #13 (CLI reference documentation) and #14 (hand platform to a partner engineer with CLI docs and have them operate independently).

## 2. Background

SOW-035.0 delivered the admin CLI module with commands for health, database, config, and baseline management. SOW-036.0 adds `design validate` and `design list` commands for JSON Schema validation of design layer YAML. Together, these provide comprehensive command-line administration for the OT simulator platform.

Currently, the only command documentation is the `printUsage()` output in `main.go` and the individual `printXxxUsage()` functions in each command handler. These provide synopsis-level help but lack:
- Extended descriptions explaining when and why to use each command
- Worked examples with realistic output
- Environment variable reference with precedence rules
- Exit code documentation for scripting
- Common workflow recipes that chain multiple commands
- OT-specific context explaining why certain operations (e.g., baseline reset) have safety implications

The target audience is IT engineers who:
- Know command-line tools but not Go
- Are familiar with Linux/macOS terminals
- May not know Modbus, PLCs, or OT monitoring concepts
- Need to operate the platform for Blastwave/Dragos/CyberVision integration testing
- Will use this documentation alongside the admin web dashboard

The milestone spec requires:
- #13: Read CLI reference documentation that explains every command, its flags, examples, and expected output -- written for engineers without Go experience
- #14: Hand the platform to a partner engineer with the admin dashboard URL and CLI docs, and have them operate it independently

### Reference Implementations

- `admin/internal/cli/health.go`: Health command implementation and output format
- `admin/internal/cli/db.go`: DB subcommands (status, validate, prune, export)

- Binary name is `admin` (not `otsim`) per td-admin-075; document accordingly but note the eventual `otsim` wrapper plan

## 3. Scope

### In Scope

- CLI reference document (`docs/admin-cli-reference.md`) covering all commands and subcommands
- Command synopsis, description, flags, examples with expected output, exit codes for every command
- Global flags reference with environment variable fallback precedence
- Environment variable reference table (OTS_DESIGN_DIR, OTS_MONITOR_CONFIG, OTS_API_ADDR, OTS_ADMIN_ADDR, OTS_HEALTH_PORTS)
- Quick start section showing common workflows for new engineers
- OT-specific notes explaining the operational significance of key operations (baseline reset, prune, health check interpretation)
- Troubleshooting section covering common error messages and their resolution

### Out of Scope

- Web dashboard documentation (the dashboard is self-explanatory; documentation for it would be a separate effort)
- YAML schema reference (JSON Schema files in `design/schemas/` are self-documenting with `description` annotations)
- Design layer authoring guide (how to create new device atoms, environments)
- Architecture documentation (ADRs serve this purpose)
- Man pages or built-in `--help` text improvements (the CLI's inline help is separate from reference docs)
- Installation or build instructions (covered by README)

## 4. Requirements

### Functional Requirements

**FR-1**: The CLI reference document is a single Markdown file at `docs/admin-cli-reference.md`. It uses standard GitHub-flavored Markdown with no external dependencies (no MkDocs, no Hugo, no Docusaurus). Engineers read it in a text editor, on GitHub, or in any Markdown viewer.

**FR-2**: The document is structured with the following top-level sections:

1. **Overview** -- what the admin CLI is, who it's for, how to get started
2. **Quick Start** -- 5-minute workflow: build, check health, view config, validate design files
3. **Global Flags** -- flags that apply to all commands, with environment variable fallback
4. **Commands Reference** -- one subsection per command with full documentation
5. **Environment Variables** -- reference table with defaults and usage
6. **Exit Codes** -- summary of exit code conventions
7. **Troubleshooting** -- common errors and resolutions

**FR-3**: Each command reference subsection follows this template:

```
### admin <command> [subcommand]

**Synopsis**
    admin [global-flags] <command> [subcommand] [flags]

**Description**
[2-3 sentences: what the command does, when to use it, what data it reads]

**Flags**
| Flag | Default | Description |
|------|---------|-------------|
...

**Examples**

    $ admin <command> [flags]
    [realistic output matching actual CLI format]

**Exit Codes**
| Code | Meaning |
|------|---------|
| 0 | [success condition] |
| 1 | [failure condition] |

**Notes**
[OT-specific context, operational warnings, related commands]
```

**FR-4**: The Quick Start section provides a complete workflow that a new engineer can follow step-by-step:

1. Build the binary: `cd admin && go build -o admin ./cmd/admin/`
2. Check platform health: `./admin health`
3. View monitoring config: `./admin config view`
4. Validate a device atom: `./admin design validate design/devices/compactlogix-l33er.yaml`
5. Validate an entire environment: `./admin design validate design/environments/greenfield-water-mfg/`
6. List all design elements: `./admin design list`
7. Check baseline status: `./admin baseline status`
8. Export events for analysis: `./admin db export --format csv --output events.csv`
9. Start the admin dashboard: `./admin web`

Each step includes the expected output so the engineer can verify their results match.

**FR-6**: Global flags documentation explains the precedence chain (CLI flag > environment variable > default) with a concrete example:

```
Flag precedence example:

    # These three are equivalent:
    $ admin --design-dir /opt/ot/design health
    $ OTS_DESIGN_DIR=/opt/ot/design admin health
    $ admin health  # uses default ./design
```

**FR-7**: The Environment Variables section provides a reference table:

| Variable | Default | Used By | Purpose |
|----------|---------|---------|---------|
| `OTS_DESIGN_DIR` | `./design` | All commands | Path to design layer directory |
| `OTS_MONITOR_CONFIG` | `./config/monitor.yaml` | db, config, web | Path to monitoring config file |
| `OTS_API_ADDR` | `localhost:8091` | health, baseline, web | Monitoring API address |
| `OTS_ADMIN_ADDR` | `:8095` | web | Admin dashboard bind address |
| `OTS_HEALTH_PORTS` | `5020,5021,...` (14 ports) | health | Comma-separated Modbus TCP ports to probe |

**FR-8**: Exit code documentation follows OT-aware conventions:
- Exit 0: all operations successful, all services reachable, all validations passed
- Exit 1: any operation failed, any service unreachable, any validation error detected
- OT note: the admin CLI follows the principle that a degraded plant is not a "healthy" condition. If 13 of 14 ports respond but one PLC is offline, `admin health` exits 1, not 0. This is intentional -- a single offline PLC may represent a loss of control for an entire process stage.

**FR-9**: The `admin baseline reset` section includes an OT-specific warning box:

> **OT Warning**: Baseline reset triggers a learning period during which anomaly detection is suspended. Any traffic observed during this window -- including malicious activity -- will be accepted as normal behavior and incorporated into the new baseline. In real OT environments, this is operationally equivalent to temporarily disabling the IDS. The `--force` flag bypasses the confirmation prompt but not the consequences.

**FR-10**: The `admin design validate` section includes examples of both success and failure output, covering:
- Single file validation (device atom): success and failure with line-level errors
- Environment directory validation: grouped output showing schema + cross-reference results
- The `--verbose` flag with schema description annotations
- The `--cross-refs-only` flag for reference-only checks

**FR-11**: The Troubleshooting section covers at minimum:
- "admin health: all ports offline" -- plant is not running, or wrong port list
- "admin db status: database not found" -- event database path configuration
- "admin config validate: cannot read file" -- wrong config path
- "admin design validate: cannot determine schema type" -- file outside design directory
- "admin baseline status: monitoring API unreachable" -- monitor is not running
- "admin web: address already in use" -- another process on port 8095

### Non-Functional Requirements

**NFR-1**: The document is under 1000 lines of Markdown. Conciseness is valued -- this is a reference, not a tutorial.

**NFR-2**: All example output is indented with 4 spaces (Markdown code block style) and matches the actual CLI output format precisely. Tab-aligned table output, key-value pair alignment, and error message format must be accurate to what the CLI produces.

**NFR-3**: The document is written at a reading level appropriate for IT engineers without OT domain knowledge. OT-specific terms (Modbus, PLC, baseline learning, register map variant) are explained in context when first used.

**NFR-4**: No dependencies on external documentation tools. The file is readable as raw Markdown text, rendered on GitHub, or viewed in any Markdown-capable editor.

**NFR-5**: The document references the binary as `admin` (not `otsim`) per td-admin-075, with a note in the Overview section that a unified `otsim` wrapper may be introduced in a future release.

### Example Output Derivation

All example output is derived from reading the actual CLI implementation code:

- `admin health`: Table with columns "Service", "Status", "Details" -- port probes, API check, DB check
- `admin db status`: Key-value pairs with 24-char key width
- `admin db export`: CSV with columns from `dbutil.Export()`
- `admin config view`: Key-value pairs with 28-char key width, syslog section, environments section
- `admin design validate`: SOW-036.0 defines the output format (OK/FAIL per file, errors with line numbers)
- `admin design list`: SOW-036.0 defines the categorized table format
- `admin baseline status`: Table with columns "Device ID", "Status", "Samples", "Required", "Registers"

### OT Context Integration

The documentation integrates OT-specific context at natural points rather than in a separate glossary:

- Health check: explains that Modbus TCP ports represent physical devices, not services
- Baseline reset: explains learning period as detection blind spot
- Design validate: explains that cross-reference errors catch wiring mistakes (wrong network, missing device)
- DB export: explains that events are Modbus transaction records, useful for SIEM import
- Config view: explains poll interval, learning cycles, and their operational implications
